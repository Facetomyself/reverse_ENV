## 雷电模拟器

| 项目 | 值 |
|------|-----|
| 版本 | LDPlayer 9 |
| CLI 实际路径 | `D:\leidian\LDPlayer9\ldconsole.exe` |
| ADB 实际路径 | `D:\leidian\LDPlayer9\adb.exe` |
| 安装壳路径 | `C:\Program Files\ldplayer9box\`（若存在，仅作为安装目录参考） |
| 后端 | VirtualBox 6.1.50 |
| ADB 映射 | `emulator-{5554 + index * 2}`，例如 index 1 为 `emulator-5556` |
| 管控脚本 | `tools\ldplayer\ldplayer.ps1` |
| 模板实例 | `re-base` / `re-xposed` / `re-stealth` |
| 模板备份 | `storage\ldplayer-backups\*.ldbk` |
| Android 模块资产 | `tools\android-modules\` |

大文件（~2GB），不纳入仓库。RE 实例管理优先使用 `skill\ldplayer-control\scripts\`，底层原子操作使用 `tools\ldplayer\ldplayer.ps1`。

当前 LDPlayer 采用模板实例模型：

| 模板 | 能力 | 用法 |
|------|------|------|
| `re-base` | Root + Kitsune Mask v27.0 + Frida 17.15.3 + mitmproxy CA | 低干扰动态调试、基础抓包 |
| `re-xposed` | `re-base` + LSPosed v1.9.2 + JustTrustMe v2 | 日常 App 逆向默认模板 |
| `re-stealth` | `re-xposed` + Hide My Applist V3.6.1 + Shamiko v0.7.5 | Root/模块隐藏要求更高的目标 |

真实目标 App 不直接装进模板。新项目用：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "D:\reverse_ENV\skill\ldplayer-control\scripts\re-init.ps1" -Project myapp -Template re-xposed
```

模板 verified 备份位于 `storage\ldplayer-backups\`；APK/ZIP 模块本体位于 `tools\android-modules\` 并由 `.gitignore` 排除，只跟踪资产清单。

## ruyipage 指纹浏览器

| 项目 | 值 |
|------|-----|
| Python 依赖版本 | `ruyiPage==1.2.46` |
| MCP 适配版本 | `ruyi-mcp 0.1.1` |
| 浏览器 | Firefox 151.0a1（定制指纹浏览器） |
| 协议 | WebDriver BiDi（非 CDP） |
| Side-by-side runtime | `D:\reverse_ENV\tools\ruyipage\runtimes\151-proxy\firefox\firefox.exe` |
| Release asset SHA256 | `f82151f9f197b528b36fb461cf106bc6825c0f752efb5c5d762c34f7529055f6` |
| BuildID | `20260702113527` |
| 当前项目配置 | `.mcp.json` 与 `.codex/config.toml` 已指向 `tools\ruyipage\runtimes\151-proxy\firefox\firefox.exe` |
| 用途 | 反检测浏览器自动化、过 Cloudflare/验证码 |

`151-proxy` 已通过 BiDi launch/quit、UA/viewport/timezone/locale/orientation、DOM click、launch/runtime Trace entries 增长和 stop 后冻结，以及真实 HTTP 认证代理与 percent-encoded 凭据验证。3 个 headed fingerprint profile 未复现 Issue #20；SOCKS5 因当前供应商无对应产品只完成 offline contract，待有可用供应商时补真实出口门禁。浏览器二进制位于 `.gitignore` 排除的 `tools\ruyipage\runtimes\`，不纳入仓库；治理说明见 `tools\ruyipage\README.md`。

`ruyi_trace_start`、`ruyi_trace_stop`、`ruyi_trace_get_results` 输出的是 RuyiPage/WebDriver BiDi JSON Trace，不是 C++ DOMTrace。`151-proxy` 没有 `RUYI_DOMTRACE.txt` marker，实际验证也不生成 DOMTrace，不能直接替换下述 trace kernel。

## ruyiTrace DOM 追踪

| 项目 | 值 |
|------|-----|
| 版本 | v1.2（Firefox 151 trace kernel） |
| 内核路径 | `tools\ruyitrace\firefox\firefox.exe` |
| 追踪机制 | `MOZ_DOM_TRACE=1` → C++ 层 Hook DOM API |
| 输出格式 | NDJSON |
| 启动约束 | `ruyitrace.ps1` 同时设置 `MOZ_DISABLE_LAUNCHER_PROCESS=1` |
| 多进程输出 | Firefox 生成 `<output>_<PID>.ndjson` 分片，脚本退出后合并到 `-Output` |
| 行数限制 | `-Limit` 可选；按 Firefox 进程限制 NDJSON 行数 |
| 分析脚本 | `tools\ruyitrace\trace_analyzer.py` |
| 大小 | ~340MB（C++ 层 Hook，页面脚本不可检测）

`tools\ruyitrace\firefox\` 继续作为 DOMTrace 专用 runtime。它与 `tools\ruyipage\runtimes\151-proxy\` 分层维护，不共享“已支持 DOMTrace”的结论。

## reqable-mcp 抓包数据 MCP

| 项目 | 值 |
|------|-----|
| 版本 | 0.3.2 |
| 仓库 | https://github.com/ElonJask/reqable-mcp |
| 路径 | `tools\reqable-mcp\`（源码 + editable install） |
| CLI | `.venv\Scripts\reqable-mcp.exe mcp` |
| 用途 | 查询 Reqable 抓取的 HTTP/WebSocket 流量 — 请求搜索/域名统计/API 分析/代码生成 |

工作流：
1. 在 [Reqable](https://reqable.com) 桌面端配置 Report Server → `http://127.0.0.1:18765/report`
2. Reqable 抓包 → HAR 数据推送到 reqable-mcp
3. Claude Code 通过 MCP 工具查询分析流量（`reqable_*` 前缀，17 个 tools）

安装：`pip install -e tools\reqable-mcp`（已装在 venv，editable 模式）。

MCP 配置已写入 `.mcp.json`，前缀 `reqable_*`。
