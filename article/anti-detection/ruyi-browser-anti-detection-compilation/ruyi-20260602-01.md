##  运行配置

配置文件：

text


    C:\safari\fp.txt

支持的代理配置项：

ini


    http_proxy=username:password@proxy-host:proxy-port
    proxy=username:password@proxy-host:proxy-port
    proxy_url=username:password@proxy-host:proxy-port

    proxy_exclude_hosts=localhost,127.0.0.1
    proxy_ignore_hosts=localhost,127.0.0.1

读取优先级：

  1. ` http_proxy  `

  2. ` proxy  `

  3. ` proxy_url  `

也就是说，优先使用  ` http_proxy  ` ，如果没有写这个字段，再尝试读取  ` proxy  ` 和  ` proxy_url  ` 。

关闭代理的写法：

ini


    http_proxy=0
    http_proxy=false
    http_proxy=none
    http_proxy=disable

代理地址可以不写协议头。例如：

ini


    http_proxy=username:password@proxy-host:proxy-port

MiniBrowser 会自动补成：

text


    http://username:password@proxy-host:proxy-port

##  整体链路

HTTP 密码代理的完整流程是：

text


    fp.txt
      -> MiniBrowser 读取代理配置
      -> 拆出 username/password
      -> 代理 URL 去掉账号密码，只保留 http://host:port
      -> 调用 WKWebsiteDataStoreEnableCustomNetworkProxySettingsWithCredentials
      -> UIProcess 把代理配置传给 WebKit
      -> IPC 序列化传到 NetworkProcess
      -> CurlContext 给 libcurl 设置 CURLOPT_PROXY / CURLOPT_PROXYUSERNAME / CURLOPT_PROXYPASSWORD / CURLOPT_PROXYAUTH
      -> libcurl 访问目标网站时通过 HTTP 密码代理发起连接

核心原则：

text


    代理地址和代理密码分开保存、分开传递。

不要长期把密码放在代理 URL 里。

##  修改文件

###  1\.  ` Tools/MiniBrowser/win/WebKitBrowserWindow.cpp  `

这个文件负责从  ` fp.txt  ` 读取代理配置，并把代理信息传给 WebKit。

主要修改点：

  * 读取  ` http_proxy  ` /  ` proxy  ` /  ` proxy_url  ` 。

  * 自动给没有协议头的代理地址补  ` http://  ` 。

  * 从  ` username:password@host:port  ` 里拆出账号和密码。

  * 把  ` m_proxy.url  ` 改成不带账号密码的安全 URL。

  * 保存代理 host 和 port，用于后续认证挑战匹配。

  * 调用新增的带账号密码 API。

  * 当 WebKit 收到代理认证挑战时，只对当前配置的代理 host/port 自动填入账号密码。

这里必须检查：

  * 是否是代理认证：  ` WKProtectionSpaceGetIsProxy(protectionSpace)  `

  * host 是否是当前代理 host

  * port 是否是当前代理 port

这样可以避免把代理账号密码错误地发给普通网站。

##  为什么要这样改

最开始的问题是：

text


    HTTP 密码代理访问 HTTPS 网站时，需要在 CONNECT 阶段完成代理认证。

HTTPS 网站访问流程大概是：

text


    浏览器 -> HTTP 代理：CONNECT www.example.com:443
    代理 -> 浏览器：要求认证
    浏览器 -> 代理：发送 Proxy-Authorization
    代理 -> 目标网站：建立隧道
    浏览器 -> 目标网站：开始 TLS 握手和 HTTPS 请求

如果代理账号密码没有正确传给 libcurl，HTTPS 访问就容易失败，表现为：

text


    CurlErrorDomain
    Error code 28
    Failed sending data to peer
    Proxy authentication failed
    Provisional Navigation Failure

其中  ` Error code 28  ` 是 curl timeout，可能是代理认证问题，也可能是代理线路到目标网站不稳定。

###  为什么不直接把密码放在 URL 里

例如：

text


    http://username:password@proxy-host:proxy-port

这样虽然简单，但不够干净。

风险：

  * URL 字符串可能出现在日志里。

  * URL 字符串可能通过 IPC、调试信息、错误信息传播。

  * WebKit 的 URL 对象可能对 URL 做规范化处理。

  * 后续代码看见 URL 时，不容易区分这是代理地址还是带敏感信息的字符串。

所以现在的设计是：

text


    URL 只保存代理地址。
    账号密码单独保存。
    认证方式单独保存。

也就是：

text


    proxy.url      = http://proxy-host:proxy-port
    proxy.user     = username
    proxy.password = password
    proxy.auth     = Basic

###  为什么要同时处理 AuthenticationChallenge

libcurl 使用  ` CURLOPT_PROXYUSERNAME  ` 和  ` CURLOPT_PROXYPASSWORD  `
后，通常可以直接完成代理认证。

但 WebKit 也有认证挑战回调：

C++


    didReceiveAuthenticationChallenge

如果代理服务器返回认证挑战，UIProcess 也需要知道如何回答。

所以 MiniBrowser 保留：

C++


    m_proxyUsername
    m_proxyPassword
    m_proxyHost
    m_proxyPort

然后只在确认是当前代理服务器的挑战时，才自动提交代理凭据。

这样既能兼容 WebKit 的认证流程，又不会把代理密码发给普通网站。
