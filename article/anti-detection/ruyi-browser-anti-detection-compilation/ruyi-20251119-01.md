在编写完了mojom之后，实际上进行IPC通信的模板代码都生成完毕了，在编译之后的out目录下会有相关的h和cpp文件：

  *   *   *   *   *   *   *   *   *   *   *   *   *   *


    class  RuyiJsExtensionProxy    : public RuyiJsExtension { public:  using InterfaceType = RuyiJsExtension;
      explicit RuyiJsExtensionProxy(mojo::MessageReceiverWithResponder* receiver);
      bool Log(const std::string& message, std::string* out_modified_message) final;
      void Log(const std::string& message, LogCallback callback) final;
     private:  mojo::MessageReceiverWithResponder* receiver_;};

这里的代码都是样板代码，也就是根据我们的mojom标准生成的：

  *


    // ruyi/public/mojom/ruyi_render_browser.mojom.h is auto generated by mojom_bindings_generator.py, do not edit

生成了这么多文件，我们如果不用那就没人用了，即便你加到BUILD.gn构建系统当中，chromium自己也不会用的。

所以我们需要自己编写cpp和h文件来使用这些东西，直接继承上边的样板代码，并且仿写：

  *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *


    #pragma once
    #include <string>#include "base/logging.h"#include "mojo/public/cpp/bindings/receiver.h"#include "ruyi/public/mojom/ruyi_render_browser.mojom.h"
    class RuyiJsExtensionImpl : public ruyi::mojom::RuyiJsExtension { public:
      explicit RuyiJsExtensionImpl(      mojo::PendingReceiver<ruyi::mojom::RuyiJsExtension> receiver);  ~RuyiJsExtensionImpl() override;
      void Log(const std::string& message,ruyi::mojom::RuyiJsExtension::LogCallback callback) override;  bool Log(const std::string& message,std::string* modified_message_out) override;

     private:  mojo::Receiver<ruyi::mojom::RuyiJsExtension> receiver_;};


至于这里为什么写了两个log呢？

因为实际上在生成同步sync代码的时候，本身mojom会生成同步和异步两个接口，而且同步代码实际上还是调用了异步：

  *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *   *


    void RuyiJsExtensionAsyncWaiter::Log(    const std::string& message, std::string* out_modified_message) {  base::RunLoop loop;  proxy_->Log(      std::move(message),      base::BindOnce(          [](base::RunLoop* loop,             std::string* out_modified_message,             const std::string& modified_message) {*out_modified_message = std::move(modified_message);            loop->Quit();          },          &loop,          out_modified_message));  loop.Run();}
    std::string RuyiJsExtensionAsyncWaiter::Log(    const std::string& message) {  std::string async_wait_result;  Log(std::move(message),&async_wait_result);  return async_wait_result;}

loop.run会阻塞当前事件，只能是异步里边的quit调用后才会退出，所以需要实现同步和异步两份代码。

  * ` base::RunLoop loop;  `

    * 创建一个事件循环（run loop）。后面通过  ` loop.Run()  ` 阻塞当前执行流并处理当前线程的任务队列（message loop），直到有人调用  ` loop.Quit()  ` 。

  * ` proxy_->Log(std::move(message), base::BindOnce(...));  `

    * 使用生成的  ` proxy_  ` （也就是  ` RuyiJsExtensionProxy  ` ）发起异步 IPC 调用。

    * ` std::move(message)  ` 把字符串所有权移动给 proxy 的调用序列。

    * 第二个参数是  ` base::OnceCallback  ` 回调（  ` BindOnce  ` ），当服务端通过 callback 返回结果时，这个回调会被调用。


当然，写完之后，还要加入BUILD.gn的构建系统，对于我们自己的服务端代码的BUILD，只要把cpp和相关依赖写入即可：

  *   *   *   *   *   *   *   *   *   *   *   *   *   *   *


    source_set("ruyi_mojo_impl") {  sources = [    "ruyi_mojo_impl.cpp",  ]
      deps = [    "//base",    "//mojo/public/cpp/bindings",  ]

      deps += [    "//ruyi/public/mojom:mojom",  ]}

写完之后，这个BUILD这里我直接加入到了browser的BUILD.gn当中，就可以在browser主进程中调用了。
到此为止，实际上服务端（Browser）要做的事已经完毕了，之后就是做好mojom注册和renderer端的消息发送。
