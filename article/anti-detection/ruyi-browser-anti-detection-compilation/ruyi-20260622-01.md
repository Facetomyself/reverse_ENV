1\. Skill 定位

xiaojianbang-auto-reverse  是一个面向 Android 逆向研究的 Skill，核心定位是：

●  Android  .so  检测链分析

●  Native 反调试、反 Frida、Root、模拟器、Hook、完整性/CRC 检测分析

●  SIGKILL  、  SIGSEGV  、  SIGTRAP  、  SIGABRT  、  BRK  、  exit_group
等闪退、退出的定位

●  匿名内存的定位

●  加密/壳化/自解密  .so  的 dump/fix 与 IDA 分析、OLLVM/控制流混淆还原辅助

●  各种工具、小肩膀定制系统能力的集成

它不是单一脚本，也不是只负责跑 Frida 的工具，而是一套把动态证据、静态分析、内存 dump、内核无痕验证、patch 和实验记录串起来的工作流。

适用范围限定为自有样本、用户明确授权样本、研究环境和调试环境。目标边界不清时，不执行可能导致未授权访问、隐蔽控制或数据外传的操作。

2\. 核心能力总览

### 2.1 Android Native 检测链分析

Skill 可以围绕 Android App 中的 native 层  .so  建立检测链视图，重点分析：

●  JNI_OnLoad

●  RegisterNatives

●  constructor 初始化逻辑

●  dlopen  /  android_dlopen_ext

●  direct syscall

●  libc / linker / libart 调用

●  反调试逻辑

●  反 Frida 逻辑

●  Root 检测

●  模拟器检测

●  Hook / Xposed / 注入检测

●  文件、属性、命令、procfs、maps、fd 等环境探测

●  .text  CRC、自身完整性校验、libc/libart/linker 完整性校验

分析时要求区分 helper 函数和真实检测入口，避免把工具函数、字符串初始化函数、日志函数误判为检测点。

### 2.4 syscall-filter 工作流

xiaojianbang-syscall-filter  用于从 syscall
层捕获关键行为。新版流程中，它既能做路径类检测伪装，也承担闪退/崩溃/退出定位和匿名执行来源定位；在闪退、崩溃、退出场景下是硬门禁，不允许用 Frida
动态试错替代。

主要能力：

●  路径检测 trace：按目标 UID 捕获
faccessat/faccessat2/openat/openat2/newfstatat/readlinkat/statx/execve/name_to_handle_at
等路径类 syscall，记录调用者 so、线程、路径和分类。

●  路径存在性伪装：命中 ROOT / FRIDA / XPOSED / AOSP 规则时可统一返回  -ENOENT
，让文件存在性检测认为目标不存在。

●  退出/信号 trace：  exitmon=on  后记录  exit  、  exit_group  、  kill  、  tkill  、
tgkill  、  rt_sigqueueinfo  、  rt_tgsigqueueinfo  、  pidfd_send_signal
等，只记录不拦截。

●  内存执行来源 trace：  memmon=on  后记录
mmap/mprotect/pkey_mprotect/memfd_create/prctl/clone/clone3/ptrace/wait4/waitid
等事件，定位匿名 RX/RWX、  memfd  执行段、VMA 命名和 watcher 线程。

●  内核态地址解析：  resolve=on  时在进程死亡前解析  pc/lr -> so!offset  、  anon: +  、
[anon:name]+offset  或  memfd:name!offset  。

●  全量调试：  dump=on  打印目标 UID 的全量路径 syscall；  memdump=on  打印全量内存类
syscall，用于排查默认高价值过滤漏项。

●  关键证据输出：  logs/  _resolved.log  是主要阅读对象，包含路径命中、退出/信号事件、内存事件和调用者解析结果。

它不依赖小肩膀定制系统，但需要通用设备条件：

●  Android root/su

●  Android arm64 或匹配目标 ABI

●  APatch/KernelPatch

●  KPM 加载能力

●  设备端  /data/local/tmp/kpatch  或等价 KPM 管理入口

●  superkey，可通过  XJB_KP_SUPERKEY  或脚本配置覆盖

### 2.8 内核无痕 HWBP hook / stealth-hook

Skill 支持  xiaojianbang-stealth-hook  ，用于强反 Frida、强  .text  校验或 inline hook
不稳定时的无痕验证。

主要能力：

●  按  pid + so + offset  live trace

●  捕获函数入参

●  捕获返回值

●  替换返回值

●  修改 X0-X7 参数

●  query 当前 hook 状态

●  unhook 清理断点

它的特点是：

●  不修改目标进程用户态代码

●  不注入 so

●  不创建目标进程可疑映射

●  适合验证窄 patch 候选和低频检测函数

设备要求：

●  Android arm64

●  GKI 5.4+

●  root/su

●  APatch/KernelPatch

●  KPM 加载能力

注意：当目标是“过检测、让 Frida 注入后 App 正常运行、Frida 不被发现”时，Skill 禁止把 HWBP/stealth-hook
作为主流程。此时 HWBP 只能作为辅助验证手段，用来确认参数、返回值、窄 patch 或 syscall 证据。分析算法类任务中，如果目标有强 Frida
检测，HWBP/stealth-hook 才是主力验证手段；若 HWBP 功能不支持所需复杂逻辑，再回到 Frida 过检测路线。

### 2.9 eCapture TLS 抓包

Skill 支持 Android arm64 侧 eCapture，用于 TLS 明文或 keylog 采集。

### 2.10 Java / Kotlin 层辅助分析

虽然 Skill 重点是 Android Native，但也要求 Java/Kotlin 层使用 jadx 进行辅助分析。

典型用途：

●  查找  System.loadLibrary

●  查找  System.load

●  查找 native 方法声明

●  查找壳入口

●  查找业务调用路径

●  查找证书 pinning、代理检测、VPN 检测等 Java 层逻辑

要求：

●  jadx 打开 dex 时关闭 checksum 校验。

●  找不到 jadx 时，先查 PATH、项目目录、已有记录和常见安装路径。

●  本轮确实需要 jadx 且仍未命中时，必须做宿主机全盘搜索并记录范围和结果；全盘仍找不到后，才询问用户路径或确认回退方案。

3\. 小肩膀定制系统能力

小肩膀定制系统是 Skill 中一组特殊能力的前置环境。它不是所有工具的必要条件，但会影响脱壳、native 注册监听、任意 so
注入和系统级防检测适配能力。

### 3.1 定制系统使用前提

每个会话首次涉及以下能力时，必须先确认当前连接设备是否为“小肩膀定制系统”：

●  整体加固脱壳

●  抽取式加固脱壳

●  native 函数注册监听

●  任意 so 注入

●  定制系统内置 Apatch root

●  dexfix_runner.py  合并抽取式脱壳产物

确认前不能启用这些能力。

如果用户确认设备不是定制系统，或无法确认，则按通用流程推进，例如：

●  Frida

●  MemDumper

●  syscall-filter

●  stealth-hook

●  IDA

●  OLLVM_Deobfuscator

●  eCapture

其中  xiaojianbang-syscall-filter  和  xiaojianbang-stealth-hook
不属于定制系统能力。它们只依赖通用前置条件：root/su、APatch/KernelPatch、KPM、arm64/GKI 等。

### 3.2 定制系统的系统级防检测适配

小肩膀定制系统已经做了部分系统级环境适配。分析时需要把这些能力视为环境既有条件，不能误写成目标 App 的检测绕过成果。

包括：

●  user 版本编译

●  适配 SELinux Enforcing

●  集成 GMS 谷歌全家桶

●  替换 AOSP test-keys 为自定义密钥

●  设备信息伪装为商用 Pixel 6

●  bootloader 信息伪装为已锁定

●  AOSP 专有 App 替换为 Google App，例如输入法、图库、相机、搜索、浏览器、WebView 等

●  WebView 强制可调试

●  一定程度防 USB 调试检测

●  一定程度防 WiFi 调试检测

●  一定程度防开发者选项检测

●  Java 层防 VPN 检测

注意：这些是系统侧伪装，不代表目标 App 没有自己的 root、设备指纹、VPN、调试或完整性检测。App 自身检测链仍然要按常规流程分析。

### 3.3 定制系统开箱即用功能

### 3.3.1 默认开启 adb

定制系统默认开启 adb，便于调试、日志采集、文件拉取和工具协作。

### 3.3.2 破解系统签名校验

定制系统支持修改 App 后无需重签名即可安装。这对样本调试、patch 验证和快速复测有帮助。

### 3.3.3 native 函数注册监听

定制系统可以在 native 函数注册时通过 logcat 输出  RegisterNatives  相关日志，帮助快速定位：

●  so 名称

●  so 基址

●  Java 方法签名

●  native 函数地址

●  相对 so 基址的 offset

这类日志可以直接对应 IDA 中的函数偏移，减少手动 hook  RegisterNatives  的工作量。

### 3.3.4 默认整体加固脱壳

整体脱壳得到的是完整 dex，通常可以直接拉回用 jadx 分析。用 jadx 打开时仍需要关闭 checksum 校验。

### 3.4 定制系统需手动启用的能力（AI可自动操作）

### 3.4.1 Apatch 内核级 root

定制系统内置 Apatch root 能力，需要输入超级密钥启用。

这个密钥同样是 KernelPatch/APatch 相关工具的 superkey，可通过  XJB_KP_SUPERKEY  覆盖。

### 3.4.2 任意批量 so 注入

定制系统支持将 so 放入指定目录后由 App 自动加载。适合授权调试环境下进行辅助库加载、验证和工具协作。

该能力属于定制系统能力，使用前必须确认设备类型。
