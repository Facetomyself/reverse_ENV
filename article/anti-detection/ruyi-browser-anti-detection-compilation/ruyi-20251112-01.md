##  一、研究目的简述

###  原文核心

> By compiling carefully chosen computational tasks into Wasm and measuring
> their execution characteristics, we can generate fine-grained hardware-level
> fingerprints that go beyond what traditional JavaScript-based methods
> provide.

###    如意解读

论文探讨了使用  ** WebAssembly（Wasm）  ** 作为浏览器指纹识别的新途径。通过将特定计算任务编译为 Wasm
并测量其执行特征，可以生成比传统 JavaScript 指纹  更精细的、反映硬件层差异的指纹  。

传统指纹识别依赖：

  * Canvas / WebGL（图形渲染差异）

  * AudioContext（音频波形差异）

  * Navigator / Plugins（系统信息差异）

  * ....

但这些层面容易被指纹浏览器伪造，像如意2024年在清华出版社出版的《Chromium指纹浏览器开发教程》这本书里，把这些指纹改了个遍。


WebAssembly 能深入  CPU 微架构层（如缓存、流水线、时钟偏移）  ，难以伪造。
所以这篇论文的研究者是要验证  ** “Wasm 执行时间差异能否成为新型硬件指纹”  ** 。

我觉得很有可能，新的CPU架构层指纹，这是wasm执行时间推断出来的，而不是单纯魔改浏览器接口能做到的。  二、研究方法（Methodology）

###  原文核心

> We designed a set of 20 microbenchmarks in C and compiled them to
> WebAssembly. Each benchmark isolates a specific CPU behavior (e.g., floating
> point addition, integer division, memory access, etc.). We then measure
> execution time distributions using high-resolution timers available to Wasm.

###  如意解读

研究者设计了 20 个  ** 微型性能测试程序（microbenchmarks）  ** ，用 C 语言实现后编译成 Wasm 模块。
每个程序针对一种  特定的 CPU 行为  （如浮点加法、整数除法、内存访问等），并使用高精度计时器  测量其执行时间分布  。

实验涵盖：

  * 200 台真实设备

  * 14 种不同 CPU 型号

  * 5 种主流浏览器（Chrome、Firefox、Edge、Safari、Brave）


这是一个典型的  ** 硬件层时序指纹（timing fingerprint）  ** 方案。
核心思想：

> 不读取浏览器属性，而是利用代码执行“速度差异”识别硬件。

WebAssembly 拥有接近原生性能的执行环境，不受 JavaScript 引擎优化影响，可捕捉：

  * CPU 微架构差异（cache line、pipeline、branch predictor）

  * 浏览器 JIT 策略差异

  * 操作系统调度延迟特征


这让它成为“  指纹浏览器最难伪造的层  ”，实际上也没法伪造，如意现在没想到好办法。

##  三、实验结果（Results）

###  原文核心

> Our evaluation shows that the Wasm-based fingerprint achieves 93.1% device
> classification accuracy across identical browser configurations, and 87.3%
> accuracy even when browser settings or OS locale are changed.

> Compared with JavaScript-based performance fingerprints, our method provides
> 2.3× more entropy (14.5 bits vs 6.3 bits).

###  如意解读

实验表明：

  * 基于 WebAssembly 的指纹在相同浏览器配置下可达到  ** 93.1% 的设备区分准确率  ** ；

  * 即使更改浏览器设置或系统区域语言，仍保持  ** 87.3% 准确率  ** ；

  * 与基于 JavaScript 的性能指纹相比，本方法信息熵提高约  ** 2.3 倍（14.5 bits vs 6.3 bits）  ** 。


那么我觉得可以认为：

  1. Wasm 指纹能识别  ** 相同浏览器、不同设备，也就是跨设备  ** ；

  2. 不依赖浏览器配置或语言环境，  脱离传统JS接口  ；

  3. 熵值更高，意味着能更精确区分设备。

换句话说：

> 即使一个指纹浏览器伪造了所有传统 Canvas、WebGL、Navigator 数据，只要底层 CPU 不同或虚拟机模拟性能不一致，Wasm
> 测试依然能识别出“真设备 vs 模拟环境”。


##  四、防伪与可检测性分析

###  原文核心

> We tested popular anti-fingerprinting browsers (e.g., Tor Browser, Brave
> with Shields, and commercial fingerprint browsers) and found that they
> provide limited or no protection against Wasm-based timing fingerprints.
> Some even amplify the signal due to their internal sandboxing overhead.

###  如意解读

研究者测试了几种主流防指纹浏览器（如 Tor、Brave、部分商业指纹浏览器），发现：

  * 它们对基于 Wasm 的时序指纹  几乎没有防护  ；

  * 某些浏览器的沙盒隔离反而强化了这种信号（增加可识别性）。


这对风控非常关键：

  * 目前许多“反指纹浏览器”通过修改 JS API 属性欺骗检测，但无法伪造底层硬件时序；

  * WebAssembly 在沙盒中执行，但性能仍暴露出 CPU 缓存层差异；

  * 一些浏览器如 Tor 的隔离机制（sandbox）使得延迟分布更独特，反而成为新的特征。

这意味着：

> ** Wasm 时序特征  ** 可成为风控系统识别指纹浏览器或虚拟机伪装的一种强力手段。


##  五、如意对风控和检测的思考（Anti-Fraud & Fingerprint Evasion）

角度  |  启示
---|---
** 指纹识别系统设计  ** |  在 Canvas / Audio / WebGL 基础上，可增加 Wasm 时序测试模块，用来检测 CPU 真伪、虚拟化特征、沙盒执行差异。
** 指纹浏览器检测  ** |  若浏览器环境存在非原生硬件（如云环境、虚拟机、Docker、RDP），其 Wasm 任务的执行分布与真实设备不同，可作为高置信度识别信号。
** 反指纹浏览器局限  ** |  即便屏蔽 JS API 或修改 UA，Wasm 的执行延迟仍暴露真实 CPU/内存差异。
** 风控策略建议  ** |  将 Wasm 微基准测试嵌入风控 SDK，收集执行时间分布，用机器学习模型（如 SVM/Random Forest）区分正常 vs 模拟环境。
** 对抗方向（浏览器研发）  ** |  若想防御此类检测，可：1）为 Wasm 引入随机噪声计时器；2）在浏览器层屏蔽高精度计时器；3）虚拟化 CPU 行为一致化。


##  六、总结


学术界研究的新一代指纹，它不看你报什么设备信息，而是  ** 测你 CPU 真跑得怎么样  ** 。我觉得吧，是借助浏览器，而脱离浏览器的新一代硬件指纹。


因此：

  * 对风控方：这是未来识别虚拟环境和指纹浏览器的强信号。

  * 对对抗方：需要伪造的不仅是 API，而是  ** 底层硬件执行特征  ** 。
