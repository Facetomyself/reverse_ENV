##  请求顺序

###  1\. 获取页面首包

本地先请求：

  *


    GET https://www.xxxxxx.com/wealth/jglc39/index.html

需要保存和解析：

  * HTML。
  * 初始  ` Set-Cookie  ` 。
  * 页面里的 challenge 配置。
  * 页面里的 hidden input。
  * 外链 JS，尤其是瑞数 challenge 脚本和业务脚本。
  * localStorage/sessionStorage 相关初始状态。

###  2\. 生成 P Cookie

页面加载后，瑞数 challenge 逻辑会生成或刷新：

  *


    pXlaX0mT0vLDP=<encrypted-cookie-value>

这个 cookie 的生成不是简单 hash。它由多段环境字段、状态字段、动态字节数组和 AES 封包组成。

当前一键脚本通过本地固定环境模型执行站点 JS，得到满足目标页面请求链的 P Cookie。最近一次 fresh run 中：

  *


    finalPLength = 428

###  3\. 获取 token

业务请求前先发 token POST。这个请求通常没有业务 body，但 URL、Cookie、Header 仍需要满足瑞数校验。

结果返回同样是  ` vgL7Kuyn  ` 加密封包。本地解密后提取 32 位 token：

  *


    tokenSeedLength = 32

之后业务 POST 带上：

  *


    easysite7token: <32-hex-token>

###  4\. 获取业务数据

业务 POST 使用同一个目标入口：

    POST https://www.cebwm.com/eportal/ui?7ibksfL8=<encrypted-url-param>

业务 body 是：

  *


    vgL7Kuyn=<encrypted-business-body>

明文业务查询参数中可以看到类似结构：

  *


    inputQueryStr=&pageKeyStr=lccp14&currentPage=1&isGrlcStr=2&qclx=&cpFlbs=0

服务端响应仍是：

  *


    vgL7Kuyn=<encrypted-business-response>

##  加密和封包结构

这个站点当前链路的核心不是 RSA/SM2，也不是 Wasm。主体结构是：

  *   *   *   *   *   *


    明文字节-> AES-128-CBC 加密-> 拼协议前缀 / IV / 状态字节-> CRC32-> 自定义 Base64 字母表编码-> 输出成 7ibksfL8 / vgL7Kuyn / P Cookie

可以把外层封包抽象成下面的伪代码：

  *   *   *   *   *   *   *   *   *   *   *   *


    function pack(name, bodyBytes):    crc = CRC32(bodyBytes)    packet = U32BE(crc) + bodyBytes    encoded = customBase64(packet)    return name + "=0" + encodedfunction unpack(name, text):    encoded = removePrefix(text, name + "=0")    packet = customBase64Decode(encoded)    crc = U32BE(packet[0:4])    body = packet[4:]    assert crc == CRC32(body)    return body

其中  ` 0  ` 是当前版本号，  ` U32BE  ` 表示 4 字节大端整数。

封包层和加密层不要混淆：


###  AES 层

对称加密使用：


    AES-128-CBC

本地实现入口：


    encryptIY() decryptIY() decryptVgL7KuynResponse()

###  key wrapper

站点 JS 传入的 key 不是总是直接 16 字节明文 key，而可能是 wrapper 形式。当前本地逻辑会做一次归一化：


    wrapper bytes -> 读取尾部 xor 字节 -> 去掉尾部 5 字节 -> 前面字节逐个 xor -> 得到 AES key

也就是本地函数：


    normalizeIYKey()

更具体地说，key 归一化规则是：

  *   *   *   *   *   *


    if keyBytes.length == 16:    aesKey = keyByteselse:    xorByte = keyBytes[keyBytes.length - 1]    body = keyBytes[0 : keyBytes.length - 5]    aesKey = body.map(byte => byte ^ xorByte)

尾部 5 字节中，最后 1 字节是 xor byte，中间 4 字节常被用作时间或状态字段。这个 wrapper 机制让同一类 AES key 在站点 JS
里不是直接裸露为 16 字节。

###  CRC32 层

` 7ibksfL8  ` 和  ` vgL7Kuyn  ` 外层都有 CRC32 校验：

  *


    body = protocol-prefix + encrypted-bytes packet = crc32(body) + body encoded = customBase64(packet)

解包时先校验：


    crcOk = true

最近 fresh run 的业务响应解密结果：


    responseCrcOk = true plainLength = 9068

CRC32 使用标准反射多项式：


    0xedb88320

计算流程：

  *   *   *   *


    crc = 0xfffffffffor byte in body:    crc = table[(crc ^ byte) & 0xff] ^ (crc >>> 8)crc = crc ^ 0xffffffff

###  自定义 Base64

外层不是标准 Base64，而是站点自定义字母表。所有  ` 7ibksfL8  ` 、  ` vgL7Kuyn  ` 、P Cookie
都需要经过这一层编码或解码。

当前字母表：

  *


    qrcklmDoExthWJiHAp1sVYKU3RFMQw8IGfPO92bvLNj.7zXBaSnu0TC6gy_4Ze5d

它和标准 Base64 的差异：

  * 字符顺序不同。
  * 使用  ` .  ` 和  ` _  ` 。
  * 没有标准  ` =  ` padding。
  * 编码时按 3 字节一组拆成 4 个 6-bit index。
  * 解码时必须按这个字母表反查 index，不能直接用  ` Buffer.from(x, "base64")  ` 。

本地实现入口：

  *   *   *   *


    customBase64Encode()customBase64Decode()parse7ibksfL8()parseVgL7Kuyn()

###  7ibksfL8

` 7ibksfL8  ` 是 URL query 参数：


    7ibksfL8=0<custom-base64-crc-packet>

它保护的是目标接口 URL 的请求参数状态。

当前 POST URL 参数的 body 由两部分组成：


    IYParamPrefix + AES(encryptedUrlPlain)

其中 prefix 不是固定 0 字节填充，而是从状态明文中抽取出来的动态前缀：

  *


    [1, 0] + statePlain[14 : 14 + 42 + encryptedLength / 16 + extra] + encryptedLength

这意味着只知道业务明文还不够，还要有同一轮状态明文  ` statePlainBytes  ` 才能构造正确的 URL 后缀。

###  vgL7Kuyn

` vgL7Kuyn  ` 既用于请求 body，也用于响应 body：


    vgL7Kuyn=0<custom-base64-crc-packet>

请求 body 里承载业务查询参数。响应 body 里承载业务 JSON。

请求体构造：

  *   *   *   *   *


    businessPlain-> AES-128-CBC-> IYParamPrefix-> CRC32 + customBase64-> vgL7Kuyn=0...

响应体解密：

  *   *   *   *   *   *   *


    vgL7Kuyn=0...-> customBase64Decode-> CRC32 校验-> 跳过 responsePrefixLength-> AES-128-CBC 解密-> 从明文字节中查找 JSON 起点 "{"-> JSON.parse

当前业务响应的前缀长度：

  *


    responsePrefixLength = 53

当前响应明文不是从第 0 字节直接就是  ` {  ` ，所以解密后还要扫描 JSON 起点。当前本地逻辑要求最终 JSON 至少包含：

  *   *


    pageresultList

##  P Cookie 结构

P Cookie 名称：


    pXlaX0mT0vLDP

生成链路可以抽象为：

  *   *   *   *   *   *   *   *   *   *


    环境字段-> 状态字段-> seed8-> segment48-> raw74-> third segment-> cookie plain bytes-> AES-128-CBC-> 自定义 Base64 + CRC-> pXlaX0mT0vLDP

当前本地实现中相关概念：


P Cookie 的检测目的：

  * 区分真实页面执行链和直接脚本请求。
  * 绑定当前页面 challenge。
  * 绑定会话 cookie。
  * 绑定部分环境字段和 storage 状态。
  * 控制请求阶段是否允许进入业务接口。

###  P Cookie 字节布局

当前本地模型把 P Cookie 明文拆为：

  *   *   *   *   *   *   *   *   *


    plainArg0 = CRC32(payload) + payloadpayload =  0x02  0x08  seed8  0x30  segment48  0x40  segment64

字段含义：


最终：


    pXlaX0mT0vLDP = version0 + customBase64(AES-128-CBC(plainArg0))

注意：P Cookie 这里没有再套  ` name=CRC32+body  ` 这一层，而是版本号  ` 0  ` 后直接接自定义 Base64 编码后的
AES 输出。

###  segment48

` segment48  ` 是一段 48 字节环境/状态片段。当前样例中它表现为稳定段，和 cookie 第三段构造强相关。

在本地实现里，  ` segment48  ` 还会参与第三段构造：

  *   *   *   *


    property57Encode(raw74)-> 前 16 字节与 segment48[0:16] xor-> AES-128-CBC-> segment64

###  raw74

` raw74  ` 是第三段构造前的 74 字节结构化数组。它不是随机数组，而是由多个 tag/length/payload 小字段拼成。

本地模型中的关键字段：


当前构造要求  ` raw74.length === 74  ` ，长度不对时直接说明 cookie 第三段输入错了。

###  property57 编码

第三段不是直接 AES(raw74)，中间还有一层  ` property57Encode  ` 。它是一个按表编码的 bitstream 变换：

  *   *   *   *   *


    raw74 byte-> 查 PROPERTY57_TABLE 得到 code/bits-> 按 bit 写入 bitstream-> 不足 8 bit 时用 fillByte=184 补齐-> 输出字节数组

这个编码层是 P Cookie 复算里比较容易漏掉的点。只看 AES 和 Base64 会发现长度不对或第三段不一致。  ....
