##  一、目标确认

●  目标站点  ：  https://www.mihuashi.com/stalls

●  目标接口  ：  https://www.mihuashi.com/api/v1/stalls/preview?topic_count=4

●  目标参数  ：请求头中的  M-S  （签名）和  M-T  （时间戳）

●  最终目标  ：用 Node.js 补环境还原签名算法

##  二、分析工具与数据来源

使用 RuyiTrace 工具对浏览器进行 DOM/JS API 调用日志记录，日志目录：


C:\Users\Administrator\Desktop\RuyiTrace\log\

共产生 9 个 NDJSON 格式的 trace 文件，其中：

●  trace_*_31220.ndjson  （46698行）—— mihuashi.com 主页面进程，  核心分析对象

●  trace_*_11948.ndjson  （785行）—— mozilla.org 的无关进程

为什么选择 31220？  通过查看文件首行的  trace_init  记录和后续调用栈中的域名，确认 31220 是 mihuashi.com
的主渲染进程。

##  三、定位 M-S 头的设置位置

###  3.1 搜索 setRequestHeader

在 trace 日志中搜索  setRequestHeader  ，找到 XHR 请求设置 header 的位置（第 8895-8907 行）：


{"type":"call","interface":"XMLHttpRequest","member":"setRequestHeader",
"args":["Accept","application/json, text/plain, */*"]}
{"type":"call","interface":"XMLHttpRequest","member":"setRequestHeader",
"args":["Authorization","Bearer null"]}
{"type":"call","interface":"XMLHttpRequest","member":"setRequestHeader",
"args":["M-S","3F9aJFJZweno_oVRtVCO+RWLM1mUV1GOB9Wb40CZtMGarRjMwYjM5cTO0gTbV9GTvZGO4QHW39WIYFSMSpFZtYlVSd2ZzR0ZSJVTzxDP"]}
{"type":"call","interface":"XMLHttpRequest","member":"setRequestHeader",
"args":["M-T","1778571128"]}
{"type":"call","interface":"XMLHttpRequest","member":"setRequestHeader",

关键发现  ：

●  M-T  =  1778571128  （秒级 Unix 时间戳）

●  M-S  = 104 字符的签名字符串

●  调用栈来自  isObject.BpF7pdk3.js  （axios 封装层）

###  3.2 确认 headers 对象的组装

在第 8841 行找到完整的 headers 对象：


{
"Accept": "application/json, text/plain, */*",
"Content-Type": "undefined",
"Authorization": "Bearer null",
"M-S": "3F9aJFJZweno_oVRtVCO+...",
"M-T": "1778571128",
"Web-Version": "frontend"
}

为什么要找这个？  确认 M-S 和 M-T 是在同一个拦截器中一起设置的，说明它们的生成逻辑在同一处。

###  4.1 找到 M-S 值首次出现的位置

结论  ：签名算法封装在 Rust 编译的 WebAssembly 模块  mhs_fe_sign_bg.DLpTGLRB.wasm  中。

###  4.3 确认时间戳的生成

在第 8736-8737 行：


{"type":"call","interface":"Date","member":"now","return":1.77857e+12,
"stack":[{"func":"getNow"},{"func":"Y"},{"func":"timeGetter","file":"http.BzJ4_4Bj.js"}]}
{"type":"call","interface":"Math","member":"floor","args":[1.77857e+09],"return":1778571128,
"stack":[{"func":"getNow"},{"func":"Y"},{"func":"timeGetter","file":"http.BzJ4_4Bj.js"}]}

结论  ：  M-T = Math.floor(Date.now() / 1000)  ，即秒级时间戳。

##  五、分析 WASM 的环境依赖

###  5.1 WASM 读取 favicon URL

第 7712-7715 行，WASM 在初始化时查询 DOM：


{"type":"call","interface":"TextDecoder","member":"decode",
"args":[{"0":108,"1":105,"2":110,"3":107,...}],
"return":"link[rel*='icon']"}

{"type":"call","interface":"Document","member":"querySelector",
"args":["link[rel*='icon']"],
"return":"[Object HTMLLinkElement]"}

{"type":"call","interface":"Element","member":"getAttribute",
"args":["href"],
"return":"https://js-assets.mihuashi.com/mhs-assets/legacy/favicon.ico"}

为什么 WASM 要读 favicon？  这是一种反爬虫手段——将页面特征（favicon
URL）作为签名密钥的一部分。如果在非浏览器环境中运行，没有正确的 DOM 结构，签名就会失败。
