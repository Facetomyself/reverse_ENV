##  一、JNI 是什么：Java 和 Native 的通信协议

###  1.1 问题场景

在 jadx 中，你经常会看到这样的 Java 方法声明：


    public class NativeLib {
        static { System.loadLibrary("native"); }

        public static native String encrypt(String input, String key);
        public static native byte[] sign(byte[] data);
    }


` native ` 关键字告诉 JVM：这个方法的实现不在 Java 字节码中，而是在 Native 库（SO 文件）里。当 Java 代码调用 `
NativeLib.encrypt("hello", "key123") ` 时，ART 虚拟机需要找到对应的 C/C++ 函数来执行。

JNI 就是定义「Java 方法如何找到并调用 Native 函数」的规范。它规定了 Native 函数的命名规则、参数传递方式、类型映射关系等。理解
JNI 的工作方式，是分析 Java-Native 混合应用的基础。

###  1.2 JNIEnv：Native 代码的 Java 世界入口

每个 Native 函数的第一个参数都是 ` JNIEnv* ` ——一个指向 JNI 函数表的指针。通过这个指针，Native 代码可以执行几乎所有
Java 操作：创建 Java 对象、调用 Java 方法、访问 Java 字段、抛出 Java 异常等。

` JNIEnv* ` 的结构是一个函数指针表（vtable），包含 230 多个函数指针。每个指针对应一个 JNI 操作。这个表的布局是 JNI
规范强制规定的——不管 Android 版本如何变化，第 N 个位置的函数永远是同一个操作。

类比一下：JNIEnv 函数表就像 Linux 内核的 syscall 表——内核里 syscall N 对应固定的系统调用号；JNIEnv 的
vtable 也是这样，索引 N 对应固定的 JNI 操作。开发者无法改变这个映射，逆向者就能用索引直接定位实现，不依赖符号表是否被 strip。

下面这张表先扫一眼有个印象就行——本篇真正反复用到的只有 **215 RegisterNatives** ；其他几个是为了让你看清「vtable
是一张连续的表、函数按规律连排」。等读到第 4 章 反向追踪那节，FindClass / GetMethodID / Call*Method
这几行才会派上用场。表里的索引以 AOSP ` libnativehelper/include_jni/jni.h ` 的 `
JNINativeInterface ` 结构体声明顺序为准：

###  1.3 两种注册方式的深度对比

为什么会有两种？简单说：静态注册是 JVM
自动建立映射的默认方式，写起来省心；动态注册是开发者「主动隐藏函数对应关系」的高阶手段，在反逆向场景里几乎是标配。下面拆开看。

先说静态注册——JVM 自动建立映射的默认方式。它通过 **函数命名规则** 来对应：JVM 看到 Java 类 `
com.example.NativeLib ` 里有 ` native String encrypt(String, String) ` ，就去 SO
的导出表查找名为 ` Java_com_example_NativeLib_encrypt ` 的函数。找到就建立映射，后续每次 Java 层调 `
encrypt ` ，ART 直接跳到这个 Native 函数。

这种方式的优势是 **省事** （开发者只管写函数，命名按规则就行）；劣势是 **函数名等于答案** ——对逆向者来说，看一眼 SO 的导出表 `
Java_com_example_NativeLib_encrypt ` 就直接知道「这是 NativeLib.encrypt 的实现」。

类比一下： **静态注册像是把名字直接刻在门牌上** ，谁来都能找到；反逆向团队当然不会满足于此。

再说动态注册——开发者主动设计的「隐藏映射」方式。它在 SO 的 ` JNI_OnLoad ` 函数里通过 ` RegisterNatives ` API
**运行时手动建立** ：开发者可以给 Native 函数取任意名字 ` sub_1234 ` 、 ` a ` 、甚至 **根本不导出** ，然后等 SO
加载时通过 ` RegisterNatives ` 告诉 JVM「Java 方法 X 对应的 Native 函数在地址 Y」。

类比一下： **动态注册像是 App 跑起来后，才在前台小黑板上贴一张映射表** ——而且地址可以填任意混淆函数，导出表里什么都看不见。


    // 动态注册的示例代码（C 层）
    static JNINativeMethod methods[] = {
        // { Java方法名, 签名, Native函数指针 }
        {"encrypt", "(Ljava/lang/String;Ljava/lang/String;)Ljava/lang/String;", (void*)real_encrypt_func},
        {"sign",    "([B)[B",                                                   (void*)real_sign_func},
    };

    JNIEXPORT jint JNI_OnLoad(JavaVM *vm, void *reserved) {
        JNIEnv *env;
        vm->GetEnv((void**)&env, JNI_VERSION_1_6);  // 拿 JNI 1.6 规范的 env

        jclass clazz = env->FindClass("com/example/NativeLib");
        env->RegisterNatives(clazz, methods, sizeof(methods)/sizeof(methods[0]));

        return JNI_VERSION_1_6;  // 返回值告诉 JVM「我用的是 JNI 1.6」,Android 常见 1.4/1.6
    }


在这段代码中， ` real_encrypt_func ` 和 ` real_sign_func ` 可以是 SO 内部的任意函数——它们不需要遵循 `
Java_ ` 前缀规则，也不需要被导出。在 SO 的导出表中你可能完全看不到它们，但它们确实是 Java ` native ` 方法的实现。

这就是为什么你在第21篇 第 3.1 节 用 ` enumerateSymbols ` 扫 SO 时，有时候找不到任何 ` Java_ `
开头的符号——App 使用了动态注册。要找到真正的 Native 函数地址，光靠符号表枚举不够，得 Hook ` RegisterNatives ` 。

##  二、Hook RegisterNatives：揭开动态注册的面纱

###  2.1 定位 RegisterNatives 函数

` RegisterNatives ` 是 ART 虚拟机内部的一个 C++ 函数。我们的脚本要 hook
它，得先拿到它在内存里的绝对地址。两条路径，优先走第一条、不行再退到第二条：

**先试符号表** ——ART 的核心实现在 ` libart.so ` 里，如果符号没被 strip，直接遍历 ` libart.so `
的符号表找名字带「RegisterNatives」的就行：


    function findRegisterNatives() {
        var addr = null;
        var artModule = Process.findModuleByName("libart.so");
        if (!artModule) return null;

        artModule.enumerateSymbols().forEach(function(sym) {
            // 查找包含 RegisterNatives 但不包含 CheckJNI 的符号
            // (CheckJNI 是 ART 同时编译的「带参数检查的包装版本」,
            //  名字里也含 RegisterNatives 字样;我们要的是真正干活的实现入口,
            //  所以剔掉 CheckJNI 这一版)
            if (sym.name.indexOf("RegisterNatives") !== -1 &&
                sym.name.indexOf("CheckJNI") === -1 &&
                !addr) {
                addr = sym.address;
                console.log("[*] RegisterNatives 通过符号找到: " + addr);
                console.log("    符号名: " + sym.name);
            }
        });

        return addr;
    }


**符号被 strip 怎么办？走 vtable 偏移兜底。** 商业反检测 App 自带的 libart.so / 自编 AOSP / 部分 OEM
老版本 ROM 都会把 ART 符号 strip 掉，这时 ` enumerateSymbols ` 拿到的全是匿名地址，搜不到 `
RegisterNatives ` 字样。但还有救—— ` RegisterNatives ` 在 JNIEnv 函数表里 **永远是第 215 个**
（JNI 规范强制规定的固定槽位），数过去拿地址就行：


    function findRegisterNativesViaVtable() {
        // 获取 JNIEnv 指针
        var env = Java.vm.getEnv();
        var envPtr = env.handle;

        // JNIEnv* 指向一个函数指针表（vtable）
        // 第一次解引用得到 vtable 的基地址
        var vtablePtr = Memory.readPointer(envPtr);

        // RegisterNatives 在 vtable 中的索引是 215
        // 每个函数指针的大小是 Process.pointerSize（64位系统是8字节）
        var registerNativesPtr = Memory.readPointer(
            vtablePtr.add(215 * Process.pointerSize)
        );

        console.log("[*] RegisterNatives 通过 vtable 找到: " + registerNativesPtr);
        return registerNativesPtr;
    }


这条兜底路径有个直观类比： **就算把百货大楼里所有商店招牌都拆了，只要你知道「五楼是家电区」，数楼层一样能找到 — vtable 第 215 个槽位永远是
RegisterNatives，跟招牌（符号名）在不在没关系** 。索引 215 是 JNI 规范规定的，所有 Android 版本一致。

###  2.2 完整的 RegisterNatives 监控脚本

下面把脚本拆成四段：① 定位 ` RegisterNatives ` 地址；② Hook 上 ` onEnter ` 、解析 `
JNINativeMethod[] ` ；③ 配套的 JNI 签名翻译；④ 把三段装配起来跑。读起来按 ① → ② → ③ → ④ 顺序，跑起来 Frida
注入后立即定位、立即挂 hook。

**第一段：定位 RegisterNatives**


    // hook_register_natives.js (1/4) — 定位
    // 优先用 libart.so 的符号表；符号被 strip 时回退到 vtable 偏移
    function findRegisterNatives() {
        var artModule = Process.findModuleByName("libart.so");
        if (artModule) {
            var found = null;
            artModule.enumerateSymbols().forEach(function(sym) {
                if (!found &&
                    sym.name.indexOf("RegisterNatives") !== -1 &&
                    sym.name.indexOf("CheckJNI") === -1) {
                    found = sym.address;
                }
            });
            if (found) return found;
        }

        // 回退方案：vtable 偏移（索引 215，JNI 规范固定）
        try {
            var env = Java.vm.getEnv();
            var vtable = Memory.readPointer(env.handle);
            return Memory.readPointer(vtable.add(215 * Process.pointerSize));
        } catch(e) {
            return null;
        }
    }


定位返回的是 ` RegisterNatives ` 函数指针。下一步把 ` Interceptor.attach ` 挂到这个地址，解析它的四个参数 `
(env, jclass, methods*, count) ` 。

**第二段：Hook 与解析 JNINativeMethod 数组**


    // hook_register_natives.js (2/4) — Hook 与解析
    function attachRegisterNatives(addr) {
        Interceptor.attach(addr, {
            onEnter: function(args) {
                var jclass = args[1];
                var methodsPtr = args[2];
                var methodCount = args[3].toInt32();

                // 通过 JNI 反射拿 Java 类名：把 jclass cast 成 Class 对象再 .getName()
                var className = "(未知)";
                try {
                    Java.perform(function() {
                        className = Java.cast(jclass, Java.use("java.lang.Class")).getName();
                    });
                } catch(e) {
                    className = "class@" + jclass;
                }

                console.log("\n╔══════════════════════════════════════════");
                console.log("║ RegisterNatives: " + className);
                console.log("║ 注册 " + methodCount + " 个 native 方法");
                console.log("╠──────────────────────────────────────────");

                // JNINativeMethod 内存布局：{ name*, signature*, fnPtr* } —— 三个指针连排
                var structSize = Process.pointerSize * 3;

                for (var i = 0; i < methodCount; i++) {
                    var entry = methodsPtr.add(i * structSize);
                    var name = Memory.readUtf8String(Memory.readPointer(entry));
                    var sig = Memory.readUtf8String(Memory.readPointer(entry.add(Process.pointerSize)));
                    var fnPtr = Memory.readPointer(entry.add(Process.pointerSize * 2));

                    // 计算 Native 函数所在模块的偏移（能直接在 IDA 里 G 跳）
                    var moduleInfo = Process.findModuleByAddress(fnPtr);
                    var modName = moduleInfo ? moduleInfo.name : "???";
                    var offset = moduleInfo
                        ? "0x" + fnPtr.sub(moduleInfo.base).toString(16)
                        : "???";

                    console.log("║");
                    console.log("║ [" + (i + 1) + "] " + className + "." + name);
                    console.log("║     签名: " + sig);
                    console.log("║     地址: " + fnPtr);
                    console.log("║     位置: " + modName + " + " + offset);
                    console.log("║     解析: " + parseJniSignature(sig));
                }

                console.log("╚══════════════════════════════════════════\n");
            }
        });
    }


` parseJniSignature ` 是个辅助函数，把 ` (Ljava/lang/String;I[B)V ` 这种 JNI 签名翻译成人类可读形式
` (String, int, byte[]) → void ` 。下面单独写。

**第三段：JNI 签名翻译**


    // hook_register_natives.js (3/4) — 签名翻译
    function parseJniSignature(sig) {
        var types = {
            "Z": "boolean", "B": "byte", "C": "char", "S": "short",
            "I": "int", "J": "long", "F": "float", "D": "double", "V": "void"
        };

        // 解析单个类型符号；支持任意维度数组前缀（如 [[B → byte[][]）
        function parseOne(s, i) {
            var dims = 0;
            while (s[i] === '[') { dims++; i++; }
            var t;
            if (types[s[i]]) {
                t = types[s[i]];
                i++;
            } else if (s[i] === 'L') {
                var end = s.indexOf(';', i);
                t = s.substring(i + 1, end).replace(/\//g, '.').replace("java.lang.", "");
                i = end + 1;
            } else {
                t = s[i] || "?";
                i++;
            }
            for (var d = 0; d < dims; d++) t += "[]";
            return { type: t, next: i };
        }

        try {
            var params = [];
            var i = 1;  // 跳过开头的 '('
            while (i < sig.length && sig[i] !== ')') {
                var p = parseOne(sig, i);
                params.push(p.type);
                i = p.next;
            }
            var ret = parseOne(sig, i + 1);  // 跳过 ')' 后解析返回类型
            return "(" + params.join(", ") + ") → " + ret.type;
        } catch(e) {
            return sig;
        }
    }


**第四段：装配入口**


    // hook_register_natives.js (4/4) — 装配
    // 用法: frida -U -f <包名> -l hook_register_natives.js --no-pause
    (function() {
        console.log("[*] 开始查找 RegisterNatives...");
        var addr = findRegisterNatives();
        if (!addr) {
            console.log("[!] 无法定位 RegisterNatives");
            return;
        }
        console.log("[*] RegisterNatives @ " + addr);

        attachRegisterNatives(addr);

        console.log("[*] RegisterNatives 监控已启动");
        console.log("[*] 等待 App 加载 Native 库...\n");
    })();


Frida 注入后立即执行：找到 ` RegisterNatives ` → 挂 hook → 等待 App 调用 ` System.loadLibrary
` 触发 ` JNI_OnLoad ` 里的 ` RegisterNatives ` 。一旦触发，onEnter 就会把所有动态注册的方法连同它们的真实
SO 偏移一起吐出来。

> **spawn vs attach** ── 上面这条路径以 ` frida -f <包名> ` spawn 模式为前提。Frida 在 App
> 第一条指令执行前就挂上 hook，所以能截到 ` JNI_OnLoad ` 里的全部 ` RegisterNatives ` 。 **attach
> 模式则不行** ：App 已经跑过初始化， ` JNI_OnLoad ` 一波早过完，onEnter 只能截到运行时延迟加载的 SO（看 第 6 章
> 输出节选可对比）。同主题的真机案例见 ` scripts/frida22/outputs/qq_register_natives_full.txt `
> ——同一段脚本在 Pixel 5 上 spawn QQ 音乐 30 秒截到 7 次 ` RegisterNatives ` / 94 个方法。
>
> **Android 14 spawn 环境坑** ── frida 16.7.19 + Android 14 spawn 模式会在 `
> libframework-connectivity-jni.so ` dlopen 失败处闪退（namespace 隔离
> bug，与本篇脚本无关）。修法是拦 ` android_dlopen_ext ` 把该 SO 重定向到 `
> /apex/com.android.tethering/lib64/ ` 绝对路径，完整 patch 见 `

**拓展：同一目标的两种写法**

上面这版接近一百行，带回退、带 parseJniSignature 翻译、带 module 为 null
的防御——「教科书写法」。实战里你会看到更短的版本，像下面这样的：


    // 简洁写法 —— 适合个人快速分析
    Module.enumerateSymbolsSync("libart.so").forEach(function(sym) {
        if (sym.name.indexOf("art") >= 0 &&
            sym.name.indexOf("JNI") >= 0 &&
            sym.name.indexOf("RegisterNatives") >= 0 &&
            sym.name.indexOf("CheckJNI") < 0) {
            Interceptor.attach(sym.address, {
                onEnter: function (args) {
                    var clazz = Java.vm.tryGetEnv().getClassName(args[1]);
                    var p = ptr(args[2]);
                    var count = parseInt(args[3]);
                    for (var i = 0; i < count; i++) {
                        var name = Memory.readCString(Memory.readPointer(p.add(i * 3 * Process.pointerSize)));
                        var sig  = Memory.readCString(Memory.readPointer(p.add((i * 3 + 1) * Process.pointerSize)));
                        var fn   = Memory.readPointer(p.add((i * 3 + 2) * Process.pointerSize));
                        var mod  = Process.findModuleByAddress(fn);
                        console.log(clazz + "." + name + sig + " → " + mod.name + " + " + fn.sub(mod.base));
                    }
                }
            });
        }
    });


20 多行，核心三件套（符号查找 + ` Interceptor.attach ` \+ 按 3 倍指针步长读 ` JNINativeMethod `
）和教科书版完全一样， **原理无差** 。但有 5 处工程化差异决定了它们各自最适合的场景：

维度  |  简洁写法  |  教科书写法  |  实际影响
---|---|---|---
**符号 strip 回退** |  无，strip 后直接废  |  vtable[215] 偏移兜底  |  加固 App / 自编 AOSP / 部分 OEM strip 后的 libart 上，简洁版找不到地址
**module 为 null 防御** |  无，直接 ` mod.name ` 会 throw  |  ` mod ? name : "???" ` 三元判断  |  加固 App 把 native 函数解密到匿名 mmap 段时（360 / 易盾常见），简洁版本次 onEnter 后续 method 全断
**类名获取** |  ` Java.vm.tryGetEnv().getClassName() ` 直拿，不进 Java.perform  |  ` Java.perform + Java.cast + getName() ` |  简洁版快一个数量级；代价是用了 Frida 非公开 API，大版本升级有变动风险
**JNI 签名翻译** |  无，直接吐 ` (Ljava/lang/String;[B)V ` |  ` parseJniSignature ` 翻成 ` (String, byte[]) → void ` |  老手脑算签名快；新人和复杂签名场景需要翻译
**偏移格式** |  默认十进制大数  |  加 ` 0x ` 前缀的 hex 字符串  |  教科书版可直接 IDA 按 G 跳；简洁版要手动转 hex

**取舍的工程化视角** ：

  * **个人快速分析** → 简洁写法。短、快、改造门槛低，标准 ROM + 标准 libart 上够用。
  * **加固 App / strip 后的 libart / 写给别人用** → 教科书写法。两条致命路径（strip 回退 + module 防御）决定能否跑通。
  * **兼得二者优点** ：把简洁版的 ` tryGetEnv().getClassName() ` 抄进教科书版替掉 ` Java.cast ` 那段（性能 + 简洁双赢）；把教科书版的 vtable 回退 + module 防御抄进简洁版（覆盖 strip 和加固两个致命场景）。15 分钟做完，各取所长。

写代码不是审美比赛，选哪种取决于「这段脚本要服务谁」。本篇主线代码偏向教科书写法，正是为了让从未碰过 JNI 动态注册的读者照搬就能跑通。等你熟了，自己
daily 用的脚本完全可以瘦身到 20 行。

##  三、自动 Hook 动态注册的 Native 方法

到这里你已经拿到所有 native 方法的 SO 偏移，可以挨个手写 ` Interceptor.attach ` 去 hook。 **有更省事的做法**
：在 ` RegisterNatives ` 被调用的那一刻，顺手把每个注册过来的方法都自动挂上 hook，不用回头再写一遍。


    // auto_hook_registered.js
    // 自动 Hook 所有通过 RegisterNatives 注册的 Native 方法

    (function() {
        var targetClassName = "com.example.app.NativeLib";  // 替换为你关注的类

        // 沿用 第 2.2 节 第一段的 findRegisterNatives 函数(把它整段拷过来即可)
        var registerNativesAddr = findRegisterNatives();
        if (!registerNativesAddr) {
            console.log("[!] 无法定位 RegisterNatives");
            return;
        }

        Interceptor.attach(registerNativesAddr, {
            onEnter: function(args) {
                var methodsPtr = args[2];
                var methodCount = args[3].toInt32();

                // 获取类名
                var className = "";
                try {
                    Java.perform(function() {
                        className = Java.cast(args[1], Java.use("java.lang.Class")).getName();
                    });
                } catch(e) {}

                // 只对我们关心的类进行自动 Hook
                if (className !== targetClassName) return;

                var structSize = Process.pointerSize * 3;

                for (var i = 0; i < methodCount; i++) {
                    var entry = methodsPtr.add(i * structSize);
                    var name = Memory.readUtf8String(Memory.readPointer(entry));
                    var sig = Memory.readUtf8String(Memory.readPointer(entry.add(Process.pointerSize)));
                    var fnPtr = Memory.readPointer(entry.add(Process.pointerSize * 2));

                    // 为每个注册的方法设置 Hook
                    // 使用闭包保存当前迭代的变量
                    (function(methodName, methodSig, funcPtr) {
                        try {
                            Interceptor.attach(funcPtr, {
                                onEnter: function(args) {
                                    console.log("\n[JNI] " + className + "." + methodName +
                                                " 被调用");
                                    // 通用参数打印（从 args[2] 开始是实际参数）
                                    for (var j = 2; j < 6; j++) {
                                        try {
                                            console.log("  arg" + (j-2) + ": " + args[j]);
                                            var str = Memory.readUtf8String(args[j], 64);
                                            if (str) console.log("    → " + str);
                                        } catch(e) {}
                                    }
                                },
                                onLeave: function(retval) {
                                    console.log("  返回: " + retval);
                                }
                            });
                            console.log("[*] 自动 Hook: " + methodName + methodSig);
                        } catch(e) {
                            console.log("[!] Hook 失败: " + methodName + " - " + e.message);
                        }
                    })(name, sig, fnPtr);
                }
            }
        });
    })();


注意这段代码中使用了 **IIFE（Immediately Invoked Function Expression）** 闭包模式—— `
(function(methodName, methodSig, funcPtr) { ... })(name, sig, fnPtr) `
。这是处理「循环里注册的延迟回调」必备的技巧，原因是 JavaScript ` var ` 是函数级作用域而非块级：循环结束后， ` name ` 、 `
sig ` 、 ` fnPtr ` 都会停留在最后一次迭代的值；之后任何被注册函数的 ` onEnter `
触发时，拿到的都是同一份（最后一组）信息。IIFE 把当前迭代的值作为参数传进新函数，等于为每个 Hook 回调单独留了一份变量副本。

> Frida 的 QuickJS 引擎支持 ES6，更简洁的写法是把 ` var ` 换成 ` let ` （ ` let `
> 是块级作用域，每次循环迭代会创建新绑定），就可以省掉 IIFE。本篇为了兼容更老的 Frida 版本和示意作用域问题，仍用 ` var ` \+
> IIFE 写法。

##  四、Native 回调 Java：反向追踪

到这里 Java → Native 的拦截路径已经完整了：hook ` RegisterNatives ` 拿映射、自动 hook 每个 native
方法的入口、参数解析也搞定。但 **JNI 其实是双向桥** ——Native 代码也能反过来调 Java
方法，这条路在加密、签名、事件回调里是常态，逆向时漏看会丢掉一半信息。

在 SO 里，反向调用的典型代码长这样：


    // Native 代码调用 Java 方法
    jclass utilsClass = env->FindClass("com/example/Utils");
    jmethodID encodeMethod = env->GetStaticMethodID(utilsClass, "base64Encode", "([B)Ljava/lang/String;");
    jstring result = (jstring) env->CallStaticObjectMethod(utilsClass, encodeMethod, dataArray);


这段代码在 Native 层查找 Java 类 ` com.example.Utils ` ，获取其静态方法 ` base64Encode ` 的
ID，然后调用这个方法。这种模式在 Native 加密实现中很常见——加密逻辑在 Native 层完成，但编码、网络请求等辅助操作仍然调用 Java
层的方法。

注意上面三行 C 代码，正好对应下面三节的三个 hook 脚本——它们不是三个独立工具，而是「Native 调
Java」这条标准动作链上的三个观察点，各自代价不一样：

C 代码动作  |  对应 hook  |  索引  |  看到什么  |  代价
---|---|---|---|---
` env->FindClass("...") ` |  第 4.1 节 hook FindClass  |  6  |  Native 想拿哪些 Java 类  |  输出量小，可全程开
` env->GetStaticMethodID(..., "...") ` |  第 4.2 节 hook GetMethodID  |  33 / 113  |  Native 想调哪些方法  |  输出中等，建议过滤类名
` env->CallStaticObjectMethod(...) ` |  第 4.3 节 hook Call*Method  |  34 / 37 / 49 / 61 / 114 / ...  |  实际参数 + 返回值  |  输出爆炸，必须 backtrace 过滤

**实战推荐顺序** ： **先开 FindClass 摸地图** → 看 Native 关心哪些 Java 类； **再开 GetMethodID
拿菜单** → 看它想调什么方法； **最后只对感兴趣的 methodID 开 Call*Method** →
看实际参数。这样代价递增、信息越来越具体，而不是上来就 hook Call*Method 被输出淹没。

##  五、JNI 类型转换实战

在 Native Hook 中处理 JNI 类型是最常遇到的技术困难。这里总结几种最常见的类型转换场景。

先讲一条 **贯穿全节的规矩** ：JNI 里凡是「Get 出来一段数据」的函数，都得搭配对应的 ` Release ` 用完通知 JVM——比如 `
getStringUtfChars ` 配 ` releaseStringUtfChars ` 、 ` getByteArrayElements ` 配 `
releaseByteArrayElements ` 。原因是 JVM 实现不一定真给你内部指针，可能给的是一份拷贝；无论哪种，你不 release
就会内部锁不释放或拷贝内存泄漏。 **Get 了什么就 Release 什么，这是 JNI 的对账规则** ，下面每段代码都遵守。

###  5.1 读取 jstring 参数


    // 在 Interceptor.attach 的 onEnter 中
    function readJString(jstr) {
        if (jstr.isNull()) return "null";
        var env = Java.vm.getEnv();
        var strPtr = env.getStringUtfChars(jstr, null);
        var str = Memory.readUtf8String(strPtr);
        env.releaseStringUtfChars(jstr, strPtr);
        return str;
    }

    // 使用：
    onEnter: function(args) {
        var input = readJString(args[2]);  // args[2] 是 jstring 类型的参数
        console.log("输入: " + input);
    }


> 如果 onEnter 触发的线程没有 attach 到 JVM（少数 Native 自起线程的场景）， ` Java.vm.getEnv() ` 会返回
> null 或抛错。稳妥写法是 ` var env = Java.vm.tryGetEnv() ||
> Java.vm.attachCurrentThread(); ` ，第 17 篇反调试章节给出过更详细的版本。

###  5.2 读取 jbyteArray 参数


    function readJByteArray(jarray) {
        if (jarray.isNull()) return null;
        var env = Java.vm.getEnv();
        var len = env.getArrayLength(jarray);
        var ptr = env.getByteArrayElements(jarray, null);

        var bytes = Memory.readByteArray(ptr, len);
        env.releaseByteArrayElements(jarray, ptr, 0);

        // 转换为十六进制字符串
        var hex = [];
        var view = new Uint8Array(bytes);
        for (var i = 0; i < view.length; i++) {
            var b = view[i].toString(16);
            hex.push(b.length === 1 ? "0" + b : b);
        }
        return hex.join("");
    }


###  5.3 更简单的方式：在 Java 层 Hook

对于 JNI 接口函数（Java 声明的 ` native ` 方法），在 Java 层 Hook 永远是更简单的选择——Frida
会自动处理所有类型转换：


    Java.perform(function() {
        var NativeLib = Java.use("com.example.app.NativeLib");

        // 在 Java 层 Hook native 方法，参数自动转换为 JS 类型
        NativeLib.encrypt.implementation = function(input, key) {
            console.log("input: " + input);   // 自动从 jstring 转为 JS string
            console.log("key: " + key);
            return this.encrypt(input, key);  // 返回值也自动转换
        };

        NativeLib.sign.implementation = function(data) {
            // data 是 byte[]，在 Frida 中是一个可以遍历的数组
            console.log("data length: " + data.length);
            return this.sign(data);
        };
    });


**什么时候必须在 Native 层 Hook 而不能用 Java 层？**

当你要分析的不是 JNI 接口函数本身，而是它内部调用的子函数时。比如 ` encrypt ` 的 JNI 接口函数在内部调用了 ` aes_encrypt
` → ` key_expansion ` → ` sub_bytes ` 这一系列算法函数——这些子函数不对应任何 Java 方法，只能在 Native
层通过地址 Hook。

##  六、综合实战：追踪一个动态注册的加密 SO

![JNI
JNI 综合分析流程

把本篇的 RegisterNatives 拦截和第21篇的地址定位手段整合成一个完整的分析流程。脚本按时序分三段：

  1. **第一段** 钩 ` System.loadLibrary ` ，看哪些 SO 被加载、导出表里是否还有 ` Java_ ` 前缀（判断走静态注册还是动态注册）。
  2. **第二段** 钩 ` RegisterNatives ` ，对每个注册的方法吐出 Java 类名 / 方法名 / SO 偏移，并立刻给它挂一个 onEnter/onLeave 简单日志。
  3. **第三段** 在 Java 层也挂 hook（参数自动转 JS 类型，比 Native hook 更易读）。

**第一段：监控 SO 加载**


    // jni_analysis.js (1/3) — 监控 SO 加载
    console.log("[*] JNI 综合分析脚本启动\n");

    Java.perform(function() {
        var System = Java.use("java.lang.System");
        System.loadLibrary.implementation = function(lib) {
            console.log("[SO加载] System.loadLibrary: " + lib);
            this.loadLibrary(lib);
            // 加载完成后立刻枚举这个 SO 的导出函数
            var soName = "lib" + lib + ".so";
            var mod = Process.findModuleByName(soName);
            if (mod) {
                var exports = mod.enumerateExports()
                    .filter(function(e) { return e.type === "function"; });
                var jniExports = exports.filter(function(e) {
                    return e.name.indexOf("Java_") === 0;
                });
                console.log("  导出函数: " + exports.length + " 个");
                console.log("  JNI 函数: " + jniExports.length + " 个");
                if (jniExports.length === 0) {
                    console.log("  → 没有 Java_ 前缀的导出，可能使用了动态注册");
                }
            }
        };
    });


> **三层加载路径的关系** ： ` System.loadLibrary("nesec") ` 内部其实最终也是调 `
> dlopen("libnesec.so") ` 加载 SO 的—— **只是绕了 Java → Native 的桥** 。所以钩 `
> loadLibrary ` 截到「通过 Java 路径加载」的、钩 ` System.load(全路径) ` 截到「用全路径而非短名加载」的、钩 `
> dlopen ` 截到「Native 代码自己 dlopen」的——三层互补。本段只覆盖第一层短名加载；要全面监控得把三处一起
> hook（完整版见配套脚本 ` scripts/frida22/recipe/hook_with_loadlibrary.js ` ）。

**第二段：监控 RegisterNatives 并自动 Hook**


    // jni_analysis.js (2/3) — RegisterNatives + 自动 Hook
    (function() {
        var addr = null;
        var art = Process.findModuleByName("libart.so");
        if (art) {
            art.enumerateSymbols().forEach(function(s) {
                if (!addr && s.name.indexOf("RegisterNatives") !== -1 &&
                    s.name.indexOf("CheckJNI") === -1) {
                    addr = s.address;
                }
            });
        }
        if (!addr) {
            try {
                var env = Java.vm.getEnv();
                var vt = Memory.readPointer(env.handle);
                addr = Memory.readPointer(vt.add(215 * Process.pointerSize));
            } catch(e) {}
        }
        if (!addr) return;

        Interceptor.attach(addr, {
            onEnter: function(args) {
                var methodsPtr = args[2];
                var count = args[3].toInt32();
                var ss = Process.pointerSize * 3;

                var className = "";
                try {
                    Java.perform(function() {
                        className = Java.cast(args[1], Java.use("java.lang.Class")).getName();
                    });
                } catch(e) {}

                console.log("\n[RegisterNatives] " + className + " (" + count + " 个方法)");

                for (var i = 0; i < count; i++) {
                    var e = methodsPtr.add(i * ss);
                    var name = Memory.readUtf8String(Memory.readPointer(e));
                    var sig = Memory.readUtf8String(Memory.readPointer(e.add(Process.pointerSize)));
                    var fn = Memory.readPointer(e.add(Process.pointerSize * 2));
                    var mi = Process.findModuleByAddress(fn);
                    var offset = mi ? "0x" + fn.sub(mi.base).toString(16) : "?";

                    console.log("  " + name + sig + " → " +
                                (mi ? mi.name : "?") + " + " + offset);

                    // 自动给每个注册的方法挂日志 hook（用 IIFE 捕获当前迭代的变量）
                    (function(n, f) {
                        try {
                            Interceptor.attach(f, {
                                onEnter: function(a) { console.log("[Native] " + n + " 被调用"); },
                                onLeave: function(r) { console.log("[Native] " + n + " 返回: " + r); }
                            });
                        } catch(err) {}
                    })(name, fn);
                }
            }
        });

        console.log("[*] RegisterNatives 监控已启动");
    })();


到这里 Native 偏移和 onEnter/onLeave 日志都有了。如果想看完整的参数语义（jstring 自动转 JS string、byte[]
自动转数组），还得回到 Java 层 hook。

**第三段：Java 层 native 方法 Hook**


    // jni_analysis.js (3/3) — Java 层 hook

    // 加载到 ART 里, Java.use 直接抛 ClassNotFoundException.
    // setTimeout 5 秒是图省事 (覆盖大多数 App 的冷启动);
    // 更稳的做法: 在第二段 onEnter 看到目标类名时再触发本段 hook.
    setTimeout(function() {

                    return result;

                console.log("[*] Java 层 native 方法 Hook 成功");
            } catch(e) {

##  总结

本篇核心： **用 hook` RegisterNatives ` （索引 215）拦下 SO 加载时的动态注册行为 ** ，把 Java native
方法 → SO 偏移的隐藏映射全部摊开。这些偏移直接拿到 IDA / Ghidra 里 ` G ` 跳就是反汇编。再配合 第 4 章 的 `
FindClass ` / ` GetMethodID ` / ` Call*Method ` 三件套，Native 反过来调 Java
的反向流也能截到——这就是 JNI 双向桥的完整拼图。

落到实战：

**下次再遇到一个不明 SO 的 native 方法，推荐工作流** ：

  1. **静态侦察** — 用 ` strings ` 或 ` nm -D ` 看 SO 有没有 ` Java_ ` 前缀符号
     * **有** → 静态注册，直接 ` enumerateExports ` 找 ` Java_xxx ` 函数，Interceptor 挂上即可
     * **没有** → 动态注册场景，进 2
  2. **动态拦截** — ` frida -f <pkg> ` spawn 模式启动 + hook ` RegisterNatives ` （本篇 第 2.2 节 主脚本）
     * 拿到每个 native 方法的 Java 类名、JNI 签名、SO + 偏移
  3. **深入分析** — 用 SO 偏移在 IDA / Ghidra 里 ` G ` 跳过去看反汇编/伪 C
     * 如果 SO 被 strip / 函数名混淆，定位手段切换到第 21 篇多策略兜底框架（符号表 → 特征码 → 字符串/调用点交叉引用）
  4. **反向监控** （可选） — 如果发现 Native 函数内部又调了 Java 方法，挂上 ` FindClass / GetMethodID / Call*Method ` 三件套（本篇 第 4 章）看完整双向流
  5. **加固反检测** （进阶） — 如果 hook 触发一次后就没反应了，看是否有 ` UnregisterNatives ` （索引 216）解绑后重注册的反检测套路

按这条流程走， **绝大多数「动态注册场景」都能定位到关键加密/签名函数** 。本篇真机验证（QQ 音乐 spawn 截 7 次
RegisterNatives / 94 个方法）走的就是这条路径。
