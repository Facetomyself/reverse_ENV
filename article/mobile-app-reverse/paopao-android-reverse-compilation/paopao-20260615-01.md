##  一、为什么需要懂 ARM 汇编

###  1.1 Frida Native Hook 的知识缺口

前面讲的 Native Hook 代码中有这样的写法：


    Interceptor.attach(funcAddr, {
        onEnter: function(args) {
            console.log("arg0: " + args[0]);  // 第一个参数
            console.log("arg1: " + args[1]);  // 第二个参数
        }
    });


你知道 ` args[0] ` 是第一个参数， ` args[1] ` 是第二个。但问题是：

  * 这些参数是怎么传递的？为什么 ` args[0] ` 就是第一个参数？
  * 如果参数超过 8 个怎么办？
  * ` args[0] ` 里面存的到底是值本身还是一个指针？
  * 返回值在哪里？
  * IDA 中看到的 ` STP X29, X30, [SP, #-0x30]! ` 是什么意思？

要回答这些问题，你需要理解 ARM64 的寄存器和调用约定。

###  1.2 本篇的范围

本篇 **只讲与 Frida 实战直接相关的部分** ：

  * 寄存器：哪些寄存器存什么（参数/返回值/栈指针）
  * 调用约定：函数调用时参数怎么排列
  * 常见指令：IDA 中最常见的 20 条指令

##  二、ARM64 寄存器

###  2.1 通用寄存器

ARM64（AArch64）有 31 个 64 位通用寄存器，命名为 X0 到 X30：

寄存器  |  用途  |  Frida 访问
---|---|---
**X0 - X7** |  函数参数 / 返回值  |  ` this.context.x0 ` \- ` this.context.x7 `
**X8** |  间接结果地址 / 系统调用号  |  ` this.context.x8 `
**X9 - X15** |  临时寄存器（调用者保存）  |  ` this.context.x9 ` \- ` this.context.x15 `
**X16 - X17** |  过程间调用临时寄存器  |  ` this.context.x16 ` \- ` this.context.x17 `
**X18** |  平台寄存器（Android 上保留）  |  ` this.context.x18 `
**X19 - X28** |  被调用者保存的寄存器  |  ` this.context.x19 ` \- ` this.context.x28 `
**X29 (FP)** |  帧指针（Frame Pointer）  |  ` this.context.fp `
**X30 (LR)** |  链接寄存器（返回地址）  |  ` this.context.lr `

###  2.2 特殊寄存器

寄存器  |  全名  |  用途  |  Frida 访问
---|---|---|---
**SP** |  Stack Pointer  |  栈指针，指向当前栈顶  |  ` this.context.sp `
**PC** |  Program Counter  |  程序计数器，指向当前执行的指令  |  ` this.context.pc `
**NZCV** |  Condition Flags  |  条件标志位（零/负/进位/溢出）  |  —

###  2.3 32 位子寄存器

每个 64 位 X 寄存器的低 32 位可以通过 W 前缀访问：


    X0 = [高32位 | 低32位(W0)]


在 IDA 中如果你看到 ` W0 ` 而不是 ` X0 ` ，说明代码在操作 32 位数据（如 ` int ` ）。

###  2.4 关键记忆

对于 Frida 逆向，你只需要记住这几点：


    X0-X7  = 函数的前 8 个参数（同时 X0 也是返回值）
    X29    = 帧指针（FP）
    X30    = 返回地址（LR）
    SP     = 栈顶指针


这就是为什么 Frida 的 ` args[0] ` 对应 X0、 ` args[1] ` 对应 X1——Frida 直接从 ARM64
的参数寄存器中读取。

##  三、函数调用约定（AAPCS64）

###  3.1 参数传递规则

ARM64 的标准调用约定（AAPCS64）规定：

**整数 / 指针参数** ：前 8 个通过 X0-X7 传递。第 9 个及以后通过栈传递。

**浮点参数** ：前 8 个通过 D0-D7（浮点寄存器）传递。

**返回值** ：通过 X0 返回（64 位以内）。128 位返回值用 X0 + X1。


    // C 函数
    int add(int a, int b, int c) {
        return a + b + c;
    }

    // ARM64 调用时的寄存器状态:
    // X0 = a (第一个参数)
    // X1 = b (第二个参数)
    // X2 = c (第三个参数)
    // 返回值在 X0 中


    // JNI 函数的特殊情况
    jstring Java_com_example_NativeLib_encrypt(JNIEnv *env, jclass clazz, jstring input, jstring key) {
        // X0 = env      (JNIEnv 指针，JNI 固定参数)
        // X1 = clazz    (类引用，JNI 固定参数)
        // X2 = input    (Java 传入的第一个参数)
        // X3 = key      (Java 传入的第二个参数)
        // 返回值 (jstring) 在 X0 中
    }


在 Frida 中：


    Interceptor.attach(funcAddr, {
        onEnter: function(args) {
            // args[0] = X0 = env
            // args[1] = X1 = clazz
            // args[2] = X2 = input (我们关心的)
            // args[3] = X3 = key   (我们关心的)
        },
        onLeave: function(retval) {
            // retval = X0 = 返回的 jstring
        }
    });


###  3.2 栈帧

当函数被调用时，它需要保存一些信息到栈上：

  * 调用者的返回地址（LR / X30）
  * 调用者的帧指针（FP / X29）
  * 需要保存的被调用者保存寄存器（X19-X28）
  * 局部变量

函数序言（Prologue）：


    STP X29, X30, [SP, #-0x30]!   ; 栈上预留 0x30 字节，保存 FP 和 LR
    MOV X29, SP                    ; 设置新的帧指针


函数尾声（Epilogue）：


    LDP X29, X30, [SP], #0x30     ; 恢复 FP 和 LR，回收栈空间
    RET                            ; 返回（跳转到 LR 指向的地址）


> **在 IDA 中** ：几乎每个函数的开头都是 ` STP X29, X30, [SP, #-N]! ` ，结尾都是 ` LDP X29, X30,
> [SP], #N ` \+ ` RET ` 。看到这个模式就知道这是一个函数的边界。

##  四、常见 ARM64 指令速查

> 下面四张是速查表。第一遍读只需先记最常用的 6 条： ` MOV ` （赋值）、 ` LDR/STR ` （读写内存）、 ` BL ` （函数调用）、
> ` CMP + B.EQ/B.NE ` （条件跳转）、 ` STP/LDP ` （成对存取寄存器）、 ` RET `
> （返回）。其余遇到再回来查表即可，不必死记。

###  4.1 数据移动

指令  |  示例  |  含义
---|---|---
` MOV ` |  ` MOV X0, X1 ` |  X0 = X1
` MOV ` |  ` MOV X0, #42 ` |  X0 = 42
` MOVZ ` |  ` MOVZ X0, #0x1234 ` |  X0 = 0x1234（零扩展）
` MOVK ` |  ` MOVK X0, #0x5678, LSL #16 ` |  X0 的 bit[16:31] 设为 0x5678（保留其余位）
` LDR ` |  ` LDR X0, [X1] ` |  X0 = *(X1)（从内存加载）
` LDR ` |  ` LDR X0, [X1, #8] ` |  X0 = *(X1 + 8)
` STR ` |  ` STR X0, [X1] ` |  *(X1) = X0（存到内存）
` STR ` |  ` STR X0, [X1, #8] ` |  *(X1 + 8) = X0
` LDP ` |  ` LDP X0, X1, [X2] ` |  同时加载两个寄存器
` STP ` |  ` STP X0, X1, [X2] ` |  同时存储两个寄存器
` ADRP ` |  ` ADRP X0, #page ` |  加载页地址（用于寻址全局变量）

###  4.2 算术运算

指令  |  示例  |  含义
---|---|---
` ADD ` |  ` ADD X0, X1, X2 ` |  X0 = X1 + X2
` ADD ` |  ` ADD X0, X0, #1 ` |  X0 = X0 + 1
` SUB ` |  ` SUB X0, X1, X2 ` |  X0 = X1 - X2
` MUL ` |  ` MUL X0, X1, X2 ` |  X0 = X1 * X2
` AND ` |  ` AND X0, X1, #0xFF ` |  X0 = X1 & 0xFF
` ORR ` |  ` ORR X0, X1, X2 ` |  X0 = X1 | X2
` EOR ` |  ` EOR X0, X1, X2 ` |  X0 = X1 ^ X2（XOR）
` LSL ` |  ` LSL X0, X1, #4 ` |  X0 = X1 << 4
` LSR ` |  ` LSR X0, X1, #4 ` |  X0 = X1 >> 4（逻辑右移）

###  4.3 比较和跳转

指令  |  示例  |  含义
---|---|---
` CMP ` |  ` CMP X0, #0 ` |  比较 X0 和 0（设置标志位）
` CMP ` |  ` CMP X0, X1 ` |  比较 X0 和 X1
` CBZ ` |  ` CBZ X0, label ` |  如果 X0 == 0 则跳转
` CBNZ ` |  ` CBNZ X0, label ` |  如果 X0 != 0 则跳转
` B ` |  ` B label ` |  无条件跳转
` B.EQ ` |  ` B.EQ label ` |  相等则跳转（CMP 后）
` B.NE ` |  ` B.NE label ` |  不相等则跳转
` B.GT ` |  ` B.GT label ` |  大于则跳转
` B.LT ` |  ` B.LT label ` |  小于则跳转
` BL ` |  ` BL function ` |  调用函数（B + 保存返回地址到 LR）
` BLR ` |  ` BLR X8 ` |  通过寄存器间接调用函数
` RET ` |  ` RET ` |  返回（跳转到 LR）

###  4.4 NOP


    NOP                            ; 指令编码 0xD503201F（小端写入字节 1F 20 03 D5）- 什么都不做


NOP 在 Frida 中 patch 代码时很常用——如果你想跳过某条指令（如一个检测判断），可以用 NOP 覆盖它（按小端写入字节数组 ` [0x1f,
0x20, 0x03, 0xd5] ` ）。

##  五、在 Frida 中操作寄存器

###  5.1 读取寄存器


    Interceptor.attach(funcAddr, {
        onEnter: function(args) {
            // 通过 args 读取（推荐，语义更清晰）
            console.log("参数1: " + args[0]);
            console.log("参数2: " + args[1]);

            // 通过 context 直接读取寄存器（等价）
            console.log("X0: " + this.context.x0);
            console.log("X1: " + this.context.x1);

            // 读取特殊寄存器
            console.log("SP:  " + this.context.sp);
            console.log("LR:  " + this.context.lr);  // 返回地址
            console.log("PC:  " + this.context.pc);
            console.log("FP:  " + this.context.fp);

            // 通过 LR 判断是谁调用了这个函数
            var caller = DebugSymbol.fromAddress(this.context.lr);
            console.log("调用者: " + caller);
        },
        onLeave: function(retval) {
            // retval 就是 X0
            console.log("返回值: " + retval);
            console.log("X0:     " + this.context.x0);  // 同一个值
        }
    });


> **PAC 注意** ：在 ARMv8.3+ 且启用了指针认证（PAC， ` pac-ret ` 分支保护）的设备 / 系统库上， ` LR `
> 高位会带签名位，直接 ` DebugSymbol.fromAddress(this.context.lr) ` 可能解析失败或落到错误地址。此时要先
> strip 掉认证位（对地址做高位掩码，或用 Frida 提供的去认证手段）再解析。现代高版本 Android 的系统库正逐步启用 PAC，遇到「LR
> 解析不出符号」时优先怀疑这一点。

###  5.2 修改寄存器


    Interceptor.attach(funcAddr, {
        onEnter: function(args) {
            // 修改参数
            args[0] = ptr(1);  // 将第一个参数改为 1

            // 等价写法
            this.context.x0 = ptr(1);
        },
        onLeave: function(retval) {
            // 修改返回值
            retval.replace(ptr(0));  // 返回 0

            // 等价写法
            // this.context.x0 = ptr(0);  // 直接改 X0 也可以
        }
    });


###  5.3 读取栈上的参数

当函数参数超过 8 个时，多余的参数在栈上：


    // 假设函数有 10 个参数
    Interceptor.attach(funcAddr, {
        onEnter: function(args) {
            // 前 8 个在寄存器中
            var arg0 = args[0];  // X0
            var arg7 = args[7];  // X7

            // 第 9 个和第 10 个在栈上
            // SP 指向栈顶，栈上参数从 [SP] 开始
            var arg8 = this.context.sp.readPointer();           // [SP + 0]
            var arg9 = this.context.sp.add(8).readPointer();    // [SP + 8]

            console.log("arg8 (栈上): " + arg8);
            console.log("arg9 (栈上): " + arg9);

            // 等价写法：args[8]、args[9] 也能直接读——Frida 会自动算到栈上，
            // 与手动取 SP 等价；手动读 SP 主要是帮你理解它们物理上在哪
        }
    });


##  六、IDA/Ghidra 伪代码阅读技巧

###  6.1 IDA 中的函数视图

在 IDA 中打开 SO，定位到目标函数，按 **F5** 生成伪代码（Hex-Rays 反编译器）。伪代码是一种"介于 C 和汇编之间"的可读格式：


    // IDA 生成的伪代码示例
    __int64 __fastcall sub_4A8C(__int64 a1, __int64 a2, __int64 a3, int a4)
    {
        // a1 = X0（第一个参数）
        // a2 = X1（第二个参数）
        // a3 = X2（第三个参数）
        // a4 = W3（第四个参数，32位 int → 用 W3）

        void *v8;
        v8 = malloc(a4 + 16);
        memcpy(v8, a3, a4);

        // ... 加密逻辑 ...

        return v8;  // 返回值在 X0
    }


**` sub_4A8C ` ** ：IDA 自动命名的函数（4A8C 是函数的偏移地址）。你在 Frida 中用 `
module.base.add(0x4A8C) ` 就能定位到它。

**` __fastcall ` ** ：调用约定标记。ARM64 几乎都是 fastcall（寄存器传参）。

**参数命名** ： ` a1 ` = 第一个参数 = X0， ` a2 ` = 第二个 = X1，依次类推。

###  6.2 从 IDA 伪代码到 Frida Hook


    // IDA 伪代码：
    __int64 sub_4A8C(const char *input, int input_len, const char *key, char *output)
    {
        // 加密 input，使用 key，结果存入 output
        // 返回输出长度
    }


对应的 Frida Hook：


    var mod = Process.findModuleByName("libnative.so");
    var funcAddr = mod.base.add(0x4A8C);

    Interceptor.attach(funcAddr, {
        onEnter: function(args) {
            // 根据 IDA 伪代码的参数类型解读 args
            this.input = args[0];                    // const char *input → 指针
            this.inputLen = args[1].toInt32();        // int input_len → 整数
            this.key = args[2];                       // const char *key → 指针
            this.output = args[3];                    // char *output → 指针

            console.log("输入: " + this.input.readUtf8String(this.inputLen));
            console.log("密钥: " + this.key.readUtf8String());
        },
        onLeave: function(retval) {
            var outLen = retval.toInt32();
            console.log("输出 (" + outLen + " bytes):");
            console.log(hexdump(this.output, { length: outLen }));
        }
    });


###  6.3 Ghidra 的等价操作

Ghidra 是免费的 IDA 替代品：

  1. 打开 SO 文件：File → Import File
  2. 分析：Analysis → Auto Analyze
  3. 定位函数：Navigation → Go To → 输入偏移地址 ` 0x4A8C `
  4. 反编译：Window → Decompile（或双击函数名）

Ghidra 的伪代码格式和 IDA 略有不同，但参数和返回值的对应关系是一样的。

###  6.4 识别加密函数的特征

在 IDA/Ghidra 的反汇编中，加密函数通常有以下特征：

  * **大量 XOR 运算** （ ` EOR ` 指令 / ` ^ ` 操作符）
  * **循环结构** （固定轮数的 for 循环，如 10 轮 = AES-128）
  * **查表操作** （通过数组下标访问常量表 = S-Box）
  * **位移操作** （ ` LSL ` / ` LSR ` / ` ROR ` ）
  * **调用了` memcpy ` 、 ` malloc ` ** （输入/输出缓冲区操作）

##  七、ARM32 与 Thumb 模式速览

虽然现代 Android 设备几乎都是 ARM64，但仍有部分 App 只提供 32 位 SO（ ` armeabi-v7a ` ）。在 64 位设备上这些
SO 通过兼容层运行。

###  7.1 ARM32 vs ARM64 关键差异

|  ARM32 (AArch32)  |  ARM64 (AArch64)
---|---|---
寄存器  |  R0-R15（16个，32位）  |  X0-X30（31个，64位）
参数传递  |  R0-R3（前4个参数）  |  X0-X7（前8个参数）
返回值  |  R0（32位）/ R0+R1（64位）  |  X0
栈指针  |  R13 (SP)  |  SP
链接寄存器  |  R14 (LR)  |  X30 (LR)
程序计数器  |  R15 (PC)  |  PC
指令大小  |  4字节(ARM) / 2字节(Thumb)  |  固定4字节

###  7.2 Thumb 模式与地址 +1 规则

ARM32 有两种指令集： **ARM** （32位指令）和 **Thumb** （16位指令，代码更紧凑）。现代 Android 的 32 位 SO
几乎都使用 Thumb 模式编译。

**关键规则** ：Thumb 函数的地址最低位为 1。在 IDA 中看到的地址如 ` 0x1234 ` 如果是 Thumb 函数，在 Frida
中需要写成 ` 0x1235 ` （地址 +1）。


    // ARM32 Thumb Hook 示例
    var mod = Process.findModuleByName("libnative.so");
    var offset = 0x1234;  // IDA 中看到的偏移

    // 如果是 Thumb 函数，地址需要 +1
    var funcAddr = mod.base.add(offset + 1);  // 注意 +1

    Interceptor.attach(funcAddr, {
        onEnter: function(args) {
            // ARM32 只有 R0-R3 传参
            console.log("arg0 (R0): " + args[0]);
            console.log("arg1 (R1): " + args[1]);
            console.log("arg2 (R2): " + args[2]);
            console.log("arg3 (R3): " + args[3]);
            // 第5个及以后的参数在栈上
        }
    });


> **如何判断是否是 Thumb 函数？** 在 IDA 中，最可靠的判据是反汇编窗口里该地址的 ` T ` 段寄存器值为 1（左侧 ` T=1 `
> 行），或代码被按 16 位 Thumb 指令解码（指令更短）；也可在 ` Edit → Segments ` 中查看该段的 ARM/Thumb 属性。在
> Ghidra 中，ARM 和 Thumb 代码段的反汇编会有不同的指令格式。

##  总结

**ARM64 寄存器** ：X0-X7 传递前 8 个参数，X0 也是返回值，X29 是帧指针，X30 是返回地址，SP 是栈指针。

**调用约定** ：JNI 函数的 X0=env、X1=this/class，实际参数从 X2 开始。超过 8 个的参数在栈上。

**常见指令** ： ` MOV ` =赋值、 ` LDR/STR ` =读写内存、 ` BL ` =函数调用、 ` CMP+B.EQ/B.NE `
=条件跳转、 ` STP/LDP ` =保存/恢复寄存器对、 ` RET ` =返回。

**Frida 中** ： ` this.context.x0 ` 到 ` this.context.x7 ` 对应 args[0]-args[7]。通过
` this.context.lr ` 可以快速判断调用者。通过 ` this.context.sp ` 可以读取栈上的额外参数。

**IDA/Ghidra 伪代码** 中的 ` a1 ` \- ` a8 ` 对应 X0-X7，函数名 ` sub_偏移 ` 中的偏移可以直接在 Frida
中 ` module.base.add(偏移) ` 定位。
