##  一、问题起点：抓包遇到黑盒

使用 Charles Proxy 对某音乐 App Android 客户端进行 HTTPS 抓包时，发往  `
https://u6.y.qq.com/cgi-bin/musics.fcg  ` 的 POST 请求呈现出以下异常特征：

  1. ** 请求体是二进制数据  ** ，无法直接读取 JSON；
  2. ** 响应体同样是二进制  ** ，非明文；
  3. 请求头携带特殊字段：

字段  |  示例值  |  含义
---|---|---
` M-Encoding  ` |  ` m1  ` |  自定义编码方案
` sign  ` |  ` [base64 字符串]  ` |  请求签名
` mask  ` |  ` [base64 字符串]  ` |  签名掩码
` x-sign-data-type  ` |  ` json  ` |  签名数据类型标识

这条请求乍看像是"重加密黑盒"，而实际分析后发现：  ** 混淆层并不厚，真正的难点在签名核心  ** 。

本文的分析目标：

  1. 定位请求体编码算法及实现代码；
  2. 定位响应体解码路径；
  3. 定位  ` sign  ` 和  ` mask  ` 的生成逻辑；
  4. 编写 Frida Hook 脚本，实时捕获明文请求与响应。

###  2.4 请求构建主流程

核心类  ` com/xxx/xxxmusicplayer/network/business/b.java  ` 的  ` g0()  ` 调用链：


    g0()
     ├── Y(xVar)          → 构建请求体 JSON（明文 byte[]）
     ├── n.c(this, body)  → 签名：调用 MERJni.calc() 生成 sign/mask，写入 HTTP 头
     └── T(body)          → M-Encoding：Deflate 压缩 + 5 字节随机前缀


####  2.4.1 签名生成 —  ` n.java  ` （SignRequestHelper）

` com/xxx/xxxmusicplayer/network/business/n.java  ` ：


    // 拼接签名所需参数
    byte[] bytes = joinToString(ct, appVersion, uid, qq, timestamp, udid, "android");

    // 调用原生签名库
    String calcRet = MERJni.calc(content, bytes);

    // 解析返回值（空格分隔的两段 Base64）
    String[] split = calcRet.split(" ");
    xVar.addHeader("sign", split[0]);         // 请求签名
    xVar.addHeader("mask", split[1]);         // 签名掩码
    xVar.addHeader("x-sign-data-type", "json");


` params  ` 字段格式固定：


    ct&cv&uid&qq&timestamp&udid&android

各字段含义：

字段  |  含义
---|---
` ct  ` |  客户端类型（Android=11）
` cv  ` |  客户端版本号
` uid  ` |  用户 UID
` qq  ` |  QQ 号码
` timestamp  ` |  毫秒级时间戳
` udid  ` |  设备唯一标识
` android  ` |  平台标识

####  2.4.2 MERJni — 原生签名库接口

` com/xxx/xxxmusic/modular/framework/encrypt/logic/MERJni.java  ` ：


    public class MERJni {
        // 加载原生签名库 libmer.so
        public static void a() { System.loadLibrary("mer"); }

        // JNI native 方法，实现在 libmer.so 中
        public static native String calc(byte[] content, byte[] params);
    }


关键特征：

  * 加载  ` libmer.so  ` （大小约 2.3MB，内嵌完整 BoringSSL 变体）；
  * ` calc()  ` 通过  ` RegisterNatives  ` ** 动态注册  ** ，无标准  ` Java_com_tencent_...  ` 导出符号；
  * JNI 签名：  ` ([B[B)Ljava/lang/String;  ` 。

####  2.4.3 M-Encoding 编码 —  ` Request.T()  `

` com/xxx/xxxmusicplayer/network/base/Request.java  ` ：


    public void T(byte[] bArr) {
        if (this.f41306a.h() == 1) {  // h() 返回编码类型，1 = m1
            // 第一步：Deflate 压缩
            byte[] compressed = zipper.b.c(bArr);

            // 第二步：添加 5 字节随机前缀（每字节值 0~99）
            byte[] encoded = zipper.a.Companion.a(compressed);

            if (encoded != null) {
                this.f41306a.addHeader("M-Encoding", "m1");
                bArr = encoded;
            }
        }
    }


** 5 字节随机前缀实现  ** （  ` com/xxx/xxxmusiccommon/cgi/zipper/a.java  ` ）：


    public byte[] a(byte[] source) {
        byte[] prefix = new byte[5];
        for (int i = 0; i < 5; i++) {
            prefix[i] = (byte) random.nextInt(100);  // 范围 0~99
        }
        byte[] result = new byte[source.length + 5];
        System.arraycopy(prefix, 0, result, 0, 5);
        System.arraycopy(source, 0, result, 5, source.length);
        return result;
    }


> ** 结论  ** ：M-Encoding m1 本质上是"Deflate 压缩 + 随机混淆头"，  ** 不是密码学加密  ** 。任何人只要跳过前
> 5 字节再 inflate，都能还原明文。

###  2.5 响应解码路径

` b.java  ` 的  ` Q()  ` 方法处理所有响应：


    Q(response)
     ├── i0(response)       → 判断编码类型
     │    ├── 检查 Content-Encoding 头（gzip / brotli）
     │    └── 检查 encoding type（f==5 或 f==0）
     ├── Z(body)            → 验证前 5 字节是否全为 0x00
     └── b0(body, 5, type)  → 解压分发
          ├── type=1 → Inflate（Deflate 解压）
          ├── type=2 → Brotli 解压
          └── type=0 → GZIP 解压


响应格式判别规则：

响应格式  |  特征  |  解码方式
---|---|---
明文 JSON  |  ` {  ` 开头  |  直接读取
Gzip 压缩  |  ` 1f 8b  ` 开头  |  gzip 解压
M-Encoding  |  ` 00 00 00 00 00  ` 开头  |  跳过 5 字节 + Inflate
JCE/Protobuf  |  其他二进制  |  专用二进制协议解码

###  2.6 静态分析结论


    请求编码流程：
      明文 JSON
        → MERJni.calc(body, params) 生成 sign + mask（写入 Header）
        → Deflate 压缩
        → 前置 5 字节随机数据
        → 设置 M-Encoding: m1 Header
        → 经 HTTPS TLS 发送

    响应解码流程：
      接收字节流
        → 检查 Content-Encoding 与编码类型标志
        → M-Encoding：验证前 5 字节为 0x00，跳过后 Inflate
        → Gzip：直接解压
        → 得到明文 JSON


##  三、动态分析：Frida 踩坑全纪录

初始目标是直接 Hook Java 层方法（  ` b.Y()  ` 、  ` b.b0()  ` 、  ` MERJni.calc()  ` 、  `
Request.T()  ` ）。然而一路走下来，遇到了 12 个工程级问题——这些问题比算法本身更费时间。

###  问题 1：Android 13 + Frida 17.8.2 Java Bridge 崩溃

** 现象  ** ：  ` Java.perform()  ` 报错——


    Unable to find copied methods in java/lang/Thread


** 原因  ** ：Frida 17.8.2 的 Java bridge 与 Android 13 ART 运行时不兼容。

** 尝试方案（均失败）  ** ：

方案  |  结果
---|---
` Java.performNow()  ` |  同样失败
` setTimeout  ` 延迟注入  |  无效
spawn 模式  |  无效
` enumerateClassLoaders  ` |  无效
降级到 frida 16.2.5  |  不同错误：  ` TypeError: t is not a function  `

** 结论  ** ：Android 13 上 Java bridge 完全不可用 → 升级到 Android 15。


###  问题 2：Android 15 + Frida 17.8.2 Java Bridge SIGSEGV

** 现象  ** ：升级到 Android 15 后，  ` Java.perform()  ` 或  ` Java.use()  `
导致目标进程崩溃——


    signal 11 (SIGSEGV), code 0 (SI_USER)
    backtrace:
      #00 libart.so nterp_op_return_object
      #01 base.vdex ConstraintAnchor.getDependents
      ...


** 分析  ** ：崩溃发生在 ART nterp 解释器内，属于 Frida Java bridge 与 Android 15 ART 的  **
根本性不兼容  ** 。验证：只跑  ` Process.enumerateModules()  ` 时进程完全正常。

** 各方案结果  ** ：

方案  |  结果
---|---
` Java.scheduleOnMainThread  ` \+  ` Java.perform  ` |  SIGSEGV
` Java.performNow  ` （spawn 模式，早期注入）  |  类加载 OK，  ` enumerateClassLoaders  ` 时崩溃
` Java.deoptimizeEverything()  ` |  SIGSEGV
attach 模式  |  ` Java.perform  ` 直接崩溃

** 最终决策：放弃 Java bridge，改用纯 Native Hook 方案。  **


###  问题 3：Python API 中 Java bridge 为 undefined

** 现象  ** ：  ` session.create_script(js)  ` 中  ` typeof Java === "undefined"
` 。

** 原因  ** ：frida CLI REPL 自动加载 Java bridge；Python API 的  ` create_script()  `
在 17.8.2 中  ** 不再自动加载  ** 。

** 解决  ** ：改用 frida CLI + 脚本文件方式（  ` frida -U -f ... -l script.js  ` ）。


###  问题 4：Frida 17.8.2 Module API 变更

** 现象  ** ：  ` Module.findExportByName("libssl.so", "SSL_write")  ` 报  `
TypeError: not a function  ` 。

** 原因  ** ：17.8.2 中  ` Module  ` 是构造函数，  ` findExportByName  ` 变为实例方法。


    // 旧版 API（已废弃）
    Module.findExportByName("libssl.so", "SSL_write")

    // 新版 API
    Process.getModuleByName("libssl.so").findExportByName("SSL_write")


** 解决  ** ：封装兼容帮助函数——


    function findExport(mod, func) {
        try {
            return Process.getModuleByName(mod).findExportByName(func);
        } catch(e) {
            return null;
        }
    }


###  问题 5：MERJni.calc() 函数无法通过符号定位

** 现象  ** ：  ` libmer.so  ` 有 2800+ 个导出符号，搜索  ` calc  ` 只找到 OpenSSL 内部函数  `
rsa_multip_calc_product  ` 。

** 原因  ** ：  ` MERJni.calc()  ` 通过  ` JNI_OnLoad()  ` →  ` RegisterNatives()
` 动态注册，没有标准  ` Java_com_tencent_...  ` 导出符号，运行时地址随 ASLR 变化。

** 解决  ** ：Hook ART 的  ` RegisterNatives  ` ，拦截 JNI 方法注册事件——


    Interceptor.attach(registerNativesAddr, {
        onEnter: function(args) {
            var nMethods = args[3].toInt32();
            var methods = args[2];
            for (var i = 0; i < nMethods; i++) {
                // JNINativeMethod 结构体，每项 24 字节
                var name = methods.add(i * 24).readPointer().readUtf8String();
                if (name === "calc") {
                    // 偏移 16 处为函数指针
                    var fnPtr = methods.add(i * 24 + 16).readPointer();
                    hookCalc(fnPtr);  // 对实际地址安装 Interceptor
                }
            }
        }
    });


** 实际输出  ** ：


    [!] calc registered: ([B[B)Ljava/lang/String; @ 0x7xxx...xxxx


###  问题 6：calc() 返回值是 jstring 引用

** 现象  ** ：  ` calc()  ` 返回  ` jstring  ` （JNI 对象引用），直接打印只显示指针地址。

** 解决  ** ：通过 JNI 函数表（vtable）手动调用  ` GetStringUTFChars  ` ——


    onLeave: function(retval) {
        var env = this.jniEnv;
        var vtable = env.readPointer();
        // GetStringUTFChars = JNI 函数表第 169 项（索引从 0 起）
        var getStrChars = new NativeFunction(
            vtable.add(169 * 8).readPointer(),
            'pointer', ['pointer', 'pointer', 'pointer']
        );
        var cstr = getStrChars(env, retval, ptr(0));
        var result = cstr.readUtf8String();
        // result = "sign_base64 mask_base64"
    }


###  问题 7：GetByteArrayElements 触发 abort

** 现象  ** ：读取  ` params  ` 字节数组时，调用  ` GetByteArrayElements  ` （JNI index
184）报  ` Error: abort was called  ` 。

** 解决  ** ：改用  ` GetByteArrayRegion  ` （JNI index 200），将数组内容复制到预分配的 native
缓冲区——


    var getByteArrRegion = new NativeFunction(
        vtable.add(200 * 8).readPointer(),
        'void', ['pointer', 'pointer', 'int', 'int', 'pointer']
    );
    var pBuf = Memory.alloc(pLen + 1);
    getByteArrRegion(env, this.paramsArr, 0, pLen, pBuf);
    var paramsStr = pBuf.readUtf8String(pLen);


> ** 原理  ** ：  ` GetByteArrayElements  ` 可能返回指向 GC 托管内存的直接指针，使用不当会触发 ART 内部
> abort；  ` GetByteArrayRegion  ` 是安全复制语义，不存在此问题。


###  问题 8：SSL Hook attach 后零流量

** 现象  ** ：attach 到已运行进程后，60 秒内 SSL_write/SSL_read 零触发。

** 原因  ** ：HTTP keepalive 长连接已建立；该 App 后台闲置时不发送 fcg 请求，新 Hook 不会触发已建立连接上的回调。

** 解决  ** ：改用  ** spawn 模式  ** ，重启 App，捕获启动阶段的全部新连接——


    frida -U -f com.xxx.xxxmusic -l hooks/target_decrypt.js


###  问题 9：二进制请求体导致 readUtf8String 失败

** 现象  ** ：M-Encoding 编码后的请求包含 HTTP headers（文本）+ body（二进制）。对整个 SSL_write
buffer 调用  ` readUtf8String()  ` 抛异常，导致所有  ` musics.fcg  ` 请求被漏掉。

** 解决  ** ：扫描  ` \r\n\r\n  ` 分隔符，将 headers 和 body 分开处理——


    function findHeaderEnd(buf, len) {
        for (var i = 0; i < len - 3; i++) {
            if (buf.add(i).readU8()   === 0x0d &&
                buf.add(i+1).readU8() === 0x0a &&
                buf.add(i+2).readU8() === 0x0d &&
                buf.add(i+3).readU8() === 0x0a) {
                return i + 4;  // 返回 body 起始偏移
            }
        }
        return -1;
    }

    // headers: buf.readUtf8String(headerEnd) → 纯文本，安全读取
    // body:    buf.add(headerEnd)            → 二进制，走 inflateData 解压


###  问题 10：Memory.writePointer API 已移除

** 现象  ** ：  ` Memory.writePointer(ptr, value)  ` 报  ` TypeError: not a
function  ` 。

** 原因  ** ：Frida 17.8.2 移除了  ` Memory.writePointer  ` ，改为实例方法。


    // 旧写法（已废弃）
    Memory.writePointer(stream.add(0), inputPtr)

    // 新写法
    stream.add(0).writePointer(inputPtr)


###  问题 11：z_stream 结构体大小

** 现象  ** ：zlib  ` inflateInit2_  ` 返回错误码，inflate 失败。

** 解决  ** ：arm64 上  ` z_stream  ` 大小为  ** 112 字节  ** （非 128
字节）。修正参数并在调用前清零结构体——


    // 清零 z_stream 结构体（分配 128 字节，实际使用 112）
    for (var z = 0; z < 128; z += 8) stream.add(z).writeU64(0);

    var ret = inflateInit2Fn(stream, wbits, zlibVersion, 112);


###  问题 12：目标 App 加载了两个 SSL 库

** 现象  ** ：只 Hook  ` libssl.so  ` 时漏掉部分网络流量。

** 分析  ** ：目标 App 同时使用系统  ` libssl.so  ` 和 Cronet 内嵌的  `
stable_cronet_libssl.so  ` ，两个库都需要 Hook。

** 解决  ** ：对两个模块都安装 Interceptor：


    ["libssl.so", "stable_cronet_libssl.so"].forEach(function(libName) {
        var sslWrite = findExport(libName, "SSL_write");
        var sslRead  = findExport(libName, "SSL_read");
        if (sslWrite) Interceptor.attach(sslWrite, sslWriteCallbacks);
        if (sslRead)  Interceptor.attach(sslRead,  sslReadCallbacks);
    });


##  四、最终方案：纯 Native Hook 三点架构

受 Android 15 Java bridge 完全不可用的限制，最终采用  ** 纯 Native Interceptor  ** 方案。


Frida 纯 Native Hook 架构

###  4.1 三大 Hook 点

####  Hook 1：SSL_write / SSL_read

  * 同时 Hook  ` libssl.so  ` 和  ` stable_cronet_libssl.so  ` ；
  * 用 SSL 连接指针（  ` ssl_ptr  ` ）区分  ` musics.fcg  ` /  ` musicu.fcg  ` ；
  * 请求体：检测  ` \r\n\r\n  ` 切分 headers/body，二进制 body 执行 inflate 解压；
  * 响应体：按首字节特征（  ` 7b  ` =JSON、  ` 1f 8b  ` =gzip、  ` 00 00 00 00 00  ` =M-Encoding）分支处理。

####  Hook 2：RegisterNatives

  * Hook  ` libart.so  ` 的  ` RegisterNatives  ` 符号；
  * 遍历注册方法列表，匹配  ` name === "calc"  ` ；
  * 提取函数指针，动态安装 calc Interceptor。

####  Hook 3：calc() 本体

  * 通过 JNI vtable 安全读取参数和返回值；
  * params：  ` GetByteArrayRegion  ` （index 200）读取；
  * content 长度：  ` GetArrayLength  ` （index 171）获取；
  * 返回值：  ` GetStringUTFChars  ` （index 169）读取 jstring。

###  6.1 请求编码（Python 伪代码）


    import zlib
    import random
    import base64

    def encode_musics_request(json_body: str, ct: str, cv: str,
                               uid: str, qq: str, udid: str) -> tuple:
        """
        目标 App musics.fcg 请求编码
        返回: (encoded_body, sign, mask)
        """
        body_bytes = json_body.encode('utf-8')

        # Step 1: 签名计算（通过 libmer.so native 实现）
        timestamp = current_time_ms()
        params = f"{ct}&{cv}&{uid}&{qq}&{timestamp}&{udid}&android"
        sign_result = MERJni_calc(body_bytes, params.encode('utf-8'))
        sign, mask = sign_result.split(' ')

        # Step 2: Deflate 压缩（保留 zlib header 78 9c）
        compressed = zlib.compress(body_bytes)

        # Step 3: 添加 5 字节随机前缀（每字节范围 0~99）
        prefix = bytes([random.randint(0, 99) for _ in range(5)])
        encoded_body = prefix + compressed

        return encoded_body, sign, mask


###  6.2 响应解码（Python 伪代码）


    import gzip
    import zlib

    def decode_musics_response(body: bytes) -> str:
        """
        目标 App musics.fcg 响应解码
        """
        if body[:1] == b'{':
            # 明文 JSON，直接读取

        else:
            # Protobuf / JCE 二进制协议，需要专用解码器
            returnNone

##  八、安全评估

###  8.1 各防护层强度

机制  |  类型  |  安全强度  |  评注
---|---|---|---
M-Encoding m1  |  Deflate + 随机前缀  |  ** 弱  ** |  仅混淆，任何人都能解压还原
sign + mask  |  libmer.so native 签名  |  ** 中  ** |  算法在 SO 内，需逆向 SO 才能伪造
HTTPS TLS 1.3  |  传输层加密  |  ** 强  ** |  标准传输层保护
CFF 混淆  |  tmesec_llvm 编译器  |  ** 高  ** |  有效阻碍静态反编译
内联 Crypto  |  MD5 不走导出 API  |  ** 中  ** |  增加动态追踪难度
RSA 设备密钥  |  本地 .rsa 文件  |  ** 高  ** |  密钥与设备绑定，难以通用复制
时间戳绑定  |  毫秒精度  |  ** 中  ** |  防重放攻击，但存在一定时间窗口

###  8.2 主要发现总结

  1. ** M-Encoding 不是加密  ** ：只是 Deflate 压缩 + 随机前缀，可轻松还原，不提供机密性保护；
  2. ** 签名才是核心  ** ：  ` sign  ` 和  ` mask  ` 由  ` libmer.so  ` 原生计算，防止请求被篡改或伪造；
  3. ** 三级 CGI 端点设计合理  ** ：  ` musicu  ` （通用）/  ` musics  ` （签名）/  ` musicw  ` （JCE）按功能和安全需求分层；
  4. ** 响应格式多样  ** ：JSON 明文、Gzip、M-Encoding、Protobuf/JCE 并存，需要分支处理；
  5. ** MD5 链路硬编码盐值  ** ：  ` 4cc5449d300d06092a86  ` 固定可提取，是已知弱点；
  6. ** RSA 设备绑定是强保护  ** ：本地 .rsa 密钥与设备绑定，大幅提升复制门槛。

###  8.3 技术难点复盘

本次分析最大的技术挑战不在于加密算法本身，而在于  ** Frida 工具链与 Android 15 ART 的兼容性问题  ** 。Java bridge
完全不可用迫使我们采用纯 Native Hook 方案，通过 SSL 层拦截 + JNI 函数表手动调用实现等效功能。
