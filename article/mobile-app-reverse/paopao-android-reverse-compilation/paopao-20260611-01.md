##  一、Gadget 是什么，免 Root 注入怎么做

###  1.1 Gadget vs frida-server：两条注入路

前面所有篇章都假设你有 Root： ` frida-server ` 以 Root 权限运行，用 ` ptrace ` 把 ` frida-agent `
注入目标进程。需要 Root，是因为 Android 用 SELinux 与 UID 隔离禁止一个普通 App 进程 ptrace 另一个 UID 的
App 进程（同 UID、或自己 fork 的子进程不在此限）。


    有 Root：电脑 (frida-tools) ←→ 手机 (frida-server) → ptrace 注入 → 目标 App 进程


Frida Gadget 走的是另一条路：把 ` frida-gadget.so ` 作为共享库 **打包进 APK** ，让 App 启动时自己 `
loadLibrary ` 加载它。Gadget 在 App 进程内部把 Frida 引擎跑起来，功能和 frida-agent 完全一样，但 **不需要
ptrace，也就不需要 Root** 。


    免 Root：电脑 (frida-tools) ←→ App 进程内自己加载的 Gadget (frida-gadget.so)


打个比方：frida-server 像从外面破门而入的特工（需要 Root 这把万能钥匙）；Gadget
像事先藏进快递包裹、被合法带进大楼的特工，不用刷门禁，因为它本就是货物的一部分。

这个比方很漂亮，也正是后面要被现实修正的地方：包裹进门要过安检（签名校验），而你为了把特工塞进包裹，已经把封条撕了。

###  1.2 三种配置模式

Gadget 的行为由一个 JSON 配置文件控制。配置文件名必须和 Gadget 的 SO 同名、扩展名换成 ` .config.so ` ，注意这是
Android 的打包约定（ ` lib/ ` 下只打包以 ` lib ` 开头、 ` .so ` 结尾的文件），文件 **内容其实是纯文本 JSON**
。Frida 官方文档对四种 interaction 都有定义：listen、connect、script、script-
directory。本篇实战用到其中两种：Listen 与 Script（Connect 仅作了解）。

**Listen 模式** （最常见）：Gadget 起一个端口等电脑连。


    {
      "interaction": {
        "type": "listen",
        "address": "0.0.0.0",
        "port": 27042,
        "on_port_conflict": "pick-next",
        "on_load": "wait"
      }
    }


` on_load ` 为 ` wait ` 时 App 启动后暂停、等 Frida 连上再继续；为 ` resume ` 则正常启动、随时可连。

**Script 模式** （离线跑固定脚本，本篇 ZygiskFrida 就用它）：


    {
      "interaction": {
        "type": "script",
        "path": "libhook.js.so",
        "on_change": "reload"
      }
    }


` path ` 是脚本文件路径（相对路径就放在和 Gadget 同目录），Gadget 启动即自动执行，不需要电脑连接。注意 script 模式
**只支持` path ` 引用外部脚本 ** ，不能把脚本内容内联进 JSON。

**Connect 模式** ：Gadget 主动连电脑上的 frida-portal，适合远程网络的设备。

###  1.3 免 Root 注入的三种方式——以及它们共同的死结

免 Root 把 Gadget 塞进 App，常见三种做法：

  1. **APK 重打包 + 改 smali** ：apktool 反编译，把 ` libgadget.so ` 放进 ` lib/arm64-v8a/ ` ，在入口类（ ` Application.attachBaseContext ` 或主 Activity 的 ` onCreate ` ）的 smali 里加一句 ` System.loadLibrary("gadget") ` ，再重打包、重签、安装。
  2. **LIEF 改 SO 依赖** ：用 LIEF 在 App 已有的某个 SO 的依赖列表里加上 ` libgadget.so ` ，加载它时 Gadget 被当依赖自动拉起，不用改 smali（前提是 App 至少有一个 SO）。
  3. **Script 模式打包独立脚本** ：把固定 Hook 脚本（如 SSL Pinning 绕过）和 Gadget 一起打进 APK，安装即用。

这三种方式底层都干同一件事： **修改 APK** 。而只要改了 APK，就必须 **重签名** （原签名私钥不在你手上）。重签名之后，APK
的签名证书就变了。

记住这一点： **免 Root 注入 = 改 APK = 重签 = 签名变了。**
这是下面整个故事的起点。签名变了之后能不能糊弄过签名校验，是另一场仗，第三节专打它。

##  二、墙 A：免 Root 重打包，撞上签名校验

###  2.1 目标侦察

先认识目标。从设备拉下 base.apk 静态看一眼：

维度  |  实测
---|---
版本 / SDK  |  9.2.80，minSdk 23，targetSdk 33
架构  |  arm64-v8a，单 base.apk（无 split），约 182 MB
Application  |  ` com.netease.nis.wrapper.MyApplication ` ← 网易易盾壳入口
Native 库  |  ` lib/arm64-v8a/ ` 下 180 个 ` .so ` ，含 ` libnesec.so ` （易盾安全 SDK）、 ` libmsaoaidsec.so ` （MSA 移动安全联盟）、一大批 React Native（hermes / fbjni / folly）、agora、cronet
extractNativeLibs  |  true（安装时会把 SO 解压到磁盘 ` lib/ ` 目录）

` libnesec.so ` 一出现就说明，这是个有反调试、反 Frida、签名校验的加固 App。Application 写的是 `
com.netease.nis.wrapper.MyApplication ` ，也就是易盾壳的 wrapper，真实业务 dex
在运行时解密，静态只看得到壳。

###  2.2 走常规路线，然后闪退

按第 1.3 节方式一来：apktool 反编译 → 在 ` MyApplication.attachBaseContext ` 顶部注入 `
loadLibrary("gadget") ` → 用自己的 keystore 重签 → 安装启动。

结果：进程在 **Gadget 脚本都还没加载** 时就 ~1.5 秒闪退。logcat 和 tombstone 给出铁证：


    D POISON  : APK Signature:d6e5dbbf5f3abe969190df07c37cef74
    F libc    : Fatal signal 11 (SIGSEGV) ... tid (Init Thread #1)

    tombstone（自底向上读调用顺序）:
      #03 libpoison.so + 0x7af40        → 调 exit()
      #02 libc exit + 24                → exit() 触发 __cxa_finalize
      #01 libc __cxa_finalize           → 跑 atexit 析构表
      #00 org.apache.http.legacy.jar + 0x448d0  → 坏掉的析构指针 → SEGV_ACCERR


读这段崩溃要自底向上： ` libpoison.so ` （易盾的签名校验/反篡改模块）算出当前 APK 的签名证书 MD5 = ` d6e5dbbf… `
，发现和它写死的期望值对不上，于是 **主动调` exit() ` 自杀 ** 。那个 SIGSEGV 不是「崩」，只是 ` exit() `
退栈时一个坏掉的 atexit 析构指针的副作用（ ` exit ` 本是 noreturn，被搅乱了）。

界面上还会弹一个 ` com.netease.nis.wrapper.NEDialog ` ，文案是「请前往官方渠道下载正版
APP」。这行字才是真正的死因告示牌，方法学一节会回头看它。

###  2.3 机制：取证书 → 算 MD5 → 比对写死值

精确量一下两边的签名证书：


    原始网易证书：CN=LiangJian, O=CloudMusic, RSA2048，DER 915 字节，MD5 da6b069da1e2982db3e386233f68d76d
    我的重签证书：DER 711 字节，MD5 d6e5dbbf5f3abe969190df07c37cef74  ← 与 POISON 日志一致


校验点在 ` libpoison + 0x7add8 ` （打印 ` "APK Signature:%s" ` 的 native 方法），它 **从
Java 经 JNI 调进来** （backtrace 全是 libart 帧）、 **不读 base.apk** （没有
openat）。也就是说它等价于走 ` getPackageInfo ` 取签名（旧标志走 ` GET_SIGNATURES.signatures[0] `
，targetSdk 28+ 的新壳多改用 ` signingInfo.getApkContentsSigners() ` ，见第三节）→ 拿到证书 DER
→ MD5 → 和写死的 ` da6b… ` 比对。自杀点在 ` libpoison + 0x7af40 ` （ ` bl exit ` ）。

再做个对照就清楚了。我做了一个干净的 round-trip 控制版：apktool 反编后零逻辑改动、不注入任何
Gadget，只重打包重签。它崩得一模一样，同样打出 ` POISON: APK Signature:d6e5 ` ，同样停在 ` libpoison +
0x7af40 ` 自杀。

这个对照很重要：控制版 **根本没有 Gadget** ，照样死。所以触发条件是 **重签本身** ，跟 Gadget、跟注入手法 **一点关系都没有**
。

###  2.4 为什么签名 spoof 也救不回来

「那我 Hook 一下签名校验，让它读到原证书不就行了？」四条路全堵死，逐条试过：

拦截思路  |  结果  |  死因
---|---|---
Java Hook ` getPackageInfo ` 返回原证书  |  X  |  Gadget 极早期（+0.2s，主 looper 还没跑）， ` Java.perform ` 回调根本不执行（实测 ` Java.available=true @+0.02s ` ，但 perform 回调从不启动）；而校验 +0.2s 就跑完—— **赛跑必输**
native Hook ` MD5_Final ` 改写结果  |  X  |  5 个 crypto 模块的 ` MD5_Final ` 全 Hook 上， **从没命中** → libpoison 不走 libcrypto，用自己加密代码里的 MD5
inline 比较强制相等（Hook memcmp/strcmp）  |  X  |  带 libpoison-caller 过滤， **零命中** → 比较是 **内联在加密代码** 里的
JNI 字节数组层 swap 成原证书  |  X  |  我的 cert 711B、原证书 915B， **长度不等** ，没法在缓冲区原地覆盖

签名校验做得又早（早于 Java 桥可用）、又 native（不经标准库）、又
inline（在加密代码里），窗口期内没有一个干净的拦截点。这正是签名校验作为加固手段的强处。第 25 篇会专门讲签名校验绕过的更多角度。

##  三、场景转变：绕过签名校验是一场时间竞赛

撞到这一步，与其继续死磕，不如停下来想个更本质的问题： **重签让签名变了，那把签名校验 Hook 掉、让它读到原签名，不就行了？**

这个想法本身没错，签名校验绕过本来就是 Frida 的常规活。能不能成，就看两件事，而且都由 App
自己的实现说了算：一是时间上的赛跑，二是有没有够得着的拦截点。

  * **竞赛** ：Gadget 必须在签名校验执行 **之前** 就位（Hook 挂上）。Gadget 靠你注入的 ` loadLibrary ` 拉起来，最早也就在壳 Application 的早期；校验要是比它先跑完，你还没上场就输了。
  * **拦截点** ：就算来得及，你还得在校验的数据流里找到一个能 Hook 的点（Java 的 ` getPackageInfo ` 、native 的 MD5、比较函数……）。校验若把这些点都做成内联、加密、自实现，你无处下手。

按这两条，签名校验 App 分成截然不同的两类：

|  校验 **晚** / 在 Java 层 / 延迟到登录·首次请求  |  校验 **早 + native + inline + 加密** （易盾这类顶级壳）
---|---|---
Gadget 来得及就位？  |  来得及（校验在 onCreate 之后甚至联网时才跑）  |  来不及（+0.2s 跑完，Gadget 的 Java 桥还没起）
找得到拦截点？  |  找得到（ ` getPackageInfo ` 是标准 API）  |  找不到（自实现 MD5 + 内联比较 + 代码段运行时解密）
免 Root Gadget Hook 绕过  |  **可行，而且是常规操作** |  这局必输

**左边这一类是大多数 App。** 免 Root Gadget 跑起来后，Hook 掉 ` getPackageInfo ` 、把签名换回原证书即可：


    // Gadget 脚本里：把重签后的签名「换回」原版，糊弄 Java 层签名校验
    Java.perform(function () {
        const PM  = Java.use('android.app.ApplicationPackageManager');
        const Sig = Java.use('android.content.pm.Signature');
        const ORIG = Sig.$new('308203...');   // 原版 APK 里抠出的证书 hex
        PM.getPackageInfo.overload('java.lang.String', 'int').implementation = function (pkg, flags) {
            const info = this.getPackageInfo(pkg, flags);
            if (info.signatures && info.signatures.value) info.signatures.value = [ORIG];
            return info;
        };
    });


对「Java 层取签名 → 比对」的 App，这就够了，所以 **免 Root Gadget 能绕签名校验，这不是新鲜事** 。（targetSdk
较新、改用 ` PackageInfo.signingInfo.getApkContentsSigners() ` 取签名的 App，把 `
signingInfo ` 一起 spoof 即可，原理一样。）签名校验绕过的更多角度（native 层、 ` PackageManagerService
` 层）见第 25 篇。

**问题在右边这一类。** 易盾把竞赛和拦截点 **两头都堵死了** （第 2.4 节已逐条验证）：校验在 +0.2s 就跑完，Gadget 的 `
Java.perform ` 回调那时根本没启动；校验又在 native + inline + 加密代码里，连 native 早 Hook
都找不到干净落点。所以对网易云，免 Root Gadget **实际** 糊弄不过去。

> 拎清边界： **这是「实现层不可行」，不是「逻辑上不可能」。**
> 易盾是靠工程堆料把「可绕」做成了「实际不可绕」，不是签名校验天然不可绕。严格说还有两条没被逻辑堵死的路：一是用 LIEF 把 Gadget 设成壳 SO
> 的依赖，让它在壳 ` JNI_OnLoad ` 之前加载、抢在校验前就位；二是直接静态 patch 掉壳里的校验，而不靠运行时 Hook。易盾因为 SO
> 运行时解密加自校验，把这两条也基本堵了，但能不能更早、能不能静态抹，仍是随壳强弱而变的攻防。

收个口。免 Root Gadget 对软目标（不校验签名、或只在 Java 层校验的 App）完全可用，那是它的主场；对易盾、梆梆、360
这类把校验做得又早又深的顶级加固壳，免 Root 这条路实际走不通。对网易云，与其在签名校验上死磕，不如换个思路：保留原版 APK 不动，签名还是 `
da6b ` 、墙 A 根本不出现，用 Root 在进程外把 ` .so ` 注进去。

##  四、工具登场：ZygiskFrida

###  4.1 它解决什么

ZygiskFrida 是一个 **Zygisk 模块** （Magisk 的 Zygote 注入框架）。它在 Zygote fork 出 App 进程的
**极早期** 就拿到执行权，然后 ` dlopen ` 把 Gadget 加载进目标进程。关键在于：

  * **注的是原版 App** ：完全不碰 base.apk，签名仍然是 ` da6b ` ， **第二节那堵签名墙直接不存在** 。
  * **无 ptrace、无 memfd** ：是 linker 在进程内 ` dlopen ` （ ` TracerPid=0 ` ），不像 frida-server 走 ` ptrace ` \+ ` memfd ` 。
  * **` mremap ` 隐藏 ** ：把 Gadget 从 ` /proc/self/maps ` 重映射抹掉（实测 ` grep -ic frida /proc/<pid>/maps ` → ` 0 ` ）。注意它只骗读 ` /proc/maps ` 的检测；直接遍历 linker solist 链表的检测照样能发现——这正是第 5.1 节的墙 C 慢层。

所以 ZygiskFrida 做的事就一句话：Root 把 Gadget 注进原版 App，绕开签名校验，还自带隐身。这台设备已经装好 Magisk
### 30.7 + Zygisk + ZygiskFrida v1.9.0(lico-n)。

###  4.2 配置长什么样

ZygiskFrida 的配置都在 ` /data/local/tmp/re.zyg.fri/ ` 下，三个文件：


    /data/local/tmp/re.zyg.fri/
    ├── config.json            # 注谁、注什么
    ├── libgadget.so           # 线程名 patch 过的 Gadget
    └── libgadget.config.so    # Gadget 的 script-mode 配置，指向 recipe 脚本


` config.json ` 指定目标和要注入的库：


    {
      "targets": [
        {
          "app_name": "com.netease.cloudmusic",
          "enabled": true,
          "start_up_delay_ms": 0,
          "injected_libraries": [
            { "path": "/data/local/tmp/re.zyg.fri/libgadget.so" }
          ]
        }
      ]
    }


Gadget 加载后会用 ` dladdr ` 找 **和自己同目录、同名前缀** 的 ` libgadget.config.so ` （第 1.2 节的
script 模式），所以注入路径和 config 必须放一起、名字前缀一致，否则脚本不会跑。 ` libgadget.config.so ` 里写：


    { "interaction": { "type": "script", "path": "/data/local/tmp/re.zyg.fri/ncm.js", "on_change": "reload" } }


` ncm.js ` 就是你的 recipe 脚本，支持热重载。

###  4.3 线程名 patch：把 Frida 的指纹抹掉

Gadget 默认会起 ` gum-js-loop ` 、 ` gmain ` 等线程，模块名里也有 ` frida-gadget ` ，而很多反调试会扫
` /proc/<pid>/task/*/comm ` 找这些名字。所以这里的 Gadget 做了线程名等长改写：


    # 等长替换，长度必须相等（否则破坏 ELF 布局）
    for old, new in [(b"gum-js-loop\x00", b"ndk_worker1\x00"),
                     (b"gmain\x00",       b"gmaio\x00"),
                     (b"frida-gadget\x00", b"nemediacodec\x00")]:
        data = data.replace(old, new)
    # 校验：strings libgadget.so | grep -axE "gum-js-loop|gmain|frida-gadget"  → 应为空


实测注入后 ` /proc/<pid>/task/*/comm ` 只剩 ` ndk_worker1 ` 、 ` gmaio ` 、 `
nemediacodec ` 加 RN 的 ` mqt_js ` ，没有任何 gum/gmain/frida 字样。

###  4.4 先装原版，墙 A 消失


    DEV=1C132FDEE003SU; PKG=com.netease.cloudmusic
    # music163.9.2.80.apk 带 META-INF/NETEASE.RSA，是网易原签 APK
    adb -s $DEV uninstall $PKG
    adb -s $DEV install --no-incremental music163.9.2.80.apk
    # 验证证书：cert MD5 应为 da6b069d...（= libpoison 期望值）


装上原版后启动，logcat 里 ` POISON: APK Signature:da6b… ` = 期望值 → libpoison
不退出、不弹「请前往官方渠道下载正版 APP」。第二节那堵签名墙， **靠「不重签」直接绕过，而不是靠 Hook 绕过** 。

##  五、墙 C：装原版后的运行时反调试

墙 A 消失，剩下的才是这个 App 真正的反 Frida 防线，而且这次没有签名噪声干扰，看得最干净。它分两层。


网易云运行时反调试时序图

> 上图把本节的时序拍平在一条时间轴上： **裸 Gadget 先稳住（破慢层）→ 以「点同意」为分水岭 → 带 Java Hook 的 recipe
> 在此被杀、native-only 的 recipe 安然跨过** 。下面分慢层、快层、winning recipe 三小节展开。

###  5.1 慢层：MSA 构造函数遍历 solist，撞到毒指针自爆

裸 Gadget（零 Hook）注进原版、点「同意」继续初始化 → **+4.4 秒，` libmsaoaidsec.so ` （MSA）加载完立刻
SIGSEGV ** ：


    fault addr 0xb0202000015b2  in tid (licationWrapper)
      #00 libmsaoaidsec.so + 0x20d10   ← ldrb w8,[x20,0x188]，x20 = 毒指针
      #01 + 0x11f64    #02 + 0x95f8    #03 + 0x14824（.init_array 构造函数）


调用链 ` 0x14824 → 0x95f8 → 0x11f64 → 0x20bdc → 0x20d10 ` ：MSA 在 ` 0x20bdc ` 处
**遍历 linker 的 solist 链表** （注意，它直接走链表，不读 ` /proc/maps ` ，所以 ` mremap `
隐藏对它无效），找到 agent 的 soinfo，然后故意去 deref 一个毒指针把自己崩掉。这是典型的「注入名检测」：发现进程里多了不该有的 `
.so ` 就自爆。

**破法** ：MSA 这个 SO **没有加密** ，可以直接静态 patch，把 ` 0x20bdc ` 处改成「直接返回 0」，检测器就废了：


    偏移 0x20bdc 写入：00 00 80 d2 c0 03 5f d6     # mov x0,#0 ; ret
    补丁版校验 md5 = 00c476dd6447adfc429c3008fa0f8e5e


然后 root 替换 **解压在磁盘上的那份 lib** （不动 base.apk，所以签名校验读到的还是 ` da6b ` ，不受影响）：


    LIB=$(adb -s $DEV shell "su -c 'find /data/app -name libmsaoaidsec.so | grep cloudmusic'")
    adb -s $DEV push patched_libmsaoaidsec.so /data/local/tmp/_msa.so
    adb -s $DEV shell "su -c 'cat /data/local/tmp/_msa.so > \"$LIB\"; chmod 644 \"$LIB\";
      chown system:system \"$LIB\"; chcon u:object_r:apk_data_file:s0 \"$LIB\"'"


` chcon u:object_r:apk_data_file:s0 ` 这步 **不能漏** ，否则 SELinux 标签不对，App
直接起不来。补丁后裸 Gadget **撑过 MSA 加载 → 进 MainActivity（完整主页），存活 33 秒以上、零崩溃** 。慢层破了。

###  5.2 快层：点「同意」后认 Java/ART Hook，inline-svc 静默杀

给存活的 Gadget 加一个 **Java 层 Hook** （ ` NeteaseMusicUtils.serialdata(String,
String) ` ，这是网易云的签名算法入口，+0.3s 装上）。结果很有意思：

  * 在 **同意页** 它 **活了 84 秒** 没事，还在冷启动突发里 **抓全了 serialdata 的输入输出** 。
  * 一旦 **点了「同意」** ，随即 ` has died: fg TOP ` ， **没有任何崩溃信号** 。这种「干净地死、libc 的 exit/abort Hook 都看不见」的死法，是 **inline-svc** （直接内联系统调用，绕过 libc）杀的。

这揭示一条 **同意门控** 的时序铁律：

时机  |  安全 SDK 状态  |  Java Hook 命运
---|---|---
点同意 **前** |  受合规门控，检测未武装  |  在同意页活 84 秒没事
点同意 **后** |  快层武装，开始认 Java/ART Hook（蹦床 / ART 入口篡改 / 硬件断点）  |  inline-svc 杀

###  5.3 winning recipe：只 native hook 非安全库

既然快层认的是「Java/ART Hook」和「钩安全库」，那就 **两个都不碰** ：换成 **纯 native SSL Hook** ，钩
conscrypt 的 ` SSL_write ` / ` SSL_read ` （这是非安全库），把所有 Java Hook 全删掉。

结果：A13 上 ** >2 分钟在线（实测 144 秒，跑到 LoginActivity）+ API 明文持续捕获 ** ，App 全程功能正常。核心
hook 如下，可直接拷进 ` ncm.js ` 跑（完整脚本——含按 host 过滤、SNI 提取、 ` SSL_get_session ` 关联——见第
17 篇第 8 节）：


    // 设计三原则：不碰 Java、不碰安全库 .text、只钩非安全库的明文读写点
    // 1. 不要 Java.perform 早期 hook —— 极早期 Java 桥不可用，且点同意后会被杀
    // 2. 不要钩 libnesec / libmsaoaidsec / libpoison 等安全库 .text
    // 3. 只在 conscrypt 的非安全库（libssl.so）里钩明文读写点
    function dump(tag, buf, len) {
        if (len <= 0) return;                      // SSL_read 出错/EOF 时返回 ≤0
        console.log(`\n[${tag}] ${len} bytes`);
        console.log(hexdump(buf, { length: Math.min(len, 0x200), ansi: false }));
    }

    // SSL_write(ssl, buf, num)：明文在 onEnter 的 buf 里（加密前），num 即长度
    Interceptor.attach(Module.findExportByName("libssl.so", "SSL_write"), {
        onEnter(args) { dump("TLS →", args[1], args[2].toInt32()); }
    });

    // SSL_read(ssl, buf, num)：buf 要等函数返回才填好，真实长度 = 返回值（非 num）
    Interceptor.attach(Module.findExportByName("libssl.so", "SSL_read"), {
        onEnter(args) { this.buf = args[1]; },
        onLeave(ret)  { dump("TLS ←", this.buf, ret.toInt32()); }
    });


> 注：上面的全局两参形式 ` Module.findExportByName(name, sym) ` 在本篇 Frida 16.7.19
> 上正确；Frida 17+ 已弃用它，改用 `
> Process.getModuleByName("libssl.so").findExportByName("SSL_write") ` 。

这条 recipe 抓到的真实样本（节选）：


    # serialdata 算法 I/O（冷启动突发，path + params 明文）
    /api/ad/start/info/upload   {"firstActive":"true","oaid":"","orpheus":"","header":"{}","e_r":true}
    /api/sp/flow/status/v2      {"deviceid":"bnVsbAkwMjowMDowMDowMDowMDowMAk...","header":"{}","e_r":true}

    # API 明文（native SSL hook，>2min 在线期间）
    POST /yd/nonlogin/client/initSdk.do   Host: sdk.reg.163.com   yd-version: 2.4.9.19
    POST /client/api/uploadStartUpInfo.do Host: crash.163.com
    HTTP/1.1 200  Set-Cookie: SID=...; JSESSIONID=...


到这里实战目标全部达成：进 App、长存活、抓到签名算法 I/O、抓到 API 明文。

##  六、Gadget 并不比 frida-server 隐蔽

初稿我写过一句结论：「Gadget 没有 ptrace/memfd 的注入结构，所以躲过了秒杀 frida-server 的 A13
检测」。这句话听起来很顺，也是网上常见说法。 **做对照基线时当场翻车。**

把 ZygiskFrida 对网易云的注入 **关掉** ，在 **同一台 A13、同一原版、同一会话** 里直接跑 frida-
server（spawn）：

对照实验  |  结果
---|---
frida-server spawn 原版（停在同意页）  |  活 >30 秒
frida-server spawn 原版 + 点同意  |  活 >45 秒，进 LoginActivity
同上， **禁用 Shamiko 后重测** |  仍活 >45 秒，进 LoginActivity

**带 memfd/ptrace 的 frida-server，活得和无这些结构的 Gadget 一样好；禁不禁 Shamiko 都一样。**
结论被自己的对照实验证伪：

  * 所谓「A13 注入结构检测」这堵墙（ **墙 B** ）， **这次根本没观测到** 。第 17 篇第 9 节判过「Pixel 6 Pro/A13 失败、frida ~1 秒被杀、recipe 不可移植」，在干净设备态下并不复现，最可能是当时设备被反复测试进入了退化态、被持久化标记过。 **本篇等于推翻了那条 A13 死刑判决。**
  * **Gadget 对 frida-server 没有任何可证明的存活优势。**

为什么裸 agent（不管 Gadget 还是 frida-server）都能活？用「两层痕迹」就能解释。Frida 的痕迹分两层：注入层（memfd /
ptrace / 线程名 / 模块名）和运行时插桩层（Interceptor 蹦床 / ART 入口篡改 / gum 引擎内存）。这台 A13
的快层只认第二层里的蹦床，既不认第一层，也不认「光是 gum 引擎在场」。所以裸 agent 没下任何 Hook、没有蹦床，能活；一旦下 Java/ART
Hook 或钩安全库 ` .text ` 产生了蹦床，就死。winning recipe 能成立，正是因为它精确避开了快层唯一在看的东西。

那么真正让实战跑通的功臣是谁？是「不重签（绕墙 A）+ MSA 静态补丁（破慢层）+ native-only
recipe（避开快层）」这三样，和「Gadget vs frida-server」「用不用 Shamiko」都无关。

##  七、选择指南与总结

###  7.1 更新后的决策表

场景  |  推荐方案  |  关键限制
---|---|---
软目标（不校验自身签名）  |  免 Root Gadget 重打包  |  唯一适合免 Root 的场景
加固大厂 App（有签名校验）+ 有 Root  |  **ZygiskFrida 注原版** （不重签）  |  需破运行时反调试（如 MSA 补丁）
同上，想要现成 attach/spawn  |  **frida-server attach/spawn 原版** |  与 ZygiskFrida 存活等价，无隐身劣势
只要还原签名算法、批量出参  |  抓一次明文 I/O → 转 **unidbg 离线** |  终局：没真机、没 frida、无 on-device 反制可检测

###  7.2 三类典型需求该走哪条路

  * **「我要还原 serialdata 签名算法、批量生成」** → 真机抓一次明文输入输出（本篇 Java Hook 抢冷启动突发），拿到 path / params / sign 样本，转 unidbg 离线复现，彻底脱离真机反制。
  * **「我要在真机上长期抓这个 App 的流量」** → Root + **不重签** （ZygiskFrida 注原版 或 frida-server attach 原版）+ MSA 补丁 + native SSL recipe；Gadget 和 server 随便选，存活一样。
  * **「我没有 Root，只能改 APK」** → 对易盾这类签名校验 App **此路不通** （第二、三节）；只能转抓包（若无证书锁）/ Web 侧 / unidbg，或者换一台有 Root 的设备。

###  7.3 总结

这次实战真正学到的，不是「免 Root Gadget 怎么注」的操作步骤，而是 **一次被现实逼出来的场景迁移** ：

  * **注入这个核心，要先看场景。** 免 Root Gadget 是个好工具，软目标是它的主场；对加固大厂 App，免 Root = 重签 = 撞签名墙，能不能 Hook 糊弄过去要看实现（普通 App 绕得过，易盾这类顶级壳又早又深，实际绕不过去），所以对网易云真实可行的是 Root + 不重签（ZygiskFrida / frida-server）。
  * **签名校验这个核心，是免 Root 甩不掉的债。** 它早、native、inline，窗口期没有干净拦截点；而「不重签」从根上让它失去意义，绕过它最好的办法就是不去碰它。签名校验绕过的更多角度见第 25 篇。
  * **跑通之后还有第二堵墙** （运行时反调试）：慢层（MSA solist-walk 自爆）靠静态补丁破，快层（认 Java/ART Hook）靠 native-only recipe 避开。
  * **最后别忘了证伪自己。** Gadget 比 frida-server 隐蔽，是个没复跑对照基线就下的结论；一旦补上对照，神话就破了。
