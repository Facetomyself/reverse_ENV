##  前言

在移动安全研究和逆向工程领域，动态分析工具是不可或缺的利器。Frida 作为目前最流行的动态插桩框架（Dynamic Instrumentation
Framework），其  ` Interceptor  ` 模块已被广泛使用——无论是 Hook Java 层方法还是 Native
函数，Interceptor 都能轻松胜任。然而，当我们需要更精细的  ** 指令级追踪能力  ** 时，Frida 内置的  ** Stalker  **
引擎才是真正的"杀手锏"。

Stalker 是 Frida 内置的  ** 代码追踪引擎（Code Tracing Engine）  **
，能够实现指令级别的跟踪，捕获每一条执行的指令、每一次函数调用和返回。想象一下：当你面对一个被 OLLVM 混淆的加密函数，Interceptor
只能告诉你"函数被调用了、参数是什么、返回值是什么"，而 Stalker
可以像放慢镜头一样，逐条回放函数内部执行的每一条汇编指令，记录每一个寄存器的值变化。

本文将从原理到实战，全面介绍 Stalker 的使用方法和技巧。


##  一、Stalker 概述

###  1.1 什么是 Stalker

Stalker 是 Frida 提供的  ** 线程级代码追踪引擎  ** ，它能够：

  * ** 跟踪线程执行  ** ：捕获目标线程执行的每一条指令
  * ** 记录函数调用  ** ：追踪所有 CALL/RET 指令
  * ** 实时代码插桩  ** ：在指令执行前后插入自定义逻辑
  * ** 修改执行流程  ** ：通过 transform 回调动态修改代码行为

简单来说，如果 Interceptor 是在高速公路入口/出口设置的检查站，那么 Stalker
就是一架全程跟拍的无人机——它能看到车辆在高速上的每一个动作：变道、加速、转弯，一览无余。

###  1.2 底层原理

Stalker 基于  ** 动态重编译（Dynamic Recompilation）  ** 机制工作，这种机制也被称为  ** copy-on-exec
** ：


Stalker 动态重编译工作原理

具体流程如下：

  1. ** 拦截执行  ** ：当目标线程即将执行一个基本块（Basic Block，即一段没有分支的连续指令序列）时，Stalker 拦截执行
  2. ** 分配新内存  ** ：Stalker 分配一块新的内存区域
  3. ** 复制并重定位  ** ：将原始基本块的指令复制到新位置，在复制过程中进行  ** 指令重定位（Relocation）  ** ，处理 PC 相关寻址（如  ` adr  ` 、  ` adrp  ` 、PC-relative load 等）
  4. ** 插入插桩代码  ** ：在指令副本的适当位置插入事件记录代码、回调调用代码等
  5. ** 修改跳转  ** ：修改基本块末尾的分支指令，使其跳回 Stalker 引擎，以便对下一个基本块进行同样的处理
  6. ** 执行副本  ** ：线程转而执行插桩后的代码副本

> ** 什么是基本块（Basic Block）？  **
> 基本块是编译原理中的一个基础概念，指一段连续的指令序列，其中只有第一条指令是跳转目标（入口），只有最后一条指令是分支/跳转指令（出口）。换句话说，进入基本块后，程序一定会顺序执行到最后一条指令，中间不会有跳转。

这种 copy-on-exec 机制的巧妙之处在于：  ** 它不修改原始代码  **
，因此不会被简单的代码完整性校验检测到。已编译的基本块会被缓存在内存中，避免重复编译带来的开销——第二次执行同一个基本块时，直接执行缓存的副本即可。

###  1.3 支持架构

架构  |  支持状态  |  典型场景
---|---|---
AArch64  |  完整支持  |  Android/iOS 手机和平板
ARM (32-bit)  |  支持  |  旧版 Android 设备
Intel x64  |  完整支持  |  桌面应用、模拟器
IA-32 (x86)  |  支持  |  32位桌面应用

> ** 提示  ** ：在实际逆向工作中，AArch64 是最常用的架构。现代 Android 设备（Android 7.0+）和所有 iOS
> 设备基本都是 64 位 ARM。如果你的目标 App 仍然只提供 32 位 SO，可以考虑使用 64 位设备的 32 位兼容模式运行。

###  1.4 Stalker vs Interceptor

Interceptor vs Stalker 对比

特性  |  Interceptor  |  Stalker
---|---|---
粒度  |  函数级  |  指令级
性能开销  |  低  |  较高
使用场景  |  Hook 特定函数  |  追踪执行流程
修改能力  |  修改参数/返回值  |  修改任意指令
数据量  |  少量精确数据  |  大量追踪数据
适用阶段  |  已知目标函数  |  探索未知逻辑

** 最佳实践  ** ：先用 Interceptor 定位关键函数，再用 Stalker
深入分析函数内部的指令执行流程。这种"先粗后细"的策略能在效率和精度之间取得最佳平衡。

###  2.1 Stalker.follow(threadId, options)

这是 Stalker 最核心的 API，用于开始跟踪指定线程。它接受线程 ID 和一个配置对象，配置对象中可以指定要捕获的事件类型和回调函数。


    // Stalker.follow - 开始跟踪指定线程
    // threadId: 目标线程的 ID（可通过 Process.enumerateThreads() 或 this.threadId 获取）
    Stalker.follow(threadId, {
        // 事件配置：指定要捕获哪些类型的事件
        events: {
            call: true,    // 捕获 CALL 指令（函数调用，如 BL/BLR）
            ret: false,    // 捕获 RET 指令（函数返回）
            exec: false,   // 捕获所有执行的指令（慎用！数据量极大，可能导致卡死）
            block: false,  // 捕获基本块执行（每次进入一个基本块时触发）
            compile: false // 捕获基本块编译（仅在首次编译时触发，适用于代码覆盖率收集）
        },

        // 方式一：接收原始事件数据（二进制格式，数据更完整）
        onReceive(events) {
            // events 是 ArrayBuffer，包含 GumEvent 结构体数组
            // 需要使用 Stalker.parse() 解析为可读格式
            const parsed = Stalker.parse(events, {
                annotate: true,  // 在结果中标注事件类型（call/ret/exec/block/compile）
                stringify: true  // 将 NativePointer 格式化为十六进制字符串（减少 send() 的序列化开销）
            });
            console.log(JSON.stringify(parsed));
        },

        // 方式二：接收调用统计摘要（更高效，适合快速了解调用分布）
        // 注意：onReceive 和 onCallSummary 是互斥的，只能选择其中一个！
        // onCallSummary(summary) {
        //     // summary 是一个对象：{ "0x7fff12345": 42, "0x7fff12346": 10, ... }
        //     // key 为调用目标地址，value 为在统计周期内的调用次数
        // },

        // 高级功能：transform 回调（后续第四节详细介绍）
        // transform(iterator) { ... }
    });


** 重要提示  ** ：  ` onReceive  ` 和  ` onCallSummary  ` 只能二选一。如果你不需要某种回调，不要传空函数（如
` onReceive() {}  ` ），因为即使回调是空的，Stalker 仍然会为其生成事件数据，造成不必要的性能开销。

###  2.2 Stalker.unfollow(threadId)

停止跟踪指定线程。如果省略参数，则停止跟踪当前线程：


    // 停止跟踪指定线程
    Stalker.unfollow(targetThreadId);

    // 停止跟踪当前线程（省略参数时默认当前线程）
    Stalker.unfollow();


###  2.3 Stalker.flush()

立即刷新事件缓冲区，将已缓冲的事件数据发送到  ` onReceive  ` /  ` onCallSummary  ` 回调：


    // 强制刷新缓冲区，不等待下一个 queueDrainInterval 周期
    Stalker.flush();


默认情况下，Stalker 每  ** 250ms  ** 自动刷新一次（由  ` queueDrainInterval  `
控制）。也就是说，事件并不是实时发送的，而是先积攒在缓冲区中，达到时间间隔或缓冲区满时才批量发送。调用  ` flush()  `
可以立即获取最新数据，这在跟踪结束时（  ` unfollow  ` 之前）尤其重要——确保最后一批事件不会丢失。

###  2.4 Stalker.garbageCollect()

在安全时机回收已释放的代码块内存：


    // 典型用法：先 unfollow，等待一小段时间，再回收内存
    Stalker.unfollow(threadId);

    // 等待 50ms，确保线程已经完全离开插桩代码区域
    setTimeout(() => {
        Stalker.garbageCollect();
    }, 50);


** 为什么需要这个 API？  ** 当  ` unfollow()  ` 被调用时，被跟踪的线程可能仍在执行最后一段插桩代码（因为 unfollow
是异步生效的）。如果此时立即释放这些代码块的内存，线程就会跳转到已被回收的内存地址，导致段错误崩溃。  ` garbageCollect()  `
通过延迟释放机制，确保在安全时机才释放内存，避免这种竞态条件（Race Condition）。

###  2.5 Stalker.exclude(range)

排除指定内存范围，Stalker 不会跟踪进入该范围的代码执行。这是  ** 性能优化最重要的手段  ** ：


    // 排除系统库，大幅降低噪音和性能开销
    // 排除 libc — 标准 C 库的函数调用（如 malloc、memcpy）不需要追踪
    const libc = Process.getModuleByName("libc.so");
    Stalker.exclude(libc);

    // 排除 linker — 动态链接器的内部逻辑与分析目标无关
    const linker = Process.getModuleByName("linker64");
    Stalker.exclude(linker);

    // 排除 Frida 自身的 agent — 避免追踪 Frida 内部代码导致死循环
    const fridaAgent = Process.getModuleByName("frida-agent-64.so");
    Stalker.exclude(fridaAgent);


> ** 比喻  ** ：如果不使用  ` exclude  ` ，就像在整个城市安装了监控摄像头来追踪一个人——你会淹没在海量无关的监控画面中。使用  `
> exclude  ` 就是只在目标人物可能出没的区域安装摄像头，既节省资源又减少噪音。

###  2.6 Stalker.invalidate(address)

使指定地址所在基本块的已编译缓存失效，强制 Stalker 在下次执行该基本块时重新编译：


    // 使当前线程中某地址的缓存失效
    Stalker.invalidate(ptr("0x12345678"));

    // 使指定线程中的缓存失效
    Stalker.invalidate(threadId, ptr("0x12345678"));


这比  ` unfollow()  ` \+ 重新  ` follow()  ` 要高效得多，因为后者会丢弃  ** 所有  ** 已缓存的编译结果，而
` invalidate  `
只影响一个基本块。适用于需要动态调整某个基本块的插桩策略时——比如在分析过程中发现某个基本块是关键的加密循环，想要添加更详细的追踪。

###  2.7 Stalker.addCallProbe(address, callback)

在指定地址设置调用探针（Call Probe），当被跟踪的线程执行到该地址的 CALL 指令时，会  ** 同步调用  ** 回调函数：


    // 在目标地址设置探针，每次被调用时打印第一个参数
    const probeId = Stalker.addCallProbe(targetAddr, function(args) {
        console.log("Call to target, arg0:", args[0]);
    });

    // 不再需要时移除探针
    Stalker.removeCallProbe(probeId);


> ** 与 Interceptor.attach 的区别  ** ：Call Probe 只在 Stalker 跟踪的线程中生效，而
> Interceptor.attach 对所有线程生效。Call Probe 的优势是可以与 Stalker 的其他功能（如
> transform）协同工作。

###  2.8 Stalker.parse(events, options)

解析  ` onReceive  ` 收到的原始二进制事件数据，将 ArrayBuffer 转换为可读的数组格式：


    onReceive(events) {
        const parsed = Stalker.parse(events, {
            annotate: true,   // 在结果中标注事件类型（call/ret/exec/block/compile）
            stringify: true   // 将 NativePointer 转为十六进制字符串（减少后续 send() 的开销）
        });
        // 解析结果示例（annotate 模式）：
        // call 事件：["call", "0x7fff1234", "0x7fff5678", 0]  — [类型, 源地址, 目标地址, 调用深度]
        // ret 事件：["ret", "0x7fff5678", "0x7fff1238", 0]    — [类型, 源地址, 返回地址, 调用深度]
        send({ type: "stalker-events", payload: parsed });
    }


###  2.9 配置属性


    // 事件队列容量（可容纳的事件数量上限）
    // 当队列满时会立即触发 onReceive/onCallSummary，不等待 drainInterval
    Stalker.queueCapacity = 16384;  // 默认值

    // 事件队列自动排空间隔（毫秒）
    // Stalker 每隔这么长时间就把积攒的事件发送给回调
    Stalker.queueDrainInterval = 250;  // 默认值 250ms
    // 设为 0 可禁用自动排空，完全由你手动调用 Stalker.flush() 控制

    // 信任阈值 — 控制对自修改代码（Self-Modifying Code, SMC）的检测
    Stalker.trustThreshold = 1;   // 默认值：基本块执行 1 次后信任（不再检查是否被修改）
    // -1 = 永不信任（每次执行都重新编译，非常慢，但能处理自修改代码/VMP 保护）
    //  0 = 立即信任（第一次编译后就不再检查，性能最好）
    //  N = 执行 N 次后信任

###  3.2 最小示例：跟踪函数调用

创建文件  ` stalker_basic.js  ` ：


    // stalker_basic.js — 跟踪目标函数内部的所有 CALL 指令
    // 场景：监控 libc 的 open 函数执行过程中调用了哪些子函数

    // 获取 libc 中 open 函数的地址
    const openAddr = Module.findExportByName("libc.so", "open");

    // 使用 Interceptor 作为触发器：当 open 被调用时启动 Stalker
    Interceptor.attach(openAddr, {
        onEnter(args) {
            // 读取 open 的第一个参数（文件路径）
            const path = args[0].readUtf8String();

            // 只对包含特定关键词的路径启动追踪，避免噪音
            if (path && path.includes("target_file")) {
                console.log(`[*] open("${path}") — 开始 Stalker 追踪`);

                // 跟踪当前线程（this.threadId 是触发 onEnter 的线程 ID）
                Stalker.follow(this.threadId, {
                    events: {
                        call: true,    // 只关心函数调用事件
                        ret: false,
                        exec: false,
                        block: false,
                        compile: false
                    },

                    // 使用 onCallSummary 获取调用统计（比 onReceive 更高效）
                    onCallSummary(summary) {
                        console.log("[*] 调用统计：");
                        // summary 的 key 是目标地址，value 是调用次数
                        for (const [addr, count] of Object.entries(summary)) {
                            // 尝试解析地址所属的模块和符号名
                            const module = Process.findModuleByAddress(ptr(addr));
                            const name = module ? module.name : "unknown";
                            const func = DebugSymbol.fromAddress(ptr(addr));
                            console.log(`    ${addr} (${name}!${func.name || "?"}) => ${count} 次`);
                        }
                    }
                });
            }
        },

        onLeave(retval) {
            // 函数返回时停止跟踪
            Stalker.unfollow(this.threadId);
            Stalker.flush();  // 确保最后一批事件被发送
        }
    });

###  3.3 使用 onReceive 获取详细事件

当你需要更详细的事件信息（如调用深度、源地址和目标地址）时，使用  ` onReceive  ` ：


    // stalker_events.js — 获取详细的调用和返回事件

    function stalkThread(tid, duration) {
        console.log(`[*] 开始跟踪线程 ${tid}，持续 ${duration}ms`);

        Stalker.follow(tid, {
            events: {
                call: true,   // 捕获函数调用
                ret: true,    // 捕获函数返回（配合 call 可以还原调用链）
                exec: false,
                block: false,
                compile: false
            },

            onReceive(events) {
                // 解析原始二进制事件数据
                const parsed = Stalker.parse(events, {
                    annotate: true,   // 标注事件类型
                    stringify: true   // 指针转字符串
                });

                for (const event of parsed) {
                    // annotate 模式下的事件格式：
                    // call 事件：["call", fromAddr, toAddr, depth]
                    // ret 事件：["ret", fromAddr, toAddr, depth]
                    if (event[0] === "call") {
                        const target = event[2];  // 调用目标地址
                        const sym = DebugSymbol.fromAddress(ptr(target));
                        if (sym.name) {
                            console.log(`  CALL => ${sym.moduleName}!${sym.name}`);
                        }
                    }
                }
            }
        });

        // 指定时长后自动停止跟踪
        setTimeout(() => {
            Stalker.unfollow(tid);
            Stalker.flush();
            // 延迟回收内存，避免竞态崩溃
            setTimeout(() => Stalker.garbageCollect(), 50);
            console.log(`[*] 线程 ${tid} 跟踪结束`);
        }, duration);
    }

    // 获取主线程并跟踪 3 秒
    const mainThread = Process.enumerateThreads()[0];
    stalkThread(mainThread.id, 3000);


###  3.4 常见坑点

  1. ** 线程选择  ** ：不要随意跟踪主线程（UI 线程），Stalker 会显著降低执行速度，可能导致 App 卡死甚至触发 ANR（Application Not Responding）。建议先用  ` Process.enumerateThreads()  ` 了解线程分布，选择目标工作线程
  2. ** 时机控制  ** ：在  ` Interceptor.attach  ` 的  ` onEnter  ` 中启动 Stalker，  ` onLeave  ` 中停止，可以精确控制跟踪范围，避免追踪无关代码
  3. ** 事件回调互斥  ** ：  ` onReceive  ` 和  ` onCallSummary  ` 只能二选一，传两个会导致只有一个生效
  4. ** 资源清理  ** ：  ` unfollow  ` 后务必在安全时机（延迟 50ms 左右）调用  ` garbageCollect  ` ，否则可能出现内存泄漏


##  四、transform 回调 — Stalker 的核心能力

` transform  ` 是 Stalker 最强大的功能，它允许你在  ** 基本块编译阶段  ** 介入，逐条检查和修改指令。如果说事件回调（  `
onReceive  ` /  ` onCallSummary  ` ）是"被动观察"，那么  ` transform  ` 就是"主动参与"。

###  4.1 工作原理

当 Stalker 遇到一个新的基本块需要编译时，会调用  ` transform  ` 回调。在回调内部，你通过 iterator
遍历基本块中的每条指令，可以：

  * ** 遍历基本块中的每条指令  ** ：获取地址、助记符、操作数等信息
  * ** 选择保留或丢弃指令  ** ：调用  ` iterator.keep()  ` 保留，不调用则丢弃
  * ** 在指令前后插入自定义代码  ** ：使用  ` iterator.putCallout()  ` 插入 JavaScript 回调
  * ** 读取指令的详细信息  ** ：地址、助记符（如  ` mov  ` 、  ` bl  ` 、  ` ret  ` ）、操作数字符串等

###  4.2 基本结构


    Stalker.follow(threadId, {
        // transform 在每个基本块首次编译时被调用
        // 编译结果会被缓存，同一基本块不会重复调用 transform
        transform(iterator) {
            // iterator.next() 前进到下一条指令，返回指令对象
            let instruction = iterator.next();

            do {
                // instruction 对象包含以下属性：
                // instruction.address — 指令在原始代码中的地址（NativePointer）
                // instruction.mnemonic — 指令助记符，如 "mov", "bl", "ret", "ldr"
                // instruction.opStr — 操作数字符串，如 "x0, x1", "#0x10"
                // instruction.size — 指令的字节长度

                // 必须调用 keep() 保留原始指令到输出
                // 如果不调用 keep()，该指令会被丢弃（相当于 NOP 掉）
                iterator.keep();
            } while ((instruction = iterator.next()) !== null);
            // next() 返回 null 表示已遍历完当前基本块的所有指令
        }
    });


** 如果不提供 transform 回调，Stalker 的默认行为等价于  ** ：


    // 默认行为：原封不动地保留所有指令
    transform(iterator) {
        while (iterator.next() !== null)
            iterator.keep();
    }


###  4.3 iterator API

方法  |  说明
---|---
` iterator.next()  ` |  前进到下一条指令，返回指令对象或 null（遍历结束）
` iterator.keep()  ` |  将当前指令写入输出（保留指令），不调用则丢弃该指令
` iterator.putCallout(callback)  ` |  在当前位置插入一个 JavaScript 回调函数调用
` iterator.putChainingReturn()  ` |  插入提前返回（让 Stalker 回到调度循环）
` iterator.memoryAccess  ` |  返回  ` 'open'  ` 表示当前位置可以安全插入代码（不在 exclusive 序列中）

###  4.4 使用 putCallout 插入回调

` putCallout  ` 是最常用的 transform 操作，它在指定位置插入一个 JavaScript 回调。回调接收  ` context  `
参数，包含当前 CPU 所有寄存器的状态（在 AArch64 上包括 x0-x28、sp、lr、pc 等）：


    // 示例：在每个 BL（分支链接/函数调用）指令前插入回调，记录调用目标和参数
    const targetModule = Process.getModuleByName("libtarget.so");
    const moduleBase = targetModule.base;
    const moduleEnd = moduleBase.add(targetModule.size);

    Stalker.follow(threadId, {
        transform(iterator) {
            let instruction = iterator.next();

            // 判断当前基本块是否属于目标模块
            // 注意：这个判断放在 do-while 外面，因为基本块内所有指令属于同一模块
            const isTargetModule = instruction.address.compare(moduleBase) >= 0 &&
                                   instruction.address.compare(moduleEnd) < 0;

            do {
                // 只对目标模块内的 BL 指令插桩
                if (isTargetModule && instruction.mnemonic === "bl") {
                    // 在 BL 指令执行前插入回调
                    iterator.putCallout((context) => {
                        // context 提供当前 CPU 寄存器的完整快照
                        console.log(`BL at ${context.pc}, target: ${instruction.opStr}`);
                        // 在 AArch64 调用约定中，x0-x7 是前 8 个参数
                        console.log(`  x0=${context.x0}, x1=${context.x1}`);
                    });
                }

                iterator.keep();  // 保留原始指令
            } while ((instruction = iterator.next()) !== null);
        }
    });


###  4.5 条件性插桩：只关注特定范围

在实际分析中，我们通常只关心目标 SO 库内部的指令执行，系统库的代码不需要追踪。通过地址范围判断可以大幅减少插桩点：


    // 只对目标 SO 库的代码进行插桩，忽略其他模块
    const targetModule = Process.getModuleByName("libtarget.so");
    const moduleStart = targetModule.base;
    const moduleEnd = moduleStart.add(targetModule.size);

    Stalker.follow(threadId, {
        transform(iterator) {
            let instruction = iterator.next();
            const blockStart = instruction.address;

            // 判断当前基本块是否属于目标模块
            // 只需检查基本块起始地址即可，同一基本块内的指令连续排列
            const isInTarget = blockStart.compare(moduleStart) >= 0 &&
                               blockStart.compare(moduleEnd) < 0;

            do {
                if (isInTarget) {
                    // 在目标模块中的每条指令前插入追踪回调
                    // 注意：对每条指令都插入回调会产生巨大的性能开销
                    // 实际分析中建议只对关键指令（如 bl、ret、cmp）插桩
                    iterator.putCallout((context) => {
                        console.log(`[${context.pc}] ${instruction.mnemonic} ${instruction.opStr}`);
                    });
                }
                iterator.keep();
            } while ((instruction = iterator.next()) !== null);
        }
    });


###  4.6 ARM/AArch64 注意事项

在 ARM 架构上，需要特别注意  ** exclusive load/store 指令序列  ** 。这类指令用于实现原子操作（如互斥锁、CAS
操作），在  ` ldxr  ` （exclusive load）和  ` stxr  ` （exclusive
store）之间不能插入任何额外的内存访问或分支指令，否则会导致 exclusive monitor 被清除，进而导致无限重试或崩溃：


    transform(iterator) {
        let instruction = iterator.next();

        do {
            // 检查是否可以安全插入代码
            // 'open' 表示当前不在 exclusive load/store 序列中
            // 在 exclusive 序列中插入 putCallout 会导致崩溃！
            const canInsertCode = iterator.memoryAccess === 'open';

            if (canInsertCode && instruction.mnemonic === "ret") {
                // 只在安全位置插入回调
                iterator.putCallout((context) => {
                    console.log(`RET at ${context.pc}, x0=${context.x0}`);
                });
            }

            iterator.keep();
        } while ((instruction = iterator.next()) !== null);
    }


> ** 什么是 Exclusive Load/Store？  ** 在 ARM 架构中，  ` ldxr  ` /  ` stxr  `
> 指令对用于实现无锁的原子操作。  ` ldxr  ` 加载一个值并设置 exclusive monitor，  ` stxr  `
> 尝试写入并返回是否成功。如果在两者之间有任何干扰（包括中断、其他内存访问），  ` stxr  ` 会失败，程序需要重试。Stalker 的  `
> putCallout  ` 会插入函数调用，这会破坏 exclusive monitor，因此必须检查  ` memoryAccess  ` 。


##  五、实用技巧与性能优化

Stalker 的指令级追踪不可避免地带来性能开销，但通过合理的优化策略，可以将开销控制在可接受的范围内。


Stalker 性能开销对比

###  5.1 使用 Stalker.exclude 排除无关模块

这是  ** 最重要的优化手段  ** ，效果立竿见影：


    // 策略一：排除已知的系统库和框架
    const excludeList = [
        "libc.so",                  // C 标准库
        "libm.so",                  // 数学库
        "libdl.so",                 // 动态链接相关
        "liblog.so",                // Android 日志
        "linker64",                 // 动态链接器
        "libandroid_runtime.so",    // Android 运行时
        "libart.so",                // ART 虚拟机
        "libbase.so",               // Android 基础库
        "frida-agent-64.so"         // Frida 自身（必须排除，否则可能死循环）
    ];

    for (const name of excludeList) {
        try {
            const mod = Process.getModuleByName(name);
            Stalker.exclude(mod);
            console.log(`[*] 已排除: ${name}`);
        } catch (e) {
            // 模块不存在（比如在不同设备上库名不同），忽略
        }
    }

    // 策略二（更激进）：只保留目标模块，排除其他所有模块
    // 适用于明确知道只需要分析某个特定 SO 的场景
    const targetModuleName = "libtarget.so";
    for (const mod of Process.enumerateModules()) {
        if (mod.name !== targetModuleName) {
            Stalker.exclude(mod);
        }
    }


###  5.2 精确选择事件类型

不同的分析场景需要不同的事件类型，选错事件类型会导致要么数据不够、要么性能爆炸：


    // !! 不要这样做 — exec 事件数据量巨大，通常一秒钟就能产生数百万条事件
    Stalker.follow(tid, {
        events: { call: true, ret: true, exec: true, block: true, compile: true },
        onReceive(events) { /* ... */ }
    });

    // === 按需选择正确的事件类型 ===

    // 场景1：只需要知道调用了哪些函数及频率 — 用 onCallSummary（最高效）
    Stalker.follow(tid, {
        events: { call: true },
        onCallSummary(summary) { /* { 地址: 次数 } */ }
    });

    // 场景2：需要代码覆盖率 — 只开 compile（最小数据量）
    // compile 事件只在基本块首次编译时触发，不会重复
    Stalker.follow(tid, {
        events: { compile: true },
        onReceive(events) { /* ... */ }
    });

    // 场景3：需要完整调用链（含调用深度）— call + ret
    Stalker.follow(tid, {
        events: { call: true, ret: true },
        onReceive(events) { /* ... */ }
    });


###  5.3 合理使用 flush


    // 方案1：减小排空间隔（数据更实时，但回调更频繁）
    Stalker.queueDrainInterval = 100;  // 100ms，默认 250ms

    // 方案2：禁用自动排空，完全手动控制
    // 适合精确分析：在目标函数执行完毕后手动 flush
    Stalker.queueDrainInterval = 0;

    // 在关键时机手动 flush
    Interceptor.attach(keyFuncAddr, {
        onLeave() {
            Stalker.flush();  // 确保函数执行期间产生的所有事件都已发送到回调
        }
    });


###  5.4 线程管理策略


    // 策略1：精确跟踪 — 只在目标函数执行期间追踪
    // 优点：追踪范围精确，数据量可控
    Interceptor.attach(targetFunc, {
        onEnter() {
            this.stalking = true;
            Stalker.follow(this.threadId, { /* options */ });
        },
        onLeave() {
            if (this.stalking) {
                Stalker.unfollow(this.threadId);
                Stalker.flush();
                this.stalking = false;
            }
        }
    });

    // 策略2：定向跟踪 — 找到特定工作线程进行持续跟踪
    const threads = Process.enumerateThreads();
    for (const thread of threads) {
        // 通过线程名、状态或其他标识来选择目标线程
        console.log(`Thread ${thread.id}: state=${thread.state}`);
    }


###  5.5 大量数据分批传输

当追踪产生大量数据时，直接在 JS 端通过  ` send()  ` 传输可能造成瓶颈。采用分批传输策略：


    // Frida 脚本端：分批积攒事件，达到阈值后批量发送
    let eventBuffer = [];
    const BATCH_SIZE = 1000;

    Stalker.follow(tid, {
        events: { call: true },
        onReceive(events) {
            const parsed = Stalker.parse(events, { stringify: true });
            eventBuffer = eventBuffer.concat(parsed);

            // 达到批次大小时发送
            if (eventBuffer.length >= BATCH_SIZE) {
                send({ type: "stalker-batch", events: eventBuffer });
                eventBuffer = [];
            }
        }
    });


Python 端接收并写入文件：


    import frida
    import json

    def on_message(message, data):

                with open("stalker_trace.jsonl", "a") as f:

###  5.6 Stalker + Interceptor 配合

这是最常用的实战模式——用 Interceptor 作为"触发器"，在特定条件下才启动 Stalker：


    // 先用 Interceptor 确认目标函数被调用，再用 Stalker 深入分析
    // 场景：某个函数在 App 启动时会被调用数百次，但只有第 3 次是我们关心的
    let callCount = 0;

    Interceptor.attach(targetAddr, {
        onEnter(args) {
            callCount++;

            // 只在第 3 次调用时启动 Stalker 分析（避免前两次的干扰数据）
            if (callCount === 3) {
                console.log("[*] 第3次调用，启动 Stalker 深度追踪");
                Stalker.follow(this.threadId, {
                    transform(iterator) {
                        let insn = iterator.next();
                        do {
                            // 只追踪函数调用指令
                            if (insn.mnemonic === "bl" || insn.mnemonic === "blr") {
                                iterator.putCallout((ctx) => {
                                    console.log(`  -> ${insn.mnemonic} at ${ctx.pc}`);
                                });
                            }
                            iterator.keep();
                        } while ((insn = iterator.next()) !== null);
                    }
                });
            }
        },

        onLeave(retval) {
            if (callCount === 3) {
                Stalker.unfollow(this.threadId);
                Stalker.flush();
            }
        }
    });


##  六、应用场景

###  6.1 代码覆盖率收集（Coverage）

代码覆盖率是 Stalker 最经典的应用场景之一。通过收集哪些基本块被执行过，然后导出为  ** drcov 格式  ** （DynamoRIO
的覆盖率数据格式），可以在 IDA 的 Lighthouse 插件或 Ghidra
中实现可视化的覆盖率着色——执行过的代码会被高亮显示，未执行的代码保持灰色。


    // coverage.js — 收集代码覆盖率，输出 drcov 格式

    const targetModule = Process.getModuleByName("libtarget.so");
    const moduleBase = targetModule.base;
    const moduleSize = targetModule.size;
    const modulePath = targetModule.path;

    // 使用 Map 存储已执行的基本块：offset -> size
    // 用 Map 而不是 Set 是因为需要同时记录偏移和大小
    const coveredBlocks = new Map();

    Stalker.follow(Process.getCurrentThreadId(), {
        events: {
            compile: true  // 只需要 compile 事件即可！
            // compile 在基本块首次被编译时触发一次，天然去重
            // 比 block（每次执行都触发）高效得多
        },

        onReceive(events) {
            // 不需要 annotate（节省解析开销）
            const parsed = Stalker.parse(events, { annotate: false, stringify: false });

            for (const event of parsed) {
                // compile 事件格式：[blockStartAddr, blockEndAddr]
                const start = event[0];
                const end = event[1];

                // 只记录目标模块内的基本块
                if (start.compare(moduleBase) >= 0 &&
                    start.compare(moduleBase.add(moduleSize)) < 0) {
                    // 计算相对于模块基址的偏移（IDA/Ghidra 使用偏移来匹配）
                    const offset = start.sub(moduleBase).toInt32();
                    const size = end.sub(start).toInt32();
                    coveredBlocks.set(offset, size);
                }
            }
        }
    });

    // 在合适的时机导出覆盖率数据（如分析完目标操作后手动调用）
    function exportDrcov() {
        Stalker.flush();

        // 生成 drcov v2 格式的文件头
        let output = "DRCOV VERSION: 2\n";
        output += "DRCOV FLAVOR: frida\n";
        output += `Module Table: version 2, count 1\n`;
        output += "Columns: id, base, end, entry, checksum, timestamp, path\n";
        output += `0, ${moduleBase}, ${moduleBase.add(moduleSize)}, 0, 0, 0, ${modulePath}\n`;
        output += `BB Table: ${coveredBlocks.size} bbs\n`;

        // 将基本块数据打包发送到 Python 端
        const blocks = [];
        for (const [offset, size] of coveredBlocks) {
            blocks.push({ offset, size, moduleId: 0 });
        }

        send({
            type: "coverage",
            format: "drcov",
            header: output,
            blocks: blocks
        });
    }

    // 通过 RPC 暴露导出函数，Python 端可以调用 script.exports.export_coverage()
    rpc.exports = {
        exportCoverage: exportDrcov
    };


###  6.2 函数调用链追踪

还原完整的函数调用链，特别适合分析  ** 被混淆或通过间接调用（如函数指针、虚函数表）调用  ** 的代码，这类调用在静态分析中很难追踪：


    // calltrace.js — 追踪完整的函数调用链（带缩进显示层级关系）

    const targetModule = Process.getModuleByName("libtarget.so");
    const base = targetModule.base;
    const end = base.add(targetModule.size);
    let callDepth = 0;  // 跟踪调用深度，用于输出缩进

    function startCallTrace(threadId) {
        Stalker.follow(threadId, {
            events: { call: true, ret: true },

            onReceive(events) {
                const parsed = Stalker.parse(events, {
                    annotate: true,
                    stringify: true
                });

                for (const event of parsed) {
                    const type = event[0];
                    const from = event[1];  // 调用/返回的源地址
                    const to = event[2];    // 调用/返回的目标地址

                    if (type === "call") {
                        // 根据调用深度添加缩进，直观展示调用层级
                        const indent = "  ".repeat(callDepth);
                        const sym = DebugSymbol.fromAddress(ptr(to));
                        // 有符号名就显示符号名，没有就显示偏移（如 sub_1a2c）
                        const name = sym.name || `sub_${ptr(to).sub(base).toString(16)}`;
                        console.log(`${indent}=> ${name} (from: ${from})`);
                        callDepth++;
                    } else if (type === "ret") {
                        callDepth = Math.max(0, callDepth - 1);
                    }
                }
            }
        });
    }

    // 在目标函数被调用时启动调用链追踪
    Interceptor.attach(Module.findExportByName("libtarget.so", "process_request"), {
        onEnter() {
            callDepth = 0;  // 重置调用深度
            startCallTrace(this.threadId);
        },
        onLeave() {
            Stalker.unfollow(this.threadId);
            Stalker.flush();
        }
    });


###  6.3 算法分析与还原

跟踪加密/签名函数的执行，记录关键寄存器和内存变化。这是逆向分析中最有价值的应用——通过观察算术/逻辑运算指令的输入输出，可以逐步还原算法逻辑：


    // algo_trace.js — 追踪算法执行，记录寄存器和内存操作

    const signFunc = Module.findExportByName("libtarget.so", "generate_sign");
    const targetModule = Process.getModuleByName("libtarget.so");

    Interceptor.attach(signFunc, {
        onEnter(args) {
            // 记录函数输入（作为后续验证的基准数据）
            this.inputPtr = args[0];
            this.inputLen = args[1].toInt32();
            console.log(`[*] generate_sign 输入: ${this.inputPtr.readByteArray(this.inputLen)}`);

            // 启动 Stalker 追踪算法内部的每一步运算
            Stalker.follow(this.threadId, {
                transform(iterator) {
                    let insn = iterator.next();
                    // 判断是否在目标模块内
                    const inTarget = insn.address.compare(targetModule.base) >= 0 &&
                                     insn.address.compare(targetModule.base.add(targetModule.size)) < 0;

                    do {
                        if (inTarget && iterator.memoryAccess === 'open') {
                            // 关注所有算术和逻辑运算指令 — 这些是加密算法的核心
                            // eor=异或, and=与, orr=或, lsl/lsr/ror=移位/旋转
                            // add/sub=加减, mul=乘法, adc=带进位加法
                            const dominated = ["eor", "and", "orr", "lsl", "lsr", "ror",
                                              "add", "sub", "mul", "adc"];

                            if (dominated.includes(insn.mnemonic)) {
                                // 保存指令信息（闭包捕获）
                                const addr = insn.address;
                                const mnemonic = insn.mnemonic;
                                const opStr = insn.opStr;

                                iterator.putCallout((context) => {
                                    // 记录运算前的寄存器状态
                                    // 通过对比运算前后的值，可以推断每一步的计算逻辑
                                    console.log(`[${addr}] ${mnemonic} ${opStr}`);
                                    console.log(`  x0=${context.x0} x1=${context.x1} x2=${context.x2} x3=${context.x3}`);
                                });
                            }
                        }

                        iterator.keep();
                    } while ((insn = iterator.next()) !== null);
                }
            });
        },

        onLeave(retval) {
            Stalker.unfollow(this.threadId);
            Stalker.flush();
            console.log(`[*] generate_sign 返回: ${retval}`);
        }
    });


###  6.4 反调试/反检测绕过

通过 Stalker 追踪反检测函数内部的执行路径，定位关键的条件判断分支，然后精确地修改寄存器值来绕过检测：


    // anti_detect_bypass.js — 追踪并绕过检测逻辑

    const checkFunc = Module.findExportByName("libtarget.so", "check_integrity");
    const targetModule = Process.getModuleByName("libtarget.so");

    Interceptor.attach(checkFunc, {
        onEnter() {
            console.log("[*] 检测函数被调用，启动 Stalker 分析执行路径...");

            Stalker.follow(this.threadId, {
                transform(iterator) {
                    let insn = iterator.next();
                    const inTarget = insn.address.compare(targetModule.base) >= 0 &&
                                     insn.address.compare(targetModule.base.add(targetModule.size)) < 0;

                    do {
                        if (inTarget) {
                            // 记录所有条件跳转，分析程序在检测时走了哪个分支
                            const branchInsns = ["b.eq", "b.ne", "b.gt", "b.lt",
                                               "cbz", "cbnz", "tbz", "tbnz"];

                            if (branchInsns.includes(insn.mnemonic)) {
                                const addr = insn.address;
                                const mnemonic = insn.mnemonic;

                                iterator.putCallout((context) => {
                                    // NZCV 是条件标志寄存器，决定了条件跳转的走向
                                    console.log(`[BRANCH] ${addr}: ${mnemonic} — NZCV=${context.nzcv}`);
                                });
                            }

                            // 精确绕过示例：如果发现 CMP 指令在比较 frida-server 的默认端口号
                            // 0x6a69 = 27241 = frida-server 默认端口
                            if (insn.mnemonic === "cmp" && insn.opStr.includes("#0x6a69")) {
                                console.log(`[!] 发现端口检测: ${insn.address}`);
                                // 通过 putCallout 修改寄存器，使比较结果不匹配
                                iterator.putCallout((context) => {
                                    // 将比较的寄存器值改为 0，使 CMP 结果改变
                                    context.x0 = ptr(0);
                                });
                            }
                        }

                        iterator.keep();
                    } while ((insn = iterator.next()) !== null);
                }
            });
        },

        onLeave(retval) {
            Stalker.unfollow(this.threadId);
            Stalker.flush();
            console.log(`[*] 检测函数返回值: ${retval}`);
        }
    });

###  6.6 Native 层逆向

SO 库中的关键函数定位和 JNI 调用链还原。对于动态注册（通过  ` RegisterNatives  ` ）的 JNI
函数，静态分析很难找到对应关系，Stalker 可以在运行时追踪完整的调用链：


    // native_trace.js — SO 库深度逆向分析

    const targetSo = Process.getModuleByName("libnative.so");

    // 排除其他所有模块以减少噪音（激进策略）
    Process.enumerateModules().forEach(mod => {
        if (mod.name !== "libnative.so") {
            Stalker.exclude(mod);
        }
    });

    // 预定义 JNI 函数表中常用函数的索引（用于识别 JNI 调用）
    const jniEnv = Java.vm.tryGetEnv();
    if (jniEnv) {
        const jniFunctions = {
            "FindClass": 6,
            "GetMethodID": 33,
            "CallObjectMethod": 34,
            "GetStringUTFChars": 169,
            "NewStringUTF": 167,
            "GetArrayLength": 171,
            "GetByteArrayElements": 184,
        };

        console.log("[*] 目标 SO 已加载，等待 JNI 调用...");
    }

    // 自动发现所有 JNI 导出函数并设置追踪
    const exports = targetSo.enumerateExports();
    for (const exp of exports) {
        // JNI 函数的命名规则：Java_ 开头，或 JNI_OnLoad
        if (exp.name.includes("Java_") || exp.name === "JNI_OnLoad") {
            console.log(`[*] 发现 JNI 函数: ${exp.name} @ ${exp.address}`);

            Interceptor.attach(exp.address, {

##  七、实战案例：逆向分析某 App 的签名算法

以下通过一个完整的实战案例，演示如何使用 Stalker 逐步逆向分析 App 的请求签名算法。整个过程严格遵循"先粗后细"的分析策略。


Stalker 实战分析六步工作流

###  7.1 目标描述

目标 App 在每个 HTTP 请求中携带一个  ` X-Sign  ` 头，值为 32 位十六进制字符串（看起来像 MD5）。通过初步分析（抓包 +
Java 层 Hook），签名在 Native 层生成，关键 SO 为  ` libsecurity.so  ` 。

###  7.2 Step 1：定位签名函数

第一步先从 Java 层入手，通过 Hook OkHttp 的请求构建器，找到签名添加的位置，再通过堆栈回溯定位到 Native 入口：


    // step1_locate.js — 通过 Java 层 Hook 定位 Native 入口

    Java.perform(() => {
        // Hook OkHttp 的 Request.Builder.addHeader 方法
        // 当 App 添加 X-Sign 头时会触发
        const RequestBuilder = Java.use("okhttp3.Request$Builder");

        RequestBuilder.addHeader.implementation = function(name, value) {
            if (name === "X-Sign") {
                console.log(`[*] X-Sign = ${value}`);
                // 打印 Java 层的完整调用堆栈
                // 通过堆栈可以看到是哪个类的哪个方法添加了这个 Header
                console.log(Java.use("android.util.Log").getStackTraceString(
                    Java.use("java.lang.Throwable").$new()
                ));
            }
            return this.addHeader(name, value);
        };
    });


通过堆栈回溯，确认签名由  ` com.example.SecurityUtil.generateSign  ` 调用，底层是 JNI 函数  `
Java_com_example_SecurityUtil_nativeSign  ` 。

###  7.3 Step 2：使用 Stalker 追踪 Native 签名函数

知道了 Native 函数入口后，用 Stalker 追踪其内部的完整执行过程：


    // step2_stalker_trace.js — 追踪 nativeSign 函数内部的所有调用和关键运算

            Stalker.exclude(mod);
        }
    });

        "Java_com_example_SecurityUtil_nativeSign");

    console.log(`[*] nativeSign @ ${nativeSign} (offset: 0x${nativeSign.sub(soBase).toString(16)})`);

    // 用于存储追踪记录
    const traceLog = [];

    Interceptor.attach(nativeSign, {
        onEnter(args) {

            const input = args[2];          // jstring input
            const timestamp = args[3].toInt32();  // jint timestamp

            console.log(`[*] nativeSign 被调用, timestamp=${timestamp}`);

            Stalker.follow(this.threadId, {
                events: { call: true, block: true },

                transform(iterator) {
                    let insn = iterator.next();
                    const blockAddr = insn.address;
                    // 只追踪 SO 内部的代码

                            // 记录所有函数调用（BL=直接调用, BLR=间接调用/寄存器调用）

                                const mn = insn.mnemonic;
                                const op = insn.opStr;

                                iterator.putCallout((context) => {
                                    let targetStr = "";
                                    if (mn === "blr") {
                                        // 间接调用：目标地址在寄存器中（如 blr x8）
                                        const regName = op;  // 如 "x8"
                                        const target = context[regName];
                                        const sym = DebugSymbol.fromAddress(target);
                                        targetStr = sym.name ? `${sym.moduleName}!${sym.name}` : `${target}`;
                                    } else {
                                        targetStr = op;  // 直接调用：操作数就是目标地址
                                    }

                                    traceLog.push({
                                        pc: addr.sub(soBase).toString(16),
                                        type: "call",
                                        target: targetStr,
                                        x0: context.x0.toString(),
                                        x1: context.x1.toString()
                                    });
                                });
                            }

                            // 记录关键算术运算（用于后续算法还原）
                            if (["eor", "ror", "lsl", "lsr"].includes(insn.mnemonic)) {
                                const addr = insn.address;
                                const mn = insn.mnemonic;

                                iterator.putCallout((context) => {
                                    traceLog.push({
                                        pc: addr.sub(soBase).toString(16),
                                        type: "alu",
                                        op: mn,
                                        x0: context.x0.toString(),
                                        x1: context.x1.toString()
                                    });
                                });
                            }
                        }

                        iterator.keep();
                    } while ((insn = iterator.next()) !== null);
                }
            });
        },

        onLeave(retval) {
            Stalker.unfollow(this.threadId);
            Stalker.flush();

            console.log(`[*] nativeSign 返回`);
            console.log(`[*] 共记录 ${traceLog.length} 条追踪记录`);

            // 发送到 Python 端进行分析
            send({ type: "trace", log: traceLog });
            traceLog.length = 0;  // 清空，为下一次调用做准备
        }
    });


###  7.4 Step 3：通过 transform 在关键指令处记录内存

经过 Step 2 的分析，假设发现 offset  ` 0x1234  ` 处有一个 XOR 循环。现在精确追踪这个循环的每次迭代，记录 XOR
操作的输入和输出：


    // step3_memory_trace.js — 在关键点记录内存内容

    const securitySo = Process.getModuleByName("libsecurity.so");
    const soBase = securitySo.base;
    // 根据 Step 2 分析结果确定的 XOR 循环地址范围
    const xorLoopAddr = soBase.add(0x1234);  // XOR 循环起始地址
    const xorLoopEnd = soBase.add(0x1280);   // XOR 循环结束地址

    const xorRecords = [];

    Interceptor.attach(Module.findExportByName("libsecurity.so",
        "Java_com_example_SecurityUtil_nativeSign"), {
        onEnter(args) {
            Stalker.follow(this.threadId, {
                transform(iterator) {
                    let insn = iterator.next();
                    const blockAddr = insn.address;

                    // 只对 XOR 循环所在的基本块进行精细追踪
                    // 这是 Step 2 "先粗" -> Step 3 "后细" 的体现
                    const inXorLoop = blockAddr.compare(xorLoopAddr) >= 0 &&
                                      blockAddr.compare(xorLoopEnd) < 0;

                    do {
                        if (inXorLoop && insn.mnemonic === "eor" &&
                            iterator.memoryAccess === 'open') {
                            const addr = insn.address;
                            iterator.putCallout((context) => {
                                // 记录 XOR 操作前的寄存器和栈状态
                                xorRecords.push({
                                    pc: addr.sub(soBase).toString(16),
                                    x0: context.x0.toString(16),
                                    x1: context.x1.toString(16),
                                    x2: context.x2.toString(16),
                                    // 读取栈上的局部变量（可能存储中间结果）
                                    stack_0: context.sp.readPointer().toString(16),
                                    stack_8: context.sp.add(8).readPointer().toString(16)
                                });
                            });
                        }
                        iterator.keep();
                    } while ((insn = iterator.next()) !== null);
                }
            });
        },

        onLeave(retval) {
            Stalker.unfollow(this.threadId);
            Stalker.flush();

            console.log(`[*] XOR 循环执行了 ${xorRecords.length} 次迭代`);
            for (const rec of xorRecords) {
                console.log(`  [${rec.pc}] x0=${rec.x0} ^ x1=${rec.x1} => x2=${rec.x2}`);
            }

            send({ type: "xor-trace", records: xorRecords });
            xorRecords.length = 0;
        }
    });


###  7.5 Step 4：分析 trace 数据，还原算法

Python 端收集 Stalker 产生的追踪数据，进行自动化分析：


    # step4_analyze.py — 分析 Stalker trace 数据

    import frida
    import json
    import sys

    trace_data = []
    xor_data = []

    def on_message(message, data):
        """处理 Frida 脚本发来的消息"""
        if message["type"] == "send":
            payload = message["payload"]
            if payload["type"] == "trace":
                trace_data.extend(payload["log"])
            elif payload["type"] == "xor-trace":
                xor_data.extend(payload["records"])

    def analyze_trace():
        """分析追踪数据，还原算法流程"""

        # 1. 分析函数调用链
        print("\n=== 调用链分析 ===")
        calls = [x for x in trace_data if x["type"] == "call"]
        for call in calls:
            print(f"  0x{call['pc']}: CALL {call['target']} (x0={call['x0']}, x1={call['x1']})")

        # 2. 统计算术/逻辑操作的频率分布
        print("\n=== ALU 操作分析 ===")
        alu_ops = [x for x in trace_data if x["type"] == "alu"]
        op_counts = {}
        for op in alu_ops:
            op_counts[op["op"]] = op_counts.get(op["op"], 0) + 1
        for op, count in sorted(op_counts.items(), key=lambda x: -x[1]):
            print(f"  {op}: {count} 次")

        # 3. 分析 XOR 循环，尝试提取密钥
        print("\n=== XOR 循环分析 ===")
        if xor_data:
            keys = set()
            for rec in xor_data:
                x0 = int(rec["x0"], 16) & 0xFF
                x1 = int(rec["x1"], 16) & 0xFF
                x2 = int(rec["x2"], 16) & 0xFF
                # 验证 XOR 关系：如果 x0 ^ x1 == x2，则 x1 是 key 字节
                if x0 ^ x1 == x2:
                    keys.add(x1)
                    print(f"  0x{rec['pc']}: {x0:#04x} ^ {x1:#04x} = {x2:#04x}")
            print(f"\n  提取到的 XOR key bytes: {[hex(k) for k in sorted(keys)]}")

    pid = device.spawn(["com.example.target"])
    session = device.attach(pid)

    with open("step2_stalker_trace.js") as f:
        script = session.create_script(f.read())

    script.on("message", on_message)

    device.resume(pid)

    input("按回车触发签名请求后分析...\n")
    analyze_trace()


###  7.6 Step 5：验证还原结果

根据 Stalker 追踪数据推断出的算法逻辑，用 Python 复现并验证：


    # step5_verify.py — 用 Python 复现签名算法并验证

    import hashlib
    import struct

    def generate_sign(data: bytes, timestamp: int, key: bytes) -> str:
        """
        还原的签名算法（基于 Stalker 追踪数据推断）：
        1. 输入数据与固定 key 进行循环 XOR
        2. 拼接 timestamp（小端序 4 字节）
        3. 计算 MD5 哈希
        """
        # Step 1: XOR with key（循环密钥异或）
        xored = bytearray(len(data))
        for i in range(len(data)):
            xored[i] = data[i] ^ key[i % len(key)]

        # Step 2: Append timestamp（小端序写入 4 字节整数）
        xored += struct.pack("<I", timestamp)

        # Step 3: MD5
        return hashlib.md5(bytes(xored)).hexdigest()

    # 使用 Frida 抓到的实际输入/输出数据验证
    test_data = b"test_request_body"
    test_timestamp = 1679900000
    # 从 Stalker XOR 追踪中提取的密钥字节
    test_key = bytes([0x5A, 0x3C, 0x7E, 0x1B, 0x9D, 0xF2, 0x48, 0xA6])

    result = generate_sign(test_data, test_timestamp, test_key)
    print(f"Python 计算结果: {result}")
    # 将结果与 Frida 抓到的 X-Sign 值对比
    # 如果完全一致，说明算法还原成功！


##  八、Stalker 局限与注意事项

###  8.1 性能开销

指令级追踪的性能开销不可避免，下表列出了不同场景下的大致减速倍数：

场景  |  大致减速倍数  |  适用建议
---|---|---
仅 block/compile 事件  |  2-5x  |  代码覆盖率收集
call + ret 事件  |  5-10x  |  调用链分析
transform + putCallout  |  10-50x  |  关键位置精确追踪
exec 事件（全指令）  |  50-100x+  |  极短时间内的精确分析

** 缓解策略  ** ：

  * 使用  ` Stalker.exclude()  ` 限制追踪范围——这是效果最显著的优化
  * 在 transform 中精确过滤，只在关键位置（如特定偏移的指令）插入 callout
  * 使用  ** CModule  ** 将性能关键的回调用 C 语言实现（见 9.2 节）
  * 控制追踪时长，不要长时间全量追踪——用 Interceptor 精确控制 follow/unfollow 的时机

###  8.2 自修改代码（SMC）

如果目标代码会在运行时修改自身（常见于加壳/VMP/代码混淆保护），需要调整信任阈值：


    // 处理自修改代码
    Stalker.trustThreshold = -1;  // 永不信任，每次执行基本块都重新编译
    // 注意：这会显著降低性能，仅在确认存在 SMC 时使用

    // 或者设为 0：立即信任（忽略自修改）
    // 适用于代码虽然有解密过程，但解密后不会再变化的场景
    Stalker.trustThreshold = 0;


> ** 何时怀疑存在 SMC？  ** 如果你发现 Stalker 追踪的指令与 IDA 中看到的反汇编不一致，或者 Stalker
> 在某些基本块崩溃，可能是代码在运行时被解密/修改了。

###  8.3 多线程竞态

  * ` Stalker.follow()  ` 跟踪的是  ** 特定线程  ** ，不会自动跟踪该线程创建的子线程
  * 多个线程同时被 follow 时，各自的 transform 回调是独立的，不会相互干扰
  * ` onReceive  ` /  ` onCallSummary  ` 回调是在 Frida 的  ** JS 线程  ** 中串行执行的，不存在并发问题

###  8.4 ARM Thumb/ARM 模式切换

在 32 位 ARM 上，代码可能在 ARM（32 位指令）和 Thumb（16 位指令）模式之间切换。确保传给 Stalker 的地址考虑了 Thumb
位——地址的最低位为 1 表示 Thumb 模式：


    // 在 32 位 ARM 上，注意 Thumb 位
    const addr = Module.findExportByName("libtarget.so", "target_func");
    // addr 可能包含 Thumb 位（最低位为 1），Stalker 会自动处理
    // 不需要手动清除 Thumb 位


> ** 为什么要关注这个？  ** 在 64 位 AArch64 上不存在 Thumb 模式，因此只有在分析 32 位 ARM
> 代码时需要注意。如果你的目标设备和 App 都是 64 位的，可以跳过此节。

###  8.5 Stalker 被检测的风险

Stalker 的动态重编译机制可能被高级的反分析保护检测到：

  * ** 代码完整性校验  ** ：App 可能计算代码段（.text section）的 hash。虽然 Stalker 不修改原始代码，但某些更激进的检测方式可能检查代码执行路径的完整性
  * ** 时间检测  ** ：Stalker 显著增加执行时间（至少 2-5x），可被精确的计时器（如  ` clock_gettime  ` ）检测到。某些 App 会在关键操作前后设置时间窗口
  * ** 内存特征  ** ：Stalker 分配的代码块有特定的内存布局特征（大量可执行的匿名 mmap 区域），可通过读取  ` /proc/self/maps  ` 来检测

###  9.1 Stalker + IDA Lighthouse（覆盖率可视化）

将 Stalker 收集的覆盖率数据导出为 drcov 格式，然后在 IDA 中通过 Lighthouse 插件加载，实现代码覆盖率的可视化着色：


    # ida_coverage.py — 生成 Lighthouse 兼容的 drcov 二进制文件

    import struct

    def write_drcov(module_name, module_base, module_size, module_path, blocks, output_path):
        """

        """
        with open(output_path, "wb") as f:

###  9.2 Stalker + CModule（高性能 C 回调）

当 JavaScript 回调的性能开销无法接受时，可以使用 Frida 的  ** CModule  ** 将 transform 和事件处理逻辑用 C
语言实现。C 回调直接在目标进程的上下文中执行，没有 JS ↔ Native 的跨边界调用开销：


    // cmodule_stalker.js — 使用 CModule 实现高性能 Stalker 回调

    const cm = new CModule(`
    #include <gum/gumstalker.h>
    #include <stdio.h>

    // 这些变量由 JS 端设置（通过 CModule 构造函数的第二个参数）
    extern guint64 module_start;
    extern guint64 module_end;
    extern guint64 call_count;

    // C 语言版本的 transform 回调
    // 直接操作 Stalker 的 C API，没有 JS 桥接开销
    void
    transform (GumStalkerIterator * iterator,
               GumStalkerOutput * output,
               gpointer user_data)
    {
      cs_insn * insn;

      while (gum_stalker_iterator_next (iterator, &insn))
      {
        // 只对目标模块内的代码进行插桩
        if (insn->address >= module_start && insn->address < module_end)
        {
          // 在 RET 指令前插入 C 回调
          if (insn->id == ARM64_INS_RET)
          {
            gum_stalker_iterator_put_callout (iterator,
                on_ret, NULL, NULL);
          }
        }

        gum_stalker_iterator_keep (iterator);
      }
    }

    // C 语言版本的回调函数 — 直接处理，无 JS 调用开销
    static void
    on_ret (GumCpuContext * cpu_context,
            gpointer user_data)
    {
      call_count++;
      // 在 C 层直接处理数据，避免频繁的 JS 回调
      if (cpu_context->x[0] != 0)
      {
        // 只记录非零返回值（减少输出量）
        printf ("RET x0=0x%llx pc=0x%llx\\n",
            (unsigned long long) cpu_context->x[0],
            (unsigned long long) cpu_context->pc);
      }
    }
    `, {
        // 将 JS 端的变量传递给 CModule
        module_start: ptr(targetModule.base),
        module_end: ptr(targetModule.base.add(targetModule.size)),
        call_count: Memory.alloc(8)  // 分配 8 字节作为计数器
    });

    // 使用 CModule 导出的 transform 函数
    Stalker.follow(threadId, {
        transform: cm.transform,
        data: ptr(0)
    });


> ** 性能对比  ** ：使用 CModule 的 transform 回调比纯 JavaScript 版本通常快  ** 5-10 倍  **
> ，因为省去了每次 putCallout 时从 Native 调用 JS 引擎的开销。在需要对大量指令进行插桩时，CModule 是必须的。

###  9.3 Stalker + Python 数据分析

对大规模 trace 数据进行自动化分析。当追踪数据量达到数万甚至数十万条记录时，人工查看不现实，需要自动化分析框架：


    # trace_analyzer.py — 自动化 trace 数据分析框架

    import json
    from collections import Counter, defaultdict

    class StalkerTraceAnalyzer:
        def __init__(self, trace_file):
            """从 JSONL 文件加载追踪数据"""
            self.events = []
            with open(trace_file) as f:
                for line in f:
                    self.events.append(json.loads(line))

        def call_frequency(self, top_n=20):
            """分析函数调用频率 — 找出最常被调用的函数（可能是循环体内的热点）"""
            targets = Counter()
            for ev in self.events:
                if ev.get("type") == "call":
                    targets[ev["target"]] += 1

            print(f"\n=== Top {top_n} 调用目标 ===")
            for target, count in targets.most_common(top_n):
                print(f"  {count:6d}x  {target}")

        def call_sequence(self):
            """提取函数调用的时间序列 — 还原执行的先后顺序"""
            calls = [ev for ev in self.events if ev.get("type") == "call"]
            print(f"\n=== 调用序列（共 {len(calls)} 次调用）===")
            for i, call in enumerate(calls[:50]):  # 只显示前 50 个
                print(f"  [{i:3d}] {call['target']}")

        def detect_loops(self):
            """检测循环模式 — 通过 PC 重复出现来识别循环"""
            pc_sequence = [ev["pc"] for ev in self.events if "pc" in ev]
            print("\n=== 热点地址（可能的循环）===")
            pc_counts = Counter(pc_sequence)
            for pc, count in pc_counts.most_common(10):
                if count > 5:  # 执行超过 5 次的地址很可能在循环内
                    print(f"  0x{pc}: {count} 次执行")

        def data_flow(self, register="x0"):
            """追踪特定寄存器值的变化 — 观察数据如何在算法中流动"""
            values = []
            for ev in self.events:
                if register in ev:
                    values.append((ev.get("pc", "?"), ev[register]))

            print(f"\n=== {register} 值变化追踪 ===")
            for pc, val in values[:30]:
                print(f"  [{pc}] {register} = {val}")

    # 使用示例
    analyzer = StalkerTraceAnalyzer("stalker_trace.jsonl")
    analyzer.call_frequency()    # 哪些函数被调用最多？
    analyzer.call_sequence()     # 调用的先后顺序是什么？
    analyzer.detect_loops()      # 有哪些循环？
    analyzer.data_flow("x0")    # x0 寄存器的值如何变化？


##  十、总结

###  Stalker 使用流程总结


    1. 确定目标 → 用 Interceptor/Java Hook 定位关键 Native 函数
             ↓
    2. 配置环境 → Stalker.exclude 排除无关模块
             ↓
    3. 初步追踪 → call/block 事件，了解整体调用结构
             ↓
    4. 深入分析 → transform + putCallout，在关键位置记录状态
             ↓
    5. 数据分析 → Python 端自动化分析 trace 数据
             ↓
    6. 算法还原 → 根据 trace 数据还原逻辑，编写验证代码


###  核心要点

  1. ** 先粗后细  ** ：从宏观的调用统计开始，逐步深入到指令级追踪。不要一上来就开  ` exec: true  `
  2. ** 精确限定范围  ** ：善用  ` Stalker.exclude()  ` 和地址过滤，避免信息过载和性能崩溃
  3. ** 合理选择事件  ** ：按需开启事件类型——  ` compile  ` 适合覆盖率，  ` call  ` 适合调用链，  ` exec  ` 仅作为最后手段
  4. ** transform 是核心  ** ：掌握  ` putCallout  ` 和指令过滤，可以实现几乎任何分析需求
  5. ** 注意性能  ** ：始终在精度和性能之间找到平衡，必要时使用 CModule 将热点逻辑用 C 实现
  6. ** 配合使用  ** ：Stalker 不是孤立使用的，与 Interceptor、Java Hook、IDA、Python 分析工具配合效果最佳
