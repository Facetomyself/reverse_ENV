##  一、签名机制：为什么必须重签

###  1.1 一个字节改动，为什么就要重新签名

拿到一个 APK，用 ` apktool d ` 解开，改一行字符串， ` apktool b ` 打回来，装机——装不上：


    adb install app_repack.apk
    Performing Streamed Install
    adb: failed to install app_repack.apk:
      Failure [INSTALL_PARSE_FAILED_NO_CERTIFICATES:
      Failed collecting certificates for /data/app/vmdl.../base.apk:
      Attempt to read a length of size 78 at offset 32]


因为你改坏了签名。APK 签名不是 "一个额外的文件挂在 APK 外面"，它是对 APK 内所有文件内容的密码学哈希 + RSA
签名。改了任何一个字节，哈希对不上，签名验证失败， ` PackageManager ` 拒绝安装。

Android 有三代签名方案叠在一起，新的兼容老的：

  * **v1（JAR Signature）** ：Android 起就有。原理是 Java 的 JAR 签名—— ` META-INF/MANIFEST.MF ` 里逐个文件记 SHA-256, ` META-INF/CERT.SF ` 是 MANIFEST 的签名摘要， ` META-INF/CERT.RSA ` 是 CERT.SF 用私钥 RSA 签名后的 PKCS#7 结构。只保护 APK 内单个文件的完整性，不保护整体 zip 结构——所以早年的 Janus 漏洞 CVE-2017-13156 能在 APK 头前面塞一个 dex,v1 校验不到。
  * **v2（APK Signature Scheme v2, Android 7.0+）** ：把整个 APK 文件当成字节流 hash，签名结果放在 zip 中央目录之前的 **APK Signing Block** —— 一个专门的 chunk 结构，chunk ID ` 0x7109871A ` 。这是当前主流校验用的。
  * **v3（Android 9.0+）** ：v2 的扩展，支持签名密钥轮换 ` SigningCertificateLineage ` —— 上一版本用密钥 A 签，下一版本换密钥 B，可以做无缝过渡。chunk ID ` 0xF05368C0 ` 。
  * **v4（Android 11.0+）** ：为增量安装设计 ` adb install --incremental ` ，签名放在独立的 ` .apk.idsig ` 文件里。日常逆向基本不用碰。

` apksigner sign --ks my.keystore ` 默认会同时打 v1 + v2 + v3，安装时任一版本能过就行。

关键点： v2/v3 的哈希覆盖整个 APK zip 结构 —— 你甚至不能修改 zip 里文件的顺序、不能改 zip 头里的时间戳。任何一个字节变了，v2
hash 就变，签名就废。

###  1.2 keystore 生成 + apksigner 用法

生成一个自签 keystore，一次的事：


    keytool -genkeypair -v \
      -keystore my.keystore \
      -alias mykey \
      -keyalg RSA -keysize 2048 \
      -validity 10000 \
      -storepass 123456 -keypass 123456 \
      -dname "CN=Test, OU=Test, O=Test, L=Test, ST=Test, C=CN"


给一个 APK 重签：


    # 用 build-tools 里的 apksigner —— 不是 jarsigner,jarsigner 只能签 v1
    export PATH="$ANDROID_HOME/build-tools/34.0.0:$PATH"

    apksigner sign \
      --ks my.keystore --ks-pass pass:123456 \
      --out app_signed.apk \
      app_repack.apk

    apksigner verify --print-certs app_signed.apk
    # Verifies
    # Verified using v1 scheme (JAR signing): true
    # Verified using v2 scheme (APK Signature Scheme v2): true
    # Verified using v3 scheme (APK Signature Scheme v3): true
    # Signer #1 certificate DN: CN=Test, OU=Test, O=Test, ...


` Verifies ` 就是 "三代签名都对"。这时 ` adb install ` 能过。

###  1.3 但是——安装能过 ≠ App 就认

装上去只是 ` PackageManager ` 认了 "这个 APK 有合法签名"。至于签名主体是谁 —— 是原开发者的 keystore 还是你的自签
—— ` PackageManager ` 只关心 "同一个包名不能升级到不同签名"，不关心 "这个签名是谁"。

App 内部的业务代码才是关键。它会在启动的某个时刻问系统：

> "我现在装的这个包，签名 hash 是什么？"

然后跟自己代码里硬编码或者服务端下发的 "预期签名 hash" 对比。你的自签 hash 不在预期列表里，业务代码触发拒绝分支 —— 闪退、Toast、静默
crash、业务功能不可用……各种拒绝方式。

这就是签名校验绕过的核心难题：不是让 ` apksigner verify ` 过关，而是让业务代码相信当前签名是原版。

###  1.4 五档校验光谱

按业务代码检查签名的复杂度，把常见做法分成五档，越往下越难打：

档次  |  实现方式  |  典型代码  |  常见于
---|---|---|---
第 1 档  |  Java 层直调 PackageManager  |  ` getPackageInfo(pkg, GET_SIGNATURES).signatures[0] ` |  导流工具、内容浏览、免登录 App —— 大厂裸奔产品也在这一档
第 2 档  |  反射 + 字符串常量加密  |  ` Class.forName(base64("android.content.pm...")) ` |  用了第三方混淆 SDK 但 **不做服务端签名 gate** 的普通商业 App
第 3 档  |  Native 层 JNI 校验  |  JNI ` env->GetObjectField ` 拿 signatures，或直接 ` pread ` 读 signing block  |  客户端把签名字节塞进业务请求、 **服务端做 gate** —— 电商、音乐、社交等有账号 / 付费闭环的 App
第 4 档  |  Java + Native 双链路交叉  |  Java 一遍、Native 一遍，两边比对  |  风控 / 合规要求高、本地 + 服务端双验 —— 大厂内容平台、直播打赏、部分金融理财类
第 5 档  |  加固壳内嵌校验 + 分层触发  |  加固 wrapper 里 ` libnesec ` / ` libpoison ` 分层 SGN check + 反调试 + 分批解密  |  核心资产敏感（版权 / 游戏经济 / 风控数据 / 合规数据）、有预算上完整加固 —— 覆盖网易易盾、360 加固、腾讯乐固、爱加密的客户群，跨行业都有

判断在哪一档，三步走完：


    # 1. dex 里搜签名读取,第 1、2 档在这里
    jadx -d out target.apk
    grep -rE "GET_SIGNATURES|GET_SIGNING_CERTIFICATES|getPackageInfo" out/sources/ | head

    # 2. native so 列表,第 3、4 档在这里
    unzip -l target.apk | grep -oE 'lib/arm64-v8a/lib[^.]+\.so' | sort -u
    # 关注：libsentry / libargus / libtoken / libsecure / lib*guard / lib*shield 这类命名

    # 3. 加固特征,第 5 档在这里
    unzip -l target.apk | grep -iE "libnesec|libpoison|libjiagu|libsecshell|libDexHelper|libtup|libtosprotection"
    # 出现 libnesec/libpoison = 网易加固；libjiagu = 360；libsecshell = 腾讯乐固
    # libtup = 通付盾；libtosprotection = 爱加密


###  1.5 五档在真实 App 里的印证


签名校验五档光谱

理论最容易变成空谈。同一台 Pixel 5（Android 14, Magisk root），把五个真实 App 分别走一次 "最简重打包 → 装机 →
看在哪一档" 的流程：

档次  |  真实样本  |  版本  |  换签后现象  |  加载器/校验特征
---|---|---|---|---
第 1 档  |  抖音极速版 ` com.ss.android.ugc.aweme.lite ` |  2024  |  直接可用，视频/直播/新人红包全部触发  |  无 native 校验 so
第 1 档  |  小宇宙播客 ` app.podcast.cosmos ` |  2.63.0  |  直接可用，推荐流全部加载  |  无 native 校验 so
第 3 档  |  QQ 音乐 ` com.tencent.qqmusic ` |  13.8.1  |  首页显示 "糟糕发生错误了"，其它页面正常  |  Java 层读签名字节参与请求，服务端拒服务
第 4 档  |  小红书 ` com.xingin.xhs ` |  8.69.0  |  启动 3 秒内 ` Fatal signal 6 SIGABRT ` 自杀  |  ` libsentry.so ` bytehook 全 signal + native 直读 signing block
第 5 档  |  网易云音乐 ` com.netease.cloudmusic ` |  9.2.80  |  500ms 内自杀，splash 都不出  |  Nesec 加固 + Tinker + 多层联防

两条比档次数字更实用的分界线：

  * **是否主动自杀** ：第 1、3 档不自杀，要么直接可用要么服务端拒；第 4、5 档 ` SIGABRT ` / ` System.exit ` 让 App 死掉 —— 后者留给你调试的时间窗口极短，反馈周期从秒级恶化到每次崩溃从头再来。
  * **签名读取的物理层** ：全在 Java 层第 1、2 档的， ` Signature ` 对象替换就够；下沉到 native 第 3 档以上，Java hook 从原理上就绕不过 —— 比如小红书的 ` libsentry.so ` 直接 ` pread ` 读 APK signing block chunk ` 0xF05368C0 ` ,Java 层任何 hook 都触及不到那条路径。

厂商规模不是决定因素。字节的极速版几乎裸奔、腾讯的 QQ 音乐比小红书弱一档、独立开发者的小宇宙完全不设防 —— **业务模式（签名是否参与商业闭环）**
才是决定档次的因素。

后面第二节讲第 1、2 档的 Java 层打法，第三节讲第 3、4、5 档的 native 层，四、五节是 QQ 音乐第 3 档和网易云音乐第 5
档的完整实战。

##  二、Java 层

Java 层是签名校验的第一站。所有的读签名操作最终都要经过 Android Framework 的 ` PackageManager ` API ——
无论业务代码用直调、反射、加密调用哪种姿势，最终都会 dispatch 到同一批系统方法。这一节把这批方法讲清楚，再讲怎么 hook。

###  2.1 Android 里读签名的完整 API 集

四条主要路径：


    // 路径 1：传统 API,Android <= 8
    PackageInfo pi = pm.getPackageInfo(pkg, PackageManager.GET_SIGNATURES);   // flag = 0x40
    Signature[] sigs = pi.signatures;
    byte[] bytes = sigs[0].toByteArray();      // → X.509 DER 编码,几百字节
    int hash = sigs[0].hashCode();             // = Arrays.hashCode(mSignature)
    String hex = sigs[0].toCharsString();      // → 大写 hex 字符串

    // 路径 2：Android 9+ 引入的 SigningInfo, 推荐用法
    PackageInfo pi = pm.getPackageInfo(pkg, PackageManager.GET_SIGNING_CERTIFICATES);  // flag = 0x08000000
    SigningInfo si = pi.signingInfo;
    Signature[] cur = si.getApkContentsSigners();          // 当前签名
    Signature[] hist = si.getSigningCertificateHistory();  // 历史签名, v3 密钥轮换
    boolean hasM = si.hasMultipleSigners();

    // 路径 3：直接反射读 Signature 内部字段
    Field f = Signature.class.getDeclaredField("mSignature");
    f.setAccessible(true);
    byte[] raw = (byte[]) f.get(sigs[0]);      // 底层字节数组

    // 路径 4：算 hash 再对比, 这一步在业务代码里, 不是系统 API
    MessageDigest md = MessageDigest.getInstance("SHA-256");
    byte[] h = md.digest(bytes);
    String hexHash = bytesToHex(h);            // 跟白名单里的 "abc..." 对比


` Signature ` 对象的内部结构：


    public class Signature {
        private final byte[] mSignature;      // 关键字段, 就是这个 byte[]
        private volatile int mHashCode;       // hashCode 缓存
        private volatile String mStringRef;   // toCharsString 缓存
        // ...
    }


` mSignature ` 是私有 final 字段，构造时确定，之后不可修改。 ` toByteArray() ` 返回它的 ` .clone() `
； ` hashCode() ` 是 ` Arrays.hashCode(mSignature) ` ； ` toCharsString() ` 是把 `
mSignature ` 转 hex。所有读法归根到底是读 ` mSignature ` 这个字段。

###  2.2 业务代码里的四种常见校验姿势

**姿势 A：直接算 hash 比对**


    public class SignatureCheck {
        private static final String[] WHITELIST = {
            "d6e5dbbfa87f3d0e88bfe6ad1a5f9c22"    // MD5 of orig cert.der
        };

        public static boolean verify(Context ctx) throws Exception {
            PackageInfo pi = ctx.getPackageManager().getPackageInfo(
                ctx.getPackageName(), PackageManager.GET_SIGNATURES);
            byte[] bytes = pi.signatures[0].toByteArray();
            String md5 = md5Hex(bytes);
            for (String w : WHITELIST) if (w.equalsIgnoreCase(md5)) return true;
            return false;
        }
    }


` jadx ` 反编能看到白名单常量、比对逻辑、 ` GET_SIGNATURES ` 标志 —— 一目了然。第 1 档就是这个样子。

**姿势 B：反射 + 常量加密**


    // 类名、方法名都 base64
    String pmCls = decrypt("YW5kcm9pZC5jb250ZW50LnBtLlBhY2thZ2VNYW5hZ2Vy");   // android.content.pm.PackageManager
    String getPi = decrypt("Z2V0UGFja2FnZUluZm8=");                            // getPackageInfo
    Method m = Class.forName(pmCls).getMethod(getPi, String.class, int.class);
    Object pi = m.invoke(pm, ctx.getPackageName(), 0x40);
    // ... 白名单 MD5 也不明文
    byte[] expected = decrypt("d6e5dbbfa87f3d0e88bfe6ad1a5f9c22");


多几层混淆，本质没变。 **反射调用的最终目标仍然是` PackageManager.getPackageInfo ` ** ，只要 hook
那个系统方法，反射调用还是会被 dispatch 过来。第 2 档就是这个样子。

**姿势 C：把签名字节塞进请求**


    public class RequestBuilder {
        public Request build(Context ctx, String api) {
            Request.Builder b = new Request.Builder().url(api);
            try {
                PackageInfo pi = ctx.getPackageManager().getPackageInfo(
                    ctx.getPackageName(),
                    PackageManager.GET_SIGNING_CERTIFICATES);
                byte[] certBytes = pi.signingInfo.getApkContentsSigners()[0].toByteArray();
                String sigHash = sha256Hex(certBytes);
                b.addHeader("X-App-Signature", sigHash);   // 服务端认这个 header
            } catch (Exception e) {}
            return b.build();
        }
    }


客户端本地不做校验，只是把签名 hash 作为请求参数发给服务端，服务端在业务 API 层拒服务。这就是 QQ 音乐的档次，第 3 档 —— App
能正常启动，只是拉不到数据。

**姿势 D：Java + Native 交叉**


    public class HybridCheck {
        static { System.loadLibrary("myjni"); }
        private static native byte[] getExpectedCert();  // native 返回预期 cert 字节

        public static boolean verify(Context ctx) throws Exception {
            // Java 层读一次
            byte[] javaBytes = ctx.getPackageManager()
                .getPackageInfo(ctx.getPackageName(), 0x40)
                .signatures[0].toByteArray();
            // Native 层给出预期
            byte[] expected = getExpectedCert();
            // Java 层对比
            if (!Arrays.equals(javaBytes, expected)) return false;

            // Native 层再读一次, native 内部 pread 直读 APK signing block
            return nativeVerify();
        }
        private static native boolean nativeVerify();
    }


Java hook ` PackageManager ` 只覆盖第一次读取， ` nativeVerify ` 内部再读一次绕过 Java 层 ——
这就是第 4 档，Java hook 拦不住。

###  2.3 Frida 三种打法

适用范围先说清楚：下面的三种打法只对第 1、2 档 Java 层校验、没有 native 分支的完全有效。对第 3 档以上，Java hook 覆盖不到
native 那条路径 —— 绝不是 "一段脚本解决所有 App 签名"。

三种打法通常叠着用，不是 "选一个"。

**打法 A：` Signature.toByteArray() ` 返回原版字节 **


    Java.perform(function () {
        var Signature = Java.use('android.content.pm.Signature');
        var ORIG_BYTES = /* 从原版 APK 提取的 cert.der 字节, 见 第 2.4 节 */;
        var ORIG_JAVA = Java.array('byte',
            ORIG_BYTES.map(function (b) { return b > 127 ? b - 256 : b; }));

        Signature.toByteArray.implementation = function () {
            return ORIG_JAVA;
        };
    });


这一层单独用往往不够，QQ 音乐案例证明， 详见第四节：

  * App 在启动早期读一次 ` toByteArray() ` ，把字节缓存到某字段 —— 下次业务用缓存，你后来 attach 的 hook 完全不 fire。
  * App 直接反射读 ` Signature.mSignature ` 私有字段 —— ` toByteArray ` hook 拦不到。
  * App 只算 ` Signature.hashCode() ` —— ` hashCode() ` 内部是 ` Arrays.hashCode(mSignature) ` ，你 hook 了 ` toByteArray ` 但 ` mSignature ` 字段本身还是自签的字节。

所以打法 A 是入口叶子，不是保护面。要覆盖全，接着做打法 B。

**打法 B：` PackageManager.getPackageInfo ` 整体替换 ` Signature ` 对象 **

不是 "替换 signatures 数组元素的某个字段"，是 **把整个` Signature ` 对象替换成用原版字节 ` $new ` 出来的新对象
** —— 这样后续任何读取方式， ` toByteArray ` / ` hashCode ` / 反射 ` mSignature ` / `
toCharsString ` ，拿到的都是一致的原版。


    Java.perform(function () {
        var Signature = Java.use('android.content.pm.Signature');
        var ORIG_JAVA = Java.array('byte',
            ORIG_BYTES.map(function (b) { return b > 127 ? b - 256 : b; }));

        var PM = Java.use('android.app.ApplicationPackageManager');
        PM.getPackageInfo.overloads.forEach(function (ov) {
            ov.implementation = function () {
                var pi = ov.apply(this, arguments);
                if (pi !== null) {
                    var newSig = Signature.$new(ORIG_JAVA);     // 用原版字节 new
                    var arr = Java.array('android.content.pm.Signature', [newSig]);
                    try { pi.signatures.value = arr; } catch (e) {}
                    // Android 9+ signingInfo, GET_SIGNING_CERTIFICATES 走这条
                    try {
                        var si = pi.signingInfo.value;
                        if (si !== null) {
                            var fDet = si.getClass().getDeclaredField('mSigningDetails');
                            fDet.setAccessible(true);
                            var det = fDet.get(si);
                            if (det !== null) {
                                var fSig = det.getClass().getDeclaredField('mSignatures');
                                fSig.setAccessible(true);
                                fSig.set(det, arr);
                            }
                        }
                    } catch (e) {}
                }
                return pi;
            };
        });

        // SigningInfo 直接调用也拦一次
        try {
            var SI = Java.use('android.content.pm.SigningInfo');
            SI.getApkContentsSigners.implementation = function () {
                return Java.array('android.content.pm.Signature', [Signature.$new(ORIG_JAVA)]);
            };
            SI.getSigningCertificateHistory.implementation = function () {
                return Java.array('android.content.pm.Signature', [Signature.$new(ORIG_JAVA)]);
            };
        } catch (e) {}
    });


易错点：早年很多教程只写 ` pi.signatures.value = [origSig] ` 就完事

  * 只覆盖了 ` GET_SIGNATURES ` （ ` 0x40 ` ）走的旧路径 —— Android 9+ 主流用 ` GET_SIGNING_CERTIFICATES ` （ ` 0x08000000 ` ），走 ` pi.signingInfo ` ，那才是要覆盖的字段。不覆盖 ` signingInfo ` 会让新版 App 上 "hook 装了但不生效"。
  * 反射 hook 的时候要用 ` pi.signatures.value = arr ` —— ` .value ` 是 Frida 里访问 Java 字段的写法，不能直接 ` pi.signatures = arr ` （那会 shadow 掉 pi 上的属性，不修改 Java 对象）。

**打法 C：短路` verifySignature ` 类的入口方法 **

如果 App 里签名校验有明确的 Java 入口，比如 ` com.example.app.SignatureVerifier.verify ` ，直接让它
return ` true ` ：


    Java.perform(function () {
        var Verifier = Java.use('com.example.app.SignatureVerifier');
        Verifier.verify.implementation = function () {
            return true;
        };
    });


适用面窄：

  * 需要先 ` jadx ` 反编找到入口 —— 加固/混淆后类名 ` a ` / ` b ` / ` c ` 一片，费时间。
  * 只覆盖走这个 Java 入口的校验点 —— 同 App 内其它签名读取， Bugly 崩溃上报、埋点 SDK 不走这条，可能造成 "业务通了但风控发现异常" 的状况。
  * 如果校验入口是 native 方法 ` native boolean verify() ` ，这一层根本 hook 不到。

**打法组合速查** ：

场景  |  打法
---|---
只想验证 "有没有校验"  |  只用 A
生产可用， 第 1、3 档 Java 校验  |  A + B
A + B 装了但仍报错， 且能反编入口  |  A + B + C
A + B 装了完全不生效  |  **99% 是 native 层校验** ——立刻换 第 3 章 的 native 打法

上面所有代码都不是 "通杀"。任何一段脚本都有对应的失效场景。上手时先花 15 分钟走 第 1.4 节 的三步判断，再选打法，比先写 hook 再看能不能
work 效率高得多。

###  2.4 从原版 APK 提取 cert 字节

` Signature.toByteArray() ` 返回的是 X.509 证书的 DER 编码字节。从原版 APK 提取：

    openssl x509 -in cert.pem -outform DER -out cert.der

    wc -c cert.der    # 记住这个字节数
    md5 cert.der      # 记住这个 MD5, 就是白名单可能查的那个


得到 ` cert.der ` 是几百字节的二进制。把它 base64 后嵌进 Frida 脚本：


    base64 -i cert.der | tr -d '\n'
    # 直接复制到脚本里的 CERT_B64 常量


Frida 脚本里解码：


    function b64d(s) {
        var C = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/';
        var o = [], b = 0, n = 0;
        for (var i = 0; i < s.length; i++) {
            var c = s.charAt(i);
            if (c === '=') break;
            var v = C.indexOf(c);
            if (v < 0) continue;
            b = (b << 6) | v; n += 6;
            if (n >= 8) { n -= 8; o.push((b >> n) & 0xff); }
        }
        return o;
    }
    var ORIG_BYTES = b64d(CERT_B64);


原版 cert 字节数常见范围 500-900 字节。QQ 音乐是 587 字节的 Tencent 自签根证书，小红书是 593 字节的 xingin
自签，网易云音乐 9.2.80 是 915 字节。

##  三、Native 层

Java 层 hook 打不通的时候，签名校验多半藏在 native 里。这一节讲 native 层的签名读取实现方式、怎么识别、Frida 层怎么
hook。

###  3.1 Native 读签名的两条路径

**路径 1：JNI 调 Java API**

Native 代码通过 JNI 反射调用 Java 层的 ` PackageManager.getPackageInfo ` ：


    // libcheck.so 里的代码
    JNIEXPORT jboolean JNICALL Java_com_example_Check_verify(JNIEnv* env, jclass clz) {
        jclass ctxCls = (*env)->FindClass(env, "android/content/Context");
        jmethodID getPMMid = (*env)->GetMethodID(env, ctxCls,
            "getPackageManager", "()Landroid/content/pm/PackageManager;");
        jobject pm = (*env)->CallObjectMethod(env, ctx, getPMMid);
        // ... 一路 JNI 反射拿 getPackageInfo → signatures[0] → toByteArray
        // 拿到 byte[] 后在 native 里算 hash 对比
        return match;
    }


这条路径最终还是走 Java 的 ` PackageManager ` —— 所以 第 2.3 节 的 Java hook 依然生效。native
只是拿到结果后在 native 里对比，避开了 Java 层的对比代码，但没避开 Java 层的读取代码。

**路径 2：Native 直接 pread APK 文件**

native 代码不走 JNI，直接 ` open ` / ` pread ` 打开 APK 文件，解析 ZIP 结构，找到 APK Signing
Block，从里面读出证书 chunk：


    int fd = open(apkPath, O_RDONLY);

    // 定位 zip 结构：EOCD → Central Directory → APK Signing Block
    off_t sigBlockOffset = findSigningBlock(fd);

    // APK Signing Block 结构：
    //   uint64 size_of_block
    //   { uint64 pair_size; uint32 chunk_id; bytes value; }*
    //   uint64 size_of_block (repeat)
    //   bytes[16] magic "APK Sig Block 42"
    uint32_t chunkId = APK_SIGV2_ID;  // 0x7109871A v2 或 0xF05368C0 v3
    uint8_t* certBytes = readSigningChunk(fd, sigBlockOffset, chunkId);

    // 从 chunk 里解析 X.509 cert, 算 hash
    uint8_t hash[32];
    sha256(certBytes, certLen, hash);

    if (memcmp(hash, expectedHash, 32) != 0) abort();


这条路径完全绕过 Java 层 —— Java 的 ` PackageManager ` 一次都没被调用。第 2.3 节 的所有 Java hook
打不到。

小红书的 ` libsentry.so ` 就是这种：logcat 里能看到 ` getByteBufferValueById , destApk
/data/app/xhs.apk IdValueMap = {1896449818=..., -262969152=..., ...} ` —— `
-262969152 = 0xF05368C0 ` 是 APK Signature Scheme v3 的 chunk ID，直接从 signing
block 里读的。

###  3.2 识别 native 签名校验的特征

**特征 A：加固/保护 so 的命名**


    unzip -l target.apk | grep -oE 'lib/arm64-v8a/lib[^.]+\.so' | sort -u | \
      grep -iE "sentry|nesec|poison|jiagu|secshell|jgmi|tosprotection|guard|shield|token|secure"


这类命名基本都是保护 so。看到就要警惕 native 校验。

**特征 B：logcat 里 signal 相关的日志**

App 启动时看 logcat：


    sentry_hook_tag: hooked. sym_name: sigaction
    sentry_hook_tag: [register_signal]--replace sig = 6, path = libsentry.so
    sentry_hook_tag: [register_signal]--replace sig = 11, path = libsentry.so


` sig = 6 ` 是 SIGABRT, ` sig = 11 ` 是 SIGSEGV —— 保护 so
接管了所有信号，检测到问题后走自己的处理路径而不是让 App 正常崩溃到系统层。

网易云音乐的 ` libpoison.so ` 也是类似模式，只是它更狠 —— 直接 ` Process.setExceptionHandler `
拦所有未处理异常做二次校验。

**特征 C：APK 文件被 App 主动打开**

用 Frida hook ` open ` ：


    var open_ptr = Module.findExportByName(null, 'open');
    Interceptor.attach(open_ptr, {
        onEnter: function (args) {
            try {
                var path = args[0].readCString();
                if (path && path.indexOf('/base.apk') >= 0) {
                    var lr = this.returnAddress;
                    var mod = Process.findModuleByAddress(lr);
                    send('open("' + path + '") from ' + (mod ? mod.name : '?'));
                }
            } catch (e) {}
        }
    });


看谁在打开 APK 文件 —— 如果 caller 是保护 so ` libsentry ` / ` libnesec ` / ` libpoison `
，大概率就是它在读 signing block。如果 caller 是 ` libart ` 或 ` libc ` ，可能只是 ART 加载 dex 或
ClassLoader 索引，不是签名校验。

**特征 D：App 秒退， logcat 里只有` Fatal signal 6/11 ` , 没有 Java 异常栈 **


    Process com.example.app (pid ...) has died: fg TOP
    libc: Fatal signal 6 (SIGABRT), code -1 (SI_QUEUE) in tid ...


Java 层没抛过异常就走了 native ` abort() ` —— 十有八九是 native 层校验失败。第 4 档以上的典型症状。

###  3.3 native 层 hook 三种思路

**思路 1：拦 native 层调用的 libc 出口**

如果 native 校验失败后调 ` abort ` / ` exit ` / ` _exit ` / ` _Exit ` / ` exit_group `
/ ` raise ` 之类，拦掉：


    ['abort', 'exit', '_exit', 'exit_group', '_Exit'].forEach(function (name) {
        var p = Module.findExportByName('libc.so', name);
        if (!p) return;
        Interceptor.attach(p, {
            onEnter: function (args) {
                var lr = this.returnAddress;
                var mod = Process.findModuleByAddress(lr);
                send(' libc.' + name + '(' + args[0] + ') from ' + (mod ? mod.name : '?'));
                this.skip = true;   // 阻止调用
            },
            onLeave: function (retval) {
                if (this.skip) try { retval.replace(ptr(0)); } catch (e) {}
            }
        });
    });


局限： 如果保护 so 用 ` syscall(SYS_exit_group, 0) ` 或 ` raise(SIGABRT) ` 触发 signal
陷阱，libc export 层拦不到。这时要 hook 到 ` libc ` 内部私有函数 ` __exit ` / ` __exit_group ` ：


    Process.findModuleByName('libc.so').enumerateSymbols().forEach(function (s) {
        if (s.name === '__exit' || s.name === '__exit_group') {
            Interceptor.replace(s.address, new NativeCallback(function (a) {
                send(' SKIPPED ' + s.name + '(' + a + ')');
            }, 'void', ['int']));
        }
    });


**思路 2：拦 Java 层` System.exit ` / ` Runtime.exit ` **

保护 so 有时也走 Java 出口：


    Java.perform(function () {
        Java.use('java.lang.System').exit.implementation = function (c) {
            send('[skip] System.exit(' + c + ')');
        };
        Java.use('java.lang.Runtime').exit.implementation = function (c) {
            send('[skip] Runtime.exit(' + c + ')');
        };
        Java.use('android.os.Process').killProcess.implementation = function (p) {
            send('[skip] killProcess(' + p + ')');
        };
    });


Java 层拦 + native 层拦，两边都装，覆盖面才全。

**思路 3：byte patch 关键分支跳转**

最精确的方式：反汇编保护 so，找到 "签名对比后的分支跳转" 那条指令，改成无条件跳到 "通过" 分支。

以网易云音乐 ` libpoison.so + 0x7af3c ` 为例，反汇编看到：


    0x7af3c:  2A 03 1F 2A   mov  w0, wzr         ; w0 = 0（校验失败返回 0）
    0x7af40:  ...
    ...
    0x7af04:  ...                                ; 校验通过入口


改成无条件跳到 ` 0x7af04 ` ：


    var m = Process.findModuleByName('libpoison.so');
    Memory.patchCode(m.base.add(0x7af3c), 4, function (w) {
        // ARM64 b (无条件跳转) 指令编码：0x14000000 | (imm26 & 0x3FFFFFF)
        // 相对偏移 = (0x7af04 - 0x7af3c) / 4 = -14 = 0xFFFFFFF2
        // 0x14000000 | 0x3FFFFF2 = 0x17FFFFF2
        w.writeU32(0x17FFFFF2);
    });


这种 byte patch 是最稳定的方式 —— 没有函数调用开销、不留 hook 痕迹、检测不到， ` Memory.patchCode `
修改是就地字节替换。缺点是每个 App 每个版本要单独找 offset，反汇编成本高。

**思路 4：hook` bytehook ` / ` xhook ` 的注册接口 **

保护 so 用 bytehook 接管 signal 时，我们可以在它注册之前抢先接管。但需要在 spawn 阶段就装 hook —— attach 太晚。

具体做法：hook ` dlopen ` ，检测到目标 so 加载时立即 byte patch 它的初始化代码。

###  3.4 Native hook 的一个大坑：SIGSEGV handler

很多 Frida 教程写 "拦 SIGSEGV 保护 App 不崩"：


    // 坏例子——不要这么写
    Process.setExceptionHandler(function (details) {
        send(' exception ' + details.type + ' @ ' + details.context.pc);
        return true;   // ← 拦下所有 exception, 让 App 继续
    });


这条代码是很多失败案例的根源， 详见第五节 第 5.4 节。

为什么错：Android ART 虚拟机的隐式 null check 是通过 SIGSEGV 实现的 —— ART JIT 出来的代码在很多位置省略了显式
null 检查，遇到 null 引用时靠 CPU 触发 SIGSEGV,ART 的 signal handler 收到后判断这是 "隐式 null
check 触发的"，转成 Java 层的 ` NullPointerException ` 抛出去。

你把 SIGSEGV handler 拦掉，ART 就收不到 signal，隐式 null check 全部变成未处理的 SIGSEGV —— App
立即崩溃。表现是 "装了这个 hook 反而立刻挂"。

正确做法： 如果一定要拦 SIGSEGV，只拦特定 fault address 或特定线程，其他一律放行：


    Process.setExceptionHandler(function (details) {
        // 只拦 libpoison.so 里的地址触发的异常
        var mod = Process.findModuleByAddress(details.context.pc);
        if (mod && mod.name === 'libpoison.so') {
            send(' swallow ' + details.type + ' in libpoison');
            return true;
        }
        return false;   // ← 其它放行给 ART/libc 原生 handler
    });


更好的做法：能不用 ` setExceptionHandler ` 就不用。byte patch 目标 so 里的具体分支，比拦全局 signal
稳定得多。

###  3.5 一个完整的 native hook 案例：libpoison 精简版

以网易云音乐的 ` libpoison.so ` 为例，完整的 native 层 hook 拆四块看。

**Patch 1: libpoison 里签名校验分支跳转的 byte patch**


    'use strict';

    var poisonPatched = false;
    function tryPatchPoison() {
        if (poisonPatched) return true;
        var m = Process.findModuleByName('libpoison.so');
        if (!m) return false;
        Memory.patchCode(m.base.add(0x7af3c), 4, function (w) {
            w.writeU32(0x17FFFFF2);   // b 0x7af04
        });
        poisonPatched = true;
        send(' libpoison 0x7af3c patched, base=' + m.base);
        return true;
    }
    tryPatchPoison();


先尝试同步 patch。libpoison 是延迟 dlopen 的，启动时 ` findModuleByName ` 大概率返回 null，补一层
dlopen 监听兜底：


    if (!poisonPatched) {
        var listener = Interceptor.attach(
            Module.findExportByName(null, 'android_dlopen_ext'),
            {
                onEnter: function (args) { try { this.name = args[0].readCString(); } catch (e) {} },
                onLeave: function () {
                    if (this.name && this.name.indexOf('libpoison') >= 0) {
                        setTimeout(function () {
                            if (tryPatchPoison() && listener) {
                                listener.detach();   // 装完就摘掉, 减少 hook 面
                            }
                        }, 0);
                    }
                }
            });
    }


**Patch 2: libc` HandleUsingDestroyedMutex ` 覆盖返回 0 **

加固层故意 destroy mutex 触发 bionic libc 的自杀陷阱，把这个函数替换成"返回 0":


    Process.findModuleByName('libc.so').enumerateSymbols().forEach(function (s) {
        if (s.name && s.name.indexOf('HandleUsingDestroyedMutex') >= 0) {
            Memory.patchCode(s.address, 8, function (w) {
                w.writeU32(0x2a1f03e0);   // mov w0, wzr
                w.writeU32(0xd65f03c0);   // ret
            });
            send(' HandleUsingDestroyedMutex patched');
        }
    });


**Patch 3: libc exit 系列 —— 拦掉加固失败后的 abort/exit**


    ['abort', 'exit', '_exit', 'exit_group', '_Exit'].forEach(function (name) {
        var p = Module.findExportByName('libc.so', name);
        if (!p) return;
        Interceptor.attach(p, {
            onEnter: function (args) {
                var lr = this.returnAddress;
                var mod = Process.findModuleByAddress(lr);
                send(' libc.' + name + ' from ' + (mod ? mod.name : '?'));
                this.skip = true;
            },
            onLeave: function (retval) { if (this.skip) retval.replace(ptr(0)); }
        });
    });


Java 层同步拦 exit 兜底，有些加固走 Java 出口：


    Java.perform(function () {
        Java.use('java.lang.System').exit.implementation = function (c) { send('[skip] System.exit(' + c + ')'); };
        Java.use('java.lang.Runtime').exit.implementation = function (c) { send('[skip] Runtime.exit(' + c + ')'); };
        Java.use('android.os.Process').killProcess.implementation = function (p) { send('[skip] killProcess(' + p + ')'); };
    });


这个脚本是网易云音乐实战里的最终精简版本，一共三层 patch 加上 Java 层 exit 拦截。作者 3 天写了 45 层 patch 才让
splash 出来；我们精简到这个版本让 Application init 全部走完 —— 原因是去掉了那个错误的全局 SIGSEGV
handler。详见第五节。

只能有一个解释： ` Signature.toByteArray ` 并不是签名进入请求的唯一路径。App 内部还有别的地方读签名，且这些地方没被这个
hook 覆盖。

###  4.8 补全 hook： ` PackageInfo.signatures[] ` \+ ` SigningInfo `

Android 里读签名的完整路径见 第 2.1 节，一共四条。只 hook ` toByteArray ` 覆盖不到：

  * ` pi.signatures[0] ` 直接引用 —— ` Signature ` 对象本身还是自签的。
  * ` SigningInfo.getApkContentsSigners() ` —— Android 9+ 主流用法，绕过 ` Signature.toByteArray ` 。
  * ` Signature.hashCode() ` —— 内部是 ` Arrays.hashCode(mSignature) ` ，字段本身没变。
  * 反射读 ` mSignature ` —— 直接绕过所有方法。

正确做法： **替换` Signature ` 对象本身 ** ，而不是 "读方法"。这样后续无论 App 怎么读它， ` toByteArray ` /
` hashCode ` / ` toCharsString ` / 反射 ` mSignature ` , 拿到的都是一致的原版。

最终 hook 脚本拆四块看，一块解决一层签名读取路径。

**Block 1: 全局设置 + native open 观察**


    /*
     * qq_sig_full.js —— QQ 音乐重打包签名 hook 完整方案
     */

    var L = function (s) { send(String(s)); };

    var CERT_B64 = 'MIICRzCCAbCgAwIBAgIETMVEnTANBgkqhkiG9w0BAQUFADBnMQswCQYDVQQGEwI4...';
    function b64d(s) { /* 见 第 2.4 节 */ }
    var ORIG_BYTES = b64d(CERT_B64);
    L('orig cert ' + ORIG_BYTES.length + ' bytes');

    // hook native open() 看有没有 native 层读 APK 文件, 只是观察, 不改行为
    try {
        var open_ptr = Module.findExportByName(null, 'open');
        Interceptor.attach(open_ptr, {
            onEnter: function (args) {
                var path = args[0].readCString();
                if (path && (path.indexOf('/base.apk') >= 0
                          || path.indexOf('.RSA') >= 0
                          || path.indexOf('CERT') >= 0)) {
                    L('[native] open("' + path + '")');
                }
            }
        });
    } catch (e) {}


` open ` 拦下来只 send 日志、不改， 是为了后续验证 "native 层有没有绕开 Java 读 APK"。

**Block 2:` Signature.toByteArray() ` 返回原版字节 **


    Java.perform(function () {
        var Signature = Java.use('android.content.pm.Signature');
        var ORIG_JAVA = Java.array('byte',
            ORIG_BYTES.map(function (b) { return b > 127 ? b - 256 : b; }));

        Signature.toByteArray.implementation = function () {
            L('[J] toByteArray → orig');
            return ORIG_JAVA;
        };
        L(' Signature.toByteArray hooked');


第一层叶子， 覆盖直接读字节的入口。QQ 音乐启动期读一次然后缓存， 所以必须 spawn 拦。

**Block 3:` getPackageInfo ` 整体替换 ` Signature ` 对象 **


        // Java.perform 内继续
        var PM = Java.use('android.app.ApplicationPackageManager');
        var cnt = 0;
        PM.getPackageInfo.overloads.forEach(function (ov) {
            ov.implementation = function () {
                var pi = ov.apply(this, arguments);
                if (pi !== null) {
                    cnt++;
                    if (cnt <= 10) L('[J] getPackageInfo #' + cnt);
                    try {
                        var newSig = Signature.$new(ORIG_JAVA);
                        var arr = Java.array('android.content.pm.Signature', [newSig]);
                        try { pi.signatures.value = arr; } catch (e) {}
                        // Android 9+ signingInfo → SigningDetails.mSignatures
                        try {
                            var si = pi.signingInfo.value;
                            if (si !== null) {
                                var fDet = si.getClass().getDeclaredField('mSigningDetails');
                                fDet.setAccessible(true);
                                var det = fDet.get(si);
                                if (det !== null) {
                                    var fSig = det.getClass().getDeclaredField('mSignatures');
                                    fSig.setAccessible(true);
                                    fSig.set(det, arr);
                                }
                            }
                        } catch (e) {}
                    } catch (e) { L('swap err: ' + e); }
                }
                return pi;
            };
        });
        L(' getPackageInfo hooked');


这一块是关键 —— **整体替换` Signature ` 对象 ** , 而不是 hook 读方法。同时覆盖旧路径 ` signatures ` 数组和
Android 9+ 的 ` signingInfo ` 内部字段。

**Block 4:` SigningInfo ` 直接调用兜底 **


        // Java.perform 内继续
        try {
            var SI = Java.use('android.content.pm.SigningInfo');
            SI.getApkContentsSigners.implementation = function () {
                L('[J] getApkContentsSigners');
                return Java.array('android.content.pm.Signature', [Signature.$new(ORIG_JAVA)]);
            };
            SI.getSigningCertificateHistory.implementation = function () {
                L('[J] getSigningCertificateHistory');
                return Java.array('android.content.pm.Signature', [Signature.$new(ORIG_JAVA)]);
            };
            L(' SigningInfo hooked');
        } catch (e) {}
    });   // Java.perform 结束

###  4.10 为什么 QQ 音乐能救回来：整体判断

QQ 音乐的签名校验属于 "客户端把签名字节塞进业务请求" 这一档，具体特征：

维度  |  QQ 音乐
---|---
客户端本地是否校验签名？  |  **不校验** , 不会因签名不对而拒绝启动 / 崩溃自杀 / 弹窗告警
签名读取路径  |  Java 层 ` PackageManager.getPackageInfo ` / ` SigningInfo `
是否存在 native 层直接读 APK?  |  有 ` open("base.apk") ` , 用途未定， 但从 hook 结果看不参与签名校验
签名字节是否进入业务请求？  |  **是** , 推测在 ` libTPCore-master.so ` 或 okhttp 层作为请求头 / 请求体字段的一部分
阻塞粒度  |  **仅业务数据请求** , 不阻塞启动、不阻塞 WNS 长连、不阻塞 ` SafetySdkProxy ` 上报
保护深度  |  单点校验， 服务端某处对比签名字节 / hash

拿下 QQ 音乐一共就 4 个 hook 点： ` Signature.toByteArray ` \+ `
PackageManager.getPackageInfo ` 覆盖 signatures 数组 + 覆盖 signingInfo + `
SigningInfo.getApkContentsSigners ` 。校验被外包给了服务端，客户端只是个字节搬运工，对我们来说反而是个单点故障 ——
只要客户端读到的字节是原版，服务端就认。这种设计对腾讯的好处是校验规则灰度更新不用发新版本，对我们的好处是需要覆盖的 API 面很小。

###  4.11 限制与后续方向

已知的两个限制。

微信 / QQ 登录会失败。登录 SDK 会向 QQ 或微信客户端校验回调签名，重打包的证书对不上。这跟这节讲的服务端签名校验不是一回事，不展开。

hook 不常驻。App 每次重启都要重装 hook，Frida-only 方案就这样。要脱离 Frida 得走 smali
层补丁。下面简单说下持久化的做法。

持久化到 smali 补丁：

在 apktool 反编译出来的 smali 里找所有读签名的调用点，直接改字节码：


    # 原始代码
    invoke-virtual {v3, v4, v5}, Landroid/content/pm/PackageManager;->getPackageInfo(Ljava/lang/String;I)Landroid/content/pm/PackageInfo;
    move-result-object v6

    # 改成先走原方法, 再调 utility 类替换 signatures
    invoke-static {v6}, Lcom/hook/SigPatch;->fixSignatures(Landroid/content/pm/PackageInfo;)Landroid/content/pm/PackageInfo;
    move-result-object v6


配套加一个 smali 工具类：


    public class SigPatch {
        private static final byte[] ORIG = /* 587 字节 Tencent 原证书 */;
        private static final Signature ORIG_SIG = new Signature(ORIG);

        public static PackageInfo fixSignatures(PackageInfo pi) {
            if (pi == null) return pi;
            try {
                pi.signatures = new Signature[]{ORIG_SIG};
                if (pi.signingInfo != null) {
                    Field fDet = SigningInfo.class.getDeclaredField("mSigningDetails");
                    fDet.setAccessible(true);
                    Object det = fDet.get(pi.signingInfo);
                    Field fSig = det.getClass().getDeclaredField("mSignatures");
                    fSig.setAccessible(true);
                    fSig.set(det, new Signature[]{ORIG_SIG});
                }
            } catch (Throwable t) {}
            return pi;
        }
    }


编译成 dex 塞进 APK, ` classes2.dex ` 追加， 然后 ` apktool b ` \+ ` apksigner sign ` ——
成了不依赖 Frida 的补丁 APK。

这一步没做完，主要因为：调用点数量 10+ 处，全部改一遍是纯体力活；可能还有反射调用要覆盖；每改一次要 ` apktool b ` \+ `
apksigner sign ` \+ 安装 + 手工点，反馈周期 5+ 分钟。研究目的上 Frida hook
已经足够证明模型；商用发行才值得投入这一步。

###  4.12 QQ 音乐这一场的几条经验

回头看整个 4 小时，能提炼出来的东西不算多，但都是当时真的被坑到的地方。

抓包看到 ` code:0 ` 不代表签名过了。服务端返回 ` code:0 `
只表示这个调用它收下了，不代表你的身份被认可。当时我就是被这个假象骗了一小时。判据永远是业务数据能不能正常渲染，看得见的东西才算数。

不要被 UI 局部可用误导。QQ 音乐的 "我的" 页面在未登录时看起来完整，登录按钮、空歌单、静态图标都在，我一度以为
"大部分功能没问题"。但那个页面根本不发网络请求，登录态数据都是本地渲染的。只有必须拉服务端才能出内容的页面才能当可用性判据。

系统级 http_proxy 会对不遵守它的长连接产生副作用。QQ 音乐的 WNS 走 TCP、理论上不遵守 http_proxy，但设置了 proxy
之后 WnsBinder 在 logcat 里疯狂重试，30ms 一次，跟拒服务的表象一模一样。关掉 proxy 才发现那个模块本来就正常，重试完全是
proxy 引发的次生反应。以后抓包看到某个模块 "不正常地忙"，先怀疑 proxy。

` Signature.toByteArray ` 只是叶子节点。要拦签名不是拦读方法，而是把 ` Signature ` 对象本身替换掉。 `
Signature.$new(ORIG_JAVA) ` 之后覆盖 ` pi.signatures.value ` 和 ` signingInfo `
里的私有字段，这样后续 App 用什么姿势读它 —— toByteArray、hashCode、反射 mSignature —— 拿到的都是原版。

spawn 和 attach 的差别在 QQ 音乐这个 case 上被放得很大。QQ 音乐启动期读一次签名然后缓存到字段，之后所有请求都用缓存。attach
模式装完 hook， ` toByteArray called → orig ` 一次都不出现 —— 因为它已经不再调 toByteArray 了。必须
spawn 拦启动期那 4 次调用。以后遇到 "hook 装了但一次不 fire" 的情况，先换 spawn 试。

` frida -l script.js -q ` 会 detach。CLI 加载完脚本如果没有 ` -e ` 也没有 REPL，就立刻退出，session
关闭，hook 全部丢。这一条我踩过至少两次。老实用 Python ` time.sleep(1800) ` 保活，长期调试的时候写 24 小时也没关系。

如果说这一场留下了什么最重要的东西，就是那句 "重打包之前是好的，重打包之后才坏"。跑抓包、hook WNS、猜登录态、猜 UI Fragment
，那两个小时我在拿各种"可能的原因"填空。回到这句话，其实所有假设都能立刻排除 —— 唯一变的东西就是签名，其他都没动。以后开新 case
之前，这句话应该先在心里念一遍。

###  5.1 目标概况与静态分析

###  5.1.5 libnesec 和 libpoison 的角色分工

前面提到 libnesec 是"加固主壳、含 anti-debug"，libpoison 是"加固辅助、含 SEGV 陷阱和实时校验点"。这个说法太模糊
—— 具体到签名校验这件事，两个 so 的分工到底是什么？后面 第 5.5 节 的 patch 只 patch libpoison，那 libnesec
里没有签名校验吗？

有。而且不是"分工协作"的关系 —— 是 **两条完全独立的并行校验链** ：

> **签名校验是双链路独立设计** ：libpoison（native 直读 APK signature block） + libnesec（Java 层
> ` PackageInfo.signatures[0].toByteArray() ` ）， **各自决策、各自杀进程、互不通信** 。
>
> —— 3 天时间用 4 层 hook 分别定点后确认。libpoison 单独 patch 装上、libnesec 那条 Java 校验不管、跑起来
> App 仍然按 libnesec 的 verdict 独立走杀链；反之亦然。两条链之间没有共享 verdict、没有全局变量互通。

网易在 native 层做了 **一道完整的签名校验** 、又在 Java 层 **独立做了另一道完整的签名校验** 。两道各自读签名、各自算
hash、各自决定杀不杀，中间 **没有通信** （不共享 verdict、不写全局变量互通）。要真正绕过， **两条都要打** —— 只 patch
一条，另一条会独立触发杀进程。

上面这个判断跟之前几版的"打手 + 裁判"、"libnesec 收集 libpoison verdict" 的说法完全不同 —— 之前那些是从字符串证据 +
时序日志推断出来的，后来通过分别定点 hook 直接实证了两条链互不通信。本篇之前的机制描述以最新实证为准，前面几处说过"裁判"这个词的地方全部修正。


**链路 A —— libpoison（native 直读 APK signing block）**

libpoison 在 ` serialdata ` 相关函数被调用时（App 启动早期）跑一个签名校验函数（ ` 0x7a.... ` 区），完整由 7
段组成：

段  |  地址范围  |  干什么
---|---|---
A  |  ` 0x7a.... ` |  **native 直接 open/pread APK 文件** ，找到 APK Signing Block，读出证书 chunk 字节
B  |  中间段  |  算 MD5（不走 libcrypto，自实现或经 JNI 调 Java ` MessageDigest ` ）
C  |  ` 0x7ac00 – 0x7ace8 ` |  16 字节 MD5 → 32 字符 hex
D  |  ` 0x7ad08 – 0x7adc8 ` |  释放临时资源
**E** |  **` 0x7add0 – 0x7af00 ` ** |  **15 条` bl 0x10230 ` （PLT → libc.strcmp）白名单短路链 **
F  |  ` 0x7af04 – 0x7af38 ` |  成功路径：stack canary 检查 + ` ret `
G  |  ` 0x7af3c – 0x7af44 ` |  **失败路径：` mov w0, wzr; bl exit(0); bl abort ` ** ，函数尾巴 **没有 ret** （必死）

**关键：libpoison 的证书读取走 native 路径，不经过 Java** —— 所以第二章讲的 Java 层 hook（ `
Signature.toByteArray ` / ` PackageManager.getPackageInfo ` / ` SigningInfo `
全部替换）对这条链 **完全没用** 。跟 第 1.4 节 讲的小红书 ` libsentry.so ` 是同一类实现（都是 native 直读 APK
signing block chunk）。

段 E 的白名单细节和 15 条 MD5 在 第 5.8 节 展开。这里只需要理解一件事： **libpoison 里签名校验的成败在段 G 一处出口**
。改一条 ` b 0x7af04 ` 让 15 条 strcmp 全不匹配时也跳成功路径就绕过了这一层，也就是 第 5.5 节 里的 Patch 1。

libpoison 是加密库（磁盘上 ` .text ` 是密文 + 伪造节头），要拿到上面这些偏移只能运行时 dump 已解密的内存镜像 —— 见 第
### 5.7 节 的 ptrace + process_vm_readv 方案。


**链路 B —— libnesec（Java 层独立校验，不走 native 直读）**

libnesec 里也 **有** 签名校验代码，而且不是"裁判"或"上报" —— 是 **独立的、完整的另一次校验** ，走 Java 层 API：

步骤  |  做什么
---|---
1  |  libnesec native 桥 ` MyJni.load ` 在 ` MyApplication.attachBaseContext ` 阶段被调
2  |  内部通过 JNI 反射调 Java 的 ` PackageManager.getPackageInfo(pkg, GET_SIGNATURES) `
3  |  拿到 ` signatures[0].toByteArray() ` 字节
4  |  算 hash 与内部白名单比对
5  |  不匹配 → 调 Java 层 ` NEDialog.showRiskMessage(0, "") ` → 弹"请前往官方渠道下载正版 APP" toast → 主线程 ART 自然死
5'  |  或触发 ` raise(SIGTRAP=5) ` \+ ` tgkill(pid, tid, SIGTRAP) ` 主线程自杀

**libnesec 相关字符串证据** （离线解密 dumped 内存镜像 4297 条字符串后确认）：


    0xca8f2  code0x0:detectDbg
    0xcbd11  code0x1:checkSig-Fail                    ← 签名校验失败 verdict
    0xcbee1  code0x2:detectEmulator
    0xcb0a8  signatures                                ← Java 字段名（JNI 反射用）
    0xcc693  bin/mt/apksignaturekillerplus            ← MT 管理器签名杀手插件路径
    0xccf46  bin/mt/apksignaturekillerplus/HookApplication  ← 该插件的 Hook 类名


` code0x1:checkSig-Fail ` 是 **libnesec 自己的 verdict code** （分类标签，不是从 libpoison
收来的），后两条是 **明确的反反检测** —— libnesec 硬编码扫描 ` apksignaturekillerplus ` （MT 管理器出品的
Xposed 签名绕过工具）的特征路径和类名，检测到就直接标记签名校验失败。说明网易易盾团队清楚地知道 Java 层 hook
这条路径存在，主动做了工具级特征扫描。

libnesec 除了签名校验，还有另外两条独立线（同一个 wrapper 里跑）：调试器 / frida 检测（ ` code0x0:detectDbg
` ，通过 SIGTRAP 信号探测 + ` art::Dbg::GoActive ` \+ ` TracerPid ` ）和模拟器检测（ `
code0x2:detectEmulator ` ）。它们跟本章主线不直接相关，但共享同一套杀链（ ` raise(SIGTRAP)+tgkill `
），任一 verdict 命中都杀。


**两条链完全独立、互不通信**

这是分别定点 hook 实证到的：

维度  |  链路 A（libpoison）  |  链路 B（libnesec）
---|---|---
触发时机  |  ` libpoison.serialdata ` 相关函数被调时（启动早期）  |  libnesec 在 ` MyApplication.attachBaseContext ` 被调时（跟 A 时序上大致并行）
校验对象读取  |  **native 直读 APK 文件 signing block** |  **Java` getPackageInfo(pkg, GET_SIGNATURES) → signatures[0].toByteArray() ` **
失败处理  |  ` mov w0, wzr → bl exit → bl abort ` ，函数尾没 ret， **必死** |  ` NEDialog.showRiskMessage("请前往官方渠道下载正版 APP") ` toast → ART 自然死；或 ` raise(SIGTRAP)+tgkill `
是否通知对方  |  不通知。函数体只 3 条指令负责失败： ` mov + exit + abort ` ，没写任何全局变量  |  不通知。libnesec 自己读自己算自己决定
Frida 怎么打  |  byte patch native 分支（第 5.5 节 Patch 1）  |  Java hook ` PackageManager ` / ` Signature ` / ` NEDialog.showRiskMessage ` （有 CheckJNI 陷阱，见下）

**双链路机制图** ：


网易云音乐签名校验双链路

**回答本节开头的问题** ：

  * **签名校验在网易云音乐里是双链路独立设计** 。libpoison（native 直读）和 libnesec（Java 层 ` getPackageInfo ` ） **各自完整地做一次** 、各自决定杀不杀、中间不通信。
  * **只 patch libpoison 一条链不够** —— libnesec 那条 Java 层校验不受 libpoison patch 影响，自己独立触发杀链。
  * **我们 第 5.5 节 的 3 层 patch 只覆盖了 libpoison 那条** 。让 App 能走到 Application init 完成、Splash 消失，是因为我们的 libc ` exit / abort / exit_group ` hook 把 libnesec 触发的杀链（ ` raise(SIGTRAP)+tgkill ` 或 ART 自然死路径） **部分拦住了** —— 但拦住的代价是主线程停在校验后的某个位置，进入 splash 之后的 ` NEDialog.showRiskMessage ` 弹窗或 looper 静默，主 Activity 起不来。
  * **要真正让 App 完全恢复，必须两条都打** —— libpoison 的 byte patch + libnesec 的 Java 层 hook（覆盖 ` PackageManager.getPackageInfo ` / ` Signature.toByteArray ` / ` NEDialog.showRiskMessage ` ）。第 5.5 节 只做了前一半。


**CheckJNI 陷阱（实证过的坑）**

这里有一条 **必须提醒** 的陷阱： **不能简单地在 Java 层全局 hook` Signature.toByteArray ` 返回原版字节 **
。

原因是业务代码里也有 native 代码会通过 CheckJNI 读 ` PackageInfo ` —— 如果 Java 层被 hook
返回了原版字节、native 层通过其他路径看到的是重签字节， **ART CheckJNI 检测到两次读取结果不一致，会直接 abort 杀进程** 。

这就是为什么 QQ 音乐那套" ` Signature ` 对象整体替换"方案（第四章讲的完整可复用打法） **在网易云音乐上不能照搬** —— 会直接触发
CheckJNI 报错。要打 libnesec 那条 Java 校验，需要更精细的方案： **只在 libnesec 的 JNI 调用点上返回原版**
，其他调用点不干预。具体路径见 第 5.11 节 尝试 8。

###  5.2 最简重打包：500ms 内自杀

跟 QQ 音乐一样先走最简路径：


    apktool b ncm_decoded -o ncm_repack.apk
    apksigner sign --ks my.keystore --ks-pass pass:123456 --out ncm_signed.apk ncm_repack.apk
    adb install -r ncm_signed.apk
    adb shell monkey -p com.netease.cloudmusic 1


装机现象（logcat）：


    [0.0s] I ActivityManager: Start proc ... for activity com.netease.cloudmusic/.LaunchActivity
    [0.3s] D POISON  : APK Signature:d2a93e35348b7c414dddf5caf64108ad
    [0.3s] F libc    : Fatal signal 11 (SIGSEGV), code 1 (SEGV_MAPERR), fault addr 0x0 in tid ... (InitTaskSchedul)
    [0.4s] I ActivityManager: Process com.netease.cloudmusic (pid ...) has died


App 500ms 内就自杀， splash 都没出来。跟 QQ 音乐 "首页错误但 App 存活" 完全不是一个量级。

` d2a93e35348b7c414dddf5caf64108ad ` 是我们自签 keystore cert 的 MD5，原版 keystore
cert MD5 是 ` da6b069da1e2982db3e386233f68d76d ` 。看到 POISON log 直接明文打出 hash
说明加固的校验函数其实做了这件事：读 APK cert → 算 MD5 → 打 log。这条 log 在后面的实证中是我们最重要的锚点。

Nesec 加固层的行为可以先粗略概括：

  1. ` AndroidManifest ` 里 ` application:name=com.netease.nis.wrapper.MyApplication ` 起效，加固层第一时间接管。
  2. 加固层调 native 校验 —— ` libpoison.so ` 里某个函数检测签名 hash → 不匹配就 exit 或触发 SEGV。
  3. ` Process.setExceptionHandler ` 兜底：任何 native 崩溃或未处理异常，最后一步 ` System.exit(0) ` 。

跟 QQ 音乐差别开始显现：

  * QQ 音乐的签名校验是 "服务端拒服务" —— App 能活着，只是拿不到数据。
  * 网易云音乐的签名校验是 "客户端自杀" —— App 活不过 1 秒。

###  5.3 起点：作者版 45 层 patch 的教训

社区里有一份作者写的 writeup，长期公开在若干技术论坛，记录了 3 天累计做的 45+ 层 Frida patch 才让 App **勉强看到
splash 界面** 。作者版脚本约 1491 行，覆盖：

  * SIGSEGV / SIGBUS / SIGABRT 各种 signal 拦截
  * ` dlopen ` 全 hook 用于识别加固各 so 加载时机
  * ` Process.setExceptionHandler ` 覆盖
  * ` System.exit ` / ` Runtime.exit ` / ` _Exit ` / ` exit_group ` / ` abort ` / ` raise ` / ` tgkill ` 全套 hook
  * libc ` HandleUsingDestroyedMutex ` byte patch
  * libpoison 内 30+ 处 byte patch
  * Java ` MessageDigest.digest ` / ` Signature.toByteArray ` / ` PackageManager.getPackageInfo ` 三合一
  * 反射调用点全覆盖
  * Tinker ` PatchApplication ` reflect 检测点绕过

结果： splash 能显示了，但 **主 Activity 起不来， App 停在 splash 之后** 。

我看完这份 writeup 的第一反应是：45 层不像是最优解。每加一层 patch 缓解一点点，又引入新的次生问题，堆到 45
层根本不可能反过来定位主要矛盾。这种叠加式方案得从头重来一遍，找到真正在支配这个 App 生死的那个点。

###  5.4 用 "删繁就简" 定位主要矛盾

**方法** ：不从 0 加 patch，从作者版全套开始 **一层一层往外拆** —— 看拆掉哪些 patch App 反而活得更久。

**实验设计** ：把 ` bypass_nesec.js ` 拆成能独立开关的 15 个 patch group，一组一个开关，通过 spawn
Frida 后往 script 里传 rpc 参数控制哪些开、哪些关。

**关键发现——最大的一个** ：

作者版脚本里的 **` Process.setExceptionHandler ` 全局屏蔽是主要破坏源 ** 。


    // 作者原版, 简化后
    Java.perform(function () {
        // 无条件屏蔽 Application 的 setExceptionHandler, 阻止任何自杀路径
        someObj.setExceptionHandler.implementation = function () {
            return null;   // ← 完全不设 handler
        };
    });


问题： **ART 的空指针 / 越界访问是通过 SIGSEGV + Application uncaught exception handler 走
"隐式 null check" 流程的** 。ART JIT 出来的代码在很多地方省略了显式 null check, 遇到 null 时靠 CPU 触发
SIGSEGV, signal handler 里根据 fault_addr 判断这是隐式 null check → 抛 Java
NullPointerException → 上抛到 Application 层的 exceptionHandler。

作者的 hook 把这个 handler 拦掉后， **所有本来会被正常处理的隐式 null check 都变成了未处理的 SIGSEGV** —— App
立即崩溃。

**证据** ：

  * 打开 ` setExceptionHandler ` hook：App 崩溃 ` Fatal signal 11 SEGV ` 在 libnative 侧，看似像 Nesec 校验失败。
  * 关掉 ` setExceptionHandler ` hook：libnesec 在 App 启动后 **+7.4 秒** 才被加载起来， Application init 走得更完整、走到那些依赖 libnesec 的代码点才 dlopen —— 这本身就是 App 存活的直接证据。

**教训** ：Application 全局的 signal handler 不能无条件屏蔽。如果一定要 hook，得在里面判断当前 signal
是不是要拦的、其他一律放行。

###  5.5 精简后的 3 层核心 patch

![网易云音乐三层 patch
网易云音乐三层 patch 结构

拆掉 ` setExceptionHandler ` 后，很多下游 patch 变得不必要 —— 它们本来是用来 "压制" `
setExceptionHandler ` 引发的次生崩溃。逐个删掉验证仍能启动到相同深度，最后剩下 3 层真正必要的：

**Patch 1：libpoison 内的签名校验分支跳转， byte patch**

作者定位到的关键点是 ` libpoison.so + 0x7af3c ` 附近有一段签名校验后的分支判断。原始指令：


    0x7af3c:  mov  w0, wzr        ; w0 = 0, 校验失败
    0x7af40:  ...
    0x7af04:  ...                  ; 校验通过的分支入口


改成无条件跳到 ` 0x7af04 ` ：


    var m = Process.findModuleByName('libpoison.so');
    Memory.patchCode(m.base.add(0x7af3c), 4, function (w) {
        // b 0x7af04
        // 相对偏移 = (0x7af04 - 0x7af3c) / 4 = -14 = 0xFFFFFFF2
        // 编码 0x14000000 | (imm26 & 0x3FFFFFF) = 0x14000000 | 0x3FFFFF2 = 0x17FFFFF2
        w.writeU32(0x17FFFFF2);
    });


这条 patch 是必须的 —— 去掉之后加固层直接拒绝往下走。

**Patch 2：libc` HandleUsingDestroyedMutex ` 覆盖为 "返回 0" **

Android bionic libc 在 pthread_mutex_lock 时会检查 mutex 是否已被 destroy，检测到后调 `
HandleUsingDestroyedMutex(pthread_mutex_t*) ` —— 里面调 ` __pthread_abort_msg ` 打
fatal log 然后 ` abort() ` 。

加固层某些地方会故意 destroy mutex 再 lock, 作为一种校验手段， 触发这个路径就自杀。改成直接返回 0：


    var libc = Process.findModuleByName('libc.so');
    libc.enumerateSymbols().forEach(function (s) {
        if (s.name.indexOf('HandleUsingDestroyedMutex') >= 0) {
            Memory.patchCode(s.address, 8, function (w) {
                w.writeU32(0x2a1f03e0);  // mov  w0, wzr
                w.writeU32(0xd65f03c0);  // ret
            });
        }
    });


**Patch 3：libc` exit ` / ` _exit ` / ` _Exit ` / ` abort ` 拦截 **

加固层某些 native 校验失败时会调 ` exit(0) ` 让 App 静悄悄退出， 比 abort 更隐蔽。拦掉：


    ['exit', '_exit', 'exit_group', '_Exit', 'abort'].forEach(function (name) {
        var p = Module.findExportByName('libc.so', name);
        if (!p) return;
        Interceptor.attach(p, {
            onEnter: function (args) {
                var lr = this.returnAddress;
                var lrMod = Process.findModuleByAddress(lr);
                send('libc.' + name + ' from ' + (lrMod ? lrMod.name : '?')
                   + ' arg=' + args[0]);
                this.skip = true;
            },
            onLeave: function (retval) {
                if (this.skip) retval.replace(ptr(0));
            }
        });
    });

    // 私有 __exit / __exit_group, enumerateSymbols 找
    libc.enumerateSymbols().forEach(function (s) {
        if (s.name === '__exit' || s.name === '__exit_group') {
            Interceptor.replace(s.address, new NativeCallback(function (a) {
                send('SKIPPED ' + s.name + '(' + a + ')');
            }, 'void', ['int']));
        }
    });


外加 Java 层三兄弟：


    Java.perform(function () {
        Java.use('java.lang.System').exit.implementation = function (c) {
            send('[skip] System.exit(' + c + ')');
        };
        Java.use('java.lang.Runtime').exit.implementation = function (c) {
            send('[skip] Runtime.exit(' + c + ')');
        };
        Java.use('android.os.Process').killProcess.implementation = function (p) {
            send('[skip] Process.killProcess(' + p + ')');
        };
    });


加载方式： Frida spawn + ` dev.enable_spawn_gating ` ：


    import frida, time
    device = frida.get_device_manager().add_remote_device('127.0.0.1:27042')
    device.enable_spawn_gating()   # zygote 后续 spawn 出的新 App 进程也 pause 等我们
    pid = device.spawn(['com.netease.cloudmusic'])
    session = device.attach(pid)
    scr = session.create_script(open('hybrid_minimal.js').read())
    scr.on('message', lambda m, _: print(m))
    scr.load()
    device.resume(pid)
    time.sleep(1800)   # 保活


` enable_spawn_gating ` 是关键 —— 网易云音乐有些场景会通过 ` am start ` 或 ` bindService ` 让
ActivityManager 拉起独立子进程做一些 native 初始化， ` spawn_gating ` 让这些新 zygote fork
出来的进程都 pause 等 hook 装好，而不是让它们独立跑起来先自杀。注意这条只对 zygote spawn 出来的进程有效，App 进程内部 `
fork() ` 出来的 subprocess 拦不到，需要另外单独想办法。


**开始下一节前先给这 3 层 patch 定个位** ：对照 第 5.1.5 节 的双链路图，这 3 层 + Java 兜底**全部落在链路
A（libpoison native 直读）**上 —— Patch 1 打 libpoison 的失败出口、Patch 2/3 拦 libpoison 走
libc 出去的杀链、Java 层三兄弟兜底也是拦 libpoison 触发的 Java 侧 exit。 **链路 B（libnesec Java 层
getPackageInfo + NEDialog.showRiskMessage）一层都没打** 。

这不是遗漏，是当时对机制的认识就只到"libpoison 那条"这一步 —— libnesec 那条 Java 校验的存在是后来通过分别定点 hook
实证才浮出来的（详见 第 5.1.5 节 的迭代记录）。链路 B 的对称 3 层 hook 蓝图放在 第 5.11 节 尝试 8 讲，配图 `
fig_5_11_chain_b_hooks ` 。

带着这个"只打了一半"的认知去看下一节 第 5.6 节 的结果 —— 主 Activity 起不来不是神秘失败，而是链路 B 那条校验一直没受干扰在跑， `
NEDialog.showRiskMessage ` 弹到 quota 顶或者主线程停在 signal 等待，这是最主要的嫌疑。

  * 我们版：Application 走完完整初始化， WeChat / HMS / MMKV / DataBinding, Splash 出来、Splash 消失。 **离主 Activity 只差一步** 。

  * Splash 消失后应该 startActivity 到 MainActivity，但 **没起来** 。
  * logcat 里没有 error / crash，就是静默停在那。
  * App 进程仍然存活，但没有 activity。
  * 用户看到的是黑屏 / 白屏 / 直接回到桌面。

**T+7.4s libnesec 那一行的正确解读** （跟之前几版的说法相比要更正）：

之前版本说这条 log 是"libnesec 后校验触发的时机"、"主 Activity 起不来的证据"。用分别定点 hook 实证后，这个说法是错的 ——
**libnesec 的签名校验发生在` MyApplication.attachBaseContext ` 阶段 ** （跟 libpoison
时序上大致并行，都在启动早期），不是 +7.4s。

+7.4s 是 libnesec 的 **另一个动作** —— ` libnesec.so ` 完整 dlopen 挂到进程 module 表的时机（我们的
dlopen 监听器看到的时间点）。签名校验相关代码之前已经通过 native 桥 ` MyJni.load ` 执行过一次了，dlopen 完成只是让整个
so 完全可见（多线程 anti-debug 上位、字符串解密表就位、SIGTRAP 处理器注册）。

那主 Activity 起不来的真实原因？第 5.1.5 节 讲的 libnesec Java 层校验路径 —— ` MyJni.load ` → JNI
反射 ` getPackageInfo ` → 白名单比对失败 → ` NEDialog.showRiskMessage ` 弹 toast →
主线程停下。这条 Java 层校验 第 5.5 节 完全没管，杀链被 libc exit hook 部分拦住导致进程静默留在原地。具体到 hook
位置的候选路径见 第 5.11 节 尝试 8。

到这一步为止，我的进度确实是 "作者版之后往前挪了一段"。但往前挪的 "一段" 到底有多远、加固里面到底是什么、离能用还差什么，靠 45 层 patch
那种加/减开关的方式说不清楚。要说清楚只能换工具，把 libpoison 从内存里 dump 出来做指令级反汇编。

###  5.7 深挖 libpoison：自己 dump + 自己反汇编

上面三层 patch 让 Application init 走完了，但停在 Splash 之后。这时候一个诱惑很强 —— 继续加第 4 层、第 5 层
patch，试着 "再挪一段"。作者版走的就是这条路，45 层之后停在 splash 之前。

我停了一下，反过来问自己：前面这些判断真的站得住吗？

"libpoison + 0x7af3c 是签名校验"、"改成无条件跳成功就过关"、"MD5 是 inline 实现"，这些全是从别人的 writeup
里抄来的结论，我自己一条都没独立验证过。继续加 patch 相当于在别人的二手结论上叠越来越多的假设，越叠越脆，最后到底哪儿出问题谁也说不清。

要让判断落到字节级，得自己把 libpoison 从内存里 dump 出来，自己反汇编，自己对着字节看。下面这一节就是干这件事 —— 从 "抄别人的偏移"

####  用 ptrace + process_vm_readv 从运行时内存 dump

frida spawn 这条路不通 —— agent 在 libpoison 加载前就被 kill，来不及 dump。改用底层方案 —— **root +
ptrace** 。完整可编译 C 代码 115 行，能直接 clang 交叉编译成 aarch64 static binary。

**扫读指引** ：下面 115 行第一次读只看最后 ` main() ` 里那个 ` while (time(NULL) - start <
timeout) ` 循环就够 —— polling PID → 判 libpoison 是否加载 → 调 ` do_dump ` 。上面三个 helper
` find_pid ` / ` has_libpoison ` / ` do_dump ` 都是实现细节，拷走能直接编译，不用逐行读。


    // libpoison_waitdump.c
    // 用法: libpoison_waitdump <pkg_substr> <outdir> [<timeout_sec>]

    #include <stdio.h>
    #include <stdlib.h>
    #include <string.h>
    #include <dirent.h>
    #include <fcntl.h>
    #include <unistd.h>
    #include <errno.h>
    #include <time.h>
    #include <sys/ptrace.h>
    #include <sys/wait.h>
    #include <sys/uio.h>

    // 扫 /proc 找 cmdline 包含 pkg 的 PID
    static pid_t find_pid(const char *pkg) {
        DIR *d = opendir("/proc"); if (!d) return -1;
        struct dirent *e; pid_t found = -1;
        while ((e = readdir(d))) {
            if (e->d_type != DT_DIR) continue;
            pid_t pid = atoi(e->d_name);
            if (pid <= 0) continue;
            char path[64]; snprintf(path, sizeof(path), "/proc/%d/cmdline", pid);
            int fd = open(path, O_RDONLY); if (fd < 0) continue;
            char buf[256] = {0}; read(fd, buf, sizeof(buf)-1); close(fd);
            if (strstr(buf, pkg)) { found = pid; break; }
        }
        closedir(d);
        return found;
    }

    // 判 maps 里有没有 libpoison.so
    static int has_libpoison(pid_t pid) {
        char path[64]; snprintf(path, sizeof(path), "/proc/%d/maps", pid);
        FILE *f = fopen(path, "r"); if (!f) return 0;
        char line[1024]; int hit = 0;
        while (fgets(line, sizeof(line), f)) {
            if (strstr(line, "libpoison.so")) { hit = 1; break; }
        }
        fclose(f);
        return hit;
    }

    // ATTACH + 逐段拷贝
    static int do_dump(pid_t pid, const char *outdir) {
        if (ptrace(PTRACE_ATTACH, pid, 0, 0) < 0) {
            fprintf(stderr, "PTRACE_ATTACH(%d) 失败: %s\n", pid, strerror(errno));
            return 2;
        }
        int status; waitpid(pid, &status, 0);
        fprintf(stderr, "[+] ATTACHED PID=%d status=0x%x\n", pid, status);

        char maps_path[64]; snprintf(maps_path, sizeof(maps_path), "/proc/%d/maps", pid);
        FILE *fm = fopen(maps_path, "r");
        if (!fm) { ptrace(PTRACE_DETACH, pid, 0, 0); return 4; }

        char line[1024]; int seg_idx = 0;
        while (fgets(line, sizeof(line), fm)) {
            const char *which = NULL;
            if (strstr(line, "libpoison.so"))     which = "libpoison";
            else if (strstr(line, "libnesec.so")) which = "libnesec";
            else continue;

            unsigned long start, end; char perm[8];
            if (sscanf(line, "%lx-%lx %7s", &start, &end, perm) != 3) continue;
            size_t size = end - start;

            char out[256];
            snprintf(out, sizeof(out), "%s/%s_%d_%lx_%s.bin", outdir, which, seg_idx, start, perm);
            FILE *fo = fopen(out, "wb");
            if (!fo) { seg_idx++; continue; }

            char *buf = malloc(0x10000); size_t total = 0;
            for (size_t off = 0; off < size; off += 0x10000) {
                size_t chunk = (size - off > 0x10000) ? 0x10000 : (size - off);
                struct iovec local  = { buf, chunk };
                struct iovec remote = { (void*)(start + off), chunk };
                ssize_t n = process_vm_readv(pid, &local, 1, &remote, 1, 0);
                if (n <= 0) break;
                fwrite(buf, 1, n, fo); total += n;
                if ((size_t)n < chunk) break;
            }
            free(buf); fclose(fo);
            fprintf(stderr, "  [+] %s seg #%d start=0x%lx size=0x%zx read=0x%zx perm=%s\n",
                    which, seg_idx, start, size, total, perm);
            seg_idx++;
        }
        fclose(fm);
        ptrace(PTRACE_DETACH, pid, 0, 0);
        fprintf(stderr, "[+] DETACHED %d segments\n", seg_idx);
        return 0;
    }

    int main(int argc, char **argv) {
        if (argc < 3) {
            fprintf(stderr, "用法: %s <pkg_substr> <outdir> [<timeout_sec=60>]\n", argv[0]);
            return 1;
        }
        const char *pkg = argv[1]; const char *outdir = argv[2];
        int timeout = argc > 3 ? atoi(argv[3]) : 60;

        time_t start = time(NULL); int iters = 0;
        while (time(NULL) - start < timeout) {
            iters++;
            pid_t pid = find_pid(pkg);
            if (pid < 0) continue;
            if (!has_libpoison(pid)) continue;
            fprintf(stderr, "[+] HIT iter=%d PID=%d\n", iters, pid);
            return do_dump(pid, outdir);
        }
        fprintf(stderr, "[!] TIMEOUT after %d iters\n", iters);
        return 5;
    }

    // dump_libpoison.js
    // 用 frida 在原版进程里 dump —— libpoison 从头到尾都是解密的
    'use strict';
    var OUTPATH = '/data/data/com.netease.cloudmusic/files/libpoison_dumped.bin';
    var CHUNK = 0x10000;

    function dump() {
        var m = Process.findModuleByName('libpoison.so');
        if (!m) return false;
        console.log(TAG + ' libpoison base=' + m.base + ' size=0x' + m.size.toString(16));

        var f = new File(OUTPATH, 'wb');
        var off = 0, okBytes = 0, zeroBytes = 0;
        while (off < m.size) {
            var n = Math.min(CHUNK, m.size - off);
            var buf = null;
            try { buf = m.base.add(off).readByteArray(n); } catch (e) {}
            if (buf) { f.write(buf); okBytes += n; }
            else { f.write(new Uint8Array(n).buffer); zeroBytes += n; }
            off += n;
        }
        f.close();
        return true;
    }
    setInterval(function () { if (dump()) clearInterval(iv); }, 50);

###  5.8 15 条白名单 MD5：签名校验不是单点是短路链

![libpoison 15 条 strcmp
libpoison 15 条 strcmp 白名单短路链

有了完整镜像，反汇编 ` 0x7add0 ` 到 ` 0x7af00 ` 这一段（签名校验函数的核心），看到的不是 "一次 MD5 比对"，而是 **一条
15 条 strcmp 白名单 chain** ：


    0x7add8:  adrp x1, 0x7e000          ; x1 = 0x7e000+0x7d7 = 第 1 个白名单字符串
    0x7addc:  add  x1, x1, #0x7d7
    0x7ade0:  add  x0, sp, #0x30        ; x0 = 算出的 hex_buf (32 字符 MD5 hex)
    0x7ade4:  bl   0x10230              ; strcmp(hex_buf, "d6520186...")
    0x7ade8:  cbz  w0, 0x7af04          ; w0==0(相等) → 成功路径

    0x7adec:  adrp x1, 0x7e000          ; x1 = +0x7f8 = "da6b069d..." ( 第 2 条 = 线上正式签)
    0x7adf0:  add  x1, x1, #0x7f8
    0x7adf4:  add  x0, sp, #0x30
    0x7adf8:  bl   0x10230
    0x7adfc:  cbz  w0, 0x7af04

    ... 第 3~14 条同款 chain, 每个块 16 字节 ...

    0x7aef0:  adrp x1, 0x7e000          ; x1 = +0x9a5 = "ac06dbfe..." (第 15 条)
    0x7aef4:  add  x1, x1, #0x9a5
    0x7aef8:  add  x0, sp, #0x30
    0x7aefc:  bl   0x10230
    0x7af00:  cbnz w0, 0x7af3c          ;  最后一条用 cbnz, 不等就跳失败
                                        ;   前 14 条都是 cbz, 相等就跳成功


15 个 ` bl 0x10230 ` , 每个后面跟一个 ` cbz w0, 0x7af04 ` 或最后一个 ` cbnz w0, 0x7af3c `
。前 14 条相等就 fall through 到成功；第 15 条相等则 fall through 到成功，不等才跳失败。

**为什么第 15 条用` cbnz ` 而非 ` cbz ` ** ：编译器的 fall-through 优化。如果第 15 条相等，自然 fall
through 到 0x7af04 成功；如果第 15 条也不等，跳 0x7af3c 失败。这样省了最后一条无条件跳转。

` bl 0x10230 ` 的 PLT slot 验证：


    0x10230:  adrp x16, 0xbb000           ; PLT 跳板模板
    0x10234:  ldr  x17, [x16, 0x6d8]      ; .got.plt[0xbb6d8] ← linker 填 libc.strcmp
    0x10238:  add  x16, x16, 0x6d8
    0x1023c:  br   x17                    ; → libc.so 的 strcmp


**修正** ：比对不是 inline 实现，也不是 dlsym 自查找 —— **就是标准 ARM64 PLT slot 调用 libc.strcmp**
。之前的 hook 之所以没命中，是 **hook 装得太晚** ,libpoison 加载即在 +0.05~0.1s 跑完校验， `
Process.findModuleByName("libpoison.so") ` 在 attach 早期返回 null，过滤逻辑永远 false。

####  提取 15 条白名单

####  修正 MD5 实现路径

` .rodata ` 里还有一个关键字符串，在 0x821a0：


    MD5 part of OpenSSL 1.0.2u  20 Dec 2019
    AES part of OpenSSL 1.0.2u  20 Dec 2019


**libpoison 静态链接了 OpenSSL 1.0.2u 全套** , ` MD5_Init/Update/Final/Transform `
、AES 全套都带进来，符号在 dynsym 里被 strip，但版本字符串没被剥干净。三条反向证据把 "两种可能" 钉成第三种：

  * Java MessageDigest： ` MessageDigest ` / ` getInstance ` / ` java/security ` 全 NOT FOUND
  * 散写在 .text 的 inline MD5：4 个 MD5 init 魔数 ` 0x67452301 / 0xefcdab89 / 0x98badcfe / 0x10325476 ` **0 处 raw bytes** , ` movz w?,#0x2301 + movk w?,#0x6745,LSL #16 ` **0 对**
  * dlopen libcrypto：dlopen 目标 SO 只有 ` libandroid.so ` ，没有 ` libcrypto.so `
  *  **OpenSSL 1.0.2u 整套静态链接进 libpoison.text** ： ` .rodata 0x821a0 ` 处版本字符串直接出卖了

综合前面所有实证，修正后的最终精确死链是：


    libpoison JNI native 方法
      ↓ JNI 调 ContextWrapper.getPackageManager()
                              .getPackageInfo(pkgName, GET_SIGNATURES)
                              .signatures[0].toByteArray()
      ↓ 拿到 cert DER 字节数组(原版 915 字节)
      ↓ 用静态链接的 OpenSSL 1.0.2u 算 MD5(16 字节 raw)
      ↓ 0x7ac00-0x7acec: 16 次 sprintf("%02x") 拼成 32 字符 hex
      ↓ 0x7ad04: __android_log_print("POISON", "APK Signature:%s", hex)
      ↓ 0x7add8-0x7af00: 15 条 strcmp 白名单 chain
      ↓     前 14 条: cbz → 跳 0x7af04 成功
      ↓     第 15 条: cbnz → 跳 0x7af3c 失败
      ↓ 成功路径 0x7af04: stack canary check → ret
      ↓ 失败路径 0x7af3c: mov w0, wzr → bl exit(0) @ 0x7af40

每一个偏移都对得上磁盘 dump 出来的字节。到这一步机制才算彻底摸清。

###  5.9 再战 frida：知道机制之后仍然打不通

有了精确机制，理论上有 3 个新切入点都可行：

  * **切入点 1** : ` Interceptor.attach(libpoison.base.add(0x7ade4)) ` （第 1 条 strcmp 的 bl）, ` onLeave ` 强制 ` retval.replace(0) ` → 跳过整条 15 条 chain

拿着这堆情报，我挑最干净的切入点 2 试了一发。只改 ` .rodata ` ，不动 ` .text ` ，理论上不触发安全库的代码段自校验。脚本 30

####  时间轴对照

时刻  |  事件
---|---
T+0 ms  |  ` dev.resume(pid) ` , zygote fork 出 cloudmusic
T+5~10 ms  |  frida-agent 注入， gum-js-loop 起来
T+10~50 ms  |  libnesec 通过 JNI_OnLoad 加载， 扫 ` /proc/self/maps ` \+ 线程名
T+50 ms  |  libnesec 发现 frida-agent → inline svc ` kill ` 干掉 V8 runtime
T+800 ms  |  libpoison 加载， 此时 frida 早死
T+1200 ms  |  libpoison 算 hash + strcmp 15 条全不等
T+1205 ms  |  ` bl exit(0) ` , atexit 析构表损坏 → SIGSEGV

精确机制本身决定了：任何要 "等 libpoison 加载后再装 hook" 的方案，在 frida-server 用户态注入下都是死路。切入点 1、2、3
三个都要求 hook 装在 libpoison 加载之后，在我们的设备上这都过了 frida-agent 被卸的 50 ms 死线 700 多倍。

####  磁盘静态 patch 也走不通

既然 hook 装不上，能不能直接 patch 磁盘上的 libpoison.so，把第 15 条白名单 MD5 改成重打包 hash，然后重打包 + 重签
+ 装回？


    disk = open("music163.9.2.80/lib/arm64-v8a/libpoison.so", "rb").read()
    for md5 in ["da6b069da1e2982db3e386233f68d76d",
                "ac06dbfeb05f19eac37c343b03230d07"]:
        idx = disk.find(md5.encode())
        print(f"  {md5}: {'@'+hex(idx) if idx >= 0 else 'NOT FOUND'}")


结果 15 条全 NOT FOUND。原因前面 第 5.7 节 已经验证过：libpoison 不只 ` .text ` 加密， ` .rodata `
也加密 —— 磁盘文件里那 33 字节 ASCII MD5 字符串根本不是明文，而是密文，需要 libnesec 加载时解密到内存里才是
ASCII。所以静态 patch 磁盘文件，根本找不到目标字节去改。

####  剩下的路，其实已经不是 frida 能解决的了

ZygiskFrida 注入 —— 在 zygote fork 时就把 frida-agent 加载好，远早于 libnesec init 跑起来。这样
hook 装在 libpoison 加载之前，strcmp 拦截才有机会生效。

但 ZygiskFrida 是 Magisk Zygisk 模块，不是 frida 本体的能力。这道墙最终能不能打通，取决于 Magisk 框架，不是
frida。在这一档加固面前 frida 的位置是 "hook 工具"，不是 "绕过工具"。绕过是靠 Zygisk + patched-libnesec /
patched-fs / Shamiko / DenyList 的组合，frida 只是绕过之后再用来做业务层 hook。

###  5.10 反向逆向 Toast 弹窗到死胡同

死循环过程中观察到一个之前没看见的现象 —— logcat 里反复出现：


    06-30 10:46:24.543 W NotificationService: Package com.netease.cloudmusic is above allowed toast quota,
      the following toast was blocked and discarded: TextToastRecord{b30b392 25180:com.netease.cloudmusic/u0a283
      isSystemToast=false token=android.os.BinderProxy@f5f2163
      text=请前往官方渠道下载正版APP duration=0}


每个 process 寿命内被快速排队 5 个 toast 然后被 NotificationService 限流。这说明 App 在 libpoison +
libnesec 那条链路之外， 还有一条独立的 Java 层旁路在弹这个 Toast。

####  时序的新观察

仔细看 process 启动到 SIGSEGV 之间的时间线，这条 Toast 触发比之前以为的 libpoison 校验早：

时刻  |  事件
---|---
T+0 ms  |  ` Start proc 23290:com.netease.cloudmusic `
T+50~100 ms  |  attachBaseContext + libnesec init 起
T+100 ms  |  **Package quota 撞顶，Toast 已经被弹了 5 次**
T+150~200 ms  |  MMKV initialize 之类
T+250~300 ms  |  SIGSEGV in InitTaskSchedul 死掉

**Toast 在 process 启动 100 ms 内就触发， 而且不止一次， 是一个循环在反复弹** 。这跟 libpoison +800 ms
的校验时序对不上 —— 这条 Toast 不是 libpoison 触发的， 是更早的另一条独立校验。

####  反向逆向：从 Toast 文本找 Java 调用者

Toast 是 Java 组件，触发它必须有一行 ` Toast.makeText(ctx, "请前往官方渠道下载正版APP", N).show() `
。这串字符串必然在某处可以找到 —— 反向找到它，就找到了触发位置。

按显性目标依次扫：

扫描位置  |  工具  |  结果
---|---|---
base.apk 内 21 个 classes*.dex  |  python bytes find  |  **全 NF**
` lib/arm64-v8a/*.so ` (libpoison / libnesec / libnesdk / libmsaoaidsec / libsgsecuritybody)  |  python bytes find  |  **全 NF**
resources.arsc  |  UTF-8 + UTF-16 find  |  **NF**
libpoison_dumped.bin （992 KB 运行时解密镜像）  |  bytes find + libnesec XOR key 0x81-0x88 全 8 种起始偏移  |  **明文 NF + 加密 NF**
libnesec_dumped.bin (1 MB)  |  同上  |  **NF**
base.apk 整体 （190 MB, 扫到 assets）  |  bytes find  |  **NF**

到这里所有静态位置都被排除了。 **这串字符串既不在任何 dex 业务代码里， 也不在任何 native lib 的 .rodata 里， 也不在
resources 表里， 也不在易盾壳同款 XOR 加密形态里** 。

####  进 process 内存找

字符串 logcat 都接到了完整 UTF-8 text, 说明触发那一刻字符串必然以某种形式在 cloudmusic process 内存里出现过。写了个
` mem_grep_anon ` —— ptrace ATTACH 后扫所有 anonymous + cloudmusic 相关的 r-- 段， 主要是
ART 解密后的 dex / dalvik anonymous heap / app 数据目录映射， 搜 "请前往" 这 3 字 UTF-8 序列：


    ATTACH 50 次(每次 monkey 拉起 → process 启动 → ATTACH → scan)
    扫描段总数:1500+
    hits:0


试了 **直接 ATTACH** , PID 一出现就抓， ~T+10 ms, 和 **延迟 ATTACH** , usleep 150 ms 等
Toast 触发后再抓， ~T+150 ms 两种时机 —— 都是 0 hit。

####  唯一剩下的解释

字符串的存活窗口被压扁到几乎不存在。最可能的形态：

  * libpoison, 或其加载的 secondary dex, 在 native 层把某串字节通过 ` JNIEnv->NewStringUTF(env, decoded_bytes) ` 现场构造成 jstring
  * jstring 被 ART 放进 string intern table, 几乎立刻作为参数传给 ` Toast.makeText ` → ` show ` → binder transaction → system_server
  * transaction 完成后， jstring 在 ART heap 里短暂存在 —— 然后 process 在 +200 ms 内就 SIGSEGV 死了， GC 都没机会跑

只有在 ` Toast.makeText ` 那一行调用到 binder transact 之间几十微秒的窗口， 字符串才以明文形式存在。ptrace 这种
ms 级 attach 工具根本卡不到那个窗口。

####  那 frida 呢？

Java hook ` Toast.makeText ` 的 overload, onEnter 抓 arguments + `
Thread.currentThread().getStackTrace() ` —— 这是 hook Toast 触发链最直接的方式。但回到 第 5.9
节 那个老问题：

  * frida-agent 在 T+50 ms 被 libnesec inline svc kill 干掉
  * Java.perform 在 ART vm available 时才能跑， > T+50 ms
  * **window 是负的**

` Java.perform ` 这条线在网易这种加固面前根本就装不上。

####  复盘一下

反向逆向到死胡同。证据链梳理一下就是这么几条：

静态一条都没找到。21 个 dex、5 个 lib、resources、libpoison 和 libnesec 的 dump、整个 APK
任何位置都搜过，包括用易盾同款 XOR key 加密形态找过，全 NF。

动态也抓不到。50 次 ptrace ATTACH，1500+ 段扫描，全 0 hit，无论 ATTACH 时机怎么调都一样。

结构性原因是这个字符串只在 ` NewStringUTF + Toast.makeText + binder transact `
那几十微秒的窗口里存在，native 层构造出来立刻扔给 binder，即用即销。

frida 也救不了。Java hook 装不上的根因跟第 5.9 节一样， ` Java.perform ` 比 libnesec 的 frida 检测晚
50 ms 数量级，window 是负的。

看到这里可以确认一件事：网易这一档加固不只是 "杀掉重打包 process"，它连 "想从弹窗反向找代码"
这种逆向尝试都一起堵死。静态路径全关，动态路径因为时间窗口太短也全关，从这个角度它是完整的工程封锁。

###  5.11 尝试过但没成的其他路径

除了 第 5.9 节-5.10 的正面强攻，还试过下面几条：

**尝试 1：hook Java` Signature.toByteArray ` \+ ` PackageInfo.signatures ` 替换 **
, 就是 QQ 音乐方案

结果：没有明显改善。App 仍卡在 Splash 之后。跑 v23 时 send 事件清楚显示 hook 命中并替换成功：


    [evt] {"tag": "sig_replace_Proxy"}
    [evt] {"tag": "sig_replace_Proxy"}
    [evt] {"tag": "sig_replace_APM", "flags": "40"}


3 次成功替换。但同一秒 logcat 里 POISON log 打出来的 **仍然是重签 hash** :


    D POISON: APK Signature:d2a93e35348b7c414dddf5caf64108ad


为什么 Java hook 命中 3 次但 POISON 打出来还是重签 hash? 因为 **libpoison 早于 Java hook
装上就完成了校验** 。Java.perform 在 ART VM ready 才能装，大概在 T+100~200 ms; libpoison 在
T+800 ms 加载， 但 libnesec 早在 T+50 ms 就 kill 了 frida-agent。Java hook 装上时，frida-
agent 已经死了，hook 的回调进入 dead V8 runtime，等于空跑。

结论： **网易云音乐的主 Activity 起不来根因不在 Java 签名校验路径可否被 hook 上，而在 frida-agent 是否活得过
libnesec 反 frida 检测** 。

**尝试 2：hook` startActivity ` 相关 API 追踪 **

  * ` Instrumentation.newActivity(ClassLoader, String, Intent) `
  * ` ActivityManagerService ` / ` IActivityManager.startActivity `
  * ` Activity.performStart ` / ` Activity.onCreate `

观察： ` startActivity(new Intent(this, MainActivity.class)) ` 确实被调用了、也进了 `
ActivityManager.startActivityAsUser ` 、returnCode 是 0, 成功。但 **回到 UI 层就没
Activity 出现** 。

结论：可能是加固层在 ActivityThread 的 ` handleLaunchActivity ` 里拦了，或者 MainActivity 的 dex
加载失败。要验证要 hook 到 ART 层的 ` dexFileLoad ` ，本地环境不好搞。

**尝试 3：dex 短路——修改 wrapper 直接反射加载真业务 Application**

思路：既然 wrapper ` MyApplication ` 是 Nesec 加固注入的、真业务是 ` CloudMusicApplication ` ,
我们改 wrapper 的 ` attachBaseContext ` , 让它跳过加固校验步骤， 直接反射加载真业务 App。


    # MyApplication.smali (修改后)
    .method protected attachBaseContext(Landroid/content/Context;)V
        .locals 3

完整 smali 约 300 行。

结果： Application init 完成 —— 微信 SDK / HMS / MMKV / DataBinding 全部初始化
OK，比作者版位置深了一层。这就是 第 5.6 节 的观察。但 **主 Activity 仍不起来** —— dex 短路只跨过了 wrapper
层的校验， libnesec 在 App 运行到某个业务事件时会二次校验， 我们的 dex 短路无效。

结论：dex 短路是 **必要不充分** —— 把 wrapper 里的 Nesec 校验绕过了， 但绕不过 libnesec 加载后的运行时校验。

**尝试 4：手动加载 MainActivity dex**

想法：绕过加固的动态解密，直接把 ` MainActivity ` 从原始 APK 里 extract 出来，用 ` PathClassLoader `
手动加载。

问题： ` MainActivity ` 依赖大量业务类， Fragment / ViewModel / Repository / Retrofit /
OkHttp / ..., 单独加载没意义，得连整个业务 dex 都加载。而业务 dex 又依赖 Application 提供的 context 和 DI
容器。 **不可行** 。

**尝试 5：LSPatch / Xposed hook**

尝试用 LSPatch 打成 magisk 模块方式注入。失败 —— LSPatch 需要目标 App 允许 hook （AndroidManifest
里能读到某些 flag）, 加固过的 App 通常不满足。

**尝试 6：Frida gadget 内嵌**

把 ` frida-gadget.so ` 编进 APK, 做成不依赖 frida-server 的方案。失败 —— libnesec 加载时会枚举同进程
module, 识别到 ` frida-gadget ` 相关字符串直接自杀。

**尝试 7：脱壳 + 分析 libnesec 关键函数**

尝试用 ` frida-dexdump ` 脱壳出真正的业务 dex, 然后 IDA 分析 libnesec + libpoison
找所有校验点。进展：dex 能拿出来， libpoison 的关键分支能定位到 0x7af3c, 但 **libnesec 内代码经过 OLLVM 混淆**
, 逆向工作量至少 10 天起步。没继续。

**尝试 8（未做，但路径已经清楚）：libnesec Java 层校验的 hook 方案**

![链路 B libnesec Java 层校验的 3 层 hook
链路 B libnesec Java 层校验的 3 层 hook 配方

第 5.1.5 节 讲过 libnesec 里有一条独立于 libpoison 的 Java 层校验链： ` MyJni.load ` 里 JNI 反射调
` PackageManager.getPackageInfo ` → 白名单比对 + apksignaturekillerplus 特征扫描 → 失败调
` NEDialog.showRiskMessage ` 弹 toast → ART 自然死。这条 第 5.5 节 完全没管，是主 Activity
起不来最主要的嫌疑。

**分别定点 hook** 实证过这套的可打性： **libpoison 3 层 byte patch + libnesec 的 Java 层`
NEDialog.showRiskMessage ` hook ** （这就是「4 层 hook 配方 ` bypass_sig.js ` 」），让进程"从
+1.4s 死延长到 10+ 分钟不死、所有签名校验 die 全部消失、toast 不再弹" —— 关键的一层就是给 libnesec 那条 Java
校验的失败出口打上补丁。

思路分 3 层（就是上图所示的对称补丁蓝图）：

  1. **hook` NEDialog.showRiskMessage(int, String) ` ** （最简单也最直接）—— 让弹窗不弹、失败流程不进入。整个 Java 层校验失败最终收口在这里，只要让它 no-op，杀链就断了 —— 这是 4 层配方里最关键的一层。

        Java.perform(function () {
        var NEDialog = Java.use('com.netease.xxxxxx.NEDialog');   // 具体类名以脱壳后 jadx 为准
        NEDialog.showRiskMessage.overload('int', 'java.lang.String').implementation = function (code, msg) {
            send('[skip] NEDialog.showRiskMessage(' + code + ', "' + msg + '")');
        };
    });


  2. **hook libnesec 的 JNI 反射入口` MyJni.load ` ** （更彻底）—— 让整个 Java 层校验根本不发生。这个是从 native 桥这一层拦，不进入后续 ` getPackageInfo ` → 白名单对比这条链。

  3. **精细替换` PackageManager.getPackageInfo ` ** （QQ 音乐方案的克制版）—— 关键区别是 **不能全局 hook** 。第 5.1.5 节 讲的 CheckJNI 陷阱：如果 Java 层返回原版字节，业务 native 代码通过 CheckJNI 读到的是重签字节，两次不一致会直接 abort 杀进程。所以要精细化到"只在 libnesec 的调用栈里才替换" —— 用 ` Thread.currentThread().getStackTrace() ` 判断 caller 是不是 libnesec 相关类，是才替换。

**没做的原因** ：跟 第 5.9 节 讲过的一样 —— frida-agent 在 T+50ms 被 libnesec 的 inline svc
kill 干掉，Java.perform 装上的时机 > T+50ms， **window 是负的** 。要装上上面 3 层 hook 的前提是 **用
patched-frida-server** （改了特征字符串、让 libnesec 认不出来），本篇的设备上没配这一层，所以尽管路径清楚，Frida
环境本身仍然进不去。

这条路径证明了一件事： **Frida 单点 hook 打不穿 Nesec，但不是因为"找不到 hook 点"** （hook 点是清楚的）
**，而是因为"hook 装不上去"** （frida-agent 活不过 libnesec 的反检测）。要打通只有两条路：一是用 patched-
frida-server + Magisk Hide 之类的组合先解决注入时机、再走上面 3 层 hook 配方；二是彻底放弃 Frida，走
ZygiskFrida / 静态 patch。都不在纯 Frida 的能力范围。

###  5.12 为什么最终判断这条路不通（暂时）

成本估算：

要做到跟 QQ 音乐一样 "App 完全可用"，需要：

  1. **脱壳出完整业务 dex** —— 可能要跑通 frida-dexdump 全套，2-3 天。
  2. **逆向 libnesec + libpoison 找全校验点** —— OLLVM 混淆下 10-20 天。
  3. **补 Java 层 + native 层双路径 hook** —— 3-5 天。
  4. **验证主 Activity 起来后各个业务功能** , 登录、播放、下载、评论、社区， 都可能触发单独的校验点，每个都要单独排查， **无限时间** 。
  5. **持久化到不依赖 Frida 的补丁** —— smali 改动大概 20+ 处， native 补丁涉及 5-10 处，还得处理版本升级， **长期维护成本** 。

产出评估：

如果只是证明 Frida 单点 hook 打不穿 Nesec 加固 + 深度联防， 我们已经证明了。作者花 3 天 45 层 patch 到 splash,
我们精简到 3 层 patch 到 Application init 之后， 说明加固层的对抗已经把 Frida 的实用性压到了几乎为零 —— 不是因为哪个
patch 找不到， 而是因为每个校验点都要单独 hook、hook 之间会互相干扰、任何一个漏掉整个链就断。

判断：

这条路目前不通。不是 "永远不通"，是 "投入产出比不合适"。要真做的话得从加固侧入手， 脱壳 → 完整逆向 → 补丁到非加固版本， 而不是继续加
Frida patch。

###  5.13 跟 QQ 音乐对比：为什么一个能救一个不能

![网易云音乐 9.2.80 vs QQ 音乐 13.8.1 · 8
网易云音乐 9.2.80 vs QQ 音乐 13.8.1 · 8 维度并列对比

图里 8 个维度并排看下来，可以把差别浓缩成一条： **QQ
音乐把校验放在业务请求这一步、失败也不自杀，所以留给你反反复复调试的时间窗口；网易云音乐把校验放在 Application init 期、失败直接

遇到 App 秒退不要盲加 patch。作者版 45 层的教训明摆着：每加一层 patch 都要验证进度有没有推进，没有推进的 patch
就是干扰，甚至可能反过来把 App 弄崩。反过来说，如果去掉某个 patch App 反而活得更久，那个 patch 就是问题本身 —— 我们发现的 `

交叉对比、发现前面几条推断有错、修正回来，到把 15 条白名单 MD5 全列出（5.8），到拿着精确机制再战 frida 仍然失败（5.9），到反向从
Toast 追字符串追到死胡同（5.10），到第二次交叉对比推翻"libnesec 是裁判"改成"双链路独立并行"（第 5.1.5 节 重写）——

##  六、闪退了怎么办

前面两个实战里，我们至少踩过 10 种以上 "App 装了就退" 的坑。这一节把排障流程整理成 checklist，遇到闪退时按顺序过一遍 ——
大多数情况能在半小时内定位是哪一层的问题。

###  6.1 第一步：logcat 抓到 App 死之前的最后几行

任何调试的第一步都是这个：


    # 清 logcat 缓冲
    adb logcat -c

    # 启动目标 App
    adb shell am force-stop com.example.target
    adb shell monkey -p com.example.target -c android.intent.category.LAUNCHER 1

    # 等 5-10 秒, App 大概率已经死了
    sleep 10

    # 抓最后 500 行
    adb logcat -d -t 500 > /tmp/crash.log

    # 关键关键字过滤
    grep -iE "Fatal|AndroidRuntime|died|killed|SIGSEGV|SIGABRT|Signature|com.example.target" /tmp/crash.log


四种典型模式：

**模式 A：Java` Fatal Exception ` **


    E AndroidRuntime: FATAL EXCEPTION: main
    E AndroidRuntime: Process: com.example.target, PID: xxx
    E AndroidRuntime: java.lang.RuntimeException: Unable to start activity ...
    E AndroidRuntime: Caused by: java.lang.SecurityException: Signature mismatch
    E AndroidRuntime:     at com.example.SignatureCheck.verify(SignatureCheck.java:45)


Java 层校验。看栈里的类名 ` SignatureCheck ` , 去 jadx 反编找到那个方法，然后走 第 2.3 节 打法 A/B/C。

**模式 B：native` Fatal signal ` **


    F libc    : Fatal signal 6 (SIGABRT), code -1 (SI_QUEUE) in tid xxx (com.example.target)
    F DEBUG   : Cmdline: com.example.target
    F DEBUG   : pid: xxx, tid: xxx, name: com.example.target  >>> com.example.target <<<
    F DEBUG   : signal 6 (SIGABRT), code -1 (SI_QUEUE), fault addr --------
    F DEBUG   :   #00 pc 0000000000012345  /apex/com.android.runtime/lib64/bionic/libc.so (abort+68)
    F DEBUG   :   #01 pc 000000000006abcd  /data/app/xxx/lib/arm64/libsentry.so (_ZN12sentry_crash...)


Native 层校验。看栈里的 so 名 ` libsentry.so ` —— 那就是保护 so, 需要按 第 3 章 的 native 层方法处理。

**模式 C：` Process died ` 没有原因 **


    I ActivityManager: Process com.example.target (pid ...) has died: fg TOP
    I libprocessgroup: Successfully killed process cgroup uid ... pid ...


后面 **没有 Fatal signal, 没有 Java exception** —— 就这么消失。三种可能：

  * App 调了 ` System.exit(0) ` / ` Process.killProcess ` —— 静默退出。
  * App 用 ` syscall(SYS_exit_group) ` 直接 syscall 出去， 绕过 libc export —— ` abort ` hook 拦不到。
  * 系统或 launcher 因为其他原因杀了它， 内存不足、targetSdk 兼容问题 —— **少见** 。

排查思路：装我们的 Frida hook 先拦 Java ` System.exit ` \+ ` Process.killProcess `
；如果还是死，装 ` abort ` / ` _exit ` / ` exit_group ` / ` _Exit ` 全套；再死就上 ` __exit `
/ ` __exit_group ` 私有 symbol；还死就上 syscall hook。

**模式 D：` INSTALL_PARSE_FAILED_NO_CERTIFICATES ` 或 `
INSTALL_FAILED_UPDATE_INCOMPATIBLE ` **


    Failure [INSTALL_PARSE_FAILED_NO_CERTIFICATES: ...]
    Failure [INSTALL_FAILED_UPDATE_INCOMPATIBLE: Existing package com.example.target
             signatures do not match newer version]


安装阶段就挂了 —— 不是 App 层的事。 ` NO_CERTIFICATES ` 是 apktool 打包破坏了签名， 用 ` apksigner
sign ` 重签就行； ` UPDATE_INCOMPATIBLE ` 是原版还装着且签名不同，需要 ` adb uninstall ` 后再装。

###  6.2 第二步：判断是 "原 App 的问题" 还是 "你的 hook 引起的"

装了 Frida hook 后 App 秒退，很容易以为是 "App 检测到 Frida 主动自杀"，但多数情况其实是你的 hook 破坏了 App
的正常逻辑。

判断法： **先关掉所有 hook 装机一次， 看 App 会不会退** 。


    # 不 attach Frida, 纯净装机
    adb shell am force-stop com.example.target
    adb shell monkey -p com.example.target -c android.intent.category.LAUNCHER 1


如果不装 hook 就存活， 哪怕业务功能不 work —— 问题在你的 hook。逐个 hook 关掉重新试，找到哪个 hook 是破坏源。

如果不装 hook 也秒退 —— 那才是 App 自身的自杀路径， 加固层校验、native abort, 按 第 3 章 处理。

大多数 "装了 hook 反而 App 死得更快" 的场景是：

  * 全局 ` setExceptionHandler ` 拦下所有异常 —— ART 隐式 null check 变成未处理 SIGSEGV, 第 3.4 节 详述。
  * ` Interceptor.replace ` 替换了系统方法但签名不对 —— ART 校验失败 abort。
  * Hook 数量过多导致 JIT 生成的 stub 代码放不下 CodeCache, OOM。

###  6.3 第三步：Frida spawn vs attach

spawn 模式：Frida 在 App 主线程执行第一行代码之前注入。能拦到启动早期的所有调用。


    pid = device.spawn(['com.example.target'])
    session = device.attach(pid)
    scr = session.create_script(open('hook.js').read())
    scr.on('message', lambda m, _: print(m))
    scr.load()
    device.resume(pid)   # 关键：resume 才让 App 继续


attach 模式：App 已经跑起来后 hook。启动早期的调用全部漏掉。


    pid = int(subprocess.check_output(['adb', 'shell', 'pidof', 'com.example.target']).strip())
    session = device.attach(pid)
    scr = session.create_script(open('hook.js').read())
    scr.load()


**QQ 音乐案例， 第 4.7 节-4.8** ：spawn 模式看到 4 次 ` toByteArray ` 调用， attach 模式一次都看不到
—— 因为 QQ 音乐启动早期读一次签名然后缓存到字段， 之后全用缓存。必须 spawn 才拦得到关键调用。

规则：

  * 不确定就用 spawn。反正 attach 能做的 spawn 都能做，反过来不行。
  * spawn 有额外开销 —— App 启动多几百 ms、需要处理 ` enable_spawn_gating ` 拦子进程 —— 但换来的是不漏调用。
  * debug 阶段推荐 spawn，性能测试用 attach。

###  6.4 第四步：hook 常驻保活

常见的坑： ` frida -l script.js -q com.example.target ` 加载脚本， 秒退 —— 不是 App 退， 是
Frida 退。因为 Frida CLI 加载 script 后如果没有 ` -e ` 或 REPL, 立即返回， session 关闭， hook 全部
detach。

解法： 用 Python, ` time.sleep() ` 保活。


    # keepalive.py
    import frida, sys, time
    device = frida.get_device_manager().add_remote_device('127.0.0.1:27042')

    # 或 spawn 或 attach, 二选一
    mode = sys.argv[1]   # 'spawn' or 'attach'
    target = sys.argv[2]
    script_path = sys.argv[3]

    if mode == 'spawn':
        pid = device.spawn([target])
    elif mode == 'attach':
        import subprocess as sp
        pid = int(sp.check_output(['adb', 'shell', 'pidof', target]).strip())

    session = device.attach(pid)
    scr = session.create_script(open(script_path).read())
    def on_msg(m, _):
        if m['type'] == 'send': print('[S]', m.get('payload'), flush=True)
        elif m['type'] == 'error': print('[E]', m.get('description'), flush=True)
    scr.on('message', on_msg)
    scr.load()

    if mode == 'spawn':
        device.resume(pid)

    print(f'[PY] hook installed on pid={pid}, keepalive 30min', flush=True)
    try: time.sleep(1800)
    except KeyboardInterrupt: pass


用法：


    python3 keepalive.py spawn com.tencent.qqmusic qq_sig_full.js


Ctrl-C 才退出。 **长期调试的时候， 把 duration 改成 24 小时也没关系** 。

###  6.5 第五步：Frida script 里的坑

**坑 1：` console.log ` 覆盖问题 **

Frida 里 ` console ` 是只读全局， 不能重定义：


    // 错的
    var console = { log: function (s) { send(String(s)); } };
    // TypeError: 'console' is read-only

    // 对的：定义新变量
    var L = function (s) { send(String(s)); };


**坑 2：` Java.array ` 里 byte 的正负号 **

Java ` byte ` 是 signed, -128 到 127, JS number 是 unsigned, 0-255：


    // 错的
    var arr = Java.array('byte', [255, 128, 0]);
    // 报错或数据不对

    // 对的：>127 减 256 转 signed
    var arr = Java.array('byte', bytes.map(function (b) {
        return b > 127 ? b - 256 : b;
    }));


**坑 3：字段访问` .value ` **

Frida 里访问 Java 对象的字段用 ` .value ` ：


    // 错的
    pi.signatures = arr;
    // 只是 shadow JS 属性, Java 对象没改

    // 对的
    pi.signatures.value = arr;


**坑 4：` Signature.$new ` 参数类型 **

` android.content.pm.Signature ` 有多个 constructor overload：


    // 错的, 可能触发 ambiguous constructor 错误
    var sig = Signature.$new(bytes);

    // 对的：明确 overload
    var sig = Signature.$new.overload('[B').call(Signature, bytes);
    // 或简单情况直接
    var sig = Signature.$new(Java.array('byte', bytesConverted));


**坑 5：反射改私有字段的正确姿势**


    // 反射的正确姿势
    var fDet = si.getClass().getDeclaredField('mSigningDetails');
    fDet.setAccessible(true);
    var det = fDet.get(si);
    // ...


不能用 ` si.mSigningDetails.value ` —— ` mSigningDetails ` 是私有字段， Frida 的 `
.value ` 只能访问公共字段。必须走反射。

###  6.7 反检测与 "不要装那些不该装的"

**根检测** , 不是本章重点， 但作为 checklist 提一下：

  * Magisk Hide / Zygisk DenyList 隐藏 root
  * MoveCertificate 装 CA 到 apex, Android 14+ 必需
  * 别在 App 目录下留下 frida-agent / frida-gadget 等特征文件

**Frida 检测的常见路径** ：

  * 遍历 ` /proc/self/maps ` 找 ` frida-agent ` / ` frida-gadget ` / ` gum-js-loop ` 字符串
  * 遍历 ` /proc/self/task/*/status ` 找可疑线程名， ` gum-js-loop ` 、 ` gmain ` 、 ` pool-frida `
  * ` getprop ` 查 ` ro.debuggable `
  * 尝试 ` socket().connect('127.0.0.1', 27042) ` 检测 frida-server 端口
  * 检查 ` dlopen("libfrida-agent.so") ` 是否成功

对抗：改 frida-server 名字、改端口、编译时改 symbol 名 —— 需要另开一章讲。本节的 checklist 到 App 秒退但
logcat 里 ` frida-agent detected ` 或 ` unusual thread name ` 时，往这个方向查。

网易云音乐 第 5.9 节 里的 "frida-agent 50 ms 被 inline svc kill" 就是加固层反 frida 检测机制的极端表现
—— 用 patched frida-server + Zygisk DenyList + 修改所有 gum 相关字符串， 才能把 agent 撑到超过
libpoison 加载， 单纯换端口远远不够。

###  6.8 关掉 hook 的诊断法

"装了这个 hook App 死得更快" 时， **用二分法定位哪个 hook 是破坏源** ：


    // 主 script.js
    var STAGE = 3;   // 0 = 什么都不装, 1..N = 装到第几层
    if (STAGE >= 1) { /* patch 1: libpoison byte patch */ }
    if (STAGE >= 2) { /* patch 2: HandleUsingDestroyedMutex */ }
    if (STAGE >= 3) { /* patch 3: libc exit hooks */ }
    if (STAGE >= 4) { /* patch 4: Java exit hooks */ }
    if (STAGE >= 5) { /* patch 5: setExceptionHandler */ }   // ← 这条经常是问题源


  * STAGE=0 App 秒退 → App 自身问题， 跟 hook 无关
  * STAGE=3 存活但 STAGE=5 秒退 → patch 5 是破坏源

关键原则： **去掉 patch 反而让 App 活得更久时， 那个 patch 一定是问题** 。加固对抗里这种反直觉的场景不少见 —— **默认 "加
patch" 和 "减 patch" 都要试** 。

##  七、总结

签名校验绕过从最浅的一行 ` PackageManager.getPackageInfo `
到最深的加固分层校验，档次差异极大。跨样本对下来，比档次数字更好用的是两条判据。

一条是 App 会不会自杀。第 1、3 档不自杀，要么直接可用要么服务端拒服务，你可以一直挂着调；第 4、5 档一 fail 就 ` SIGABRT ` /
` System.exit ` / ` abort ` ，反馈周期从秒级恶化到 "每次崩溃从头再来"。这两种反馈节奏差好几个数量级，直接决定了你能调多深。

另一条是签名读取在哪一层。全在 Java 层的（第 1、2 档）你把 ` Signature ` 对象整体替换 + ` signingInfo `
覆盖就够；一旦下沉到 native 层（第 3 档以上）Java hook 从原理上拦不到 —— 比如小红书的 ` libsentry.so ` 直接 `
pread ` 读 APK signing block 里的 ` 0xF05368C0 ` chunk，Java 那边任何 hook 都碰不到那条路径。

拿到一个新 App，别看到 ` getPackageInfo ` 就先写 Java hook。先花 15 分钟走一遍第 1.4
节的三步判断，确认在哪一档，再选打法。

QQ 音乐这个 case 干净。签名校验就是 "客户端读一份字节塞进业务请求"，Java 把签名对象换成原版就完事了。最费时间的不是 hook
那几十行代码，是排除各种误导 —— ` code:0 ` 的假象、 ` http_proxy ` 让 WNS 疯狂重试的次生反应、"我的"
页面看起来正常但根本不发请求。踩过这些坑的价值不比最终 hook 代码小。

网易云音乐这个 case 是失败案例。Nesec 加固把校验点分散到 native、加密字符串、SEGV 陷阱、Tinker 反射链好几个位置，Frida
单点 hook 根本覆盖不全。我做到了 Application init 走完，主 Activity 起不来，比社区流传的 45 层 patch
版本挪了一小段，但离能用还差得远。这个 case 的价值不是 "打通了"，而是把 "打不通的成本" 量化到了具体数字：

ptrace + process_vm_readv 自己 dump 出 libpoison、反汇编看到 15 条 strcmp
白名单，机制说清楚了。拿着精确机制写第 15 条白名单 MD5 的 patch，frida-agent 在 50 ms 就被 libnesec inline
svc kill 干掉，而 libpoison 加载在 800 ms，时间窗口是 16 倍的负值，结构上就不可绕。反向从 Toast 弹窗追字符串，dex
/ lib / resources / 内存扫描全 NF，静态动态都抓不到。商业级加固能让一段 UI 文本只在几十微秒的窗口里存在，任何 batch-
level 工具都摸不到。

如果真要打通，脱壳 + 完整 native 逆向数周起步。如果只是研究，我这次的失败记录已经说明白了 "Frida 单点 hook 打不穿深度联防"。6
轮迭代这一节我也没删，从推断，到自己 dump 拿证据，到交叉对比修正推断，到拿精确机制再战 frida 仍然失败，到 Toast
反查到死胡同，到再一次对比推翻"libnesec
是裁判"改成"双链路独立"，每一轮都把上一轮的几条断言推翻或精化。如果一篇逆向文章从头到尾说自己全对，八成里面有错没被发现。

最后再念一遍那句：唯一变了的是签名，问题只可能出在签名。QQ 音乐那些走弯路的时段，回到这句话就能把方向拉回来。以后开新 case
之前，这话先在心里过一遍。
