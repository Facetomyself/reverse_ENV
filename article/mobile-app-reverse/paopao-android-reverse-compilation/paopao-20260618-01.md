##  一、问题： ` base + 偏移 ` 为什么扛不住更新

未导出函数的标准定位姿势：


    var module = Process.findModuleByName("libnative.so");
    var funcAddr = module.base.add(0x4A8C);   // 0x4A8C 来自 IDA 静态分析
    Interceptor.attach(funcAddr, { /* ... */ });


` module.base ` 是运行时拿的（ASLR 每次随机，没问题），但 ` 0x4A8C ` 这个偏移是 **死的**
——它是「这个版本、这次编译、这个 ABI」的产物。下面任何一件事发生，它就失效：App
升级重编、NDK/编译器换版本、换设备拿到不同变体的包、加固重排函数顺序。

失效的代价不是报错，而是 **静默失灵** ： ` attach ` 到错误地址，可能什么都不发生，也可能崩溃。你只能重开 IDA 重找。

根因一句话： **硬编码偏移记的是「位置」，而位置是最易变的东西。**
解法是改记「内容」——函数的字节指纹、它引用的字符串、它调用的导入、它和稳定锚点的关系。这些在版本间的稳定性远高于绝对位置。

##  二、定位的本质：一张全手段排序表

把所有手段按「抗变化能力」从强到弱排开，就是一张选路图：

65% → 相对锚点 45% → 硬编码偏移

读法： **能用上面的就别用下面的** 。导出符号最稳但未导出函数没有；于是先零成本试符号表；不行就上字符串 /
调用点交叉引用（这两个最适合未导出函数）；再不行退到特征码；都没有才用相对锚点；硬编码偏移只配做最后兜底，并且必须打「会随版本失效」的警告。

第 9 章把这些组装成一个 ` resolveFunction ` 框架，第 10 章在真机上跑通并用符号表验证。

##  三、先免费试一把：符号表 ` enumerateSymbols `

###  3.1 别急着扫字节——很多函数的名字其实还在

「未导出」不等于「没名字」。ELF 有两张符号表： ` .dynsym ` （动态符号，导出/导入走它， ` findExportByName `
查的就是它）和 ` .symtab ` （完整符号表，包含 **未导出的局部函数名** ）。只要 SO 没被 strip， ` .symtab `
里就躺着大量内部函数的真名， ` Module.enumerateSymbols() ` 直接拿得到——根本不用扫字节。


    function locateBySymbol(module, nameOrRegex) {
        var hit = null;
        module.enumerateSymbols().forEach(function(sym) {
            if (hit || sym.type !== "function" || sym.address.isNull()) return;
            var ok = (nameOrRegex instanceof RegExp)
                ? nameOrRegex.test(sym.name)
                : sym.name.indexOf(nameOrRegex) !== -1;
            if (ok) hit = { name: sym.name, address: sym.address, global: sym.isGlobal };
        });
        return hit;   // { name, address, global } 或 null
    }

    // 用法：把所有名字里带 encrypt/aes/sign 的函数列出来
    module.enumerateSymbols()
        .filter(function(s){ return s.type === "function" && /encrypt|aes|sign/i.test(s.name); })
        .forEach(function(s){ console.log(s.name + " @ " + s.address + (s.isGlobal ? "" : " (local)")); });


###  3.2 顺手： ` ApiResolver ` 按 glob 批量匹配导出/导入

如果只想按通配符快速匹配导出或导入（不是局部符号）， ` ApiResolver ` 比手写循环顺手：


    var resolver = new ApiResolver("module");
    resolver.enumerateMatches("exports:libnative.so!*encrypt*").forEach(function(m) {
        console.log(m.name + " @ " + m.address);
    });
    // imports:libnative.so!*  可列出该模块的导入


###  3.3 边界

  * 重度加固 / 商业 App 的 SO 通常 strip 掉 ` .symtab ` ，这时 ` enumerateSymbols ` 只剩动态符号（≈ 导出），帮不上未导出函数——这正是你需要后面几章的时候。
  * 它零成本：一行调用，命中就省下全部字节级工作。所以永远先试它，再决定要不要动刀。
  * ` isGlobal:false ` 的就是局部符号（未导出），正是我们要的。

##  四、特征码定位（Memory.scan）

###  4.1 原理

函数头那几条指令（保存寄存器、建栈帧）和函数里的固定常量，字节序列往往跨版本稳定。把这段字节当「指纹」扫内存，就能不依赖偏移找到函数。Frida 用 `
Memory.scan ` / ` Memory.scanSync ` ，模式串支持 ` ?? ` 通配整字节、 ` 4? ` 通配半字节（nibble）：


    var pattern = "fd 7b bd a9 fd 03 00 91 ?? ?? ?? ??";   // ?? = 整字节任意


###  4.2 抠特征码：哪些字节必须通配

规则一条： **凡是带「相对偏移」或「绝对地址」的指令，整条 4 字节都通配；其余保留。**

带地址/偏移的指令编码里嵌的就是「位置」，而位置会变。ARM64 里：

  * ` BL ` / ` B ` / ` B.cond ` ：相对跳转偏移 → 整条通配（连最高字节都含偏移位）。
  * ` ADRP ` / ` ADR ` ：页地址 → 整条通配。
  * ` LDR Xt, =label ` （literal 加载）：字面量池偏移 → 整条通配。
  * ` STP ` / ` MOV ` / ` ADD #imm ` ：操作码、寄存器号、常量稳定 → 保留。

IDA 里看到：


    STP  X29, X30, [SP, #-0x30]!   ; FD 7B BD A9   ← 保留(0x30 栈帧大小编码在 BD)
    MOV  X29, SP                    ; FD 03 00 91   ← 保留
    STP  X20, X19, [SP, #0x20]      ; F4 4F 02 A9   ← 保留
    BL   sub_2100                   ; ?? ?? ?? ??   ← BL 含相对偏移,整条通配
    ADRP X8, #off                   ; ?? ?? ?? ??   ← ADRP 含页地址,整条通配


特征码： ` fd 7b bd a9 fd 03 00 91 f4 4f 02 a9 ?? ?? ?? ?? ?? ?? ?? ?? `

栈帧大小 ` 0x30 ` 编码在第一条的 ` BD ` 字节里，若不同版本局部变量变多/变少， ` BD ` 会变，可酌情再通配成 ` fd 7b ??
a9 ` ——代价是特征码变弱、易撞车。

###  4.3 代码


    function locateByPattern(module, pattern) {
        var hits = [];
        module.enumerateRanges("r-x").forEach(function(range) {     // 只扫可执行段
            hits = hits.concat(Memory.scanSync(range.base, range.size, pattern));
        });
        if (hits.length === 0) { console.log("[pattern] 未命中"); return null; }
        if (hits.length > 1)  console.log("[pattern] 命中 " + hits.length + " 处,特征码不唯一");
        return hits[0].address;
    }


###  4.4 特征码比你想的更脆

很多教程把特征码吹成「一劳永逸」，真相是它 **只对「函数体没被重新生成」有效** 。下面这些都会让字节指纹失效，而它们在版本更新里很常见：

  * 换 NDK / clang 版本：寄存器分配、指令选择、调度都可能变，连「稳定」的函数头都会变样（比如 ` STP X20,X19 ` 变成 ` STP X21,X20 ` ）。
  * 内联决策变化：同一函数这版被内联、下版独立成体，整段消失或重组。
  * PGO / LTO：跨函数优化会大改布局。

所以特征码是「中上」不是「最强」——它排在符号表、字符串/调用点交叉引用之后。当函数有可用字符串或独特调用时，优先用那两个；特征码留给「纯计算、无字符串、调用的又都是常见
libc」的函数。写特征码时多带几条稳定指令换唯一性，必要时用半字节通配（ ` 4? ` ）对单条指令做精细放行。

##  五、字符串交叉引用定位

###  5.1 原理

函数常引用唯一字符串（日志、错误信息、JSON key、算法名），字符串几乎不随版本变。流程：


    找到字符串地址 → 找到引用它的指令 → 从引用点回溯到函数入口


全程不依赖固定偏移，是未导出函数最可靠的自动定位方式之一。要做三步内存解析，正好用上第 20 篇的汇编知识。

ADRP+LDR（间接）找到引用点 0x1B38C，第三步从引用点向上回溯 BTI/PACIASP/STP 序言定位到函数入口

###  5.2 第一步：扫出字符串地址


    function stringToPattern(s) {
        return Array.prototype.map.call(s, function(ch) {
            return ("0" + ch.charCodeAt(0).toString(16)).slice(-2);
        }).join(" ");
    }

    function findStringAddress(module, str) {
        var hits = Memory.scanSync(module.base, module.size, stringToPattern(str));
        return hits.length ? hits[0].address : null;
    }


###  5.3 第二步（直接地址）：解码 ADRP + ADD

ARM64 取一个全局地址（如 ` .rodata ` 里的字符串），最常用 **ADRP + ADD** ： ` ADRP ` 算出 4KB 页基址， `
ADD ` 加上页内低 12 位偏移。所以「引用字符串的指令」= 一对 ` ADRP + ADD ` ，使 ` 页基址 + imm12 == 字符串地址 `
：


    // 把整段可执行内存读进 Uint32Array 逐指令解析(ARM64 小端,设备上字节序天然对)
    function scanCodeForRefs(module, onInsn) {
        module.enumerateRanges("r-x").forEach(function(range) {
            var buf;
            try { buf = range.base.readByteArray(range.size); } catch (e) { return; }
            var code = new Uint32Array(buf);
            for (var i = 0; i + 1 < code.length; i++) {
                if (onInsn(code, i, range.base.add(i * 4)) === true) return;   // 返回 true 提前停
            }
        });
    }

    // 解出 ADRP 的目标页基址(返回 {rd, page})
    function decodeAdrp(ins, pcAddr) {
        if (((ins & 0x9F000000) >>> 0) !== 0x90000000) return null;     // 不是 ADRP
        var rd    = ins & 0x1f;
        var immlo = (ins >>> 29) & 0x3;
        var immhi = (ins >>> 5) & 0x7ffff;
        var imm   = (immhi << 2) | immlo;
        if (imm & 0x100000) imm -= 0x200000;                            // 符号扩展 21 位
        var page  = pcAddr.and(ptr("0xfffffffffffff000")).add(imm * 0x1000);
        return { rd: rd, page: page };
    }

    // 在「ADRP+ADD 直接地址」里找指向 targetAddr 的引用,返回 ADRP 地址
    function findAdrpAddRef(module, targetAddr) {
        var result = null;
        scanCodeForRefs(module, function(code, i, pcAddr) {
            var adrp = decodeAdrp(code[i], pcAddr);
            if (!adrp) return;
            for (var k = 1; k <= 4 && (i + k) < code.length; k++) {
                var ins2 = code[i + k];
                // ADD(立即数,64位,无移位): (ins2 & 0xFFC00000) == 0x91000000
                if (((ins2 & 0xFFC00000) >>> 0) !== 0x91000000) continue;
                if (((ins2 >>> 5) & 0x1f) !== adrp.rd) continue;        // 接着同一寄存器
                var imm12 = (ins2 >>> 10) & 0xfff;
                if (adrp.page.add(imm12).equals(targetAddr)) { result = pcAddr; return true; }
            }
        });
        return result;
    }


> 性能： ` scanCodeForRefs ` 把整段可执行内存逐指令过一遍。对 libil2cpp 这种几十 MB 的大 SO，JS
> 层逐条解析会明显变慢——可先缩小到已知大致区域再扫，或把扫描逻辑下沉到 CModule（C 层）提速。

###  5.4 第二步（间接地址）：ADRP + LDR

直接地址用 ` ADD ` ，但全局指针变量 / GOT 槽很常见，用的是 ` ADRP + LDR ` ： ` ADRP ` 算页基址， ` LDR
Xt, [Xn, #imm] ` 从 ` 页基址 + imm*8 ` 这个槽里把指针读出来。所以匹配方式不同：算出槽地址，读槽，看槽里存的是不是
targetAddr：


    // 在「ADRP+LDR 间接地址」里找最终指向 targetAddr 的引用
    function findAdrpLdrRef(module, targetAddr) {
        var result = null;
        scanCodeForRefs(module, function(code, i, pcAddr) {
            var adrp = decodeAdrp(code[i], pcAddr);
            if (!adrp) return;
            for (var k = 1; k <= 4 && (i + k) < code.length; k++) {
                var ins2 = code[i + k];
                // LDR(立即数,无符号偏移,64位): (ins2 & 0xFFC00000) == 0xF9400000
                if (((ins2 & 0xFFC00000) >>> 0) !== 0xf9400000) continue;
                if (((ins2 >>> 5) & 0x1f) !== adrp.rd) continue;
                var imm12 = (ins2 >>> 10) & 0xfff;
                var slot  = adrp.page.add(imm12 * 8);                   // LDR 偏移按 8 字节缩放
                try { if (slot.readPointer().equals(targetAddr)) { result = pcAddr; return true; } }
                catch (e) {}
            }
        });
        return result;
    }


> 这两类覆盖了绝大多数引用。还有 ` ADR ` （近距离直接）、字面量池等少数变体，思路一样、解码不同，留作扩展。

###  5.5 第三步：从引用点回溯函数入口（PAC / BTI-aware）

拿到引用指令地址后，往上回溯到最近的 **函数序言** 。但新编译器加了控制流防护，序言可能不是想象中那样——回溯前必须认识这几条：

  * 标准序言： ` STP X29, X30, [SP, #-N]! ` （小端 ` FD 7B ?? A9 ` ）。
  * 大栈帧：先 ` SUB SP, SP, #N ` （ ` (ins & 0xFF0003FF) == 0xD10003FF ` ）再 ` STP ` 。
  * PAC：开了指针认证的二进制，序言第一条常是 ` PACIASP ` （ ` 0xD503233F ` ）或 ` PACIBSP ` （ ` 0xD503237F ` ），STP 在它后面。
  * BTI：开了分支目标识别的，函数入口第一条是 ` BTI c ` （ ` 0xD503245F ` ）。

所以真正的函数入口可能在 STP 之前一两条。回溯时优先认 ` BTI ` / ` PAC ` ，没有再退到 STP：


    function backtrackToPrologue(fromAddr, maxBytes) {
        maxBytes = maxBytes || 0x600;
        for (var off = 0; off < maxBytes; off += 4) {
            var a = fromAddr.sub(off);
            var ins;
            try { ins = a.readU32() >>> 0; }
            catch (e) { return null; }   // 撞上未映射页,放弃回溯,退回引用点交人工判断

            if (ins === 0xd503245f || ins === 0xd503233f || ins === 0xd503237f) {
                return a;   // BTI c / PACIASP / PACIBSP —— 这就是真入口
            }
            // STP X29,X30,[SP,#-N]! : 低16位==0x7BFD 且 最高字节==0xA9
            if ((ins & 0xffff) === 0x7bfd && ((ins >>> 24) & 0xff) === 0xa9) {
                // 入口在 STP 之上可能还有 PACIASP、BTI(顺序: BTI -> PACIASP -> STP)
                // 循环回退,吃掉所有连续的 BTI/PAC,落到最顶那条才是真入口(只回看一格会漏掉 BTI)
                var e = a;
                while (true) {
                    var p;
                    try { p = e.sub(4).readU32() >>> 0; } catch (_) { break; }
                    if (p === 0xd503245f || p === 0xd503233f || p === 0xd503237f) e = e.sub(4);
                    else break;
                }
                return e;
            }
        }
        return null;   // 没找到就 null,交人工从引用点附近判断
    }


回溯的边界，别迷信：

  * 叶子函数（不调用别的函数）可能不保存 ` X30 ` ，没有标准序言，回溯会落空——这时退回「引用点地址」让人工对照 IDA。
  * 它取「最近的序言」，万一引用点前面正好是别的小函数尾巴，可能回溯过头。
  * 最稳的函数边界其实是符号表（第 3 章）——若 ` enumerateSymbols ` 给了 ` size ` ，可直接用 ` [address, address+size) ` 判定，不必启发式回溯。回溯是「没有符号表时」的退路。

把直接 / 间接两条路串起来：


    function locateByStringXref(module, str) {
        var strAddr = findStringAddress(module, str);
        if (!strAddr) { console.log("[xref] 没找到字符串: " + str); return null; }
        var ref = findAdrpAddRef(module, strAddr) || findAdrpLdrRef(module, strAddr);
        if (!ref) { console.log("[xref] 没找到引用 " + str + " 的指令"); return null; }
        return backtrackToPrologue(ref) || ref;   // 回溯失败就返回引用点供人工判断
    }


###  5.6 选字符串的提醒

  * 要够独特： ` "error" ` 、 ` "null" ` 到处都是，毫无定位价值。选完整错误信息、专有 key、算法标识。
  * 当心被加密：有些加固把明文字符串运行时才解出， ` .rodata ` 里没有明文，扫不到——退回特征码或调用点 xref。

##  六、调用点交叉引用：按「它调用了谁」定位

纯算法函数（自研加密、签名）常常没有特征字符串，但它一定调用了某些导入——比如加密函数往往是「唯一调用 ` AES_set_encrypt_key `
的那个」。「谁调用了某导入」是定位这类函数的杀手锏。两条路，动态那条又稳又简单，优先用。

###  6.1 动态：Hook 导入，让它自报家门

Hook 那个被调用的导入函数，第一次被调用时， ` this.returnAddress ` 就落在 **调用方**
（你的目标函数）体内，回溯一下就拿到目标函数入口。比静态解码 BL 简单得多，也更稳：


    // 找「目标模块里调用了 calleeName 的那个函数」
    function locateCallerInModule(targetModule, calleeName, onFound) {
        var callee = Module.findExportByName(null, calleeName);   // 导入的真实实现
        if (!callee) { console.log("[call-xref] 找不到 " + calleeName); return; }

        var lo = targetModule.base;
        var hi = targetModule.base.add(targetModule.size);

        var listener = Interceptor.attach(callee, {
            onEnter: function() {
                var ret = this.returnAddress;
                // 只认来自目标模块的调用,过滤掉其它库对同一导入的调用
                if (ret.compare(lo) >= 0 && ret.compare(hi) < 0) {
                    listener.detach();
                    var fn = backtrackToPrologue(ret) || ret;
                    console.log("[call-xref] " + calleeName + " 的调用方函数 @ " + fn);
                    onFound(fn);
                }
            }
        });
        console.log("[call-xref] 已挂 " + calleeName + ",在 App 里触发一次相关操作...");
    }


代价是 **需要 App 真的触发一次该调用**
（运行时方法的固有特点）。但对加密/签名这种「一操作就会跑」的函数，这通常正是你想要的——顺带还确认了它确实被用到。

> 注： ` Module.findExportByName(null, name) ` 这种两参全局形式在 Frida 17 已弃用，新版改用 `
> Process.getModuleByName(name).findExportByName(sym) ` 或 `
> Module.getGlobalExportByName(sym) ` 。本篇按 16.x 写法，17+ 照此替换即可。

###  6.2 静态：解码 BL 找调用点

不想等运行时触发，可静态扫 ` BL ` ，把目标解出来，看是否指向该导入：


    // BL: (ins & 0xFC000000) == 0x94000000; imm26 = ins & 0x03FFFFFF; 目标 = pc + (sext(imm26)<<2)
    function findCallersOf(module, calleeAddr) {
        var callers = [];
        scanCodeForRefs(module, function(code, i, pcAddr) {
            var ins = code[i];
            if (((ins & 0xFC000000) >>> 0) !== 0x94000000) return;     // 不是 BL
            var imm = ins & 0x03ffffff;
            if (imm & 0x02000000) imm -= 0x04000000;                   // 符号扩展 26 位
            if (pcAddr.add(imm * 4).equals(calleeAddr)) callers.push(pcAddr);
        });
        // 每个调用点回溯到所属函数入口,去重
        var funcs = {};
        callers.forEach(function(c){ var f = backtrackToPrologue(c); if (f) funcs[f.toString()] = f; });
        return Object.keys(funcs).map(function(k){ return funcs[k]; });
    }

槽取出真实现地址再 BR 间接跳到 libcrypto.so 的真实现;所以 BL 的目标是本模块 stub,而
Module.findExportByName 给的是 libcrypto 里的真实现——findCallersOf 要匹配 stub

关键一步： ` findCallersOf ` 要的不是导入的真实现地址。导入通常先跳到本模块 ` .plt ` 里的 stub，stub 再经 GOT
间接跳到 libcrypto 等真实现——所以 ` BL ` 的直接目标是本模块的 PLT stub，不是 `
Module.findExportByName(null, name) ` 拿到的那个真实现地址。直接拿真实现地址去比对 ` BL `
一个都匹配不到。必须先把导入解析到本模块 stub 地址再喂给 ` findCallersOf ` ：


    // 把导入解析到「本模块 PLT stub 入口」:BL 真正跳的就是这个地址
    function resolveImportStub(module, name) {
        var imp = module.enumerateImports().filter(function(i){ return i.name === name; })[0];
        if (!imp || !imp.slot) return null;          // 拿到该导入的 GOT 槽地址
        var slot = imp.slot, stub = null;
        scanCodeForRefs(module, function(code, i, pcAddr) {
            var adrp = decodeAdrp(code[i], pcAddr);  // PLT stub 以 ADRP+LDR 加载 GOT 槽
            if (!adrp) return;
            for (var k = 1; k <= 3 && (i + k) < code.length; k++) {
                var ins2 = code[i + k];
                if (((ins2 & 0xFFC00000) >>> 0) !== 0xf9400000) continue;   // LDR
                if (((ins2 >>> 5) & 0x1f) !== adrp.rd) continue;
                var imm12 = (ins2 >>> 10) & 0xfff;
                if (adrp.page.add(imm12 * 8).equals(slot)) { stub = pcAddr; return true; }  // 匹配槽「地址」
            }
        });
        return stub;   // stub 入口地址,即 BL 的目标; 再 findCallersOf(module, stub)
    }


> 注意区分： ` resolveImportStub ` 比的是「ADRP+LDR 算出的槽 **地址** == GOT 槽」，第 5.4 节 `
> findAdrpLdrRef ` 比的是「槽里 **存的值** == 目标」——一个比地址、一个比内容。动态版（第 6.1 节）没有这层 PLT
> 麻烦，能跑通时优先用动态。另外， ` -fno-plt ` 编译的库没有独立 stub（调用点直接 ` ADRP+LDR(GOT)+BLR `
> ），此法不适用，也退用动态版。

##  七、相对锚点定位（兜底，知道它的局限）

目标函数附近若有稳定的导出函数当锚，且相对距离基本不变，可 ` 锚点 + delta ` ：


    function locateByAnchor(module, anchorName, delta) {
        var anchor = module.findExportByName(anchorName);
        return anchor ? anchor.add(delta) : null;
    }


它只比硬编码绝对偏移稳一点点——逻辑是「整个 SO 平移时函数间距不变」，但只要两函数之间插/删了代码（更新常发生）， ` delta `
照样失效。它是字符串、调用点、特征码都用不上时的最后兜底，注释里务必写明「仅对接近版本有效」。

##  八、ARM32 / Thumb 的差异

现代设备几乎都是 ARM64，但仍有 App 只提供 32 位 SO（ ` armeabi-v7a ` ）。前面的 ARM64 解码在 ARM32
上整套不工作，关键差异如下（更全的寄存器/调用约定见第 20 篇）：

  * 没有 ADRP：ARM32 取地址靠 ` MOVW/MOVT ` 拼 32 位立即数，或 ` LDR Rd, [PC, #imm] ` 从字面量池加载。字符串 xref 要改成解这两类，不是 ADRP+ADD。
  * 函数序言不同：Thumb 是 ` PUSH {..., LR} ` （16 位编码高字节 ` 0xB5 ` ，32 位宽版 ` PUSH.W ` 前缀 ` 0xE92D ` ）；ARM 是 ` STMFD SP!, {..., LR} ` （ ` 0xE92D.... ` ）。回溯要找这些，不是 STP。
  * Thumb 地址 +1： ` Interceptor.attach ` Thumb 函数要把地址最低位置 1（ ` base.add(offset + 1) ` ，见第 20 篇）。 ` Memory.scan ` 扫到的命中地址也要按这个规则处理。
  * 指令宽度不定：Thumb 是 16/32 位混合，不能像 ARM64 那样按 4 字节整齐切片解析，逐指令解码更麻烦。

实践建议：先用 ` Process.arch ` 分流（ ` "arm" ` vs ` "arm64" ` ），ARM32
上优先用与架构无关的手段——符号表（第 3 章，完全无关架构）、字符串地址扫描（扫的是数据字节，无关架构，只是「回溯函数头」那步要换成找 ` PUSH `
）、特征码（字节模式，换成 Thumb 的字节）。把架构相关的复杂解码（ADRP）尽量绕开。

##  九、封装：多策略逐级兜底框架

把前面各招组装成 ` resolveFunction ` ：给一组线索，按鲁棒性从强到弱逐级尝试，命中即返回，并告诉你靠哪招命中：


    // clues: { exportName, symbol, string, callsImport, pattern, anchor:{name,delta}, offset }
    function resolveFunction(moduleName, clues) {
        var module = Process.findModuleByName(moduleName);
        if (!module) { console.log("[locator] " + moduleName + " 未加载"); return null; }

        // 1) 导出符号 —— 最稳
        if (clues.exportName) {
            var p = module.findExportByName(clues.exportName);
            if (p) return report(p, "export:" + clues.exportName);
        }
        // 2) 局部符号表 —— 零成本,SO 没 strip 就有
        if (clues.symbol) {
            var s = locateBySymbol(module, clues.symbol);
            if (s) return report(s.address, "symbol:" + s.name);
        }
        // 3) 字符串交叉引用 —— 未导出函数首选
        if (clues.string) {
            var p = locateByStringXref(module, clues.string);
            if (p) return report(p, "string-xref:\"" + clues.string + "\"");
        }
        // 4) 调用点交叉引用(静态版) —— 适合纯算法函数
        if (clues.callsImport) {
            // 先解析到本模块 PLT stub:BL 跳的是 stub,不是导入真实现(见第 6.2 节)
            var stub = resolveImportStub(module, clues.callsImport);
            if (stub) {
                var fns = findCallersOf(module, stub);
                if (fns.length === 1) return report(fns[0], "call-xref:" + clues.callsImport);
                if (fns.length > 1) console.log("[locator] 调用 " + clues.callsImport + " 的函数有 " + fns.length + " 个,需进一步筛");
            }
        }
        // 5) 特征码
        if (clues.pattern) {
            var p = locateByPattern(module, clues.pattern);
            if (p) return report(p, "pattern");
        }
        // 6) 相对锚点
        if (clues.anchor) {
            var p = locateByAnchor(module, clues.anchor.name, clues.anchor.delta);
            if (p) return report(p, "anchor:" + clues.anchor.name);
        }
        // 7) 硬编码偏移 —— 兜底,明确警告
        if (clues.offset != null) {
            console.log("[locator]  用硬编码偏移兜底,版本更新后大概率失效");
            return report(module.base.add(clues.offset), "offset(fragile)");
        }
        console.log("[locator] 所有策略都没命中");
        return null;

        function report(addr, via) { console.log("[locator] 命中 @ " + addr + "  via " + via); return addr; }
    }


调用时多给几条线索，鲁棒性自动逐级兜底：


    var enc = resolveFunction("libnative.so", {
        symbol      : /encrypt/i,                              // 没 strip 时一击命中

ARM32 / Thumb 路（ ` MOVW/MOVT ` 、字面量池），上面的 ADRP 解码在这儿不适用。

几条真机踩坑（写脚本前先知道）：

  * Android 12+ 缓存应用冻结：attach 一个在后台被 freezer 冻住的进程会报 ` timed out while waiting for signal ` 。对策：spawn 新进程，或先把目标切到前台。
  * frida-server 必须 root 起（ ` su -c ` ）：否则能 ` frida-ps ` 枚举、却不能注入，attach 直接 ` unable to connect to remote frida-server: closed ` 。
  * 改非默认端口（如 ` -l 127.0.0.1:27045 ` ）能绕过不少 App 对默认端口 27042 的扫描。

##  十一、决策：什么时候用哪招

![选路决策树：从上往下逐行问，第一个「是」就用右边的手段——① 导出/JNI 静态注册 → 导出符号名（第 06 篇）；② SO 没 strip →
enumerateSymbols 局部符号；③ 引用了独特字符串 → 字符串交叉引用；④ 调用了独特导入（加密/签名）→ 调用点交叉引用（动态优先）；⑤
函数头字节稳定 → 特征码；⑥ 旁边有稳定导出锚 → 相对锚点（仅接近版本）；⑦ 一次性分析、用完即弃 →
选路决策树

两条反模式警告：

  * 别为一次性分析过度工程化：只分析一个版本、跑完就扔，直接 IDA 偏移最快。鲁棒性是给「要长期跑」的脚本投资的。
  * 别在长期脚本里硬编码偏移：要跨版本/跨设备/进 CI 的脚本，从第一天就用框架给线索，否则每次更新都在交「重找偏移」的税。

##  总结

硬编码偏移脆，是因为它记「位置」，而位置最易变。鲁棒定位的统一思路是改记「内容」，按抗变化能力从强到弱：

  * 符号表 ` enumerateSymbols ` ：零成本先试，SO 没 strip 时直接拿未导出函数的名字。
  * 字符串交叉引用：扫字符串地址 → 解 ` ADRP+ADD ` （直接）/ ` ADRP+LDR ` （指针槽）找引用 → PAC/BTI-aware 回溯到函数入口。未导出函数最可靠的自动手段之一。
  * 调用点交叉引用：按「它调用了哪个独特导入」定位纯算法函数——动态 Hook 导入读 ` returnAddress ` 最稳，静态解 ` BL ` 也行。
  * 特征码：字节指纹，带地址/偏移的指令整条通配；但跨编译器版本会脆，是中上不是最强。
  * 相对锚点：只比硬编码稍强，作兜底。

把它们组装成 ` resolveFunction ` 逐级兜底，一次写好线索就能跨版本复用—— ` via symbol/string-xref `
时偏移失效也无感，退到 ` via offset(fragile) `
才提醒你维护线索。鲁棒性不免费，按脚本生命周期决定投不投：一次性分析用偏移，长期脚本从第一天给线索。回溯函数头、静态 BL
解析都有启发式成分，拿不准时退回引用点对照 IDA，比定位到错误函数头安全。
