##  一、加固壳的本质：理解它在保护什么

###  1.1 为什么 jadx 看不到真正的代码

当 jadx 打开一个加固过的 APK， ` classes.dex ` 里通常只有壳的引导代码——一个 ` StubApplication ` 类和几段
SO 加载逻辑，App 真正的业务代码完全看不到。这是因为壳把原始 DEX 加密后藏在 APK 的某个角落（ ` assets/ `
下的伪装文件，或嵌在壳的 SO 里），运行时由壳负责解密、回填、加载。

` classes.dex ` 里的引导代码唯一职责是：在 App 启动时解密原始 DEX 并交给 ClassLoader，加载完成后把 `
Application ` 切换为真正的 Application。后续 App 的行为就和没加固时一致。

###  1.2 壳的工作流程

理解流程是选择正确脱壳时机的关键。


壳的工作流程

脱壳的核心思路就是： **在壳完成解密之后、DEX 仍然在内存中的时候，把它取出来。**

###  1.3 加固壳的代际演进

了解代际差异，可以判断脱壳的难度和该选哪个工具。

**一代壳（整体加密）** ：把整个 ` classes.dex ` 加密存储，运行时整体解密并通过 ` DexClassLoader ` / `
InMemoryDexClassLoader ` 加载。这是最简单的方案， **内存里就有完整明文 DEX，搜 magic 就能取** 。2020
年前的免费加固版本多属此类，目前仍能在维护周期长的老版本 App、内部测试构建、个别小厂方案上看到。

**二代壳（指令抽取 / 函数抽取）** ：在一代壳之上，把每个方法的字节码从 DEX 里抽走单独加密，DEX
的"骨架"（类结构、方法声明）还在，但方法体被 ` return-void ` 或单条 ` goto ` 占位。运行时 **每个方法在首次被调用前**
才解密并把字节码回填到内存 DEX 里。这种壳，搜内存只能拿到"空壳子 DEX"，必须主动让每个方法被调用一次。 **2022
年起主流加固厂商的免费版默认就是二代** 。

**三代壳（VMP / 虚拟机保护）** ：把方法字节码翻译成壳自定义虚拟机的指令集。即使你 dump 出 DEX，方法体里也只看到对壳自定义解释器的 `
native ` 跳转。分析三代壳要先逆向自定义虚拟机的解释器，复杂度极高。

**Dex2C（编译化保护）** ：和三代壳并列存在的另一条路——把关键方法翻译成 C 代码编译进 SO。从 DEX 看是 ` native `
跳转，但跳转目标不是自定义解释器，而是直接的 ARM 机器码。分析要走 SO 逆向（参考第 22-23 篇）。网易易盾
Pro、阿里聚安全企业版、几维安全等高端产品多采用 Dex2C。

##  二、主流 DEX 脱壳工具横评

本篇正文精读的是三个 Frida 生态的脱壳工具（ ` frida-dexdump ` / ` frida_dump ` / ` FART `
），因为它们的原理最能讲清楚"内存搜 magic → ART 反向定位 → CodeItem 抽取"这条完整技术演进链。但读者要知道，脱壳工具地图不止
Frida 一系——独立 App 形态的 ` BlackDex ` 星数比这三家都高，Xposed 时代的 ` DumpDex ` 至今仍是老
Android 版本首选。星数与维护状态（截至 2026-07 GitHub API）先摆全景：

**Frida 生态（本篇正文精读）** ：

---|---|---|---|---|---|---

**为什么正文只精读 Frida 三家** ：BlackDex 是一体化 App 用户看不到源码机制，学习价值低；DumpDex 依赖 Xposed（现代
Android 上迁移到 LSPosed），本系列不覆盖 Xposed 生态； ` frida_dex_dump ` 属于 FART 接力方案，机制在第 5
节讲清楚 FART 后，第 9.1 节一句话交代即可。

**怎么选** ：

  * **快脱** ：优先 BlackDex（一键 App 免脑力），或 frida-dexdump（Frida 用户）
  * **精细可控** ：frida_dump 的 dexCache 方案 · 精确定位不容误匹配
  * **二代壳指令抽取** ：FART（AOSP 版最稳）或 CYRUS-STUDIO/frida_dex_dump（不刷 ROM）
  * **Android 4-8 老 App** ：DumpDex（唯一活着的方案）
  * **三代壳 VMP / Dex2C** ：DEX 层无 meaningful 内容，转 SO 逆向（第 22-23 篇）

##  三、frida-dexdump：源码精读

###  3.1 安装与一行实战

frida-dexdump 是 hluwa（看雪资深安全研究员）维护的项目，2023-03 最后一次 push 后归档不再更新，但截至 2026-07
仍是中文圈一代壳脱壳的事实标杆。 **安装就一行 pip** ：


    pip3 install frida-dexdump


实战也是一行：


    # spawn 模式（推荐）：自动等 5s 让壳解密，dump 完即退出
    frida-dexdump -f com.target.app -d
    # attach 模式：App 已经在跑了，附加上去 dump
    frida-dexdump -F                       # 当前前台 App
    frida-dexdump -n com.target.app        # 按包名 attach
    # -d 是 deep-search，对付被故意改坏 magic 的壳
    # -o ./outdir 指定输出目录，默认 ./<包名>/


dump 出来的文件命名是 ` classes.dex ` / ` classes2.dex ` / ...，直接用 jadx 打开就能看业务代码。

接下来按"agent 端 → 主机端"的顺序看核心源码。

###  3.2 agent/src/search.ts：DEX 搜索的核心

![frida-dexdump
frida-dexdump 内存搜索管线

整个 frida-dexdump 的精华全在这一个文件里。最关键的是它 **用通配符一次性匹配所有 DEX 版本** ：


    // search.ts L115，frida-dexdump v2.0
    Memory.scanSync(range.base, range.size, "64 65 78 0a 30 ?? ?? 00").forEach(...)


` 64 65 78 0a 30 ?? ?? 00 ` 对应 ASCII ` "dex\n0XX\0" ` ，第 6、7 字节用 ` ?? `
通配——一次性命中 DEX 035 / 037 / 038 / 039 **所有正式版本** （顺带覆盖从未发布的 036 占位）。这是个关键细节：

  * DEX 035：Android 1-6 默认
  * DEX 037：Android 7（Nougat）+
  * DEX 038：Android 8.0 +
  * DEX 039：Android 9.0 +
  * DEX 036：Google 从未正式发布，035 直接跳到 037

很多自造脱壳脚本只写死 ` ... 30 33 35 00 ` （035），现代 App 编译出的 DEX 多是
037/038/039，加固壳脱出来的版本号 ≥ 编译期版本，写死 035 经常会"一个 DEX 都搜不到"。Frida 的 `
Memory.scanSync ` 支持 ` ?? ` 通配字节，这个特性必须用上。

紧跟在 magic 搜索后面的是 **双层 verify** ：


    // fast 模式：校验 string_ids_off 字段（DEX 头偏移 0x3C 处必为 0x70）
    // 真正的 header_size 字段在偏移 0x24，但 header 结构固定 0x70 字节，紧接着就是
    // string_ids 表，所以合法 DEX 的 string_ids_off = 0x70——用这个 0x70 常量做快速筛
    if (verify(match.address, range, false)) { ... }

    function verify(dexptr, range, enable_verify_maps) {
        if (dexptr.add(0x70) > range_end) return false;     // 头不能超出内存段
        if (enable_verify_maps) { /* 走 deep 路径 */ }
        else { return dexptr.add(0x3C).readUInt() === 0x70; }  // fast：string_ids 紧贴 header 尾部
    }


fast 模式只是粗校验"是不是 DEX 头"。deep 模式会校验 **map_list** ——DEX 文件末尾有一段 map_list 记录所有
section 的偏移和类型，其中有一项 ` type=0x1000 ` （DEX 规范里叫 ` TYPE_MAP_LIST ` ，代表 map_list
自身），它的 ` offset ` 字段必须等于 DEX header 中的 ` map_off ` （偏移 0x34）。这是一个自我引用的环路，伪 DEX
几乎不可能伪造正确：


    function verify_by_maps(dexptr, mapsptr) {
        const maps_offset = dexptr.add(0x34).readUInt();        // header.map_off
        const maps_size = mapsptr.readUInt();
        for (let i = 0; i < maps_size; i++) {
            const item_type = mapsptr.add(4 + i * 0xC).readU16();
            if (item_type === 4096) {                            // 0x1000 = DEX 自身
                const map_offset = mapsptr.add(4 + i * 0xC + 8).readUInt();
                if (maps_offset === map_offset) return true;     // 自我引用闭合 → 真 DEX
            }
        }
        return false;
    }


接下来是 **真实大小的反推** 。很多壳为了对抗 dump，会故意把 DEX 头的 ` file_size ` 字段（偏移 0x20）改成 0
或一个错值。frida-dexdump 不信任 ` file_size ` ，而是从 map_list 末尾反推：


    function get_dex_real_size(dexptr, range_base, range_end) {
        const dex_size = dexptr.add(0x20).readUInt();           // header.file_size
        const maps_address = get_maps_address(dexptr, range_base, range_end);
        if (!maps_address) return dex_size;                     // 拿不到 maps 就退回 file_size
        const maps_end = get_maps_end(maps_address, range_base, range_end);
        if (!maps_end) return dex_size;
        return maps_end.sub(dexptr).toInt32();                  // map_list 末尾 - DEX 起点 = 真实大小
    }


最后是 **deep_search 模式的反向搜索** 。如果壳把 DEX magic 也抹掉（替换成 ` \0\0\0\0 ` 或随机），按 magic
怎么搜都搜不到。frida-dexdump 提供了一个 plan B——按 **` 70 00 00 00 ` 搜 header_size 字段 **
，然后从命中地址往前减 0x3C 反推 DEX 起点：


    if (deepSearch) {
        Memory.scanSync(range.base, range.size, "70 00 00 00").forEach(function (match) {
            const dex_base = match.address.sub(0x3C);            // header_size 在 +0x3C 处
            if (dex_base < range.base) return;
            if (dex_base.readCString(4) != "dex\n"               // 排除 magic 还在的（前面循环已处理）
                && verify(dex_base, range, true)) {              // 用 deep verify 防误匹配
                const real_dex_size = get_dex_real_size(...);
                if (!verify_ids_off(dex_base, real_dex_size)) return;  // 再用 string/type/proto ids 偏移校验
                result.push({ "addr": dex_base, "size": real_dex_size });
            }
        });
    }


` verify_ids_off ` 是最后一道防线，校验 ` string_ids_off / type_ids_off / proto_ids_off
/ field_ids_off / method_ids_off ` （DEX 头偏移 0x3C-0x60）这五个 section 偏移都在合法范围（≥
0x70 且 < dex_size）。三层校验下来，伪 DEX 几乎不可能漏过。

还有一个排除规则：


    if (range.file && range.file.path
        && (range.file.path.startsWith("/data/dalvik-cache/") ||
            range.file.path.startsWith("/system/"))) {
        return;
    }


` /data/dalvik-cache/ ` 是 ART 的内部缓存（编译后的 OAT/CDEX 文件，magic 是 ` cdex001 `
，但内嵌的标准 DEX 段也会被 magic 搜到）， ` /system/ ` 是系统框架 DEX。 **这两类不是壳目标，不 dump 才对** 。

###  3.3 agent/src/index.ts：内存提权这一招

很多人没注意到 frida-dexdump 还做了一件关键事： **dump 之前先把目标内存的保护位改成可读** 。


    // index.ts
    function setReadPermission(base: NativePointer, size: number) {
        const end = base.add(size);
        Process.enumerateRanges("---").forEach(function (range) {     // 枚举所有 不可读 不可写 不可执行 的内存段
            if (range.base < base || range.base.add(range.size) > end) return;
            if (!range.protection.startsWith("r")) {
                Memory.protect(range.base, range.size,
                    "r" + range.protection.substr(1, 2));             // 补一个 r
            }
        });
    }

    rpc.exports = {
        memorydump: function (address, size) {
            const ptr = new NativePointer(address);
            setReadPermission(ptr, size);                             // 提权后再读
            return ptr.readByteArray(size);
        },
        ...
    };


部分高级壳会主动把 DEX 所在内存的保护位改成 ` --- ` （完全不可访问）来对抗 dump。frida-dexdump 用 ` mprotect `
直接补一个 ` r ` 权限——这是一个低成本但很关键的对抗手段。 ` mprotect ` 本身是 Linux
标准系统调用，壳很难拦截（拦了系统也跑不起来），所以这一招在大多数壳上都有效。

###  3.4 frida_dexdump/ **main** .py：磁头修复

agent 端把 DEX 字节流取下来后，主机端 Python 还要做一步 **磁头修复** ：


    # frida_dexdump/__main__.py
    def fix_header(dex_bytes):
        dex_size = len(dex_bytes)
        # 1. magic 被抹掉 → 强制写回 dex\n035\0
        if dex_bytes[:4] != b"dex\n":
            dex_bytes = b"dex\n035\x00" + dex_bytes[8:]
        # 2. file_size 写回真实长度（偏移 0x20）
        if dex_size >= 0x24:
            dex_bytes = dex_bytes[:0x20] + struct.pack("<I", dex_size) + dex_bytes[0x24:]
        # 3. header_size 强制为 0x70（偏移 0x24）
        if dex_size >= 0x28:
            dex_bytes = dex_bytes[:0x24] + struct.pack("<I", 0x70) + dex_bytes[0x28:]
        # 4. endian_tag 校正为 0x12345678 小端（偏移 0x28）
        if dex_size >= 0x2C and dex_bytes[0x28:0x2C] not in [b'\x78\x56\x34\x12', b'\x12\x34\x56\x78']:
            dex_bytes = dex_bytes[:0x28] + b'\x78\x56\x34\x12' + dex_bytes[0x2C:]
        return dex_bytes


这四个字段是 jadx / baksmali 解析 DEX 时必检的关键字段。壳故意改坏任意一个，dump 出来后 jadx 都会拒绝打开。 `
fix_header ` 强制写回标准值—— **不校验 checksum 和 signature** ，因为 jadx 默认不校验这两个字段（除非 `
-v ` 严格模式）。

至此 frida-dexdump 的核心机制全部讲完。整个工具不到 800 行代码，但每一段都有"对抗某种反 dump
手法"的具体动机——这是开源项目里少有的密度。

###  3.5 已知短板与对策

短板  |  触发场景  |  对策
---|---|---
**二代壳指令抽取无效** |  dump 出 DEX 类齐全，方法体全是 ` return-void ` |  转 FART（第五节）
**内存里有大量未解密的伪 DEX** |  壳把多个加密 DEX 段散布在内存中诱导误匹配  |  用 ` -d ` 开 deep verify；或转 frida_dump 的 dexCache 方案
**OAT/CDEX 内嵌 DEX 段也被搜到** |  dump 出大量 ` /data/dalvik-cache/ ` 路径的 DEX  |  已自动排除；如果误匹配仍多，检查 ` /data/data/<pkg>/code_cache/ ` 是否也要排除
**App 检测 frida-server** |  frida-dexdump 起不来  |  先按第 17-18 篇做反检测处理
**仓库已 archived（2023-03）** |  新版 Android 14+ 可能有兼容问题  |  fork 自维护；或换 frida_dump

##  四、frida_dump：从 ART 数据结构反向定位

frida-dexdump 走的是"全内存扫 magic"路线——优点是通用，缺点是会被假 DEX 干扰、对抹了 magic 的壳要靠
deep_search 兜底。lasting-yang 的 frida_dump 走的是另一条路： **不扫内存，直接问 ART 要 DEX 在哪** 。

仓库里两个核心脚本，分别是两种实现：

###  4.1 dump_dex.js：hook ClassLinker::DefineClass 被动拦截

>  前置声明： ` lasting-yang/frida_dump ` 仓库仍在维护（截至 2025-08），但本节代码保留原 Frida 16 API
> 写法（ ` Memory.readX ` ），Frida 17（2025-05）已经移除该系列 API，实操前需按本篇第 9.2 节迁移到 `
> ptr.readX() ` ；同时 Android 12+ 的 ArtMethod 布局重构见下文。

ART 加载任意一个类时，最终都会进入 ` art::ClassLinker::DefineClass ` ，第 5 个参数就是 ` DexFile* `
。在这里 hook 一下，每加载一次类就 dump 一次它所在的 DEX：


    // dump_dex.js 核心片段
    function dump_dex() {
        var libart = Process.findModuleByName("libart.so");
        var addr_DefineClass = null;
        var symbols = libart.enumerateSymbols();
        for (var index = 0; index < symbols.length; index++) {
            var symbol_name = symbols[index].name;
            // _ZN3art11ClassLinker11DefineClassEPNS_6ThreadEPKcmNS_6HandleINS_6mirror11ClassLoaderEEERKNS_7DexFileE...
            if (symbol_name.indexOf("ClassLinker") >= 0
                && symbol_name.indexOf("DefineClass") >= 0
                && symbol_name.indexOf("Thread") >= 0
                && symbol_name.indexOf("DexFile") >= 0) {
                addr_DefineClass = symbols[index].address;
                break;
            }
        }
        var dex_maps = {};
        Interceptor.attach(addr_DefineClass, {
            onEnter: function(args) {
                var dex_file = args[5];                                       // 第 5 个参数：DexFile*
                // DexFile 的 C++ 布局：
                //   +0:                 vtable
                //   +pointerSize:       const uint8_t* const begin_   ← DEX 起点
                //   +2*pointerSize:     const size_t size_            ← DEX 大小
                var base = ptr(dex_file).add(Process.pointerSize).readPointer();
                var size = ptr(dex_file).add(Process.pointerSize * 2).readUInt();
                if (dex_maps[base] == undefined) {
                    dex_maps[base] = size;
                    /* ... 检查 magic 开头是否为 "dex" → 写文件 ... */
                }
            }
        });
    }


关键的两个偏移 ` +pointerSize ` 和 ` +2*pointerSize ` 对应 ART 源码 `
art/runtime/dex_file.h ` 中 ` DexFile ` 类的成员布局：


    // art/runtime/dex_file.h（Android 8 / 9 / 10 / 11 布局基本一致）
    class DexFile {
        // [+0]     vtable                       (C++ 隐式)
        const uint8_t* const begin_;            // [+8]  DEX 起点（64-bit）
        const size_t size_;                     // [+16] DEX 大小
        // ...
    };


Android 12+ 未经作者实测， ` DexFile ` 布局有细节调整（12 引入 ` container_ ` 字段包装 CompactDex
容器），但 ` begin_ / size_ ` 相对 vtable 的位置仍稳定；有兼容性问题请以对应 API level 的 `
art/runtime/dex_file.h ` 源码为准。

为什么用符号匹配而不是直接拿名字？因为 C++ 函数名会被 mangle（变成 `
_ZN3art11ClassLinker11DefineClassE... ` 这种），且不同 Android 版本签名细节不同。用 indexOf
模糊匹配（ ` ClassLinker ` \+ ` DefineClass ` \+ ` Thread ` \+ ` DexFile `
四个特征词同时出现）就能跨 Android 8 / 9 / 10 / 11 通用。

这种思路的优势： **直接从 ART 内部拿真实 begin_ / size_，不会误匹配假 DEX、不依赖 magic 是否完整** 。

劣势：必须等到壳真的加载某个类才能触发。如果壳延迟加载（比如某些 Activity 进入后才加载对应类），需要在 App 里手动多点几个页面。

###  4.2 dexCache_dump.js：主动枚举 java.lang.DexCache

dexCache_dump.js 是仓库 README 主推的方案，思路更激进—— **完全不 hook，直接枚举 ART 内存里所有的`
java.lang.DexCache ` 对象 ** 。 ` DexCache ` 是 ART 给每个 DexFile 分配的元数据缓存对象，每加载一个
DEX 就会有一个对应的 DexCache 实例：


    // dexCache_dump.js
    function DexCache_dump() {
        Java.perform(() => {
            let dex_count = 1;
            Java.choose("java.lang.DexCache", {
                onMatch(instance) {
                    let classLoader = instance.classLoader.value;
                    let location = instance.location.value;
                    let dexFile = instance.dexFile.value;       // long 类型，存的是 native DexFile* 指针
                    if (classLoader) {
                        // dexFile 是 long，本身就是 native 指针的整数值
                        // 再走一次 DexFile 的内部布局：+pointerSize 是 begin_
                        let dex_ptr = ptr(dexFile).add(Process.pointerSize).readPointer();
                        let dex_size = dex_ptr.add(0x20).readU32();   // 这里直接读 DEX header.file_size
                        console.log(classLoader, location, dex_ptr, ptr(dex_size));
                        dex_count = save_dex(dex_ptr, dex_size, dex_count);
                    }
                },
                onComplete() {}
            });
        });
    }
    setImmediate(DexCache_dump);


` Java.choose("java.lang.DexCache") ` 会暂停所有 Java 线程，扫描整个 Java 堆，把所有 DexCache
实例列出来。对每个实例：

  1. 读 ` dexFile ` 字段（ ` Object ` 类型，运行时是个 ` long ` ）→ 这就是 native 端 ` DexFile* ` 的数值
  2. 走 DexFile 布局： ` *(DexFile + pointerSize) ` 拿到 ` begin_ `
  3. 从 ` begin_ + 0x20 ` 读 header.file_size 拿到 DEX 大小
  4. dump

**与 dump_dex.js 的区别** ：

  * dump_dex.js 是 **被动 hook** ——等壳加载某个类时才触发，需要 App 实际运行触发各种类加载
  * dexCache_dump.js 是 **主动枚举** ——只要 App 已经把 DEX 注册进 DexCache（壳的解密肯定已完成），不管类有没有被加载都能取到 DEX

代价是 ` Java.choose ` 在大型 App 上很慢（数千个 DexCache 实例的堆扫描可能要 10-30 秒），且有些壳会检测 `
Java.choose ` 的调用栈来反 dump。

###  4.3 frida_dump 与 frida-dexdump 选型

维度  |  frida-dexdump  |  frida_dump (dexCache)
---|---|---
思路  |  全内存扫 magic  |  从 ART 数据结构反向定位
精度  |  可能误匹配假 DEX，需 deep verify  |  拿到的都是 ART 注册过的真 DEX
速度  |  几秒  |  大型 App 10-30s
安装  |  ` pip install ` 一行  |  ` git clone ` 后手动跑
抹 magic 的壳  |  需要 ` -d ` deep_search 反推  |  完全不依赖 magic，天然免疫
反 dump 检测  |  抗 mprotect 攻击好  |  抗 ` Java.choose ` 检测差
适用  |  **默认首选** ，先跑这个  |  大量假 DEX 干扰 / 抹 magic 严重的壳，作为精确补刀

**实战上一般是** ：frida-dexdump 跑一次 → 如果 jadx 打开后类一堆但你想要的业务类找不到（说明误匹配多），换
dexCache_dump.js 跑一次确认。两个工具的 dump 结果都保留，互相对照。

##  五、FART：二代壳唯一通用方案

###  5.1 为什么 frida-dexdump 和 frida_dump 都对二代壳无效

二代壳（指令抽取）的关键操作有两步：

  1. **抽取** ：壳在 App 安装/首次启动时，把 DEX 里每个方法的 CodeItem（字节码）抽出来加密保存到独立的数据段（壳自定义格式），DEX 里方法体位置填上 ` return-void ` 或 ` goto +0 ` 占位。
  2. **回填** ：运行时， **当 ART 真的要执行某个方法** 时（具体是 ` art::ClassLinker::LoadMethod ` → ` art::ArtMethod::Invoke ` 路径上），壳触发对应方法的解密，把 CodeItem 字节码写回到 DEX 在内存中对应位置，让 ART 解释器/编译器拿到正确的字节码执行。

这种保护下，前两个工具的失败方式：

  * **frida-dexdump** ：搜到的 DEX 都是 "DEX 骨架+全部 return-void"，jadx 打开后类齐全但方法体全空。
  * **frida_dump** ：拿到的 DEX 路径是 ART 注册过的 DexFile*，但 __DexFile_ 指向的内存就是骨架 DEX _ *，方法体仍然是空——因为 dexCache 里只缓存元数据，回填发生在 method 真正执行时。

要破二代壳，必须 **主动让每个方法都被"调用"一次** ，让壳触发回填，然后 **在回填的瞬间** 抓走字节码。这就是 FART 的核心思路。

###  5.2 FART 的三步原理

![FART
FART 三步原理

FART（hanbinglengyue 提出，看雪 2018；名字是作者的谐音自嘲，无官方词源解释）的核心三步：

  1. **主动触发** ：通过反射或 ClassLoader API 主动加载每一个类、解析每一个方法。注意—— **不需要真的` invoke ` 方法 ** （参数也准备不出来），只需要让 ART 走到 ` ClassLinker::LoadMethod ` ，壳的回填 hook 就会被触发。
  2. **拦截 LoadMethod** ：在 ` art::ClassLinker::LoadMethod ` 入口插桩，每次它被调用时，从参数里拿到对应的 ` DexFile* ` 和 ` ArtMethod* ` ，记录下来。LoadMethod 返回时，壳的回填已经完成，从 ` ArtMethod ` 的 ` dex_code_item_offset_ ` 字段反向算出 CodeItem 在 DEX 内存中的位置，把这段 CodeItem 字节码 dump 出来。
  3. **离线重组** ：把所有 dump 下来的 CodeItem 片段（按 ` method_idx ` 索引）回填到骨架 DEX 中对应方法的 ` code_off ` 位置——拼出一份完整的 DEX。

FART 有两种实现形态： **AOSP 源码版** （最稳，但要刷 ROM）和 **Frida 版** （不刷 ROM，但只能搞定方法级别的回填）。

###  5.3 AOSP 版：patch 哪几个文件

仓库里的 ` FART_6.0_sourcecode.zip ` 是基于 AOSP 6.0 的完整源码包。关键 patch 集中在三处：

  1. **` art/runtime/art_method.cc ` ** — 在 ` ArtMethod::Invoke ` 入口插桩。每次任何方法被执行前，记录 ` (DexFile*, ArtMethod*, dex_code_item_offset_, dex_method_index_) ` 四元组，从内存里把对应 CodeItem 字节流 dump 到 ` /sdcard/fart/<pkg>/<size>_<tid>.bin ` ，每行一条 ` {name:..,method_idx:..,offset:..,code_item_len:..,ins:base64_encoded} ` 。
  2. **` frameworks/base/core/java/android/app/ActivityThread.java ` ** — 在 Application 完成创建后启动一个后台线程，调用 ` dalvik.system.DexFile.entries() ` 拿到当前 Application 加载的所有 DEX 里所有的类名，然后用 ` ClassLoader.loadClass() ` 主动加载每个类。 **这是"主动调用"的实质——是 loadClass 触发类解析、类解析触发方法 LoadMethod、LoadMethod 触发壳回填** 。
  3. **` art/runtime/class_linker.cc ` ** — 在 ` ClassLinker::LoadMethod ` 也插桩 dump 整个 DexFile（拿到骨架），与 art_method.cc 的 CodeItem dump 配合。

跑完后 logcat 里会输出 ` fart run over ` ，在 ` /sdcard/fart/<pkg>/ ` 下能拿到一堆 ` .dex `
（骨架）和 ` .bin ` （CodeItem 片段）。然后用仓库里的 ` fart.py ` 做离线重组。

###  5.4 Frida 版：hook LoadMethod + 主动 loadClass

Frida 版的核心文件是 ` frida_fart_hook.js ` （搭配 ` fart.so ` / ` fart64.so ` 提供 native
辅助函数）。它把 AOSP 版的 art_method.cc + class_linker.cc 两处插桩合并到 `
art::ClassLinker::LoadMethod ` 一个点上：


    // frida_fart_hook.js 核心：hook LoadMethod
    var symbols = Module.enumerateSymbolsSync("libart.so");
    for (var i = 0; i < symbols.length; i++) {
        // _ZN3art11ClassLinker10LoadMethodERKNS_7DexFileERKNS_21ClassDataItemIteratorE...
        if (symbols[i].name.indexOf("ClassLinker") >= 0
            && symbols[i].name.indexOf("LoadMethod") >= 0
            && symbols[i].name.indexOf("DexFile") >= 0
            && symbols[i].name.indexOf("ClassDataItemIterator") >= 0
            && symbols[i].name.indexOf("ArtMethod") >= 0) {
            addrLoadMethod = symbols[i].address;
            break;
        }
    }

    Interceptor.attach(addrLoadMethod, {
        onEnter: function (args) {
            this.dexfileptr = args[1];      // DexFile*
            this.artmethodptr = args[4];    // ArtMethod*
        },
        onLeave: function (retval) {
            // dump 整个 DexFile（骨架 DEX）—— 同 frida_dump 的 DefineClass 思路
            var dexfilebegin = Memory.readPointer(ptr(this.dexfileptr).add(Process.pointerSize));
            var dexfilesize = Memory.readU32(ptr(this.dexfileptr).add(Process.pointerSize * 2));
            // ... 写入 <size>_loadMethod.dex（去重，每个 DexFile 只写一次）...

            // 记录 (dexfile, artmethod) 到 map，dumpall 时按 ArtMethod 逐个 dump CodeItem
            artmethod_maps[this.artmethodptr] = new ArtMethod(
                new DexFile(dexfilebegin, dexfilesize),
                this.artmethodptr
            );
        }
    });


` LoadMethod ` 是 ART 加载每一个方法时必经的函数（壳的回填 hook 就插在它附近），第 1 个参数是 ` DexFile* ` ，第
4 个是 ` ArtMethod* ` 。这里 **只记录 (DexFile, ArtMethod) 配对，不立即 dump CodeItem**
——等所有方法的回填都完成后再统一 dump，避免 hook 自己拖慢回填。

> 上面的 ` Memory.readPointer / Memory.readU32 ` 是 Frida 16 及之前的写法。Frida
> 17（2025-05）已把 ` Memory.readX ` 系列全部移除，需改为 ` ptr.readPointer() /
> ptr.readU32() ` 。见第 9.2 节 API 迁移表。

dump CodeItem 的核心：


    function dumpcodeitem(artmethodobj) {
        var dexfilebegin = artmethodobj.dexfile.start;
        var artmethodptr = artmethodobj.artmethodptr;
        // ArtMethod 布局（Android 6-9）：
        //   +0:   declaring_class_   GcRoot<Class>
        //   +4:   access_flags_      uint32
        //   +8:   dex_code_item_off_ uint32   ← CodeItem 在 DEX 内的偏移
        //   +12:  dex_method_index_  uint32   ← method 在 DEX 内的索引
        //  Android 10+ ART 重构 ArtMethod，dex_code_item_offset_ 被并入 data_ 联合体
        //   （AOT 场景下 data_ 存的是 entry_point；仅解释器/非 AOT 时保留原语义）
        //   +8 偏移不再稳定，需按 art_method.h 对应 API level 调整
        var dex_code_item_offset_ = Memory.readU32(ptr(artmethodptr).add(8));
        var dex_method_index_ = Memory.readU32(ptr(artmethodptr).add(12));
        if (dex_code_item_offset_ > 0) {
            var codeitemstartaddr = ptr(dexfilebegin).add(dex_code_item_offset_);
            var codeitemlength = funcGetCodeItemLength(ptr(codeitemstartaddr));  // 调 fart.so
            if (codeitemlength > 0) {
                Memory.protect(ptr(codeitemstartaddr), codeitemlength, 'rwx');   // 提权（同 frida-dexdump）
                // Base64 编码 CodeItem 二进制 → 写入 <size>_<tid>.bin
                var b64content = readCStringFromNative(
                    funcBase64_encode(ptr(codeitemstartaddr), codeitemlength, base64lengthptr));
                file_handle.write("{name:ooxx,method_idx:" + dex_method_index_
                    + ",offset:" + dex_code_item_offset_
                    + ",code_item_len:" + codeitemlength
                    + ",ins:" + b64content + "};");
            }
        }
    }


为什么需要配套 ` fart.so ` ？因为 **CodeItem 的真实长度计算很复杂** ——除了 16 字节的固定头，还有 ins
字节码区、try-catch handler 数据、padding 等多种变长块。 ` GetCodeItemLength ` 的逻辑直接搬自 AOSP 的
` dex_file.cc ` ，纯 JS 写起来又长又容易出错，所以打包成 native 函数从 SO 里调。 ` Base64_encode `
同理——把二进制塞进 JSON 风格的文本文件比较省事，但 JS 端的 base64 性能差。

**关键的"主动调用"在` fart() ` 函数里 ** ：


    function fart() {
        Java.perform(function () {
            dumpall();   // 先 dump 被动 hook 期间已经回填的（App 启动到现在已经被调用过的方法）

            // 再主动 loadClass 触发剩下的方法回填
            Java.enumerateClassLoaders({
                onMatch: function (loader) { dealwithClassLoader(loader); },
                onComplete: function () {}
            });

            dumpall();   // 主动调用后再 dump 一次
        });
    }

    function dealwithClassLoader(classloaderobj) {
        Java.perform(function () {
            var basedexclassloaderobj = Java.cast(classloaderobj,
                Java.use("dalvik.system.BaseDexClassLoader"));
            var pathlistobj = Java.cast(basedexclassloaderobj.pathList.value,
                Java.use("dalvik.system.DexPathList"));
            var dexElementsobj = pathlistobj.dexElements.value;
            for (var i in dexElementsobj) {
                var elementobj = Java.cast(dexElementsobj[i],
                    Java.use("dalvik.system.DexPathList$Element"));
                var dexfileobj = Java.cast(elementobj.dexFile.value,
                    Java.use("dalvik.system.DexFile"));
                // dexFile.entries() 返回该 DEX 里所有的类名
                var enumeratorClassNames = dexfileobj.entries();
                while (enumeratorClassNames.hasMoreElements()) {
                    var className = enumeratorClassNames.nextElement().toString();
                    classloaderobj.loadClass(className);  //  这才是真正的"主动调用"
                    // loadClass 触发 ClassLinker::LoadClass
                    //   → 内部对每个方法调 ClassLinker::LoadMethod
                    //   → 我们的 hook 触发
                    //   → 壳的回填 hook 也在 LoadMethod 附近，先执行回填
                    //   → 我们 onLeave 拿到回填后的 ArtMethod
                }
            }
        });
    }


**核心一句话** ： ` classloader.loadClass(className) ` 让 ART 走完整的类解析流程 → 壳在
LoadMethod 附近做回填 → 我们的 hook 拿到回填后的 ArtMethod → 算出 CodeItem 在内存中的位置 → dump
出来。不需要真的 ` Method.invoke() ` ——loadClass 就够了。

**Frida 版与 AOSP 版的差异** ：

  * AOSP 版插桩在 ` ArtMethod::Invoke ` ，能拿到每次方法 **实际执行** 时的 CodeItem，对"运行时才解密单个方法"的高级二代壳更有效
  * Frida 版插桩在 ` ClassLinker::LoadMethod ` ，能拿到类 **首次解析** 时的 CodeItem——对"类加载时一次性回填整个类方法"的二代壳够用，但对"按方法执行粒度回填"的更高级壳无效

###  5.5 fart.py：CodeItem 离线重组

dump 完之后，你手里有：

  * 若干份 ` <size>_loadMethod.dex ` —— 各个 DexFile 的骨架（方法体仍是 ` return-void ` ）
  * 若干份 ` <size>_<tid>.bin ` —— 每行一条 ` {name:..,method_idx:X,offset:Y,code_item_len:Z,ins:base64}; `

fart.py（2300+ 行）做的事就是 **把 bin 里的 CodeItem 按 method_idx 回填到骨架 dex 对应方法的`
code_off ` 位置 ** 。核心逻辑：


    # fart.py 核心思路（简化）
    def parseinsfile():
        """解析 .bin 文件，建立 method_idx → CodeItem 映射"""
        content = open(insfilename).read()
        insarray = re.findall(
            r"{name:(.*?),method_idx:(.*?),offset:(.*?),code_item_len:(.*?),ins:(.*?)}",
            content)
        for eachins in insarray:
            method_idx = int(eachins[1])
            ins_b64 = eachins[4]
            methodTable[method_idx] = CodeItem(
                number=method_idx,
                methodname=eachins[0],
                inssize=int(eachins[3]),
                insarray=ins_b64)

    # 然后解析骨架 DEX 的所有 class_def → class_data → 每个 method 的 code_off
    # 对每个 method_idx，从 methodTable 取出对应的 CodeItem 二进制
    # 写回到 dex 文件中 code_off 偏移处覆盖原有的 return-void
    # 输出修复后的完整 dex


fart.py 同时还内置了一份 **完整的 Dalvik 字节码解析表** （dex_decode dict，256 条指令）和 LEB128
解码、CodeItem TryItem/Handler 解析等——它不只是简单回填，还能对修复后的 DEX 做指令级反汇编验证。这部分代码 2000
多行，本篇不展开。

##  七、实战：七猫小说

环境：Pixel 5 (redfin) / Android 14 (API 34) / arm64-v8a / Magisk root / Frida 主机
### 16.7.19 + 去特征 ` frida-server-patched ` / frida-dexdump 2.0.1（archived）。

七猫小说（ ` com.kmxs.reader ` ）的 base.apk classes.dex magic 是 ` dex\n035\0 ` ，没有加固
SO——是典型裸 DEX。直接跑 frida-dexdump：


    adb -s 0B201FDD4004S4 shell "am force-stop com.kmxs.reader"
    mkdir -p /tmp/dump_test/kmxs
    frida-dexdump -D 0B201FDD4004S4 -f com.kmxs.reader -d --sleep 10 -o /tmp/dump_test/kmxs


输出 96 dex found，67 秒完成，0 ERROR，磁盘产物 1.15 GB。但"96 DEX / 1.15 GB"这个数字要打折看。做严格核账：

项  |  数据
---|---
静态 ` base.apk ` 内  |  16 个 DEX / **116 MB**
frida-dexdump 产物  |  96 个 DEX 段 / **1150 MB**

看着"多出 80 个 / 10 倍膨胀"，但下一步验证暴露真相。产物里最大的 10 个 DEX 大小呈单调递减： ` classes04.dex `
### 118.6 MB / ` classes06.dex ` 109.1 MB / ` classes08.dex ` 98.8 MB ... `
classes22.dex ` 53.8 MB——每两个偶数编号大小差正好接近 10 MB。用 ` md5(尾部 1024 字节) ` 校验：


    classes04.dex 尾部 1KB md5:  0f343b0931126a20f133d67c2b018a3b
    classes06.dex 尾部 1KB md5:  0f343b0931126a20f133d67c2b018a3b
    classes08.dex 尾部 1KB md5:  0f343b0931126a20f133d67c2b018a3b
    ...
    classes22.dex 尾部 1KB md5:  0f343b0931126a20f133d67c2b018a3b   ← 全部一致


再抽字节偏移对比：


    c06[N] == c04[N + 10_000_652]   # True at N=10M/50M/100M 都对得上


这 10 个 DEX 就是同一段 ~120 MB 内存里的 OAT 缓存——ART 把多个 CompactDex 合并成一段连续内存，每段都有 `
dex\n ` magic 头。frida-dexdump 的 ` -d ` 深度扫描匹配到这 10 个 magic，每次都从当前 magic
位置读到内存段末尾， ` file_size ` 自然递减。 **后面 9 段全部是 classes04 的截取子集** （不同起点、同一尾部）。10
段合计 833 MB 磁盘占用里，独立内容只有 classes04 的 118 MB，另 9 段带来的额外磁盘冗余是 715 MB。

净 unique 账本：

|  大小
---|---
产物磁盘总量  |  1150 MB
大 DEX 段重叠冗余  |  715 MB
净 unique 内容  |  ~435 MB
静态 APK 内  |  116 MB
**净比静态多** |  **~320 MB** （动态加载的真 DEX）

也就是说——"96 DEX / 1.15 GB" 这个数字在 3-4 倍范围内是被 ` -d ` 深度扫描放大的，但净比静态多 320 MB
是真的。这部分是 Android System WebView 进程加载的 Chromium DEX（ `
Android.WebView.HistoricalApplicationExitInfo ` 字符串特征）、字节 Lynx / Compass
等热更新插件、大量第三方 SDK 的动态子 DEX——静态 APK 里完全看不到的东西。

值得记的细节：

  * DEX 035 与 039 都被搜到了——证明 ` 64 65 78 0a 30 ?? ?? 00 ` magic 通配在 Android 14 上有效；硬编码只搜 035 的话，DEX 039 那批会漏掉
  * ` size_file == size_header ` 只证明 frida-dexdump 按 header.file_size 读了那么多字节，不代表这些字节是独立的业务 DEX。判定独立性要看内容签名而非文件头
  * ` classes04.dex ` 里含 ` com.qimao.qmbook / com.qimao.qmad2 / com.kmxs.reader ` 业务包（40328 个 ` L类; ` 字符串），是真业务
  * frida-dexdump 报告 "96 dex"，但只有 ~40 段是真独立 DEX——审计时一定 md5 去重、验尾部一致性、字节偏移对比，别被"数字大"骗

` frida-dexdump -d ` 深度扫描对 ART OAT 缓存合并区会有同段内存多次重复 dump 的副作用，做统计前必须先做重叠去重，否则"1
GB dump"三分之二可能是同一段字节。这就是为什么现代脱壳工具（如 CYRUS-STUDIO/frida_dex_dump，第 9.1 节）把 `
DexFileLoader::OpenCommon ` hook 作为主路径——从 ART 数据结构反向定位 DEX 起点，天然去重。

裸 DEX App 为什么还要脱壳的真实答案——净比静态多 320 MB 的动态内容，不是"多 1 GB"。

###  8.4 反反检测层的构建

把 nuke 换成欺骗——hook AppSealing 探测的几条路径，让它每次检测都看到干净环境，走完解密：

检测类型  |  探测路径  |  hook 策略
---|---|---
Ptrace 检测  |  ` /proc/self/status ` 的 ` TracerPid ` 字段  |  hook ` open/openat ` ，遇到 status 路径重定向到伪造文件（ ` TracerPid: 0 ` ）
Frida 字符串扫描  |  ` /proc/self/maps ` 里 ` frida ` / ` gum-js ` / ` gadget ` 关键字  |  hook ` read ` ，把含黑名单的行整行替换为空格（保留长度）
线程名检测  |  ` prctl(PR_GET_NAME) ` / ` /proc/self/task/<tid>/comm ` 含 ` gum-js-loop ` |  hook ` prctl(PR_GET_NAME) ` ，把 frida 线程名改回 ` Thread-1 `
Root 路径检测  |  ` stat/access ` 探测 ` magisk ` / ` supolicy ` / ` /sbin/.magisk ` |  hook ` stat/lstat/fstat/access/faccessat ` ，含黑名单路径重定向到 ` /nonexistent `
致命信号兜底  |  ` sigqueue/alarm ` 失活  |  ` signal(SIG_IGN) ` 致命信号（兜底保险）

核心代码分五部分，逐块看。完整版见 mobikwik_full_dump.js。

**① 伪造` /proc/self/status ` （永远 TracerPid=0） **

先在沙箱写一份假 status，然后 hook ` open/openat ` ，遇到目标路径就重定向到假文件。 ` /proc/self/maps ` 和
` /comm ` 不重定向（否则 ART 会挂），改成记录 fd，交给 read hook 过滤内容：


    var FAKE_STATUS = "/data/data/com.mobikwik_new/files/.fake_status";
    var fd = openFn(Memory.allocUtf8String(FAKE_STATUS), 0x241, 0x1b6);
    writeFn(fd, Memory.allocUtf8String(
        "Name:\tmobikwik\nState:\tR\nTgid:\t" + Process.id + "\n" +
        "TracerPid:\t0\n" +
        "Uid:\t10259\t10259\t10259\t10259\n..."), 200);
    closeFn(fd);

    ['open', 'openat'].forEach(function (fname) {
        Interceptor.attach(Module.findExportByName('libc.so', fname), {
            onEnter: function (args) {
                var pathArg = (fname === 'open') ? args[0] : args[1];
                var p = pathArg.readCString();
                if (p && (p === '/proc/self/status' || p.indexOf('/status') !== -1)) {
                    pathArg.writeUtf8String(FAKE_STATUS);
                } else if (p && (p.indexOf('/proc/self/maps') !== -1 || p.indexOf('/comm') !== -1)) {
                    this.watchPath = p;
                }
            },
            onLeave: function (retval) {
                if (this.watchPath && retval.toInt32() >= 0)
                    watchFds[retval.toInt32()] = this.watchPath;
            }
        });
    });


**② 过滤` /proc/self/maps ` 里的关键字（ ` frida ` / ` magisk ` / ` gum-js ` 等） **

hook ` read() ` 。识别到之前 open 时标记的 fd 之后，把返回 buffer 内含黑名单关键字的整行替换成空格——
**保留行长不变** ，避免上层调用者觉得读到的字节数不对：


    var BLACKLIST = ['frida', 'magisk', 'gum-js', 'gadget', 're.frida', 'shamiko', 'zygisk'];
    Interceptor.attach(Module.findExportByName('libc.so', 'read'), {
        onEnter: function (args) {
            if (watchFds[args[0].toInt32()]) {
                this.targetFd = args[0].toInt32();
                this.buf = args[1];
            }
        },
        onLeave: function (retval) {
            if (!this.targetFd) return;
            var text = this.buf.readCString(retval.toInt32());
            var lines = text.split('\n');
            for (var j = 0; j < lines.length; j++) {
                var low = lines[j].toLowerCase();
                for (var k = 0; k < BLACKLIST.length; k++) {
                    if (low.indexOf(BLACKLIST[k]) !== -1) {
                        lines[j] = ' '.repeat(lines[j].length);
                    }
                }
            }
            this.buf.writeUtf8String(lines.join('\n'));
        }
    });


**③ 隐藏 frida 线程名（` gum-js-loop ` / ` gmain ` ） **

hook ` prctl(PR_GET_NAME) ` （op = 16）。取回线程名时如果匹配 frida 内部命名，改成 ` Thread-1 ` ：


    Interceptor.attach(Module.findExportByName('libc.so', 'prctl'), {
        onEnter: function (args) {
            if (args[0].toInt32() === 16) {
                this.isGet = true;
                this.buf = args[1];
            }
        },
        onLeave: function () {
            if (!this.isGet) return;
            var name = this.buf.readCString();
            if (name && /gum-js|gmain|frida/.test(name))
                this.buf.writeUtf8String("Thread-1");
        }
    });


**④ 屏蔽 root 路径探测（` magisk ` / ` supolicy ` / ` /sbin/. ` ） **

hook ` stat / lstat / access / faccessat ` 。含黑名单关键字的路径重定向到 `
/nonexistent_blocked ` ——让检测方拿到 ` ENOENT ` ：


    var FILE_BLACKLIST = ['magisk', 'frida', 'supolicy', '/sbin/.'];
    ['stat', 'lstat', 'access', 'faccessat'].forEach(function (fname) {
        Interceptor.attach(Module.findExportByName('libc.so', fname), {
            onEnter: function (args) {
                var pathArg = (fname === 'faccessat') ? args[1] : args[0];
                var p = pathArg.readCString();
                if (p && FILE_BLACKLIST.some(function (k) {
                    return p.toLowerCase().indexOf(k) !== -1;
                })) {
                    pathArg.writeUtf8String('/nonexistent_blocked');
                }
            }
        });
    });


**⑤ DefineClass hook + native 写盘**

拦 ` art::ClassLinker::DefineClass ` ，从第 5 个参数 ` DexFile* ` 反查 ` begin_ / size_
` （第 4 章的手法），用 native ` open/write/close ` 直接写文件——不走 Java
FileOutputStream，避免依赖 Java 框架初始化时机：


    var defineClassAddr = null;
    Process.enumerateModules().forEach(function (m) {
        if (m.name !== 'libart.so') return;
        m.enumerateSymbols().forEach(function (s) {
            if (s.name.indexOf('ClassLinker') !== -1
                && s.name.indexOf('DefineClass') !== -1
                && s.name.indexOf('DexFile') !== -1) defineClassAddr = s.address;
        });
    });
    var dumped = {}, dumped_count = 0;
    Interceptor.attach(defineClassAddr, {
        onEnter: function (args) {
            var dexFilePtr = args[5];                         // arg5 = const DexFile*
            var begin = dexFilePtr.add(8).readPointer();      // begin_
            var size  = dexFilePtr.add(16).readU32();         // size_
            var key = begin.toString() + '_' + size;
            if (dumped[key]) return; dumped[key] = true;
            var buf = Memory.alloc(size); Memory.copy(buf, begin, size);
            var fd = openFn(Memory.allocUtf8String(
                '/data/data/com.mobikwik_new/files/dex_' + dumped_count++ + '_' + size + '.dex'),
                0x241, 0x1b6);
            writeFn(fd, buf, size); closeFn(fd);
        }
    });


以上 ①-⑤ 拼起来是 **v3 版** ——反反检测 + 被动 DEX dump 的基础层。跑完产物 4 DEX / 17 MB，含 base.apk 主
DEX，只覆盖首屏业务。

###  8.5 迭代结果与业务覆盖

v3（8.4 的 ①-⑤）是基础层。产物 4 个 DEX / 17 MB / 首屏部分业务——4.3 秒后仍死于 kill 50025。继续加两件事：

  * **主动` enumerate ClassLoader.pathList.dexElements ` ** ：不等 DefineClass 被动触发，多波次直接从 ART 数据结构 ( ` BaseDexClassLoader → DexPathList → dexElements → DexFile.mCookie ` ) 拿已注册的所有 DEX

改完这两处得到 **v4 版** ——第二轮 dump 出 9 个 DEX / 48 MB，进程活 6+ 分钟。DEX 里 397 个 `

四条经验：

  * **anti-kill 拦不住 AppSealing 的 kill** ——AppSealing 用 raw ` svc #0 ` 走 ` rt_sigqueueinfo ` / ` rt_sigaction ` ，libc 包装全被绕。要拦得从更上游（log 决策点或 pthread 创建）
  * **anti-detection 才是脱壳的关键** ——让 AppSealing 以为环境干净、走完 sealed 解密，才有 DEX 可 dump
  * **被动 hook + 主动 enumerate 必须同时** ——单独 DefineClass 只拿到当前已加载类的 DEX，主动 ` dexElements ` 才能拿全已注册 DEX
  * **sleep / pthread_exit 拦 kill 有副作用** ——阻断决策线程会连锁卡 ART 主线程（进程活但 UI 假死，无法触发更多 lazy 解密）

###  8.9 GOT-XOR 62 槽位加密函数指针表——为什么 libc hook 不生效

### 8.8 列的 16 个因子说的是"检测什么"。到底"怎么检测"——AppSealing 用什么姿势调用 ` open ` / ` read ` / `
strstr ` 去读这些因子——才是决定 hook 能不能生效的关键。事实上就算你 hook 全部 15 个可修复因子对应的 libc 函数， `
Interceptor.attach(libc.open) ` 也是空转的：

![GOT-XOR 三层间接指针 + XOR
GOT-XOR 三层间接指针 + XOR 解密

` libcovault-appsec.so ` 里几乎不出现 libc 函数导入符号（ ` objdump -R ` 只能看到少量 libc
引用）。AppSealing 把 62 个常用 libc 函数指针存到自己的 ` .data ` 段、加了 XOR 混淆，运行时通过三层间接指针 + XOR
解密拿到真实函数地址后直接跳过去—— **完全不走 libc PLT** 。所以前面所有 `
Interceptor.attach(Module.findExportByName('libc.so', 'open'), ...) `
都是徒劳，AppSealing 从来没经过那里。

**加密结构** ：


    .data[+SLOT_OFFSET]     (outer heap ptr, 8 bytes)
        ↓ deref
    heap_outer              (inner heap ptr, 8 bytes)
        ↓ deref
    heap_inner              (encrypted ptr, 8 bytes)
        ↓ XOR with .data[+SLOT_OFFSET+8]   (32-bit blob, 8 bytes)
    real libc function pointer


**Frida 一行解密（每个槽位）** ：


    const cv = Process.getModuleByName('libcovault-appsec.so');
    const SLOT_BASE = 0x1b0c30;  // main 进程版
    const FIRST_OFF = 0x80;

    // 解出第 i 个槽位对应的 libc 函数真实地址
    function decryptSlot(i) {
        const slot = cv.base.add(SLOT_BASE + FIRST_OFF + i * 16);
        const outer = slot.readPointer();
        const inner = outer.readPointer();
        const encrypted = inner.readPointer();
        const xorKey = slot.add(8).readU64();
        return encrypted.xor(xorKey);   // ← 真实 libc 函数地址
    }


**62 槽位对照表** （ ` base + 0x1b0c30 + 0x80 + i*16 ` ，来自 REPORT 第 11.2 节）：

        Interceptor.attach(realAddr, {

###  8.10 五条 user-space 硬护栏

![五条 user-space
五条 user-space 硬护栏

即便按 8.9 hook 到 62 个真实地址，AppSealing 还有五条不可能从 user-space 突破的东西（编号对应 8.6 的三个 kill
code 映射）：

  1. **native exit 主线程不经 libc/Java/syscall** ：Thread-3 判定 tampered 后，直接内联汇编触发主线程死亡。不 exit、不 kill、不 raise、不 syscall。user-space hook 找不到入口点
  2. **检测在` .init_array ` constructor 跑 ** ：早于 JNI_OnLoad、早于 Application、早于一切可注入时机。Frida spawn 最早也是 zygote fork 之后 attach，那时 constructor 已经跑完（ **对应 8.6 kill code 50040** ——ZygiskFrida remap 输给这一时机）
  3. **Java 字节码完整性 hash 校验** ：AppSealing 保存自己 Java 类的 baseline hash 运行时比对。smali 修改立即被识别（错误码 20055），apktool 重打包路线走不通
  4. **` /proc/self/task/*/status ` 抓 Name 字段 ** ：Frida-gadget 内部 GLib 起的 ` gmain ` 线程名硬编码，改不掉（ **对应 8.6 kill code 50051** ）
  5. **1963 个间接 BLR + GOT-XOR 表 + PAC 签名** ：静态 CFG 几乎不可还原（radare2 全 invalid），二进制 patch 极难找入口（ **对应 8.6 kill code 50025** ——GOT-XOR 让 libc 层 hook 全部空转）

黑盒对抗（v1-v8c）和另一份白盒静态逆向（读完 3 个 SO 的 62 槽位 + 23 constructor + 4354
trace）两条路收敛到同一个结果：1130 mobikwik 业务类是首屏 sealed 解密的上限，进一步的动态使用不通过 user-space
bypass 达到。

###  8.11 突破的真实代价

要让 mobikwik 业务可持续运行需要 SO 二进制级 patch。按依赖顺序 + 手工估时：

  1. **(⏱ 10-15h)** 定位 libcovault 内的 native exit 指令（可能是内联 ` brk ` 或 SIGSEGV），NOP 掉——绕第一层 kill。1963 个间接 BLR + GOT-XOR + PAC 让 CFG 几乎不可还原，这一步吃逆向经验
  2. **(⏱ 5-10h)** Patch Java 层 ` getCrashReportFlag / nativeSetDeviceInfo ` 返回值 + 同步更新它们的 baseline hash——绕 20055。hash 校验的 baseline 位置本身也要挖
  3. **(⏱ 8-15h)** 在 ` .init_array[4] = fcn.0008ff94 ` 之前注入（zygote 层做，比 ZygiskFrida 更早）——让 anti-detection 装在 constructor 跑之前。要写 Zygisk 模块 + 掌握 linker 的注入时机
  4. **(⏱ 3-6h)** 改 Frida gadget 内部 GLib 线程创建（fork 源码）——让 gmain 变 Thread-1。fork gadget 编译流水线本身要花时间
  5. **(⏱ 5-10h)** Patch ` :alert_dialog ` 进程的 ` Handler.postDelayed(killProcess) ` ——拦二级 kill。要 hook Java 层 + 覆盖二级进程
  6. **(⏱ 3-5h)** 补 magiskpolicy 规则 + resetprop 修 6 个 ro.* prop——修因子 1-11。SELinux 规则调试是主要耗时

累计 34-61 小时保守估算，考虑试错和 PAC/Verified Boot 等意外因素放宽到 **40-100 小时**
。已经超出脱壳范畴，是独立的商业加固逆向工程。

如果只是要 mobikwik 的数据不是它的 UI，替代路径便宜得多：旧版 APK（2022 年前的 AppSealing 版本弱）、商业
unpacker、模拟器（BlueStacks/MEmu 部分兼容）、服务器端 RPC（第 27 篇）。

##  九、当代脱壳生态：接力工具 · API 迁移 · 兼容性

前面章节精读的三家 Frida 工具（frida-dexdump / frida_dump / FART）里，除了 frida_dump
还在维护，另外两个都停更了。这一章回答读者按前面代码跑必然遇到的 4 类现实问题： **接力方案是哪些** 、 **Frida 17 API 迁移怎么做**
、 **Android 14 兼容性变化** 、 **老一代工具兼容表** 。每个项目都有可验证 URL 和最近更新时间，内容截止 **2026-06**
。

###  9.1 FART 接力：CYRUS-STUDIO/frida_dex_dump 与 TheQmaks/clsdumper

老一代 FART 系工具的更新现状：

项目  |  URL  |  Star  |  最近更新  |  状态
---|---|---|---|---
FART (AOSP 6.0)  |  github.com/hanbinglengyue/FART  |  2.7k  |  2021 后停更  |  仅适配 Android 6/8，老 ROM
Youpk  |  github.com/Youlor/Youpk  |  ~800  |  4 commit 后停更  |  仅适配 Pixel sailfish (Android 8.1)
BlackDex  |  github.com/CodingGay/BlackDex  |  6.3k  |  2023-11 后停更  |  Android 14+ 兼容需自行打补丁

**Youpk 失败率有公开论文佐证** ：2025 年 arXiv 论文《To Unpack or Not to Unpack》（arxiv
2509.16340）实测发现 Youpk 在现代壳上 86% 的样本根本 dump 不出 DEX。

**真正接力的两个新项目** ：

**① CYRUS-STUDIO/frida_dex_dump** （github.com/CYRUS-STUDIO/frida_dex_dump，约
101，最近更新 2026-04-23）。核心思路是 hook ` libdexfile.so ` 的 `
DexFileLoader::OpenCommon ` ，配合 ` CompactDexFile ` / ` StandardDexFile `
双构造函数拦截。重点解决 **Android 9+ 的 .cdex（CompactDex）还原** ——前面提到 ART 的 OAT/cdex 缓存里
dex 是 cdex001 格式，老工具拿到也没法用 jadx 打开。这个项目还配套了 2025-05-28 的博客《使用 Frida 增强
FART》（blog.csdn.net/linchaolong/article/details/148293833），用 Frida 的 `
enumerateClassLoaders ` 弥补 FART 父委派盲区，把动态加载的 DEX 也覆盖到——是 2025-2026 最实用的 FART
补强方案。

**② TheQmaks/clsdumper** （github.com/TheQmaks/clsdumper，约 51，最近更新
2026-03-27）。打的是"9 种提取策略并联"思路—— ` Runtime → ClassLinker → DexFile ` 链路遍历 + hook
` DexFile::OpenCommon ` \+ 内存特征扫描，多路并发取最大并集。比 frida-dexdump 单一搜索路径更鲁棒，但需要的
Frida 知识门槛更高。

至于 kFART / MmmmFART / ProtectMe-Hunter / DexHunter 2.0 这些名字，在 GitHub、看雪、Google
学术上都没有对应公开仓库——不存在或仅是私有版本，不要被这些名字迷惑。

###  9.2 Frida 17 API 大重构，所有老脚本必须升级

Frida 17.0.0（2025-05-17 发布）做了一次破坏性 API 重构。对所有脱壳脚本的杀伤都是直接的：

旧 API（Frida 16 及以下）  |  新 API（Frida 17+）
---|---
` Module.findBaseAddress('libart.so') ` |  ` Process.getModuleByName('libart.so').base `
` Module.getExportByName('libart.so', 'xxx') ` |  ` Process.getModuleByName('libart.so').getExportByName('xxx') `
` Memory.readU32(ptr) ` |  ` ptr.readU32() `
` Module.enumerateExports(...) ` （callback）  |  ` module.enumerateExports() ` 返回数组

这意味着 **frida-dexdump（v2.0.1，2022-02 发布、2023-03 archived）、lasting-
yang/frida_dump、FART 自带的 Frida 脚本，在 Frida 17 上不加 polyfill 直接跑全部报`
ReferenceError ` ** 。如果你的 frida-server 已升到 17，老脚本要么 fork 改写、要么降版本到 16.x。

**正向价值是性能** ：GumJS 完成 ESM 化，单文件 agent 不再需要 webpack； `
Process.enumerateModules() ` 返回 Module 实例而不是裸字典； ` gum_memory_find_pointers()
` 引入并行 NEON/SSE2 内核——做 DEX 头特征扫描比 frida-dexdump 的 ` Memory.scan ` 快一个量级。
**真正写新脱壳脚本，建议直接用 Frida 17 + 新 API 起手** 。

###  9.3 Android 14 对脱壳的兼容性挑战

Android 14 的两个变化直击脱壳生态：

**① ART Service 接管 dexopt**
（source.android.com/docs/core/runtime/configure/art-service）——原来基于 ` installd
` \+ ` dex2oat ` 命令行的离线 hook 方案全部失效，传统"在 dex2oat 前后插桩取 DEX"的脱壳路径断了。

**② targetSdk=34 禁止可写 DEX** ——如果 App 用 ` DexClassLoader ` 加载 dex 文件且文件不是 read-
only，直接抛 ` SecurityException: Writable dex file is not allowed ` 。这堵死了"先 hook
open() 把解密后 DEX 写回磁盘再让壳加载"这条常用的脱壳辅助手法。

**③ Hidden API 收紧的累积效应** ——通过反射访问 ` BaseDexClassLoader.pathList ` 这类传统 trick 在
Android 14+ release build 上随机失败（developer.android.com/guide/app-
compatibility/restrictions-non-sdk-interfaces）。

结果就是 **老一代 ROM 级脱壳机几乎全军覆没** ：Youpk 锁死 Android 8、FART 主流分支停在 Android
10、DexHunter 停在 Android 4.4。新方案目前主要是用户态 Frida hook 库（CYRUS-STUDIO 类）和学术界的
BPFDex（IEEE 11105512，2025 年发表，用 eBPF 在 Linux 内核侧观测 DEX 加载，但无官方代码
release）。这是一个真实存在的生态空窗。

##  总结

加固壳的动作是"加密 + 运行时回填"。脱壳就是在回填后、清理前把明文 DEX 从内存里抠出来。三个老牌工具各占一层：frida-dexdump 扫内存
magic 通配 + map_list 校验 + 磁头修复，专治一代壳整体加密；frida_dump 从 ART 数据结构反查，避免 magic
误匹配；FART 主动 ` loadClass ` 触发方法体回填 + ` LoadMethod ` hook + fart.py
离线重组，专治二代壳指令抽取。

Pixel 5 / Android 14 上跑了两档实测。七猫小说是裸 DEX，frida-dexdump 一次 96 段命中 1.15 GB，但去掉大
DEX 段的重叠冗余后净 unique 只有 ~435 MB—— ` -d ` 深度扫描对 ART OAT 缓存区会同一段内存多次重复
dump，做统计前必须先做重叠去重。MobiKwik 用了 AppSealing v2.35.1，v1-v8c 加 ZygiskFrida gadget
一共 8 个版本对抗，合并去重后 35 个 DEX / 211 MB / 1130 业务类，UPI + KYC + 支付 + P2P + 钱包核心 100%
覆盖，长尾业务（cashback / bills / fastag / hotel 等）没触发 lazy 解密所以没拿到。

MobiKwik 那一档还揭示了 AppSealing 内部：62 个 libc 函数指针存自己的 ` .data ` 段 XOR
混淆，运行时解出真实地址直接跳过去，完全不走 libc PLT——所有 `
Interceptor.attach(Module.findExportByName('libc.so', ...)) ` 都是徒劳。要让 mobikwik
业务持续运行，需要在 SO 二进制层做 6 步手工 patch（NOP native exit / 改 Java hash baseline / 在 `
.init_array ` 之前注入 / 改 gadget GLib 线程名 / 拦 ` :alert_dialog ` 二级 kill / 修 6 个
ro.* prop），累计 40-100 小时。这已经跳出脱壳范畴。只是要数据不是要 UI 的话，走旧版 APK / 商业 unpacker / 模拟器 /
服务器端 RPC（第 27 篇）几个替代路径便宜得多。

三代壳（VMP）和 Dex2C 已经跳出 DEX 层——转 SO 逆向（第 22-23 篇）或在自定义解释器入口做动态分析（第 17-18
篇）。当代生态（第 9 章）三个必知点：老工具（Youpk / FART / BlackDex）全线停更或适配到 Android 8-13，接力方案是
CYRUS-STUDIO/frida_dex_dump 和 TheQmaks/clsdumper；Frida 17（2025-05）的 API 大重构让
archived 脚本不加 polyfill 直接跑就 ReferenceError， ` Memory.readX ` 全线要迁到 `
ptr.readX() ` ；Android 14+ 的 ART Service 接管 dexopt + targetSdk=34 禁写
DEX，直接堵死"hook dex2oat 命令行"和"先写盘再让壳加载"两条传统辅助路径。
