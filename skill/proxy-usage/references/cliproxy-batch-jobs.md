# Cliproxy 批量任务实战

## Session ID

- SID 只使用 `[A-Za-z0-9_]+`。
- 禁止 `-`。Cliproxy 账号参数以 `-` 分段，含连字符的 SID 可能被截断。
- 同一逻辑会话复用 SID；需要换出口时生成新的 SID。
- 验证两件事：同 SID 连续请求应保持同 IP；不同 SID 应获得不同 IP。只验证前者不够。

```python
sid = f"job{restaurant_id}_p{page}_a{attempt}"
account = build_account(user, "HK", sticky_min=30, sid=sid)
```

## 策略选择

| 任务 | 推荐策略 |
|---|---|
| 登录、验证码后连续操作 | 一个 Sticky SID 贯穿会话 |
| 匿名 SSR 分页、每页无状态 | 每页或每小批次独立 SID |
| 图片/CDN 批量下载 | 少量 Sticky worker；降速后换 SID |
| ruyi 浏览器 | 启动时固定代理；切代理需重启浏览器 |

不要把 Rotating、Sticky、多任务独立 SID 混为一谈。目标绑定 Cookie/IP 时不能每请求换 IP；无状态页面遇单 IP WAF 时不要长期复用 Sticky IP。

## 并发与进度

1. 从 4 并发开始，不直接拉满线程数。
2. 记录计划页数、已存在页数、成功页数、失败页数。
3. 每个响应先检查状态码、目标数据 marker 和验证页 marker，再写文件。
4. 以有效产物计数判断进度，不以进程存活或 CPU 占用判断。
5. 连续两个观测窗口产物数不增长时，检查 SID 碰撞、供应商限流、代理超时和目标 WAF。
6. 重启时扫描已有文件，只请求缺页；不要从头重跑。

## 失败处理

- 同一 SID 连续超时或返回验证页：换 SID/IP，避免在坏 IP 上重复三次长超时。
- 高并发首批成功后整体停滞：回退到已验证并发，不要继续加线程。
- 少数固定页失败：移到队尾，先完成其他目标，避免队头阻塞全局。
- CDN 下载先快后慢：保留已下载文件，换一组 Sticky SID 后续传。
- 供应商测试 URL 成功但目标失败：分别测试 `httpbin` 和真实目标；健康检查不能替代目标验证。

## 最小验收

- 不输出代理明文、账号或密码。
- SID 格式校验通过。
- 不同 SID 的出口轮换符合预期。
- 已落盘产物不会被覆盖或重复抓取。
- 最终数量与目标总数对账，失败项单独落 JSON。
