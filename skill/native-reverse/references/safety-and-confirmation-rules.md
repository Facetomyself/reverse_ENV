# 授权规则与边界

## 授权边界

本 Skill 仅用于自有或用户明确许可的样本、研究和调试环境。若目标边界不清，先说明边界并停止会造成未授权访问、隐蔽控制或数据外传的操作；可以继续做本地文档整理、已有日志分析、已有导出分析和无害的环境检查。

## `.so` 逆向确认规则

需要使用反编译、反汇编或逆向工具分析新的 `.so` 前，若用户没有明确说“直接分析”“继续”“已授权”或“无需确认”，默认先走确认流程：向用户说明以下内容，并等待确认后再执行工具逆向。

- 目标 `.so`。
- 分析目的。
- 计划使用工具；反编译 `.so` 文件强制使用 IDA，只有用户明确表示没有 IDA 或无法提供路径后，才允许 Ghidra/radare2/objdump 作为回退。
- 预计产物，例如导出文本、函数范围、伪代码、调用图。
- 是否需要先判断磁盘 so 加密、壳化、自解密或运行时重建；若命中，说明 dump/fix 是硬门禁，必须使用 MemDumper/frida_memdump 从运行期 so 或真实可执行段 dump/fix 后再分析，禁止直接分析磁盘 so 下结论或给 patch。

用户已经提供 IDA 导出或明确要求“直接分析导出”时，可基于该导出继续并同步记录。用户只提供 APK、so、dump 或日志但没有明确授权工具逆向新 `.so` 时，按本节确认流程走。

## 不需要额外确认的工作

- 阅读项目文档、已有实验记录、已有 IDA 导出文本、日志、脚本。
- 编写实验记录、整理检测点、创建模板或辅助脚本。
- 运行不涉及新 `.so` 反编译/反汇编的日志 grep、语法检查、文档检查。
- 分析用户提供的已有 IDA 导出、实验记录或日志。

## 工具路径规则

- 找不到 `adb`、IDA、`jadx` 等工具的位置时，先检查 `PATH`、项目 `scripts/`、`third_party/`、已有实验记录和常见安装路径。
- 仍找不到时询问用户路径，不要凭空假设路径。
- Java/Kotlin 层反编译强制使用 `jadx`；只有用户明确表示没有 `jadx` 或无法提供路径后，才允许换用其他工具，并记录用户答复和替代原因。
- `.so` 分析强制使用 IDA；只有用户明确表示没有 IDA 或无法提供路径后，才允许换用 Ghidra/radare2/objdump 等工具，并记录用户答复和替代原因。

## 记录边界

- 不把猜测写成确认。
- 不把 helper 误记为检测入口。
- 不把一次未命中写成不存在。
- 不覆盖旧记录。
- 不丢失失败轮次，失败轮次常常提供最关键的边界信息。

## Patch 边界

- patch 只要求可回退、可解释、低噪声并有验证依据，不限制必须最小化。
- 修改前保存原始文件或保留原始脚本版本。
- 对长期方案，避免大范围高频 hook。
- 对稳定性问题，回退到最后稳定 JS，先保留可运行状态，再做优化。

## 输出边界

用户要“下一步”时，给具体命令、文件和 patch 点。

用户要“文档”时，直接写入文件并说明路径。

用户要“分析”时，若涉及新 `.so` 工具逆向且用户未明确授权直接分析，先走 `.so` 逆向确认流程；若已有导出，则直接基于导出分析并同步记录。
